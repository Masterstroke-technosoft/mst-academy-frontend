02   Real-World Introduction Story

The Coinbase Interview That Changed Everything

📖 Story

In 2022, Vikram - a bright Solidity developer from Pune - applied for a Smart Contract Engineer role at Coinbase. The technical screen was a 45-minute live coding session on CoderPad. The problem: “Implement a time-locked vault where a beneficiary can withdraw only after a deadline, and the owner can deposit anytime. Include emergency withdrawal for the owner.”


Vikram panicked. He spent the first 10 minutes over-engineering the deposit mechanism, forgot to use `block.timestamp` for the deadline check, and completely omitted the emergency withdrawal. Time ran out. The interviewer politely ended the call. The feedback: “Candidate knows syntax but lacks problem-solving structure under pressure.”


Vikram didn't give up. For the next 3 weeks, he practiced the 7 classic Solidity puzzle patterns (multi-sig, time-lock, staking, voting, escrow, crowdsale, proxy upgrade) on a timer. He learned to talk through his thinking before writing a single line. He studied the evaluation rubric that interviewers use: correctness (30%), gas efficiency (20%), security considerations (30%), communication (20%).


His next interview - with a DeFi protocol - asked him to implement a staking contract with reward distribution. He solved it in 30 minutes, explained why he used `block.timestamp` for reward calculation (and its limitations), handled reentrancy with a nonReentrant modifier, and passed all test cases. He got the offer - ₹32 LPA, remote.


The difference? Structured practice, not just knowledge. This sub-module replicates that exact training.

Web3 technical interviews are fundamentally different from Web2. They test not just algorithmic thinking but security awareness, gas intuition, and EVM-specific design patterns. A candidate who writes a perfectly functional contract that silently accepts reentrancy will fail. One who catches the reentrancy bug during the interview and explains it will get hired - even if the contract isn't perfect.

03   Background & History

How Web3 Technical Interviews Evolved

2017 - 2019
The “Do you know Solidity?” era. Interviews were conversational. Questions like “What's a modifier?” or “Explain ERC-20” were sufficient. Supply of developers was so low that basic syntax knowledge often got you hired.
2020 - 2022
DeFi summer raises the bar. The hacks (Poly Network, Wormhole, Cream Finance) made security the #1 hiring filter. Live coding became standard. Typical problems: implement a simple AMM, fix a reentrancy bug in a given contract, or write a staking pool. System design rounds emerged for senior roles.
2023 - Present
Multi-stage, rubric-driven hiring. Top protocols now have: (1) a take-home assignment (2-4 hours), (2) a live coding round (45-60 min, 1-2 problems), (3) a system design round (60 min), (4) a culture/values fit. The evaluation is formalised: correctness, gas optimisation, security, and communication each have weight.
MST Ecosystem
What MST protocols test. MST-native projects often include questions about Trika POSA (validator staking), SARAL integration (how would you onboard users without seed phrases?), or WASMify (design an off-chain computation feed). Knowing the ecosystem gives you a distinct edge.
04   Beginner-Friendly Explanation

The Anatomy of a Web3 Technical Interview

Understanding the format is half the battle. A typical live coding + system design interview for a mid-level smart contract engineer follows this structure:

FIGURE 20.3-A · Standard 60-Minute Technical Interview Flow
  [0‑5 min]   Introductions & icebreaker
  [5‑10 min]  Coding Problem 1 (warm‑up) – e.g., implement a withdraw pattern
  [10‑30 min] Coding Problem 2 (main challenge) – e.g., multi‑sig or staking pool
  [30‑50 min] System Design – e.g., design a decentralised exchange on MST
  [50‑60 min] Candidate questions & closing
The coding segment is timed tightly: expect 20-25 minutes for the main problem. The system design is open-ended - they want to see your thinking process, not a perfect diagram.
🗝️ The Secret: Talk Before You Type

Interviewers don't just want working code - they want to see how you think. Narrate your approach: “First I'll define the state variables - a mapping for balances, a deadline timestamp, and an owner address. Then I'll write the deposit function, using `payable` and updating the mapping. Next, the withdraw with a `require` on deadline. Finally, an emergency withdrawal guarded by `onlyOwner`.” If you hit a syntax error, debug out loud. Communication counts for up to 30% of the score.

05   Technical Deep Dive

The 7 Must-Know Solidity Puzzle Patterns (With Solutions)

Every coding problem in a Web3 interview is a variation of one of these seven patterns. Master them, and you'll recognise the problem immediately and know the template.

1
Multi-Signature Wallet
Problem: A contract where N owners can submit transactions, and M confirmations are needed to execute.
Core logic: `mapping(uint => mapping(address => bool)) confirmations`; `executeTransaction` requires `confirmCount >= required`.
Watch for: Ensure a transaction cannot be executed twice; reset confirmations on execution. Use a struct for `Transaction` (to, value, data, executed). Gas tip: use `bytes` for calldata, not `string`.
2
Time-Locked Vault
Problem: Owner deposits; beneficiary withdraws only after a deadline. Owner can emergency-withdraw anytime.
Core: `uint deadline = block.timestamp + lockTime;` In `withdraw`, `require(block.timestamp >= deadline, "too early");`.
Security: Owner emergency withdraw must be separate and owner-only. Use `onlyOwner` modifier. Consider `block.timestamp` manipulation (miners can shift it slightly - acceptable for non-critical timing but mention it).
3
Staking Pool with Reward Distribution
Problem: Users stake tokens, earn rewards proportional to stake and time. Reward rate is fixed per second.
Core: Track `rewardPerTokenStored`, `userRewardPerTokenPaid`, and `rewards` mapping. Update rewards on every stake/unstake. Use `block.timestamp` for time calculation.
Pitfall: Reentrancy in `withdraw` - use `nonReentrant`. Ensure rewards are minted or the contract holds enough tokens. Gas: cache storage variables in memory.
4
Simple AMM (Constant Product)
Problem: A pool with two tokens. Add liquidity, swap, remove liquidity. Maintain `x * y = k`.
Core: `swap(tokenIn, amountIn, tokenOut)` calculates `amountOut = reserveOut - (k / (reserveIn + amountIn))`. Mint LP tokens proportionally.
Security: Ensure `amountOut` >= `minAmountOut` passed as parameter for slippage. Use `SafeMath` in older Solidity versions (0.8+ is safe). Beware of reentrancy if tokens are ERC-777.
5
Crowdsale / ICO
Problem: Sell tokens at a fixed price in exchange for ETH/MST. Cap the total sale. Allow owner to withdraw raised funds.
Core: Track `weiRaised`, `cap`, `rate`. `buyTokens() payable` mints tokens based on `msg.value * rate`. Require `weiRaised + msg.value <= cap`.
Trap: Must not allow purchase after cap is reached or deadline expired. Owner withdrawal should use pull pattern (not push).
6
Voting / Governance
Problem: Token holders vote on proposals. Each token = 1 vote. Proposal passes if quorum and majority reached within a deadline.
Core: `Proposal` struct (description, voteCount, deadline, executed). `vote(proposalId)` checks deadline, adds votes. `execute` requires `voteCount > quorum`.
Nuance: Prevent double-voting (track `hasVoted` mapping). Ensure proposal execution cannot be re-entered. Use snapshot of voting power to prevent flash-loan attacks.
7
Upgradeable Proxy (Minimal)
Problem: Implement a simple proxy that delegates calls to an implementation contract. Owner can upgrade the implementation.
Core: Use `delegatecall` in the fallback. Store implementation address in a specific storage slot (EIP-1967). `upgradeTo` only for owner.
Critical: `delegatecall` runs in the proxy's context - state variables must match layout. Mention OpenZeppelin Upgrades plugin as production solution.
System Design: The 4-Step Framework

In a system design round, you're asked to design a protocol from scratch - e.g., “Design a decentralised exchange on MST.” Use this structure:

1
REQUIREMENTS
Gather functional & non-functional requirements
Functional: users can swap tokens, provide liquidity, earn fees. Non-functional: handle 1000 swaps/sec, gas must be < 200k per swap, trustless, no admin keys.
2
ARCHITECTURE
Draw the high-level architecture
Smart contracts: Factory, Pair (for each token pair), Router. Frontend: React + ethers.js, connects via BridgeKey wallet. Indexing: subgraph for trade history. Oracles: maybe Chainlink for price feeds (if needed). MST specifics: use SARAL for user onboarding, WASMify for off-chain computation.
3
DATA FLOW
Describe a key flow
Swap: User sends tokenA to Router → Router identifies Pair address → calls Pair.swap() → Pair calculates output using x*y=k → transfers tokenB to user. Events emitted for indexing.
4
TRADE-OFFS
Discuss scaling, security, upgrades
Scaling: deploy on MST (high throughput, low fees). Security: audits, reentrancy guards, slippage protection. Upgradeability: proxy pattern. Risks: impermanent loss for LPs, oracle manipulation.

09   Assessment Preparation

Before You Start - The 5-Minute Drill

1
Clarify inputs/outputs
What functions are required? Who calls them? What are the edge cases? Ask before coding.
2
Write the skeleton
State variables, constructor, function signatures - get the structure right before filling logic.
3
Add access control & checks
`require`, modifiers, `nonReentrant` - security first, then logic.
4
Test mentally
Trace a deposit → withdraw flow in your head before declaring done.
5
Discuss trade-offs
Even if code is incomplete, verbalising security considerations earns major points.
10   Glossary

Key Terms - Sub-Module 20.3

Multi-Signature (Multi-Sig)
A smart contract requiring M-of-N owners to approve a transaction before it executes. Increases security for organisational funds.
Time-Lock
A smart contract pattern that restricts access to funds until a future timestamp. Used for vesting, delayed payments, or security cooling-off periods.
Staking Pool
A contract where users lock tokens to earn rewards, typically proportional to their stake and staking duration.
AMM (Automated Market Maker)
A decentralised exchange model that uses a mathematical formula (e.g., x*y=k) instead of an order book, enabling permissionless trading.
Reentrancy Attack
A vulnerability where a malicious contract calls back into the victim contract before the first invocation finishes, potentially draining funds.
Gas Efficiency
Optimising smart contract code to minimise transaction costs by using `calldata`, packing storage, and avoiding redundant operations.
System Design Interview
An open-ended discussion where the candidate designs the architecture, contracts, and infrastructure of a Web3 application, evaluated on structure and trade-off analysis.
EIP-2981 (Royalty Standard)
A standard that enables NFT marketplaces to retrieve royalty payment information, ensuring creators earn from secondary sales.
