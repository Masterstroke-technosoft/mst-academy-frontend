02   Real-World Story

The $69 Million JPEG That Changed Everything

📖 Story

On March 11, 2021, a digital artwork titled "Everydays: The First 5000 Days" by the artist Beeple sold at Christie's auction house for $69.3 million. It was a JPEG - a collage of 5,000 images Beeple had created daily for 13 years. But the buyer didn't just receive an image file. They received an NFT - a unique, verifiable token on the Ethereum blockchain that proved they owned the original artwork.


The sale sent shockwaves through the art world, the tech world, and the financial world. It wasn't the first NFT sale - CryptoPunks and CryptoKitties had paved the way - but it was the moment NFTs became undeniable. A digital file, easily copied with a right-click, had sold for tens of millions because the blockchain provided something that had never existed before: provable digital ownership and provenance.


Behind the scenes, the magic was made possible by two simple standards: ERC-721, which defined how a unique token could be created and transferred, and a metadata file stored on IPFS, which linked the token to the actual artwork. This sub-module is about that magic. You will learn exactly how Beeple's NFT worked, how to create your own, and how the ERC-721 and ERC-1155 standards are the foundation of the entire NFT economy.

03   What Is an NFT? The Concert Ticket That Cannot Be Counterfeited

Non-Fungible Tokens - Unique Digital Items on a Blockchain

An NFT (Non-Fungible Token) is a digital asset that represents ownership of a unique item or piece of content, verified by a blockchain. The term non-fungible means that each token is unique and cannot be exchanged one-for-one with another token. A ₹500 note is fungible - any ₹500 note is worth exactly the same as any other. A concert ticket for a specific seat is non-fungible - you cannot swap it for a ticket in a different row and expect the same experience.

NFTs are used to represent ownership of digital art, music, videos, virtual real estate, in-game items, domain names, and even physical assets. The blockchain records who owns each NFT, its entire history of ownership, and a link to the digital file (or metadata describing it). This provides provable scarcity, verifiable ownership, and transparent provenance - all without a central authority.

🔑 The Critical Distinction - NFT ≠ JPEG

An NFT is NOT the image itself. The image file (JPEG, PNG, GIF, etc.) is almost always stored off-chain - on IPFS, a central server, or decentralized storage. The NFT is the on-chain token that points to that file and records who owns it. When you "buy an NFT," you are buying the token that proves you own the official version of that artwork, not the image file itself. This distinction is crucial: anyone can right-click and save the image, but only one person can prove they own the NFT.

Common NFT use cases:

Digital Art: Artists sell their work directly to collectors without galleries taking commissions. Royalties can be programmed into the NFT so the artist earns a percentage every time the work is resold.

Gaming: Players truly own their in-game items. A sword purchased in one game could be used in another, or sold on a marketplace for real money. The publisher cannot take it away.

Music and Media: Musicians release limited-edition tracks as NFTs, giving fans ownership and exclusive access to content. Streaming royalties can be shared directly with token holders.

Identity and Credentials: University degrees, professional certifications, and identity documents can be issued as NFTs that are tamper-proof and instantly verifiable.

Virtual Real Estate: Platforms like Decentraland and The Sandbox sell parcels of virtual land as NFTs. Owners can build on their land, host events, and sell or rent their property.

04   The ERC-721 Standard - The Blueprint for Unique Tokens

Every Function and Event That Makes an NFT Work

ERC-721 is the standard that defines how a non-fungible token should behave on the Ethereum blockchain (and any EVM-compatible chain like MST). It was proposed in January 2018 and became the foundation of the entire NFT ecosystem. The standard specifies a set of mandatory functions and events that every ERC-721 contract must implement, ensuring that all NFTs can be viewed, transferred, and managed by any wallet or marketplace.

The core ERC-721 interface:

interface IERC721 {
    // ─── Mandatory Functions ───
    function balanceOf(address owner) external view returns (uint256);
    function ownerOf(uint256 tokenId) external view returns (address);
    function transferFrom(address from, address to, uint256 tokenId) external;
    function safeTransferFrom(address from, address to, uint256 tokenId) external;
    function safeTransferFrom(address from, address to, uint256 tokenId, bytes memory data) external;
    function approve(address to, uint256 tokenId) external;
    function getApproved(uint256 tokenId) external view returns (address);
    function setApprovalForAll(address operator, bool approved) external;
    function isApprovedForAll(address owner, address operator) external view returns (bool);

    // ─── Mandatory Events ───
    event Transfer(address indexed from, address indexed to, uint256 indexed tokenId);
    event Approval(address indexed owner, address indexed approved, uint256 indexed tokenId);
    event ApprovalForAll(address indexed owner, address indexed operator, bool approved);

    // ─── Optional (but strongly recommended) ───
    function name() external view returns (string memory);
    function symbol() external view returns (string memory);
    function tokenURI(uint256 tokenId) external view returns (string memory);
}

What each function does, in plain English:

balanceOf(owner) - Returns how many NFTs a specific address owns. Like checking how many paintings are in someone's collection.

ownerOf(tokenId) - Returns who owns a specific NFT, identified by its unique token ID. If token #42 exists, this tells you who holds it.

transferFrom(from, to, tokenId) - Transfers an NFT from one address to another. The caller must be the owner, approved for this specific token, or an authorized operator. This is the core transfer function.

safeTransferFrom - Same as transferFrom, but also checks that the recipient is capable of receiving NFTs (i.e., the recipient is either an EOA or a contract that implements the ERC-721 receiver interface). This prevents tokens from being accidentally sent to contracts that cannot handle them, which would permanently lock the NFT.

approve(to, tokenId) - Gives another address permission to transfer a specific NFT. Used when you list an NFT on a marketplace: you approve the marketplace contract to transfer that token when a buyer purchases it.

getApproved(tokenId) - Returns the address currently approved to transfer a specific token.

setApprovalForAll(operator, approved) - Gives or revokes blanket permission for another address to transfer ALL of your NFTs. Use carefully - this is how marketplaces like OpenSea manage your entire collection without requiring per-token approvals.

isApprovedForAll(owner, operator) - Checks whether an operator has blanket approval to transfer an owner's NFTs.

name(), symbol(), tokenURI(tokenId) - Metadata functions. name and symbol are human-readable identifiers (like "CryptoPunks" and "PUNK"). tokenURI returns a URL pointing to a JSON file that describes the specific NFT - its image, attributes, description, etc. This is how wallets and marketplaces display your NFT.

05   Implementing ERC-721 with OpenZeppelin - A Complete Contract

From Zero to Minted NFT in Ten Lines of Code

Writing an ERC-721 contract from scratch is complex and error-prone. Fortunately, OpenZeppelin provides a battle-tested, audited implementation that you can extend. Here is a complete, production-ready NFT contract:

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC721/ERC721.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract MyNFT is ERC721, Ownable {
    uint256 private _nextTokenId;

    constructor() ERC721("MyAwesomeNFT", "AWESOME") Ownable(msg.sender) {}

    /**
     * @notice Mints a new NFT to the specified address.
     * @param to The address that will own the newly minted NFT.
     */
    function mint(address to) public onlyOwner {
        uint256 tokenId = _nextTokenId;
        _nextTokenId++;
        _safeMint(to, tokenId);
    }

    /**
     * @notice Returns the URI where the metadata for a given token is stored.
     * @param tokenId The ID of the token.
     * @return The URI string pointing to the token's metadata JSON.
     */
    function tokenURI(uint256 tokenId) public view override returns (string memory) {
        _requireOwned(tokenId);
        // Base URI pointing to IPFS or a server hosting the metadata files
        string memory baseURI = "ipfs://QmYourMetadataFolderCID/";
        return string(abi.encodePacked(baseURI, Strings.toString(tokenId), ".json"));
    }
}

What this contract does:

Inheritance: It inherits from ERC721 (which provides all the standard functions like transfer, approve, balanceOf) and Ownable (which restricts minting to the contract owner).

Constructor: Sets the NFT's name ("MyAwesomeNFT") and symbol ("AWESOME") and records the deployer as the owner.

mint function: Creates a new NFT with a sequential token ID and assigns it to the specified address. Only the owner can mint - this creates scarcity. _safeMint is an internal OpenZeppelin function that increments the balance, emits the Transfer event, and checks that the recipient can receive NFTs.

tokenURI function: Returns the metadata URI for a specific token. In this example, it constructs a URL like ipfs://Qm.../0.json for token 0, ipfs://Qm.../1.json for token 1, etc. The actual metadata JSON files are stored on IPFS (or a server) and contain the image link, description, and attributes.

⚠️ Important - tokenURI Must Return a Valid JSON File

The tokenURI function is how every wallet, marketplace, and explorer displays your NFT. If it returns an invalid URL or the JSON file is malformed, your NFT will appear as a blank box. The JSON file must follow the ERC-721 Metadata Standard (see next section). Always test your tokenURI by visiting the URL in a browser before deploying.

06   NFT Metadata - The JSON File That Describes Your Token

What Actually Makes an NFT Display Correctly

The tokenURI function returns a URL that points to a JSON file. This JSON file contains all the information that wallets and marketplaces need to display your NFT. The standard structure is defined by the ERC-721 Metadata extension:

{
    "name": "My Awesome NFT #1",
    "description": "This is the very first NFT minted in the MyAwesomeNFT collection.",
    "image": "ipfs://QmYourImageCID/1.png",
    "attributes": [
        {
            "trait_type": "Background",
            "value": "Cosmic Purple"
        },
        {
            "trait_type": "Rarity",
            "value": "Legendary"
        },
        {
            "trait_type": "Power Level",
            "display_type": "number",
            "value": 9001
        }
    ]
}

Explanation of each field:

name: The human-readable name of this specific NFT. Often includes the collection name and the token number.

description: A longer text description of the NFT. Can include the story behind the artwork, special abilities, or terms of ownership.

image: A URL pointing to the actual image, video, or audio file. This is what wallets display. The URL can be an ipfs:// link, an https:// link, or even an on-chain SVG (though on-chain storage is expensive). IPFS is strongly recommended because it's decentralized and the content can't be changed once uploaded.

attributes (optional but critical for marketplaces): An array of trait objects that describe the properties of the NFT. Marketplaces like OpenSea use these to display rarity rankings, filter collections, and show trait statistics. Each attribute has a trait_type (the category), a value (the specific value), and optionally a display_type (which tells the marketplace how to format the value - as a number, a date, or a boost percentage).

07   IPFS - Decentralized Storage for NFT Assets

Why You Should Never Store Your NFT Images on a Central Server

IPFS (InterPlanetary File System) is a peer-to-peer network for storing and sharing files in a decentralized way. When you upload a file to IPFS, it receives a unique Content Identifier (CID) - a hash of the file's contents. The file can then be accessed via ipfs://<CID> from any IPFS gateway.

Why IPFS for NFTs? If your NFT's metadata and image are stored on a central server (like https://mywebsite.com/nft/1.json), several things can go wrong:

1. The server goes offline - your NFT becomes a broken link forever.

2. The server owner changes the file - your "Legendary" NFT could be swapped to a "Common" one.

3. The domain expires - someone else buys your domain and replaces your NFT images.

IPFS solves all of these. The CID is a cryptographic hash of the content, so if anyone changes the file, the CID changes. An IPFS link is content-addressed, meaning you cannot alter the content without changing the address. Additionally, the file is served by a network of peers, so it's resilient to single-point failures.

How to upload to IPFS:

Option 1 - Pinata (free tier): Create an account at pinata.cloud. Upload your image and JSON files. Pinata will provide ipfs:// URLs that you can use in your tokenURI.

Option 2 - web3.storage: Another free service by Protocol Labs. Provides a simple API and generous free storage.

Option 3 - Run your own IPFS node: For maximum control, you can run an IPFS node on your own server, but this requires more technical setup.

Important: Uploading a file to IPFS doesn't guarantee it stays online forever. You must pin the file to ensure it's permanently stored. Services like Pinata pin your files for you. If nobody pins a file, it may eventually be garbage-collected from the network.

🔑 The Golden Rule of NFT Metadata

Upload your metadata JSON files to IPFS first, then deploy your contract with a base URI that points to that IPFS folder. The contract should never contain the actual image data or metadata - it should only contain the IPFS CID. This keeps the contract small, cheap to deploy, and upgradeable (you can update the base URI if needed, though that compromises immutability - many projects freeze the base URI after minting is complete).

08   ERC-1155 - The Multi-Token Standard for Games and Collections

One Contract to Rule Them All - Fungible AND Non-Fungible Tokens

While ERC-721 is perfect for one-of-a-kind items, many applications need a mix of unique items and items with multiple copies. A blockchain game might have a thousand identical "health potion" tokens (fungible) and a single unique "Excalibur" sword (non-fungible). Deploying a separate ERC-721 contract for every item type would be incredibly expensive and complex. ERC-1155 solves this by allowing a single contract to manage multiple token types, each with its own supply.

The key concept: In ERC-1155, each token type has a unique id. The id can represent a fungible token (with supply > 1) or a non-fungible token (with supply = 1). All token types share the same contract address. This is dramatically more efficient for games with hundreds of item types.

// ERC-1155 interface (simplified)
interface IERC1155 {
    function balanceOf(address account, uint256 id) external view returns (uint256);
    function balanceOfBatch(address[] calldata accounts, uint256[] calldata ids) external view returns (uint256[] memory);
    function safeTransferFrom(address from, address to, uint256 id, uint256 amount, bytes calldata data) external;
    function safeBatchTransferFrom(address from, address to, uint256[] calldata ids, uint256[] calldata amounts, bytes calldata data) external;
    function setApprovalForAll(address operator, bool approved) external;
    function isApprovedForAll(address account, address operator) external view returns (bool);
}

Key differences from ERC-721:

1. Token IDs represent types, not individual items. In ERC-1155, id=1 might be "gold coins" with a supply of 10,000. balanceOf(user, 1) returns how many gold coins that user owns. In ERC-721, tokenId=1 is a single unique sword, and balanceOf(user) returns the count of all unique tokens the user owns across all IDs.

2. Batch transfers. ERC-1155 has safeBatchTransferFrom, which allows transferring multiple token types in a single transaction. This saves enormous gas when a player is moving their entire inventory.

3. Single approval model. ERC-1155 only has setApprovalForAll (no per-token approval). This is simpler for games where users frequently interact with marketplaces that need to manage all their items.

4. Metadata per token ID. ERC-1155 has a uri(uint256 id) function that returns the metadata URI for a specific token type. This is similar to ERC-721's tokenURI, but it works per token type, not per individual token.

When to use ERC-721 vs ERC-1155:

Use CaseRecommended Standard
Digital art collection with unique piecesERC-721
Collectible card game with multiple copies of each cardERC-1155
In-game currency and itemsERC-1155
Membership pass or access token (one per person)ERC-721
Event tickets with different tiers and quantitiesERC-1155
09   Complete ERC-1155 Contract Example

A Single Contract for a Game's Entire Economy

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC1155/ERC1155.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract GameItems is ERC1155, Ownable {
    uint256 public constant GOLD_COIN = 0;
    uint256 public constant SILVER_COIN = 1;
    uint256 public constant SWORD = 2; // unique – only 1 exists
    uint256 public constant SHIELD = 3; // unique – only 1 exists

    constructor() ERC1155("ipfs://QmYourBaseURI/{id}.json") Ownable(msg.sender) {
        // Mint 10,000 gold coins to the deployer
        _mint(msg.sender, GOLD_COIN, 10000 * 10**18, "");
        // Mint 50,000 silver coins to the deployer
        _mint(msg.sender, SILVER_COIN, 50000 * 10**18, "");
        // Mint the unique sword to the deployer
        _mint(msg.sender, SWORD, 1, "");
        // Mint the unique shield to the deployer
        _mint(msg.sender, SHIELD, 1, "");
    }

    /**
     * @notice Allows the owner to mint additional items.
     * @param to The recipient address.
     * @param id The token type ID.
     * @param amount The number of tokens to mint.
     */
    function mint(address to, uint256 id, uint256 amount) public onlyOwner {
        _mint(to, id, amount, "");
    }
}

In this contract, GOLD_COIN and SILVER_COIN are fungible - they have large supplies. SWORD and SHIELD are non-fungible - only one of each exists. The constructor mints initial supplies. The uri function (inherited from OpenZeppelin's ERC1155) uses a base URI where {id} is replaced with the token type ID, so each token type can have its own metadata JSON. This is how a single contract can manage an entire game's economy.

11   Glossary

Key Terms - Sub-Module 12.1

NFT (Non-Fungible Token)
A unique digital asset on a blockchain that represents ownership of a specific item. Unlike cryptocurrencies, each NFT is distinct and not interchangeable.
ERC-721
The Ethereum standard for non-fungible tokens. Defines functions like ownerOf, transferFrom, and tokenURI that every NFT contract must implement.
tokenURI
A function that returns a URL pointing to a JSON file containing the metadata (name, image, attributes) for a specific NFT. Used by wallets to display the token.
IPFS
InterPlanetary File System - a decentralized, peer-to-peer file storage network. Files are addressed by their content hash (CID), ensuring integrity and availability.
Metadata
A JSON file that describes an NFT, including its name, description, image URL, and attributes. Linked to the on-chain token via the tokenURI function.
ERC-1155
A multi-token standard that allows a single contract to manage both fungible and non-fungible tokens. Supports batch transfers and is widely used in gaming.
OpenZeppelin
A library of secure, audited smart contract templates for Solidity. Provides production-ready implementations of ERC-721, ERC-1155, and access control.
Batch Transfer
A feature of ERC-1155 that allows sending multiple token types in a single transaction, saving gas compared to multiple individual transfers.
