02   Real-World Story

The Online Rummy Site That Was Rigged for Three Years

📖 Story

Between 2018 and 2021, a popular online rummy platform in India attracted over 2 million players. The site advertised a "fair shuffle" and "certified random number generator." Players trusted the brand, the glossy interface, and the certification badges on the homepage. But in 2022, a whistleblower - a former backend engineer - leaked internal server logs that revealed the truth: the shuffle algorithm was deliberately biased. Certain VIP players received better hands. The platform's "random number generator" was nothing but a server-side script that the operations team could tweak with a configuration file. Over three years, an estimated ₹500 crore was siphoned from ordinary players to a handful of insiders.


The platform had no way to prove its fairness - because there was no proof. All the randomness happened inside a black-box server that only the company controlled. Players had to take the company's word for it. And for three years, that word was a lie.

This is what MST's gaming fairness protocols deliver.

03   The Problem of Fairness in Online Gaming

Why "Trust Me" Is Not Good Enough

In traditional online gaming - whether it's a card game, a slot machine, or a loot-box system - the entire game logic runs on a server controlled by the game operator. The operator generates random numbers, shuffles cards, determines payouts, and reports the results to the player. The player has no way to verify that any of this was done honestly.

This creates a fundamental conflict of interest. The operator profits when players lose. The operator controls the randomness. The operator has no incentive to be fair. And even if the operator is honest, there is no way to prove it. This is why online gaming is plagued by allegations of rigging, and why many players simply do not trust the platforms they use.

The solution is verifiable, decentralized randomness. If the random numbers that determine game outcomes are generated in a way that is publicly auditable - ideally, on a blockchain - then players can verify fairness themselves. They don't need to trust the operator. They can check the proof.

MST Blockchain provides a suite of three protocols specifically designed to solve different aspects of gaming fairness:

1. Klethesia Protocol - verifies the long-term payout rate (RTP) of casino games like slots and roulette.

2. SatvaShuffle Protocol - provides provably fair card shuffling for poker, rummy, and other card games.

3. FortunaX Protocol - generates decentralized, unpredictable random numbers for loot boxes, prize draws, and any application needing true randomness.

04   Klethesia Protocol - Verifiable Return-to-Player (RTP)

Proving a Slot Machine Pays Exactly What It Promises

Return to Player (RTP) is the percentage of total wagered money that a casino game returns to players over time. A slot machine with a 96% RTP should, over millions of spins, pay out 96% of all money wagered and keep 4% as the house edge. In traditional casinos, RTP is certified by independent testing labs that audit the machine's software once. But in online casinos, there is no physical machine to inspect, and the software can be changed at any time. Players have no way to know if the advertised RTP is real.

Klethesia solves this by recording every bet and every payout of a game on the MST blockchain. The smart contract accumulates the total wagered and total paid out, and continuously computes the actual RTP. This RTP is stored on-chain and is publicly visible. Anyone can query the contract and verify that the actual RTP matches the advertised RTP within statistical tolerance. If the operator tries to cheat by secretly adjusting the payout rate, the discrepancy will be visible on-chain for everyone to see.

FIGURE 12.2-A · How Klethesia Verifies RTP
Every Spin:
  Player places bet → Transaction recorded on MST
  Game contract generates result (using FortunaX for randomness)
  Payout (if any) sent to player → Transaction recorded

On‑Chain Accumulators:
  totalWagered += betAmount
  totalPaidOut += payoutAmount
  actualRTP = (totalPaidOut * 100) / totalWagered

Public Verification:
  Any player or auditor queries the contract:
    advertisedRTP = 96%
    actualRTP    = 95.94%
    Statistical test confirms: within expected variance? ✅
    If actualRTP drifts far from advertisedRTP → ❌ Red flag!
Klethesia transforms RTP from a marketing claim into a mathematically verifiable fact, updated in real time on the blockchain.

How the smart contract enforces RTP: The Klethesia contract does not just record RTP - it can enforce it. The contract can be programmed to only allow payouts that keep the overall RTP within a specified range. If the operator tries to reduce payouts to increase their edge, the contract will simply reject those transactions. The RTP becomes a hard constraint, not a promise.

// Simplified Klethesia verification – checking RTP after each bet
contract KlethesiaSlot {
    uint256 public constant ADVERTISED_RTP = 96; // 96%
    uint256 public totalWagered;
    uint256 public totalPaidOut;
    uint256 public constant MAX_DEVIATION = 3; // 3% allowed short‑term variance

    function spin(uint256 bet) public returns (uint256 payout) {
        require(bet > 0, "Bet must be positive");
        totalWagered += bet;

        // Generate verifiable random result (using FortunaX oracle)
        uint256 random = getFortunaXRandom();
        payout = calculatePayout(bet, random);

        // Klethesia check: ensure payout doesn't push RTP below acceptable range
        if (payout > 0) {
            uint256 currentRTP = ((totalPaidOut + payout) * 100) / totalWagered;
            require(currentRTP >= ADVERTISED_RTP - MAX_DEVIATION, "RTP deviation too high");
            totalPaidOut += payout;
        }

        // Transfer payout to player
        if (payout > 0) {
            payable(msg.sender).transfer(payout);
        }
    }
}
05   SatvaShuffle Protocol - Provably Fair Card Shuffling

Dealing Cards That Nobody Can Stack

Card games like poker, rummy, and bridge rely on the integrity of the shuffle. If the dealer can know the order of cards, or can manipulate the shuffle to favor certain players, the game is meaningless. SatvaShuffle provides a cryptographic guarantee that the shuffle is fair and verifiable.

The protocol works in two phases, using a commit-reveal pattern:

Phase 1 - Commit (before the game): The dealer shuffles the deck and computes a cryptographic hash of the entire shuffled deck. This hash - called the deck commitment - is published to the MST blockchain. The commitment proves that the dealer fixed the order of cards before the game began, but it does not reveal the order itself (because hashes are one-way).

Phase 2 - Reveal (after the game): The dealer reveals the original shuffled deck. Any player can now compute the hash of the revealed deck and compare it to the commitment stored on-chain. If the hashes match, the deck was not tampered with after the commitment was made. The player also verifies that the cards dealt during the game match the revealed deck order.

FIGURE 12.2-B · The SatvaShuffle Commit-Reveal Lifecycle
BEFORE THE GAME (Commit Phase):
  1. Dealer shuffles 52‑card deck → [A♠, 3♦, K♥, ... , 7♣]
  2. Dealer computes hash: deckHash = keccak256(abi.encode(deck))
  3. Dealer sends deckHash to MST smart contract → stored on‑chain
  4. Game begins. Cards are dealt from the committed deck (off‑chain for speed).

AFTER THE GAME (Reveal Phase):
  5. Dealer reveals the full deck → [A♠, 3♦, K♥, ... , 7♣]
  6. Smart contract verifies: keccak256(abi.encode(revealedDeck)) == storedDeckHash
  7. If match ✅ → Shuffle was honest. If mismatch ❌ → Dealer cheated!
  8. Players can also verify that the cards they received match the revealed order.
The commit-reveal pattern ensures that the dealer cannot change the shuffle after seeing players' bets or actions.

Why this works: The dealer cannot change the deck order after the commitment because changing even one card would completely change the hash (the avalanche effect). The players can verify the shuffle after the game ends, but they cannot predict the cards during the game because the hash reveals nothing about the input. This is the standard technique used in provably fair poker platforms, but SatvaShuffle automates it on MST with low gas costs and fast finality.

// SatvaShuffle smart contract
contract SatvaShuffle {
    struct Game {
        bytes32 deckCommitment;   // Hash of the shuffled deck
        address dealer;
        bool revealed;
        uint256[] revealedDeck;    // The actual card IDs (after reveal)
    }

    mapping(uint256 => Game) public games;
    uint256 public gameCounter;

    // Dealer commits the shuffle before the game starts
    function commitShuffle(bytes32 deckHash) public returns (uint256 gameId) {
        gameId = gameCounter++;
        games[gameId] = Game({
            deckCommitment: deckHash,
            dealer: msg.sender,
            revealed: false,
            revealedDeck: new uint256[](0)
        });
    }

    // After the game, dealer reveals the actual deck
    function revealShuffle(uint256 gameId, uint256[] calldata deck) public {
        Game storage game = games[gameId];
        require(msg.sender == game.dealer, "Only dealer can reveal");
        require(!game.revealed, "Already revealed");

        // Verify the revealed deck matches the commitment
        bytes32 computedHash = keccak256(abi.encode(deck));
        require(computedHash == game.deckCommitment, "Deck does not match commitment!");

        game.revealed = true;
        game.revealedDeck = deck;
    }

    // Any player can verify the shuffle after the game
    function verifyShuffle(uint256 gameId) public view returns (bool) {
        Game storage game = games[gameId];
        require(game.revealed, "Not yet revealed");
        bytes32 computedHash = keccak256(abi.encode(game.revealedDeck));
        return computedHash == game.deckCommitment;
    }
}

Verifying the dealt cards: The reveal step also allows players to confirm that the cards they received during the game match the revealed deck order. A player who was dealt the 5th, 12th, and 23rd cards from the deck can check that those positions in the revealed array correspond to the cards they received. If not, the dealer was dealing from a different deck than the one committed.

06   FortunaX Protocol - Decentralized Randomness

Random Numbers That Nobody Can Predict or Manipulate

Randomness on a blockchain is notoriously difficult. Everything on-chain is deterministic - smart contracts cannot generate truly random numbers because every node must produce the same result. If you use block.timestamp or blockhash as randomness, a miner or validator can subtly manipulate the block to influence the outcome. This makes traditional on-chain randomness unsuitable for games where money is at stake.

FortunaX is MST's protocol for generating verifiable, unpredictable random numbers that are resistant to manipulation. It works by aggregating entropy from multiple independent sources and using a cryptographic commitment scheme to prevent any single party from controlling the result.

How FortunaX generates randomness:

1. Entropy collection: Multiple independent oracles (called FortunaX nodes) each generate a random seed. These seeds are encrypted and submitted to the FortunaX contract.

2. Aggregation: Once enough seeds have been collected (a threshold), they are combined using a verifiable random function (VRF) - typically a BLS signature on a common input. The output of the VRF is the random number.

3. Delivery: The random number is delivered to the requesting smart contract along with a proof that can be verified on-chain. The proof ensures that the number was generated correctly from the committed seeds.

Because the seeds are committed before they are revealed, no single oracle can predict or influence the final random number. Even if some oracles are malicious, as long as the threshold of honest oracles is met, the output is unpredictable and uniformly random.

FIGURE 12.2-C · FortunaX Randomness Generation
1. Game contract requests randomness:
   gameContract.requestRandomness() → FortunaX contract

2. FortunaX contract assigns a request ID and waits for oracle responses.
   Each oracle:
     • Generates a random seed sᵢ
     • Computes commitment cᵢ = keccak256(sᵢ)
     • Submits cᵢ to FortunaX contract

3. Once threshold of commitments is reached:
   Oracles reveal their seeds sᵢ.
   Contract verifies keccak256(sᵢ) == stored cᵢ for each.

4. Aggregation:
   combinedSeed = keccak256(abi.encodePacked(s₁, s₂, ..., sₙ))
   randomNumber = VRF(combinedSeed, requestID)  ← BLS threshold signature

5. FortunaX contract calls back game contract:
   gameContract.fulfillRandomness(requestID, randomNumber, proof)
   Game contract uses randomNumber to determine loot box, spin, etc.
FortunaX combines decentralized entropy with cryptographic proofs, ensuring that no single party can influence the outcome.
// Requesting randomness from FortunaX in your game contract
contract LootBox {
    address public fortunaxContract;
    mapping(uint256 => address) public requesters;

    event BoxOpened(address indexed player, uint256 requestId);

    constructor(address _fortunax) {
        fortunaxContract = _fortunax;
    }

    function openBox() public payable returns (uint256 requestId) {
        require(msg.value >= 0.01 ether, "Insufficient fee");
        // Pay the FortunaX oracle fee (simplified)
        payable(fortunaxContract).transfer(msg.value);

        // Request randomness from FortunaX
        requestId = IFortunaX(fortunaxContract).requestRandomness();
        requesters[requestId] = msg.sender;
        emit BoxOpened(msg.sender, requestId);
        return requestId;
    }

    // Called by FortunaX contract when randomness is ready
    function fulfillRandomness(uint256 requestId, uint256 randomNumber, bytes calldata proof) external {
        require(msg.sender == fortunaxContract, "Only FortunaX");
        address player = requesters[requestId];
        require(player != address(0), "Request not found");

        // Verify the proof (ensures randomNumber was correctly generated)
        require(IFortunaX(fortunaxContract).verifyProof(randomNumber, proof), "Invalid proof");

        // Use the random number to determine loot
        if (randomNumber % 100 < 70) {
            // 70% chance – common item
            _mintCommonItem(player);
        } else if (randomNumber % 100 < 95) {
            // 25% chance – rare item
            _mintRareItem(player);
        } else {
            // 5% chance – legendary item
            _mintLegendaryItem(player);
        }

        delete requesters[requestId];
    }
}
07   Putting It All Together - A Complete Fair Gaming Platform on MST

Rummy, Slots, and Loot Boxes - All Verifiably Fair

MST's gaming fairness protocols are designed to work together. A single gaming platform can use SatvaShuffle for card games, Klethesia for slot machines, and FortunaX for loot boxes - all on the same blockchain, all benefiting from MST's low gas fees and fast finality (~3 seconds).

Why MST is the ideal chain for gaming:

Ultra-low gas fees: A single card game can involve dozens of on-chain transactions (shuffle commit, bet, deal, reveal). On Ethereum, these could cost ₹1,000+ each, making the game uneconomical. On MST, they cost a fraction of a paisa, making real-time gaming economically viable.

Fast finality: MST's ~3-second block time means that a shuffle commitment is confirmed almost instantly, and a random number request from FortunaX is fulfilled within seconds, not minutes. This is critical for a smooth player experience.

EVM compatibility: All the gaming smart contracts are written in standard Solidity and can use existing tools, libraries, and audit practices. There is no need to learn a new language or framework.

Transparency and trust: Every bet, every shuffle, every random number is recorded on a public ledger. Regulators can audit the platform without needing access to the operator's servers. Players can verify fairness independently.

09   Glossary

Key Terms - Sub-Module 12.2

Klethesia
MST protocol for on-chain Return-to-Player (RTP) verification. Records every bet and payout, allowing anyone to verify that a game's advertised payout rate is accurate.
RTP (Return to Player)
The percentage of total wagered money that a casino game returns to players over a large number of plays. A 96% RTP means the house keeps 4%.
SatvaShuffle
MST protocol for provably fair card shuffling. Uses a commit-reveal pattern where the dealer commits a hash of the shuffled deck before the game and reveals it after.
Commit-Reveal
A cryptographic pattern where a party first publishes a commitment (hash) of some data, and later reveals the data. Prevents the party from changing the data after commitment.
FortunaX
MST's decentralized randomness protocol. Combines entropy from multiple independent oracles using threshold signatures to produce unpredictable, verifiable random numbers.
VRF (Verifiable Random Function)
A cryptographic function that generates a random output along with a proof that the output was correctly computed. Used by FortunaX to ensure randomness integrity.
Threshold Signature
A signature scheme where a minimum number of participants (e.g., 3 out of 5) must collaborate to produce a valid signature. Used in FortunaX to aggregate oracle seeds.
Deck Commitment
A cryptographic hash of the shuffled deck that a dealer publishes before a card game. Proves the deck order was fixed before play began.
