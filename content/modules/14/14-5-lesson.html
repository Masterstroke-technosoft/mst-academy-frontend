02   Real-World Story

The Day Ethereum Gas Fees Broke a Billion-Dollar Protocol

📖 Story

In May 2021, during the peak of the DeFi and NFT boom, Ethereum gas prices hit an all-time high. A simple token swap on Uniswap cost over ₹15,000 in gas fees. Minting an NFT on OpenSea cost ₹25,000. For a brief period, Ethereum became unusable for anyone except whales - the very thing it was supposed to democratise was now a playground for the rich.


The root cause was not a bug in Ethereum's code. It was a fundamental architectural limitation: every single Ethereum node had to process every single transaction. All 500,000 validators were redundantly doing the same work. This is like requiring every bank teller in the world to watch every transaction at every branch before approving a single withdrawal. It is insanely secure, but insanely inefficient. The network could only handle about 30 transactions per second - and it was being asked to handle thousands.


The solution that the Ethereum community has been working towards for over five years is called sharding. Instead of every node processing every transaction, the network would be split into multiple parallel chains (shards), each processing a subset of transactions. But the original sharding plan turned out to be incredibly complex - it would require shards to communicate with each other, users to manage cross-shard transactions, and developers to build for a fragmented ecosystem.


Then, in 2020, a researcher named Dankrad Feist proposed a radical simplification: don't shard execution - shard data. Instead of splitting transaction processing across shards, just use shards to store the data that rollups need to post. Rollups would handle execution; Ethereum would handle data availability and security. This insight, which became known as Danksharding, fundamentally reshaped Ethereum's roadmap - and it is now being implemented, starting with Proto-Danksharding (EIP-4844) in March 2024. In this sub-module, you'll learn exactly how this works, and how the modular blockchain revolution it sparked is reshaping the entire industry.

03   What Is Sharding? - The Database Technique That Inspired Blockchain Scaling

Divide and Conquer - How Splitting Data Across Servers Multiplies Throughput

Sharding is a database architecture technique where a large database is split into smaller, more manageable pieces called shards. Each shard contains a subset of the data and can be stored on a different server. When a query comes in, only the server holding the relevant shard needs to process it - not every server in the system. This is how Google, Amazon, and every major internet company handle billions of users: they shard their databases so that no single machine has to hold or process everything.

In the blockchain context, sharding means splitting the blockchain into multiple parallel chains (shards), each with its own state and its own set of validators. Instead of every validator processing every transaction (as Bitcoin and Ethereum do today), each shard processes only the transactions that belong to that shard. The result is that throughput increases linearly with the number of shards: 64 shards can process roughly 64 times as many transactions as a single chain.

Sharding splits the blockchain into parallel chains, each handling a portion of the total transaction load. The Beacon Chain coordinates the shards and provides finality.

The original Ethereum sharding plan (2015-2020) envisioned 64 shards, each processing its own transactions and smart contracts. Users would have accounts on specific shards, and cross-shard communication would be handled by a central Beacon Chain. This was enormously complex: developers would need to decide which shard to deploy their contracts on, users would need to manage assets across shards, and the cross-shard communication protocol was a nightmare of edge cases and security assumptions.

The breakthrough - data sharding instead of execution sharding: Dankrad Feist's key insight was that the rollup-centric roadmap made execution sharding unnecessary. If all execution was happening on rollups anyway, Ethereum L1 didn't need to shard execution - it only needed to provide enough cheap data storage for rollups to post their compressed transaction data. This is Danksharding: the shards don't process transactions; they just store data. Rollups do the heavy lifting of execution; Ethereum provides the data availability and security backbone.

04   Proto-Danksharding (EIP-4844) - The First Step Towards Danksharding

Blobs, KZG Commitments, and the 90% Rollup Fee Reduction

In March 2024, Ethereum shipped the Cancun-Deneb upgrade, which included EIP-4844 (Proto-Danksharding). This is the first concrete step on the road to full Danksharding, and it introduced a revolutionary new transaction type: blob-carrying transactions.

What is a blob? A blob (Binary Large Object) is a chunk of data - up to 128 KB - that can be attached to a transaction. Unlike the data in regular transaction calldata, blobs are not stored permanently on Ethereum. They are stored temporarily (for about 18 days) and then deleted. This temporary storage is vastly cheaper than permanent calldata storage. Rollups can post their compressed transaction data as blobs instead of as calldata, reducing their data posting costs by 90-99%.

Why temporary storage? The data that rollups post to L1 is needed only for a short period - long enough for anyone to reconstruct the rollup's state if they need to challenge a transaction. After a few weeks, the data has served its purpose. There is no need to burden Ethereum's permanent state with it. By using temporary blob storage, rollups get the security of L1 data availability without the permanent storage cost.

EIP-4844 introduces blob storage that is cheap because it's temporary. This reduces rollup costs dramatically without compromising security.

How KZG commitments make blobs secure: A blob is 128 KB of data. The Ethereum execution layer does not store or verify the entire blob - that would defeat the purpose. Instead, the execution layer stores only a KZG commitment - a 48-byte cryptographic proof that acts as a fingerprint of the blob's contents. If a rollup later claims "the data was X," anyone can verify that claim against the KZG commitment without needing the original data. This is the cryptographic magic that makes Danksharding work: the full data is available to anyone who wants it (by downloading from beacon nodes), but the execution layer only needs to check a tiny proof.

🔑 KZG Commitments Explained Simply

Think of a KZG commitment like a cryptographic hash with superpowers. A regular hash (like SHA-256) gives you a fingerprint of data. If someone later claims "the original data was this," you can hash their claim and compare it to the fingerprint. But with KZG, you can do something much more powerful: you can prove specific properties of the data (like "this data is a valid polynomial of degree N") without revealing the data itself. This is what enables data availability sampling - proving that data is available without everyone downloading the whole thing.

05   Full Danksharding - The Endgame for Ethereum's Data Layer

64 Shards of Data, All Secured by the Beacon Chain

Proto-Danksharding (EIP-4844) supports only a small number of blobs per block (target of 3, maximum of 6). This provides enough cheap data for current rollup demand, but as rollups grow, more data capacity will be needed. Full Danksharding is the next step: splitting the data layer into 64 shards, each capable of holding multiple blobs, and using data availability sampling (DAS) to verify that all the data is available without requiring every node to download it.

Data Availability Sampling (DAS): This is the breakthrough that makes full sharding feasible. The problem with 64 shards of data is that no single node could possibly download and verify all of it. DAS solves this by letting each node randomly sample small portions of the data from each shard. Through the magic of erasure coding and KZG commitments, if a node successfully samples enough random pieces, it can be statistically certain that the entire data is available - even though it has only downloaded a tiny fraction.

The analogy: Imagine you have a 1,000-page book, and you want to verify that the entire book exists in a library without reading all 1,000 pages. If you randomly read 30 pages from different parts of the book and they all match what they're supposed to be, you can be >99.99% confident that the whole book is there. If even one page were missing, the erasure coding would make it detectable by sampling. This is how DAS allows thousands of light nodes to verify 64 shards of data collectively, without any single node shouldering the entire burden.

06   The Modular Blockchain Thesis - Unbundling the Monolith

Execution, Settlement, Consensus, and Data Availability as Separate Layers

A monolithic blockchain does everything itself: it executes transactions, reaches consensus on their order, settles finality, and stores all the data. Bitcoin and Ethereum L1 are monolithic. This works, but it means the chain can only scale as fast as its weakest component. A modular blockchain unbundles these four functions into separate, specialised layers that can be optimised independently.

The four layers of a modular blockchain stack:

1. Execution Layer: Processes transactions and computes state transitions. This is what rollups do - they execute transactions off-chain and post the results (and the data needed to verify them) to a data availability layer.

2. Data Availability (DA) Layer: Stores the transaction data and makes it available for anyone to download and verify. This is what Celestia does - it guarantees that data is published and available, without caring what the data means.

3. Consensus Layer: Orders transactions and reaches agreement on the canonical chain. In a modular stack, the consensus layer often overlaps with the DA layer.

4. Settlement Layer: Provides finality and dispute resolution. This is what Ethereum provides for its rollups - a neutral, secure court where disputes can be resolved.

07   Celestia - The First Modular Blockchain Built Only for Data Availability

A Chain That Doesn't Execute Smart Contracts - and That's the Point

Celestia (formerly called LazyLedger) is a blockchain that was purpose-built to do exactly one thing: order transactions and guarantee their data is available. It does not execute smart contracts. It does not have a virtual machine. It does not even know what the transactions it stores actually mean. It simply orders them, makes the data available, and lets other chains (sovereign rollups) handle execution.

This is a radically different philosophy from traditional blockchains. Celestia's creators argued that the hardest problem in scaling is not execution - it's data availability. Execution can be off-loaded to rollups. Consensus can be inherited from the data availability layer. But if the data isn't available, no amount of clever rollup technology can save you. By focusing exclusively on data availability, Celestia can optimise for that one thing and achieve massive throughput.

How Celestia works:

1. Rollups post transaction data to Celestia. When a rollup processes a batch of transactions, it publishes the compressed data to Celestia instead of to Ethereum. Celestia orders this data and includes it in blocks.

2. Celestia uses data availability sampling. Light nodes on Celestia can verify that data is available by randomly sampling small portions, using the same DAS technique that full Danksharding will use. This allows Celestia to have very large blocks (up to 8 MB) without requiring every node to download everything.

3. Sovereign rollups read the data and execute. Each rollup is a "sovereign" chain that reads its own data from Celestia, executes transactions, and maintains its own state. There is no central settlement layer - each rollup is responsible for its own validity rules.

4. Light nodes are first-class citizens. Celestia is designed so that even the lightest node (a mobile phone, theoretically) can verify data availability by sampling. This is in contrast to Ethereum, where running a full node requires significant hardware.

🔑 Sovereign Rollups vs. Smart Contract Rollups

On Ethereum, rollups like Arbitrum deploy a smart contract on L1 that enforces the rollup's rules. This contract verifies fraud proofs and processes withdrawals. These are smart contract rollups - the L1 enforces the rollup's validity. On Celestia, rollups are sovereign - they enforce their own rules. Celestia doesn't know or care whether a rollup's transactions are valid. The rollup's own full nodes check validity. This gives sovereign rollups more flexibility (they can fork independently, change their rules without L1 governance) but requires users to run rollup-specific nodes to verify correctness.

08   Data Availability Sampling - The Cryptographic Breakthrough Enabling Modular Chains

How Light Nodes Can Verify Massive Blocks Without Downloading Them

Data availability sampling (DAS) is the cryptographic technique that makes both Danksharding and Celestia possible. Without it, sharding would require every node to download every shard's data - defeating the purpose of sharding. With DAS, light nodes can verify data availability by downloading only a tiny fraction of the data.

How DAS works, step by step:

1. Erasure coding: The original data block (e.g., 1 MB) is expanded using erasure coding into a larger block (e.g., 2 MB). Erasure coding is like RAID for data - if any portion of the expanded data is lost, the original can be reconstructed from the remaining portions. This is the same technology that allows a scratched DVD to still play.

2. KZG commitments: A KZG commitment is generated for the entire data block. This commitment allows anyone to verify that a specific piece of data is part of the committed block, without needing the whole block.

3. Random sampling: A light node randomly selects a small number of positions in the erasure-coded data (e.g., 30 positions out of thousands) and requests just those pieces from full nodes. For each piece, the light node verifies the KZG proof that the piece is correct.

4. Statistical certainty: If all sampled pieces are valid, the light node can be >99.99% confident that the entire data block is available. The mathematics of erasure coding guarantees that if enough random samples check out, the full data must be reconstructable.

The beautiful result: A network of thousands of light nodes, each downloading only a few kilobytes of data, can collectively verify that hundreds of megabytes of data are available. No single node needs to download everything. This is how Celestia achieves scalable data availability without sacrificing decentralisation - and it's the same principle that will power Ethereum's full Danksharding.

09   Monolithic vs. Modular - The Great Debate

Which Architecture Wins, and When?

PropertyMonolithic (Ethereum L1, Solana, MST)Modular (Celestia + Rollups)
Complexity for developersLow - one chain, one set of rulesHigh - must choose DA layer, settlement layer, rollup framework
ComposabilityFull - all dApps share one stateFragmented - dApps on different rollups cannot interact atomically
Scalability ceilingLimited by hardware (single chain)Very high - can add more rollups and DA capacity independently
Security modelUnified - one consensus, one security budgetFragmented - security depends on each layer's design
User experienceSimple - one wallet, one gas tokenComplex - bridging, multiple tokens, multiple UIs
Innovation speedSlower - changes affect the whole chainFaster - each layer can innovate independently
Current maturityMature (Bitcoin, Ethereum, Solana)Early (Celestia launched 2023, ecosystem growing)

The modular approach is not inherently better - it's a trade-off. Modular chains sacrifice composability and simplicity for scalability and flexibility. Monolithic chains sacrifice scalability for simplicity and composability. Which is "better" depends on the use case. A DeFi protocol that needs atomic composability with other protocols may be better on a monolithic chain. A high-throughput gaming ecosystem may benefit from a modular architecture where each game runs its own rollup.

10   MST's Approach - A High-Throughput Monolith with Modular Ambitions

Why MST Chose the Monolithic Path - and How It Can Adopt Modularity Later

MST Blockchain is fundamentally a monolithic L1, designed to scale at the base layer. With 4,000+ TPS, ~3-second blocks, and fees a fraction of a paisa, MST delivers performance that Ethereum can only achieve through rollups - natively, without fragmentation. This is a deliberate design choice: for most applications, the simplicity of a single high-throughput chain outweighs the scalability ceiling of a modular approach.

However, MST is not ideologically opposed to modularity. Several features in MST's ecosystem point toward a future where modular components can be integrated:

WASMify: Already separates execution from the base chain - WebAssembly modules can run off-chain and be verified on-chain. This is essentially a modular execution layer.

ZK-based Communication: Uses zero-knowledge proofs for private messaging, a technology that can be extended to ZK-Rollups if scaling demands it.

Fast finality + low fees: Makes MST an ideal settlement layer for future rollups. Verifying a ZK proof on MST costs a fraction of what it costs on Ethereum.

The likely trajectory for MST is a hybrid approach: a high-throughput L1 for most applications, with the ability to support modular execution layers (via WASMify and ZK proofs) for applications that need dedicated throughput or privacy. This gives developers the best of both worlds: the simplicity of a monolithic chain for everyday use, and the flexibility of modular components when needed.
