02   Real-World Introduction Story

The Factory That Couldn't Speak Blockchain

📖 Story

Maruti Suzuki's Manesar plant runs a sophisticated SAP ERP system written in ABAP and C++. Every day, 5,000 car parts arrive from 200 suppliers. The receiving dock scans each pallet, updates the SAP inventory, and generates a three-way match between the purchase order, the goods receipt, and the invoice. This system is fast, reliable, and entirely off-chain.


The company wants to bring this data on-chain to trigger smart contract payments - paying a supplier automatically when goods are received and verified. The problem: the SAP system cannot call a smart contract directly. And even if it could, Maruti does not want to expose its entire inventory database on a public blockchain. The pricing agreements with suppliers are confidential. The production schedules are trade secrets.


Traditional oracles would require Maruti to run a Chainlink node, expose an API, and trust that the node operator won't leak the data. That's a non-starter for a publicly listed company with strict data governance.


Enter WASMify. Maruti's existing C++ inventory module - the same code that has run for years - is compiled to WebAssembly (WASM). A lightweight WASMify runtime sits at the edge of Maruti's network, watching the SAP database. When a goods receipt is recorded, the WASM module computes a cryptographic proof that "Pallet #45821 arrived at Dock 7 at 14:32 IST." The proof - but not the raw SAP data - is posted to an MST smart contract. The smart contract verifies the ZK-proof using WASMify's on-chain verifier. If valid, it releases payment to the supplier.


No SAP data ever leaves Maruti's firewall. The proof is public and verifiable, but it reveals nothing about inventory levels, pricing, or supplier identities beyond what is strictly needed. WASMify bridged a C++ ERP system to a Solidity smart contract without rewriting a single line of business logic.

WASMify solves the deepest problem in enterprise blockchain adoption: how to connect legacy systems to smart contracts without exposing confidential data and without trusting a third-party oracle.

03   Background & History

From Oracle Networks to Verifiable Computation - Why WASMify Exists

2015 - 2017
The Oracle Problem Emerges. Smart contracts are deterministic walled gardens - they cannot fetch external data. The first oracles were simple: a trusted entity would post data on-chain. Chainlink (2017) improved this by decentralising the oracle network, but the fundamental problem remained: you must trust the oracle nodes to report honestly and keep your data confidential.
2017
WebAssembly Becomes a W3C Standard. WASM emerged as a portable, sandboxed binary format that runs at near-native speed in browsers and on servers. Its key property: code written in C, C++, Rust, Go, or even Python (compiled) can run identically anywhere - on a laptop, in a browser, or in a blockchain node.
2018 - 2020
Zero-Knowledge Proofs Mature. zk-SNARKs and zk-STARKs evolved from theoretical cryptography to practical toolkits (ZoKrates, Circom, StarkWare). It became possible to prove that a computation was performed correctly without revealing the inputs. The missing piece: how to run arbitrary business logic inside a ZK proof without rewriting everything in a ZK-DSL.
2021 - 2023
ZK-WASM Research. Projects like zkWASM and RISC Zero demonstrated that a WebAssembly runtime could be implemented inside a ZK circuit. This meant any program compiled to WASM could be executed inside a ZK proof. The output would be both the computation result and a proof of correct execution - verifiable by anyone without re-running the program.
2024 - Today
WASMify on MST. MST integrates a ZK-WASM runtime directly into its validator nodes. Enterprises compile existing code (C++, Rust, Python) to WASM, run it off-chain with private data, and submit the result + ZK proof to an MST smart contract. The contract verifies the proof against an on-chain WASM bytecode hash, ensuring the correct code was executed on the claimed inputs. No oracle trusted. No data leaked. Legacy systems now speak blockchain.
💡 The WASMify Breakthrough

WASMify doesn't ask enterprises to rewrite their code. It asks them to recompile their code. A C++ inventory module that has run reliably for a decade becomes a ZK-provable computation with a single compilation target change. That's the power of the WASM standard combined with ZK proofs.

04   Beginner-Friendly Explanation

The Notary Public Analogy - How WASMify Verifies Without Seeing

📜 The Notary & The Sealed Envelope

Imagine you need to prove to a bank that you earn more than ₹50,000 per month, but you don't want to show them your exact salary slip. You take your salary slip to a notary public. The notary verifies the document, stamps it, and gives you a sealed certificate that says "Verified: income > ₹50,000." The bank trusts the notary's stamp without ever seeing the original document.

In WASMify, the salary slip is your SAP data, the notary is the ZK-WASM runtime, the sealed certificate is the ZK proof, and the bank is the MST smart contract. The smart contract verifies the cryptographic stamp without ever seeing the confidential business data.

W
WHY
Why does WASMify exist?
Because enterprises cannot expose confidential data on public blockchains, and they cannot rewrite decades of legacy code in Solidity. WASMify lets them run existing code, keep data private, and prove results on-chain - all without trusting an oracle operator.
W
WHAT
What exactly is WASMify?
WASMify is MST's off-chain computation framework with on-chain ZK verification. It consists of: (1) an off-chain WASM runtime that executes WebAssembly modules with private inputs, (2) a ZK prover that generates a cryptographic proof of correct execution, and (3) an on-chain verifier smart contract that checks the proof against the known WASM bytecode hash.
H
HOW
How does the technical flow work?
1) Developer writes a computation in C++/Rust (e.g., "verify GPS coordinates are within a delivery zone"). 2) Code is compiled to WASM bytecode. 3) The WASM bytecode hash is registered on-chain in a WASMify Registry contract. 4) Off-chain, the WASM module runs with private inputs (actual GPS data). 5) The ZK prover generates a proof: "I ran bytecode hash 0xABC with inputs that satisfy the condition, and the output is TRUE." 6) The proof is submitted to the on-chain verifier. 7) The verifier checks the proof in ~200k gas. 8) If valid, a smart contract state changes (e.g., payment released).
W
WHERE
Where is WASMify deployed?
WASMify is integrated into MST's validator nodes and available as a standalone edge runtime for enterprises. It is the backbone of MST's RWA (Real-World Asset) Orbitals, supply-chain track-and-trace, IoT data verification, and any dApp that needs trustless off-chain computation with on-chain settlement.
L
LIMITS
What are the limitations?
ZK-WASM proof generation is computationally intensive - currently, proving a complex computation may take minutes on commodity hardware, though MST is investing in GPU acceleration. The WASM module must be deterministic (no random number generation without a seeded VRF). Very large datasets (gigabytes) may not fit in the WASM linear memory limit (currently 4GB). And the on-chain verifier must be kept updated as the ZK proving system evolves.
05   Technical Deep Dive

The Four Components of WASMify - From Code to Verified Proof

1
WASM Compilation Target - Any Language to Any Chain
WebAssembly is a portable, low-level bytecode format supported by LLVM, the most widely used compiler infrastructure. This means any language that compiles to LLVM can compile to WASM: C, C++, Rust, Go (via TinyGo), Zig, and increasingly Python (via Pyodide) and .NET (via Blazor). For an enterprise, this is transformative: the exact same C++ code that runs their inventory system can be compiled to WASM with zero logic changes. The WASM binary is deterministic - given the same inputs, it produces the same outputs on any platform. This determinism is essential for ZK-proof generation, which requires that the computation be reproducible.
2
Off-Chain WASM Runtime - The Computation Engine
WASMify provides a lightweight runtime that executes WASM modules outside the blockchain. The runtime loads the WASM bytecode, injects the private inputs (from a database, an IoT sensor, or an API call), and executes the module sandboxed - the WASM code cannot access the host filesystem or network, ensuring it only operates on the provided data. The output is a simple result (e.g., a boolean, a number, a hash) plus a complete execution trace - a record of every instruction executed and every memory access. This trace is passed to the ZK prover. The runtime can be deployed on-premises (inside the enterprise firewall), on a cloud instance, or on an MST validator node, depending on the trust model.
3
ZK Prover - Turning Computation into a Verifiable Proof
The ZK prover takes the WASM bytecode, the public inputs, and the execution trace, and generates a zero-knowledge proof - typically a zk-STARK (for transparency and post-quantum security) or a zk-SNARK (for smaller proof size). The proof cryptographically attests: "The WASM program with bytecode hash H, when executed with some private inputs, produced output O, and the execution was correct according to the WASM specification." The private inputs are never revealed. The public inputs (like the WASM hash and the output) are committed in the proof. The proof itself is a few kilobytes (for SNARKs) to a few hundred kilobytes (for STARKs) and can be submitted in a single blockchain transaction.
4
On-Chain Verifier - The Smart Contract Gatekeeper
MST nodes include a precompiled contract (similar to ecrecover at 0x01) specifically for WASMify proof verification. The verifier takes three inputs: the WASM bytecode hash (stored on-chain when the module is registered), the claimed output, and the ZK proof. It runs a constant-time cryptographic verification (~200,000 gas) and returns true if the proof is valid. A smart contract can then use this result to release funds, update state, or trigger another contract. The verifier does not re-execute the WASM code - it only checks the proof. This is the magic of ZK: verification is exponentially cheaper than execution. The on-chain verifier is also used to anchor the WASM bytecode hash, ensuring that only approved modules can generate valid proofs.
🗝️ Why WASMify Replaces Traditional Oracles

A traditional oracle reports data. You must trust the oracle to report honestly. WASMify reports a proof of computation. You trust the cryptography, not the operator. Even if the WASM runtime is compromised, it cannot forge a valid proof for a different computation because the proof is tied to the specific bytecode hash registered on-chain.

07   Comparison

WASMify vs Traditional Oracles - Why ZK-Verification Wins

Feature Centralised Oracle Decentralised Oracle (Chainlink) WASMify (MST)
Trust model Trust the single operator Trust majority of nodes Trust the cryptography (ZK proof)
Data confidentiality Operator sees all data All oracle nodes see data Private inputs never leave enterprise
Computation capability Simple data relay Limited off-chain computation Full WASM: any language, complex logic
Legacy code reuse Rewrite as API Rewrite as Chainlink adapter Recompile existing C++/Rust code
Verification cost (on-chain) Low (just read storage) Low (read aggregated result) ~200k gas per proof verification
Proof of correct execution None Economic (stake slashing) Cryptographic (ZK proof)
💡 The Enterprise Decision

If your use case requires only a public price feed (like ETH/USD), a decentralised oracle is sufficient. If your use case involves confidential business logic that must be proven correct without revealing the underlying data, WASMify is the only viable option.

08   Real-World Case Studies

WASMify in Production - Where It Changes the Game

🚚
Cold Chain Pharmaceutical Logistics
A vaccine distributor uses WASMify to verify temperature compliance during transport. Their existing C++ IoT monitoring code is compiled to WASM. At each delivery checkpoint, the WASM runtime reads the temperature log from the truck's sensor, computes whether all readings stayed within the 2 - 8°C range, and generates a ZK proof. The proof is submitted to an MST smart contract that releases payment only if the proof is valid. The temperature data itself stays on the distributor's server - the buyer sees only the VALID/INVALID result with cryptographic certainty.
🏭
Automotive Supply Chain - Just-In-Time Manufacturing
An automotive OEM uses WASMify to connect their SAP MM (Materials Management) module to a smart contract-based supplier payment system. When a goods receipt is posted in SAP, a WASM module (compiled from the existing ABAP-to-C++ translation layer) generates a proof that the goods were received against a specific purchase order. The proof triggers an on-chain payment. SAP pricing data remains confidential; only the receipt confirmation reaches the chain.
🏦
Trade Finance - Document Verification Without Data Exposure
A trade finance consortium uses WASMify to verify shipping documents. A bank's document checking system (written in Python, compiled to WASM via Pyodide) runs off-chain, verifying that a bill of lading matches the letter of credit terms. The ZK proof confirms compliance without revealing the actual document contents to the blockchain. The importer's bank and exporter's bank both trust the proof, releasing funds instantly.
📡
IoT Mesh Networks - Verifiable Sensor Data for Parametric Insurance
A parametric crop insurance product uses WASMify to verify rainfall data from a network of IoT sensors across Maharashtra. The WASM module (Rust, compiled to WASM) aggregates readings from multiple sensors, computes the rainfall index, and generates a proof that the index fell below the policy threshold. The proof triggers an automatic payout to farmers. Sensor data remains on the IoT network; only the cryptographically verified index result reaches the insurance smart contract.

Top Facts - Know These Cold for the 20-Mark Assessment

01
WASMify = WASM Runtime + ZK Prover + On-Chain Verifier
Three components: off-chain execution of WebAssembly, ZK proof generation, and on-chain proof verification via a precompiled contract.
02
Any Language → WASM → ZK Proof
C, C++, Rust, Go, Python (compiled) can all produce WASM bytecode. WASMify wraps this in a ZK circuit, generating a proof of correct execution.
03
Private Inputs Never Leave the Enterprise
Only the ZK proof and the public output reach the blockchain. The verifier confirms execution correctness without knowing the inputs.
04
Bytecode Hash Anchored On-Chain
The WASM bytecode hash is registered in the WASMify Registry. The verifier only accepts proofs that reference a registered hash, preventing malicious code execution.
05
Replaces Trust with Cryptography
Unlike oracles that require trust in node operators, WASMify replaces trust with ZK proofs - economic incentives and reputation are unnecessary when math guarantees correctness.
10   Glossary

Key Terms - Sub-Module 7.5

WASMify
MST's off-chain computation framework with on-chain ZK verification, enabling legacy code to connect to smart contracts without exposing private data.
WebAssembly (WASM)
A portable, sandboxed binary instruction format that runs at near-native speed. Can be compiled from C, C++, Rust, Go, and many other languages.
ZK Proof (Zero-Knowledge Proof)
A cryptographic proof that a computation was performed correctly, without revealing the inputs. Allows verification without trust.
WASMify Registry
An on-chain smart contract storing the hashes of approved WASM modules. The verifier only accepts proofs referencing registered hashes.
ZK Prover
The off-chain component that generates a ZK proof from the WASM execution trace. Computationally intensive but run outside the blockchain.
On-Chain Verifier (Precompile)
A built-in smart contract on MST that verifies ZK proofs in ~200k gas, acting as the cryptographic gatekeeper for WASMify.
Execution Trace
A complete record of every instruction and memory operation during WASM execution, used by the ZK prover to generate the proof.
LLVM
A compiler infrastructure that supports many languages and can target WASM, enabling C/C++/Rust code to run in WASMify.
Private Input
Data fed to the WASM module that remains confidential - the ZK proof attests to correct processing without revealing this data.
Deterministic Computation
A computation that always produces the same output for the same input. Required for ZK proof generation to be reproducible.
