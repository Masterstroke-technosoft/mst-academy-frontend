MST's gaming protocols solve the fundamental trust problem in online gaming. You don't trust the casino. You trust the math - and you can verify it yourself.

03   Background & History

From "Trust Us" to "Verify Yourself" - The Evolution of Gaming Fairness

1990s - 2000s
The Black Box Era. Online casinos ran fully centralised servers. The shuffle, the RNG, the payout percentages - everything was a black box. Regulators relied on periodic audits of source code, but there was no way for a player to verify that the code running at 2 AM was the same code that was audited. Scandals were frequent and largely undetectable.
2012
Provably Fair - The First Generation. Bitcoin casinos introduced "provably fair" algorithms. The server would commit to a random seed before a game round, and the player could verify the outcome afterward. While an improvement, these systems still relied on the server generating the seed - a sophisticated operator could manipulate the seed generation itself.
2018
On-Chain Randomness - Chainlink VRF. Chainlink introduced a Verifiable Random Function (VRF) that provided on-chain randomness from a decentralised oracle network. This solved the seed manipulation problem but was expensive (high gas fees) and not natively integrated into gaming logic like shuffling or RTP calculations.
2022 - 2023
Specialised Gaming Chains. Some L1/L2 chains targeted gaming specifically, but they focused on scalability (high TPS) rather than game-specific fairness guarantees. Shuffling and RTP were left to individual game developers to implement, leading to inconsistent and often flawed implementations.
2024 - Today
MST's Gaming Protocols. MST builds fairness directly into the protocol layer. Klethesia provides on-chain RTP verification for slots. SatvaShuffle uses ZK proofs for verifiable card shuffling. FortunaX delivers a distributed randomness beacon secured by Trika POSA validators. Game developers don't need to implement fairness - they inherit it from the chain.
💡 The Protocol-Level Advantage

When fairness is built into the blockchain protocol itself, game developers cannot bypass it - even if they want to. This is fundamentally different from smart-contract-based fairness, where each developer must implement (and potentially misimplement) their own solution.

04   Beginner-Friendly Explanation

The Three Protocols - Klethesia, SatvaShuffle, and FortunaX

🎰 The Casino Manager, The Dealer, and The Lottery Machine

Klethesia is like a casino manager who publishes the slot machine's exact payout rate on a public notice board, and after every spin, posts the result and the random seed so anyone can verify it. SatvaShuffle is like a card dealer who shuffles the deck inside a glass box - you can't see the cards, but you can see the shuffle happen, and afterward you can verify the deck was standard and the shuffle was random. FortunaX is like a lottery machine that broadcasts its random number generation live, with multiple independent auditors confirming each draw.

W
WHY
Why do these protocols exist?
Because players cannot trust online gaming operators to be fair. Even licensed operators have been caught manipulating outcomes. Cryptographic proof eliminates the need for trust entirely.
W
WHAT
What does each protocol guarantee?
Klethesia: Guarantees slot machine RTP matches published rates; every spin is verifiable. SatvaShuffle: Guarantees a random permutation of a standard deck; no card manipulation possible. FortunaX: Guarantees unpredictable, unbiasable random numbers for any game mechanic.
H
HOW
How do they work technically?
Klethesia: Each spin is driven by a VRF output. The RTP is enforced by a smart contract that accumulates payouts and verifies the long-term percentage matches the advertised rate. SatvaShuffle: Uses a ZK-SNARK to prove that a shuffled deck is a valid permutation of the original deck, without revealing the deck order until after the hand. FortunaX: Combines entropy from multiple validators into a single random number using threshold BLS signatures, ensuring no single validator can predict or bias the result.
W
WHERE
Where are they applied?
Klethesia: Slot machines, crash games, roulette. SatvaShuffle: Rummy, poker, teen patti, any card game. FortunaX: Loot boxes, lottery draws, prize wheels, any game needing a single random outcome.
L
LIMITS
What are their limitations?
Klethesia's RTP verification is statistical - it requires a large number of spins to prove the advertised RTP with confidence. SatvaShuffle's ZK proofs add computational overhead (~1-2 seconds for proof generation, though verification is fast). FortunaX's distributed randomness requires validator participation; if too many validators are offline, randomness generation may be delayed (mitigated by a fallback threshold scheme).
05   Technical Deep Dive

Inside Klethesia, SatvaShuffle, and FortunaX

1
Klethesia - Provable RTP for Slot Machines
Klethesia enforces Return-to-Player (RTP) at the protocol level. Each slot machine is a smart contract that registers its advertised RTP (e.g., 96.5%). Every spin generates a random outcome using FortunaX's randomness beacon. The contract tracks cumulative bets and cumulative payouts. Over a configurable epoch (e.g., 100,000 spins), the contract verifies that the actual RTP falls within an acceptable statistical band (±1%) of the advertised rate. If the operator attempts to lower the effective RTP below the band, the contract can automatically freeze and refund players. Additionally, every spin outcome is published on-chain with a commitment to the random seed, allowing auditors to verify the fairness of individual results.
2
SatvaShuffle - ZK-Verified Card Shuffling
SatvaShuffle is a zero-knowledge shuffle protocol. Before a hand, the dealer (the game smart contract) generates a permutation of a 52-card deck using FortunaX randomness. It commits to the shuffled deck by publishing a Merkle root of the card positions on-chain. A ZK-SNARK proof is generated to attest: "There exists a permutation of the standard 52-card set that produces this Merkle root." This proof can be verified by any player for ~200k gas, guaranteeing the deck is a valid shuffle without revealing the card order. As the hand progresses, cards are revealed by publishing Merkle proofs from the committed root. After the hand, the full deck can be revealed, and players can independently recompute the Merkle root to confirm no cards were added, removed, or swapped.
3
FortunaX - Distributed Randomness Beacon
FortunaX is MST's native randomness source. It operates as an on-chain randomness beacon that produces unpredictable, unbiasable random values at regular intervals. The beacon uses a threshold BLS signature scheme: each Trika POSA validator contributes a random share (derived from a VRF seeded by the previous beacon output). Once a threshold of shares (e.g., ⅔) is collected, they are aggregated into a single BLS signature, which serves as the random output. Because no single validator knows the final output until the threshold is reached, and because the output is deterministic given the shares, the randomness is unpredictable (no validator can bias it without colluding with ⅔ of the set) and publicly verifiable (anyone can check the BLS signature against the known public keys). Game contracts request a random number by calling the FortunaX precompile, which returns the latest beacon output along with a proof of correctness.
🗝️ Why Protocol-Level Fairness Matters

If a game developer deploys on MST, they automatically use FortunaX for randomness, Klethesia for RTP (if applicable), and SatvaShuffle for card games. The chain enforces fairness - the developer cannot write a contract that bypasses these protocols. This is a far stronger guarantee than a "fairness audit" of a centralised game server.

10   Glossary

Key Terms - Sub-Module 7.6

Klethesia
MST's protocol for on-chain RTP (Return-to-Player) enforcement for slot machines, with statistical verification over epochs.
SatvaShuffle
MST's ZK-verified card shuffling protocol that commits to a deck order and proves it is a valid permutation without revealing cards.
FortunaX
MST's distributed randomness beacon using threshold BLS signatures from validators to produce unpredictable numbers.
RTP (Return-to-Player)
The percentage of total bets that a slot machine returns to players as winnings over its lifetime, e.g., 96.5%.
Merkle Root
A cryptographic commitment to a data set; changing any element changes the root, allowing efficient verification of inclusion.
ZK-SNARK
A zero-knowledge proof that is succinct and non-interactive; used by SatvaShuffle to prove a shuffle is a valid permutation.
Threshold BLS Signature
A signature scheme where multiple parties contribute shares; when a threshold is reached, a single aggregate signature is produced.
Commitment Scheme
A cryptographic primitive where a value is hidden but locked (committed) at one time, and revealed later, preventing changes after commitment.
Statistical Enforcement
Klethesia's method of ensuring RTP compliance over many spins, rather than per spin, with automated flagging on deviations.
Provably Fair
A system where players can independently verify game outcomes using cryptographic proofs, without trusting the operator.
