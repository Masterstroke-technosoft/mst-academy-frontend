02 · Real-World Story

The dApp That Nobody Could Use

📖 Story

In 2023, a team of three developers won a hackathon with a brilliant DeFi protocol on an EVM testnet. Their smart contracts were flawless - audited, gas-optimised, and ready for mainnet. They rushed to deploy the contracts and submitted their grant application. But the grant committee asked a simple question: "Can you share a link where we can try it?"


The team had no frontend. They scrambled, hacked together a React app in two days, hardcoded the contract address, and deployed it to a random Vercel link. The interface crashed on mobile, the wallet connection button didn't work on Brave browser, and there was no error handling. The grant was rejected - not because the idea was bad, but because the reviewers couldn't evaluate it.


Your capstone is your ticket to an MST grant, a job, or your own startup. A polished frontend and crystal-clear documentation are what separate a hobby project from a professional product. In this sub-module, you’ll build both.

03 · Frontend Architecture for MST dApps

React + MST SDK + BridgeKey = The Golden Stack

The frontend of a Web3 application follows a clean separation: React for UI, ethers.js (or MST’s own SDK) for blockchain communication, and the BridgeKey wallet for user authentication. Because MST is EVM-compatible, you can use the exact same libraries and patterns you would for Ethereum - but with lower fees and faster confirmations.

04 · Setting Up the React Project

From Zero to a Working Wallet Connection in 15 Minutes

We use Vite for fast builds and a clean developer experience. The following steps create a TypeScript React app, install dependencies, and connect to the BridgeKey wallet on MST mainnet.

05 · Reading & Writing to Your Deployed Contracts

Bringing Your Mainnet Contracts to the UI

After connecting the wallet, the user can call view functions (free) and send state-changing transactions (costing gas). You need the contract ABI and the address you deployed in 21.2.

Create a custom hook useContract that returns a contract instance with the signer attached.

import { ethers } from 'ethers'; import { useMemo } from 'react'; import ABI from './MyToken.json'; export function useContract(address: string, signer: ethers.Signer | null) { return useMemo(() => { if (!signer) return null; return new ethers.Contract(address, ABI, signer); }, [signer, address]); }

Then, in a component, you can read a balance or call transfer. Always handle loading and error states. Use toast notifications for pending and confirmed transactions.

💡 Pro UX Patterns

Show a spinner while the transaction is pending. Disable the button during the transaction. Display a success message with a link to the transaction on MST Explorer (https://mstscan.com/tx/{hash}). These small touches make your dApp feel professional.

06 · Deploying to Production - Vercel, Netlify & IPFS

From localhost to a Public URL

Your frontend needs to be accessible to users, judges, and grant reviewers. Here are three battle-tested options.

1
Vercel (Recommended)
Connect your GitHub repo, set the build command to npm run build and output directory to dist. Add environment variables (e.g., VITE_CONTRACT_ADDRESS) in the Vercel dashboard. Deploys in seconds with automatic SSL.
2
Netlify
Similar to Vercel. Drag-and-drop your dist folder or connect a Git repo. Netlify also supports environment variables and preview deployments for each pull request.
3
IPFS via Fleek
For maximum decentralisation, deploy your static site to IPFS. Fleek provides a gateway and automatic pinning. However, updates are slower. Use this if your dApp is fully on-chain and you want to avoid centralised hosting.
🗝️ Environment Variables in Vite

In Vite, environment variables must be prefixed with VITE_ to be exposed to the client. Store your contract address in VITE_CONTRACT_ADDRESS and access it via import.meta.env.VITE_CONTRACT_ADDRESS.

07 · Crafting the Perfect Documentation

README, Technical Specs & Demo Video - The Holy Trinity

Documentation is not an afterthought. It’s the first thing a grant committee, potential employer, or open-source contributor sees. Your capstone repository must include a README.md that answers every question before it’s asked.

✅ What a Great README Contains
  • Project name, tagline, and a live demo link
  • Architecture diagram (can be a Mermaid diagram)
  • List of smart contracts with mainnet addresses
  • Step-by-step local setup instructions
  • How to deploy to testnet/mainnet
  • Security audit summary (from 21.1)
  • Demo video embed or link
  • License & team information
❌ Common Mistakes
  • Missing live demo link - reviewers can’t test
  • Hardcoded private keys (even in examples)
  • No mention of which network the contracts are on
  • README copied from a template with placeholder text
  • No demo video - text alone rarely convinces

10 · Glossary

Key Terms - Sub-Module 21.3

BridgeKey
MST’s native browser extension wallet that injects the Ethereum provider API for dApps.
Vite
A fast frontend build tool that supports React, TypeScript, and environment variable handling with VITE_ prefix.
MST Explorer
The block explorer for MST mainnet (mstscan.com) where contracts are verified and transactions viewed.
IPFS
InterPlanetary File System - a decentralised storage network for hosting static frontends without a single server.
README
The documentation file in a repository that explains what the project does, how to run it, and where the contracts are.
Demo Video
A short screen recording that walks through the dApp’s core features, intended for investors and grant reviewers.
