02 · Real-World Deployment Story

The Night a Mainnet Launch Failed - And Why

📖 Story

In early 2024, a DeFi project built on an EVM chain prepared to launch its governance token. The team had been testing on testnet for weeks. Everything worked perfectly. They had a deploy script, a verified contract, and a waiting community. On launch day, they set the --network mainnet flag in their Hardhat command, entered the private key, and hit enter.


The transaction failed. Gas estimation had reverted because the contract’s constructor included a call to a Uniswap router address that didn’t exist on the new chain. They had hardcoded a testnet address. Worse, the private key they used belonged to a team member who was on vacation - no one else had access to the deployer address. The project launch was delayed by three days, community trust eroded, and the token lost 40% of its anticipated momentum within the first week.


This story isn’t unique. Mainnet deployment is fundamentally different from testnet. On MST, the margin for error is slim - but with the right configuration and a meticulous checklist, you can deploy confidently and become the hero who shipped on time. This sub-module gives you that checklist.

05 · Configuring Hardhat for MST Mainnet

Chain ID 4646 - The Gateway to the Real MST

The MST mainnet is identified by Chain ID 4646. Every transaction on this network requires real MST tokens for gas. The RPC endpoint and block explorer are different from testnet. Your hardhat.config.js must include the correct network configuration.

// hardhat.config.js require("@nomicfoundation/hardhat-toolbox"); require("dotenv").config(); module.exports = { solidity: "0.8.20", networks: { mst_mainnet: { url: process.env.MST_MAINNET_RPC || "https://rpc.mstblockchain.com", chainId: 4646, accounts: [process.env.PRIVATE_KEY] } }, etherscan: { apiKey: { mst_mainnet: process.env.MSTSCAN_API_KEY }, customChains: [ { network: "mst_mainnet", chainId: 4646, urls: { apiURL: "https://api.mstscan.com/api", browserURL: "https://mstscan.com" } } ] } };
🔒 CRITICAL: Never Hardcode Private Keys

Use dotenv to load PRIVATE_KEY from a .env file. Ensure .env is listed in .gitignore. For production deployments, consider using a hardware wallet connected via @nomiclabs/hardhat-ledger or a dedicated deployment account with limited funds.

06 · Acquiring Mainnet MST Tokens

How to Fund Your Deployer Wallet

Unlike testnet, there is no faucet for mainnet. You must obtain real MST tokens through one of these methods:

1
Centralised Exchange (CEX)
Purchase MST on supported exchanges (e.g., KuCoin, Gate.io) and withdraw to your deployer address. Always double-check the withdrawal network is MST Mainnet (Chain ID 4646).
2
Bridge from Ethereum / BSC
Use the official MST Bridge (bridge.mstblockchain.com) to transfer assets from Ethereum or BSC. The bridge will wrap tokens as MST-compatible assets and also provide a small amount of native MST for gas.
3
MST BuildGrant / Partner Fund
If you are part of the MST BuildGrant program, you may receive a deployment stipend in MST tokens. Check your grant agreement for details.

07 · Writing the Production Deployment Script

From Script to Mainnet - Step by Step

A professional deployment script does more than just deploy a contract. It logs every address, waits for confirmations, verifies contracts automatically, and handles errors gracefully. Below is a battle-tested script for deploying your capstone’s core contracts to MST mainnet.

// scripts/deploy_mainnet.js const hre = require("hardhat"); async function main() { const [deployer] = await hre.ethers.getSigners(); console.log("Deploying with:", deployer.address); const balance = await deployer.getBalance(); console.log("Balance:", hre.ethers.utils.formatEther(balance), "MST"); // 1. Deploy Token const Token = await hre.ethers.getContractFactory("MyToken"); const token = await Token.deploy("CapstoneToken", "CST", 1000000); await token.deployed(); console.log("Token deployed to:", token.address); // 2. Deploy Marketplace const Marketplace = await hre.ethers.getContractFactory("Marketplace"); const marketplace = await Marketplace.deploy(token.address); await marketplace.deployed(); console.log("Marketplace deployed to:", marketplace.address); // 3. Wait for a few confirmations await token.deployTransaction.wait(5); await marketplace.deployTransaction.wait(5); // 4. Verify contracts on MST Explorer if (hre.network.name === "mst_mainnet") { console.log("Verifying Token..."); await hre.run("verify:verify", { address: token.address, constructorArguments: ["CapstoneToken", "CST", 1000000], }); console.log("Token verified!"); console.log("Verifying Marketplace..."); await hre.run("verify:verify", { address: marketplace.address, constructorArguments: [token.address], }); console.log("Marketplace verified!"); } console.log("Deployment complete. Update your frontend .env with these addresses."); } main().catch((error) => { console.error(error); process.exitCode = 1; });
💡 Pro Tip: Use Hardhat Ignition (New)

Hardhat Ignition is the new declarative deployment system. It automatically handles contract dependencies, re-deployment detection, and verification. For complex multi-contract capstones, Ignition reduces script complexity significantly.

08 · Contract Verification on MST Explorer

Making Your Code Public and Auditable

Verification links your deployed bytecode to the original Solidity source code on MST Explorer (mstscan.com). Without verification, users cannot read your contract, interact with it via the explorer’s UI, or trust its logic. Verification is mandatory for any MST grant application.

Fig. 21.2-A · Contract Verification Flow
  ┌─────────────┐     ┌─────────────────┐     ┌──────────────┐
  │  Hardhat    │────▶│  MST Explorer   │────▶│  Users &     │
  │  verify     │     │  compares hash  │     │  Developers  │
  │  plugin     │     │  with on‑chain  │     │  can read    │
  └─────────────┘     └────────┬────────┘     └──────────────┘
                               │
                               ▼
                      ┌────────────────┐
                      │  Contract      │
                      │  Source Code   │
                      │  Published ✅  │
                      └────────────────┘
Verification is a one-way operation: once verified, the source code is permanently associated with the contract address.
🗝️ Manual Verification Alternative

If the Hardhat verify plugin fails (e.g., due to compiler version mismatch), you can manually verify via MST Explorer’s “Verify & Publish” page. You’ll need to provide the exact Solidity version, optimisation settings, and constructor arguments (ABI-encoded).

09 · Interacting with Your Live Contracts

First Mainnet Transactions - Reading & Writing

Once deployed and verified, you can interact with your contracts directly from the MST Explorer’s “Contract” tab. You can call view functions for free, and send state-changing transactions via your BridgeKey wallet. This is also how you test the live flow before sharing the dApp URL with users.

✅ DO
  • Test every function with a small amount of MST first.
  • Check events in the transaction receipt on MST Explorer.
  • Set up monitoring (e.g., Tenderly) for mainnet contracts.
  • Update your frontend .env with live contract addresses.
❌ DON'T
  • Never use the same private key for deployment and daily use.
  • Don't skip verification - unverified contracts look suspicious.
  • Don't forget to transfer ownership to a multisig after deployment.
  • Don't deploy at 2 AM without a team member online to verify.

Q11
DescriptiveMedium2 marks
Explain why you should wait for multiple block confirmations after deploying a contract to MST mainnet before marking it as "live".
✅ Model Answer

Even though MST has fast finality (~3 seconds), waiting for a few additional confirmations (e.g., 5 blocks) guards against rare chain reorganisations. It also ensures the transaction is widely propagated and won't be reverted due to a temporary fork. This is standard practice for high-value deployments.

2 marks: mention of chain reorgs and propagation.
Q12
DescriptiveMedium3 marks
A deployment script fails with "insufficient funds". The deployer address shows a balance of 1 MST on the explorer. List three possible reasons for the failure.
✅ Model Answer (any three)

1. The gas price or gas limit is set too high, making the total cost > 1 MST.
2. The contract's constructor is computationally expensive, requiring more gas than a simple transfer.
3. The deployer address might be on the wrong network (e.g., still connected to testnet).
4. There are pending transactions consuming the nonce, and the effective balance is lower.

3 marks: 1 per valid reason.
Q13
DescriptiveHard3 marks
What is a forked mainnet simulation, and why is it recommended before deploying to MST mainnet?
✅ Model Answer

A forked mainnet simulation uses Hardhat's hardhat node --fork <RPC_URL> to create a local copy of the mainnet state. This allows you to test deployment scripts against real contract addresses, state, and balances without spending real MST. It catches issues like incorrect external contract references or constructor logic that would fail on mainnet.

3 marks: 1 for explaining fork concept, 1 for mentioning real state, 1 for saving costs/catching errors.

Q14

The following deployment script attempts to deploy two contracts and verify them. Identify three mistakes or missing security practices, and rewrite the corrected script sections.
const hre = require("hardhat"); async function main() { const [deployer] = await hre.ethers.getSigners(); const Token = await hre.ethers.getContractFactory("Token"); const token = await Token.deploy("MyToken", "MTK"); console.log("Token:", token.address); const Staking = await hre.ethers.getContractFactory("Staking"); const staking = await Staking.deploy(token.address); console.log("Staking:", staking.address); // verification await hre.run("verify:verify", { address: token.address, constructorArguments: ["MyToken", "MTK"], }); }
✅ Solution

Mistake 1: No await token.deployed() or staking.deployed() - script continues before deployment is confirmed.
Mistake 2: Verification is attempted without waiting for block confirmations; the explorer may not have the contract bytecode yet.
Mistake 3: No network check; verification should only run on mainnet (or testnet) to avoid errors on local networks.
Missing practice: Private key is not secured (not shown, but the script might be using hardcoded keys).

Corrected snippet:

const token = await Token.deploy("MyToken", "MTK"); await token.deployed(); await token.deployTransaction.wait(5); // wait for confirmations if (hre.network.name === "mst_mainnet") { await hre.run("verify:verify", { address: token.address, constructorArguments: ["MyToken", "MTK"], }); }
5 marks: 1 per identified mistake (max 3), 2 for correct corrected code with confirmations and network check.

11 · Glossary

Key Terms - Sub-Module 21.2

Chain ID 4646
The unique network identifier for MST mainnet, used by wallets and Hardhat to prevent cross-chain replay attacks.
RPC Endpoint
The URL through which your script communicates with the MST network (e.g., https://rpc.mstblockchain.com).
MST Explorer
The block explorer for MST (mstscan.com), where contracts are verified and transactions inspected.
Hardhat Verify
Plugin that uploads source code to the explorer to prove bytecode matches the Solidity code.
Multisig Wallet
A smart contract wallet that requires multiple signatures to execute a transaction, improving security.
Forked Mainnet
A local simulation of the mainnet state used for testing deployments without real tokens.
