02 · Real-World Introduction Story

The ₹120 Crore Lesson - Why Audits Are Not Optional

📖 Story

In March 2022, a popular DeFi protocol on an EVM chain was preparing for a V2 launch. The team had built a beautiful UI, a clever staking mechanism, and a governance token. The smart contract was written by a senior developer and reviewed by two other engineers. They ran the Hardhat test suite - all green. The code was deployed to mainnet on a Friday evening.


By Monday morning, ₹120 crore (about $15 million) was gone. The attacker had exploited a subtle reentrancy vulnerability in the withdraw() function. The team had missed it because their test only checked normal withdrawal flow - not cross-contract reentrancy. Worse, they had never run a single automated security tool. No Slither. No Mythril. No fuzzing. Just manual confidence.


This is not a rare story. In 2023 alone, blockchain hacks totalled over $1.8 billion, according to Chainalysis. The vast majority of exploited contracts had never undergone a professional audit or even a basic automated scan. The MST ecosystem is safer by design - but EVM compatibility means existing Solidity vulnerabilities can still travel to MST. As an MST Blockchain Developer, your capstone project will handle real user value. You owe it to yourself and your users to harden every line of code.

03 · Why Final Hardening Matters on MST

EVM Compatibility = Attack Surface Transportability

Because MST is fully EVM-compatible, any security flaw that exists in Ethereum’s Solidity ecosystem can manifest identically on MST. The low gas fees on MST actually increase the risk - attackers can afford to try multiple exploits that would be prohibitively expensive on Ethereum. Moreover, MST’s Trika POSA provides fast finality (~3 seconds), but that also means a successful attack is irreversible almost instantly.

A robust security audit is not just a checkbox for your capstone project; it’s the difference between a funded startup and a catastrophic failure. MST’s BuildGrant requires an audit report signed by a qualified reviewer (even if self-audited with tools). The skills you learn here will be directly applicable in job interviews, where 80% of technical questions revolve around security and gas optimisation.

05 · Automated Analysis - Your First Line of Defence

Slither, Mythril, and Echidna on MST

1
Slither - Static Analysis
Slither (by Trail of Bits) is the industry-standard static analyser for Solidity. It examines your code without executing it, detecting reentrancy, missing events, incorrect modifiers, unused variables, and dangerous delegatecall patterns. Install via pip3 install slither-analyzer. Run it with slither . inside your Hardhat project. Slither outputs a ranked list of findings with severity: High, Medium, Low, Informational. For MST deployment, fix all High and Medium issues before proceeding.
2
Mythril - Symbolic Execution
Mythril explores all possible execution paths of a contract, looking for assertion violations, integer overflows, and unprotected SELFDESTRUCT calls. Unlike Slither, it doesn’t just read the code - it runs it symbolically, reaching deep states. Command: myth analyze contracts/MyContract.sol --execution-timeout 60. It can uncover bugs that a human might miss in nested if-else branches.
3
Echidna - Fuzzing / Invariant Testing
Echidna bombards your contract with random inputs and checks whether any invariant breaks (e.g., “total supply must never exceed 1 million”). You write invariants as Solidity functions with names starting with echidna_. This catches logic errors that static tools miss. For MST, we especially fuzz staking rewards and RWA token mint/burn functions.

06 · Manual Code-Review Checklist

The Human Touch - 12 Points You Must Verify

Automated tools are essential, but they cannot understand business logic. A contract may have zero tool-reported issues yet contain a fatal economic design flaw. Here is the checklist that professional auditors use for every MST dApp.

01
Reentrancy (Checks-Effects-Interactions)
All state changes must happen before external calls. Use OpenZeppelin’s ReentrancyGuard. Never rely on transfer() or send() for reentrancy protection - use CEI pattern.
02
Integer Overflow / Underflow
Solidity ≥0.8 has built-in checks, but explicit unchecked blocks can be dangerous. Verify every unchecked usage with a detailed comment.
03
Access Control
Every state-changing function must have appropriate modifiers (onlyOwner, role-based). Check for missing onlyOwner on critical setters. Avoid tx.origin for authentication.
04
Front-Running & MEV
On MST, validators can still reorder transactions. Use commit-reveal schemes for sensitive actions, and avoid large price-impact operations that can be sandwiched.
05
Unchecked Return Values
Low-level calls (.call()) return a boolean that must be checked. Failing to check can lead to silent failures. Use require(success, "call failed").
06
Denial of Service (DoS)
Avoid loops over unbounded arrays that can hit block gas limits. Use pull-over-push for payments, and never rely on a single external call to succeed for critical logic.
07
Timestamp Dependence
Block timestamps can be manipulated by validators within a small window (~30 seconds). Do not use them for precise randomness or as a sole source of truth for deadlines (use block numbers where appropriate).
08
Delegatecall & Proxy Security
If you use upgradeable contracts, ensure storage slots never clash between implementation versions. Use OpenZeppelin’s TransparentUpgradeableProxy or UUPS with caution.
09
Signature Replay Attacks
If your contract accepts off-chain signatures, include a nonce and chain ID to prevent cross-chain replay. Use EIP-712 structured hashing.
10
Rounding Errors & Precision
In financial contracts, always multiply before dividing to minimise precision loss. Use fixed-point math libraries (PRBMath) for high-accuracy calculations.
11
Event Emission
All state-changing functions should emit events for off-chain monitoring. Missing events make debugging and user-notification impossible.
12
Business Logic & Invariants
Define invariants explicitly (“token A balance + token B balance in pool must equal constant k”). Write Echidna invariants for these. Review any state transition that could break them.
07 · Gas Optimisation - Every Wei Counts

MST Gas Model and Why Optimisation Still Matters

MST’s gas fees are significantly lower than Ethereum’s during congestion (often < 0.01 MST per transfer). However, gas optimisation is still critical for three reasons:

  • Scalability: high-throughput dApps (gaming, RWA micro-transactions) can execute millions of transactions daily. Shaving 20% off each call saves real money.
  • User experience: lower gas = faster confirmation and less friction. This is part of MST’s promise.
  • MST BuildGrant criteria: grant reviewers penalise wasteful code; a gas-optimised contract signals professionalism.
❌ Common Gas Wasters
  • Using memory instead of calldata for external function parameters
  • Reading storage variables multiple times in a loop
  • Unnecessary public visibility (use external)
  • Storing large arrays of structs without packing
  • Using ++i vs i++ in loops (small but cumulative)
✅ Proven Optimisations
  • Cache storage variables into memory once per function
  • Pack multiple uint128 into a single storage slot
  • Use immutable and constant for values known at compile time
  • Short-circuit boolean evaluations (&& / ||)
  • Batch operations using multicall patterns

08 · Practical Walkthrough

Auditing a Sample Staking Contract on MST

Let’s walk through a realistic example: a simple staking pool where users deposit MST tokens and earn rewards. The contract was written by a junior developer and contains several subtle issues. We’ll run Slither, Mythril, then manually fix the problems.

// SPDX-License-Identifier: MIT pragma solidity ^0.8.19; contract StakingPool { mapping(address => uint256) public stakes; mapping(address => uint256) public rewards; address public owner; constructor() { owner = msg.sender; } function stake() external payable { stakes[msg.sender] += msg.value; } function withdraw(uint256 amount) external { require(stakes[msg.sender] >= amount, "insufficient"); stakes[msg.sender] -= amount; (bool ok, ) = msg.sender.call{value: amount}(""); require(ok, "transfer failed"); } function claimReward() external { uint256 reward = rewards[msg.sender]; rewards[msg.sender] = 0; (bool ok, ) = msg.sender.call{value: reward}(""); require(ok); } function updateReward(address user, uint256 amount) external { require(msg.sender == owner, "only owner"); rewards[user] = amount; } }
⚠️ Issues Found by Slither

Slither immediately flags: reentrancy in withdraw() (state change after external call), unchecked return value in claimReward() (no require message), and missing events for all state changes. It also warns that owner could be centralised - a governance consideration.

💡 Fixes Applied

We reorder withdraw() to follow CEI: decrease stakes before the call. We add ReentrancyGuard to both withdraw and claimReward. We emit events StakeDeposited, Withdrawal, and RewardClaimed. We also change the updateReward function to use a role-based access control or timelock if it’s a governance parameter.

After applying fixes, we re-run Slither - zero Highs, zero Mediums. We then run Echidna with an invariant that total ETH balance of the contract must always equal sum of stakes + unclaimed rewards (minus any fees). Echidna finds no violations after 10,000 transactions. The contract is now ready for mainnet.

09 · Assessment Preparation

Key Facts - Know These Cold

A
Slither’s primary advantage
Static analysis without execution; fast, detects reentrancy, missing modifiers, and incorrect pragma settings. It does not find business-logic bugs.
B
Checks-Effects-Interactions (CEI)
The golden rule: update state first, then interact with external contracts. This prevents reentrancy even without a guard (though guard is recommended).
C
Gas-optimisation mantra
Cache storage → memory, use calldata not memory for external inputs, pack variables, avoid unbounded loops.
D
MST BuildGrant Requirement
Applicants must submit a security-audit report (self-audit with tools accepted at initGrant level). The report must list all findings and their resolutions.
10 · Glossary

Key Terms - Sub-Module 21.1

Slither
Static analysis tool for Solidity that detects vulnerabilities without executing code.
Mythril
Symbolic execution engine that explores all possible code paths to find deep bugs.
Echidna
Fuzzer for EVM bytecode; tests user-defined invariants with random transactions.
CEI
Checks-Effects-Interactions pattern - update state before external calls to prevent reentrancy.
Reentrancy Guard
A modifier that prevents a function from being re-entered within the same transaction.
Gas Optimisation
Techniques to reduce the computational cost of executing smart contract functions.
MST BuildGrant
MST ecosystem grant program (up to $10,000) that requires a security-audit report for eligibility.
FortunaX
MST’s provably fair random-number generation protocol, safe for gaming and lotteries.
