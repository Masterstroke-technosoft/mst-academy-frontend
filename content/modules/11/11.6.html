05   Technical Deep Dive - Build the AMM

Step-by-Step: Your Own Constant-Product Automated Market Maker

We'll build the AMM with two tokens: Token A and Token B. For simplicity, we'll assume both tokens are already deployed (you can use any ERC-20 tokens on MST Testnet, or deploy your own mock tokens). The contract will hold reserves of both tokens and allow users to add liquidity, swap, and remove liquidity.

State Variables and Constructor

contract SimpleAMM {
    IERC20 public tokenA;
    IERC20 public tokenB;
    uint256 public reserveA;
    uint256 public reserveB;
    uint256 public totalLpShares;
    mapping(address => uint256) public lpBalances;
    uint256 public constant FEE_DENOMINATOR = 1000; // 0.3% fee

    event LiquidityAdded(address indexed provider, uint256 amountA, uint256 amountB, uint256 shares);
    event Swapped(address indexed trader, bool isAtoB, uint256 inputAmount, uint256 outputAmount);
    event LiquidityRemoved(address indexed provider, uint256 amountA, uint256 amountB, uint256 shares);

    constructor(address _tokenA, address _tokenB) {
        tokenA = IERC20(_tokenA);
        tokenB = IERC20(_tokenB);
    }
}

We store the reserves and track LP shares manually. The fee is 0.3% (3/1000 of each swap stays in the pool, growing reserves and benefiting LPs).

Adding Liquidity

When a user adds liquidity, they deposit tokens in the current reserve ratio. The contract calculates how many LP shares to mint based on the pool's total liquidity. If it's the first deposit, shares are minted based on the product of amounts; otherwise, shares are proportional to the value added relative to existing reserves.

function addLiquidity(uint256 amountA, uint256 amountB) external returns (uint256 shares) {
    require(amountA > 0 && amountB > 0, "Zero amounts");
    tokenA.transferFrom(msg.sender, address(this), amountA);
    tokenB.transferFrom(msg.sender, address(this), amountB);

    if (totalLpShares == 0) {
        shares = sqrt(amountA * amountB); // initial mint
    } else {
        uint256 shareA = (amountA * totalLpShares) / reserveA;
        uint256 shareB = (amountB * totalLpShares) / reserveB;
        shares = shareA < shareB ? shareA : shareB; // mint the smaller share to avoid dilution
    }
    require(shares > 0, "Insufficient liquidity minted");
    reserveA += amountA;
    reserveB += amountB;
    totalLpShares += shares;
    lpBalances[msg.sender] += shares;
    emit LiquidityAdded(msg.sender, amountA, amountB, shares);
}

function sqrt(uint256 x) internal pure returns (uint256) {
    uint256 z = (x + 1) / 2;
    uint256 y = x;
    while (z < y) {
        y = z;
        z = (x / z + z) / 2;
    }
    return y;
}

Swapping Tokens

The swap function implements x * y = k. For a swap from token A to token B, we calculate how much of token B the user should receive such that the product of reserves remains constant (after applying the fee). The fee is taken from the input amount before the swap calculation, effectively leaving the fee in the pool to increase reserves.

function swapAtoB(uint256 amountAIn) external returns (uint256 amountBOut) {
    require(amountAIn > 0, "Zero input");
    uint256 amountAInWithFee = (amountAIn * (FEE_DENOMINATOR - 3)) / FEE_DENOMINATOR; // 0.3% fee
    amountBOut = (reserveB * amountAInWithFee) / (reserveA + amountAInWithFee);
    require(amountBOut > 0, "Zero output");
    tokenA.transferFrom(msg.sender, address(this), amountAIn);
    tokenB.transfer(msg.sender, amountBOut);
    reserveA += amountAIn;
    reserveB -= amountBOut;
    emit Swapped(msg.sender, true, amountAIn, amountBOut);
}

Notice we follow C-E-I: the state updates (reserves) happen after the external calls (token transfers). However, since we're using the actual reserve values to compute the output before updating reserves, we must be careful. In this simple implementation, we compute the output based on the current reserves, then update them. The external call to transfer tokens from the user happens before the reserve update, but that's safe because the user already approved the transfer. The critical protection is that we don't rely on the contract's balance after an external call.

The swap function follows the constant-product formula. The fee is collected by slightly increasing k over time - the pool's total value grows, benefiting LPs.

Removing Liquidity

An LP can burn their shares to withdraw a proportional amount of both tokens from the pool.

function removeLiquidity(uint256 shares) external returns (uint256 amountA, uint256 amountB) {
    require(shares > 0 && shares <= lpBalances[msg.sender], "Invalid shares");
    amountA = (reserveA * shares) / totalLpShares;
    amountB = (reserveB * shares) / totalLpShares;
    reserveA -= amountA;
    reserveB -= amountB;
    totalLpShares -= shares;
    lpBalances[msg.sender] -= shares;
    tokenA.transfer(msg.sender, amountA);
    tokenB.transfer(msg.sender, amountB);
    emit LiquidityRemoved(msg.sender, amountA, amountB, shares);
}

Here we update state (reserves, LP balances) before the external transfer calls. This follows the C-E-I pattern and prevents reentrancy, though in this case reentrancy is unlikely because the tokens are standard ERC-20s without callbacks.

🗝️ Security Reminder

Always follow Checks-Effects-Interactions. In removeLiquidity, we check the shares, update reserves and LP balances (effects), then transfer tokens (interactions). Even if the token contract has a callback, our state is already updated, preventing reentrancy.

06   Technical Deep Dive - Build the Staking Contract

Step-by-Step: A Time-Weighted Staking Contract

The staking contract allows users to deposit an ERC-20 token and earn rewards distributed per block. Rewards are calculated using an internal accounting system that updates each user's reward debt on every action, preventing manipulation.

State Variables and Reward Logic

contract Staking {
    IERC20 public stakingToken;
    uint256 public rewardPerBlock = 100; // reward per block per token staked
    uint256 public totalStaked;
    mapping(address => uint256) public stakedBalance;
    mapping(address => uint256) public rewardDebt;
    mapping(address => uint256) public pendingRewards;
    uint256 public lastRewardBlock;
    uint256 public accRewardPerToken;

    event Staked(address indexed user, uint256 amount);
    event Unstaked(address indexed user, uint256 amount);
    event RewardClaimed(address indexed user, uint256 amount);

    constructor(address _stakingToken) {
        stakingToken = IERC20(_stakingToken);
        lastRewardBlock = block.number;
    }
}

The contract uses the concept of accumulated reward per token: accRewardPerToken tracks the total rewards that have been allocated per staked token since inception. When a user stakes or unstakes, their rewardDebt is updated to the current accRewardPerToken, and any difference is added to their pending rewards. This prevents them from claiming rewards for periods they weren't staked.

Updating Rewards

modifier updateReward(address user) {
    if (totalStaked > 0) {
        uint256 blocks = block.number - lastRewardBlock;
        accRewardPerToken += (blocks * rewardPerBlock * 1e18) / totalStaked;
    }
    lastRewardBlock = block.number;
    if (user != address(0)) {
        pendingRewards[user] += (stakedBalance[user] * (accRewardPerToken - rewardDebt[user])) / 1e18;
        rewardDebt[user] = accRewardPerToken;
    }
    _;
}

This modifier is applied to stake, unstake, and claimReward. It first updates the global accumulator based on blocks elapsed, then updates the specific user's pending rewards. The 1e18 scaling factor ensures precision.

Stake, Unstake, and Claim

function stake(uint256 amount) external updateReward(msg.sender) {
    require(amount > 0, "Zero amount");
    stakingToken.transferFrom(msg.sender, address(this), amount);
    stakedBalance[msg.sender] += amount;
    totalStaked += amount;
    emit Staked(msg.sender, amount);
}

function unstake(uint256 amount) external updateReward(msg.sender) {
    require(amount > 0 && stakedBalance[msg.sender] >= amount, "Invalid amount");
    stakedBalance[msg.sender] -= amount;
    totalStaked -= amount;
    stakingToken.transfer(msg.sender, amount);
    emit Unstaked(msg.sender, amount);
}

function claimReward() external updateReward(msg.sender) {
    uint256 reward = pendingRewards[msg.sender];
    require(reward > 0, "No rewards");
    pendingRewards[msg.sender] = 0;
    stakingToken.transfer(msg.sender, reward);
    emit RewardClaimed(msg.sender, reward);
}

Notice that in unstake we update state (balances, total) before transferring tokens - following C-E-I. The modifier updateReward runs first, updating the global and user-specific reward data.

11   Glossary

Key Terms - Sub-Module 11.6

Constant Product AMM
An AMM using x*y=k to determine swap prices, ensuring infinite liquidity and automatic price discovery.
LP Shares
Tokens minted for liquidity providers representing their proportional claim on the pool's assets.
Accumulator Pattern
A staking reward calculation method that tracks global rewards per staked token and updates user debts on each action, preventing manipulation.
Checks-Effects-Interactions
The security pattern of checking conditions, updating contract state, and only then interacting with external contracts.
