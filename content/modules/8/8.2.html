02   Real-World Story

The Developer Who Pushed His Private Key to GitHub

📖 Story

It was 2 AM. Ravi had been coding for eight hours straight, building his first DeFi dashboard on MST. The smart contract was finished, the frontend was beautiful, and he wanted to save his work. He opened his terminal, typed git add ., then git commit -m "final version", then git push. He went to sleep feeling accomplished.


When he woke up, his wallet was empty. The 0.5 ETH (about ₹1 lakh) he had in his test account was gone. An attacker had scanned GitHub for committed .env files, found Ravi's private key in plain text, and drained the wallet within hours. Ravi had forgotten one simple thing: a .gitignore file that excludes sensitive files from version control.


This is not a rare story. Thousands of private keys are leaked on GitHub every year. In blockchain development, where money is code and code is money, version control mistakes can be catastrophic. But Git itself is not the enemy - it's your greatest ally when used correctly. It records every change, allows you to collaborate safely, and can even help you recover from mistakes if you've committed properly. This sub-module will teach you to use Git like a professional, with the blockchain-specific precautions that will keep your funds and your career safe.

03   What Is Git?

Git - The World's Most Powerful Time Machine

Git is a distributed version control system. It was created in 2005 by Linus Torvalds, the same person who created the Linux operating system. Before Git, the Linux kernel project used a proprietary version control system called BitKeeper. When the free license for BitKeeper was revoked, Linus decided to build his own tool - one that would be fast, distributed, and capable of handling thousands of contributors. He built Git in a single weekend.

At its core, Git is a system that takes snapshots of your project over time. Every time you "commit," Git records the exact state of every file in your project. You can travel back to any previous commit, compare changes between versions, and create parallel timelines (branches) where you experiment without affecting the main code. This is not just useful - it is essential for any software project involving more than one person.

Git is distributed, meaning every developer has a full copy of the project's history on their local machine. There is no central server that can fail and lose everything. Even if GitHub's servers went offline permanently (unlikely, but possible), you would still have the entire project history on your laptop.

📸 The Photo Album Analogy

Think of Git like a photo album of your project. Every commit is a photograph showing exactly what the project looked like at that moment. You can flip back to any photo to see what changed. You can create a new album (branch) to experiment with a new feature. If the experiment fails, you throw away that album - the original photos are untouched. And you can share your album with others, who can add their own photos and merge them with yours.

🔬 Click to expand: How Git Stores Data - Blobs, Trees, and Commits

Git's internal database is a content-addressable file system. Every file is stored as a blob (binary large object). The structure of directories is stored as tree objects. A commit object points to a tree (the root directory state), has a parent commit (except the first commit), and contains metadata like author, timestamp, and message. All objects are identified by a 40-character SHA-1 hash of their contents. This is why Git can detect corruption - if any bit changes, the hash changes, and Git knows something is wrong.

04   Why Git for Blockchain Development?

Why Version Control Is Critical for Smart Contracts

Smart contracts are immutable. Once deployed to a blockchain, their code cannot be changed. This makes version control more important for blockchain development than for any other kind of software. When you deploy a contract, you need to know exactly which version of the code was deployed, with which compiler settings, and with which constructor arguments. Git provides this precise audit trail.

Additionally, blockchain projects are almost always open source. The culture of Web3 values transparency, and most major protocols have their code publicly visible on GitHub. This allows users to verify that the contract does what it claims, security researchers to audit the code, and other developers to learn from it. Your GitHub profile is your resume in the blockchain industry.

Furthermore, blockchain development involves many files beyond Solidity: deployment scripts, test files, configuration files (hardhat.config.js), environment variables (.env), and frontend code. Managing all these files without version control is a recipe for chaos. Git keeps everything organized and traceable.

05   Installation

Installing Git on Windows, macOS, and Linux

Git is likely already installed if you have been following this course. Verify by opening your terminal and typing git --version. If you see something like git version 2.43.0, you're ready. If not, follow the instructions below.

Windows: Download the installer from https://git-scm.com. Run the .exe file. During installation, choose the default options, but make sure "Git Bash Here" is selected (it gives you a Unix-like terminal on Windows). On the "Adjusting your PATH environment" screen, select "Git from the command line and also from 3rd-party software."

macOS: Install via Homebrew: brew install git. Or download the installer from git-scm.com. macOS may prompt you to install Xcode Command Line Tools, which includes Git.

Linux (Ubuntu/Debian): sudo apt-get install git. For other distributions, use the appropriate package manager.

🔧 Click to expand: Configuring Git - Your Identity

After installation, set your name and email. These will be attached to every commit you make:

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Verify with git config --list. This information is public on GitHub, so use a real email (or GitHub's noreply email for privacy: username@users.noreply.github.com).

06   The Core Git Workflow

The Four-Stage Journey of a File - From Working Directory to Remote Repository

FIGURE 8.2-A · The Git Workflow - Working Directory → Staging → Commit → Remote
┌─────────────────────────┐
│   WORKING DIRECTORY     │  ← You edit files here (VS Code)
│   (untracked / modified)│
└───────────┬─────────────┘
            │  git add <file>   (or git add .)
            ▼
┌─────────────────────────┐
│   STAGING AREA          │  ← Files are "staged" for the next commit
│   (index)               │     You review what will be committed
└───────────┬─────────────┘
            │  git commit -m "message"
            ▼
┌─────────────────────────┐
│   LOCAL REPOSITORY      │  ← Commit is saved in .git folder
│   (.git)                │     Complete snapshot of project at this point
└───────────┬─────────────┘
            │  git push
            ▼
┌─────────────────────────┐
│   REMOTE REPOSITORY     │  ← GitHub, GitLab, or any server
│   (origin)              │     Shared with team; backup; deployment source
└─────────────────────────┘

OTHER COMMANDS:
  git pull     = git fetch (download changes) + git merge (integrate them)
  git clone    = Download an entire remote repository to your machine
  git status   = See which files are modified, staged, or untracked
  git log      = View the history of commits
Files move through four stages. The staging area lets you control exactly which changes go into each commit.

The Working Directory is simply the folder on your computer where your project files live. When you edit a file in VS Code, you are modifying files in the working directory. Git tracks these changes but does not save them yet.

The Staging Area (Index) is a preparation zone. When you run git add <file> or git add . (add all changed files), you are telling Git: "I want these specific changes to be included in the next commit." This is powerful because it allows you to make many changes across multiple files but only commit the ones that are related to a single logical change. You should never blindly use git add . without knowing what it includes - especially in blockchain projects where sensitive files might be in the directory.

The Local Repository is where commits are permanently stored. A commit is a snapshot of the entire project at that moment, along with a message describing what changed and why. You can have hundreds of commits on your local machine without ever connecting to the internet.

The Remote Repository is a copy of your project hosted on a server - most commonly GitHub. Pushing your commits to a remote repository serves three purposes: backup (if your laptop dies, the code lives on), collaboration (others can see and contribute to your code), and deployment (services like Vercel can deploy directly from GitHub).

07   Essential Commands

The Git Commands You Will Use Every Day - In Detail

1
git init - Starting a New Project
Run git init inside any folder to turn it into a Git repository. This creates a hidden .git folder that contains all the version control data. For blockchain projects, you typically run this right after npx hardhat init. Example: mkdir my-dapp && cd my-dapp && git init.
2
git clone - Copying an Existing Project
Run git clone <url> to download an entire repository, including its full history. For example, to clone the MST Academy starter template: git clone https://github.com/mst-academy/starter-kit.git. This creates a folder with the project and sets up the remote connection automatically.
3
git status - Checking What's Changed
Run git status frequently. It shows you which branch you're on, which files have been modified, which are staged for commit, and which are untracked. In blockchain projects, always run this before committing to make sure you're not accidentally including .env or private key files.
4
git add - Staging Changes
git add <file> stages a specific file. git add . stages all changes in the current directory. For blockchain projects, use specific file names or git add contracts/ test/ scripts/ to avoid accidentally staging sensitive files. You can also use git add -p (patch mode) to review and stage individual chunks of changes interactively - extremely useful for careful commits.
5
git commit - Creating a Snapshot
git commit -m "message" creates a commit with all staged changes. Write clear, descriptive messages that explain why you made the change, not just what you did. Good: "Add ownership check to withdraw function to prevent reentrancy." Bad: "fix stuff." In blockchain, a good commit message can later help you trace when and why a security-critical change was made.
6
git push - Uploading to Remote
git push origin main uploads your commits to the remote repository (typically GitHub). Before pushing, always pull first to integrate any changes from collaborators. For blockchain projects, push frequently to ensure your work is backed up - losing a laptop should not mean losing your smart contracts.
7
git pull - Downloading and Merging
git pull is actually two commands combined: git fetch (which downloads changes from the remote) and git merge (which integrates those changes into your current branch). Always pull before starting work and before pushing. This prevents merge conflicts and keeps you synchronized with your team.
08   Branching & Merging

Branches - Working in Parallel Without Breaking Anything

A branch is an independent line of development. The default branch is usually called main (or master in older projects). When you want to add a new feature, fix a bug, or experiment, you create a new branch. This branch is a copy of the project at that moment. You can make changes, commit them, and even push the branch to GitHub without affecting the main branch at all. When your work is complete and tested, you merge the branch back into main.

FIGURE 8.2-B · Branching and Merging Visualized
main:     ●────●────●────────────●  (merge commit)
                │                 │
feature:        ●────●────●───────┘
                (create branch)   (git merge feature)

Commands:
  git branch feature-login       Create a new branch
  git checkout feature-login     Switch to that branch
  git checkout -b feature-login  Create AND switch (shortcut)
  git merge feature-login        Merge feature-login into current branch
  git branch -d feature-login    Delete branch after merging
Branches allow parallel work without conflict. The main branch always contains stable, deployable code.

Merge Conflicts. Sometimes Git cannot automatically merge two branches because the same part of the same file was changed differently. This is called a merge conflict. Git will mark the conflicting sections in the file with <<<<<<<, =======, and >>>>>>>. You must manually decide which version to keep, remove the markers, and commit the resolved file. In smart contract development, merge conflicts must be resolved with extreme care - a wrong merge could introduce a security vulnerability. Always run the full test suite after resolving a merge conflict in Solidity files.

09   .gitignore - The Guardian

Protecting Your Private Keys and Secrets

The .gitignore file is the single most important file for blockchain development security. It tells Git which files and folders to never track. Any file listed in .gitignore will not appear in git status, will not be added by git add, and will never be pushed to GitHub. This is how you keep your private keys, API secrets, and local configuration safe.

🔑 The Blockchain .gitignore Template

Create a file named .gitignore at the root of your project and add these lines:

# Environment variables (contains private keys, API keys, RPC URLs)
.env

# Node.js dependencies (huge folder, reinstall with npm install)
node_modules/

# Hardhat artifacts (compiled contracts, cache)
artifacts/
cache/

# Coverage reports
coverage/

# TypeScript build output (if using TypeScript)
dist/
typechain-types/

# Operating system files
.DS_Store
Thumbs.db

# IDE files (VS Code workspace settings can be shared, but personal settings should not)
.vscode/settings.json

The .env file is where you store environment variables like PRIVATE_KEY=0xabc123... and MST_RPC_URL=https://testnetrpc.mstblockchain.com. This file must be in .gitignore. Never, ever commit your private key to Git. Even if you delete it later, it remains in the Git history. If you accidentally commit a private key, consider it compromised and transfer all funds to a new wallet immediately.

⚠️ Click to expand: What to Do If You Accidentally Commit a Private Key

Step 1: Do not panic. Do not just delete the file and commit again - the key is still in the Git history.

Step 2: Move all funds from that wallet to a brand-new wallet immediately. The private key is compromised.

Step 3: Use a tool like git filter-branch or BFG Repo-Cleaner to rewrite the Git history and remove the sensitive file. Force-push the cleaned history to GitHub.

Step 4: Understand that if the repository was public, bots may have already scraped your key. This is why moving funds first is critical.

10   GitHub

GitHub - The Social Network for Code

GitHub is a web-based platform that hosts Git repositories and adds collaboration features on top: issues, pull requests, code reviews, actions (CI/CD), project boards, and more. While Git is the tool that runs on your laptop, GitHub is the website where you share your Git repositories with the world. It is the home of open-source software and the standard platform for blockchain projects.

Creating a Repository on GitHub. After creating a project locally with git init, you can create a repository on GitHub (click the "+" in the top right, select "New repository"), give it a name, and leave it empty (do not initialize with README if you already have a local repo). GitHub will show you the commands to connect your local repo to the remote and push your code.

# Connect your local repo to GitHub
git remote add origin https://github.com/yourusername/your-repo.git

# Rename your branch to main (if it's still "master")
git branch -M main

# Push your code
git push -u origin main

Pull Requests (PRs). A pull request is a proposal to merge one branch into another - typically a feature branch into main. It's the way team members review each other's code before it becomes part of the project. In blockchain development, every PR should be reviewed by at least one other developer, and all tests must pass before merging. This is how critical bugs are caught before deployment.

11   Hands-On Workflow

A Complete Git Session - From Zero to GitHub

# 1. Create project folder and initialize Git
mkdir my-mst-dapp
cd my-mst-dapp
git init

# 2. Create .gitignore BEFORE adding any files
echo "node_modules/" > .gitignore
echo ".env" >> .gitignore
echo "artifacts/" >> .gitignore
echo "cache/" >> .gitignore

# 3. Initialize Hardhat project
npx hardhat init

# 4. Check what will be committed
git status

# 5. Stage and commit the initial project
git add contracts/ test/ scripts/ hardhat.config.js package.json .gitignore
git commit -m "Initial Hardhat project setup"

# 6. Create a GitHub repo (on github.com), then connect and push
git remote add origin https://github.com/yourusername/my-mst-dapp.git
git branch -M main
git push -u origin main

# 7. Create a feature branch for your first contract
git checkout -b feature/hello-world

# 8. Write your contract, test it, then commit
git add contracts/HelloWorld.sol test/HelloWorld.test.js
git commit -m "Add HelloWorld contract with tests"

# 9. Push the feature branch
git push -u origin feature/hello-world

# 10. Create a Pull Request on GitHub, get review, merge into main

13   Glossary

Key Terms - Sub-Module 8.2

Git
A distributed version control system that tracks changes in source code during software development.
Repository (Repo)
A directory that contains a project's files and the entire history of changes tracked by Git.
Commit
A snapshot of the project at a specific point in time, with a message describing the changes.
Staging Area
The intermediate area where changes are prepared before being committed to the repository.
Branch
A parallel version of the repository, allowing independent development without affecting the main line.
Merge
The operation of integrating changes from one branch into another.
.gitignore
A file that specifies intentionally untracked files that Git should ignore (e.g., .env, node_modules).
GitHub
A web-based platform for hosting Git repositories with collaboration features like Pull Requests and Issues.
Pull Request (PR)
A proposal to merge one branch into another, enabling code review and discussion before integration.
Remote
A version of the repository hosted on a server (e.g., GitHub). The default remote is usually named "origin."
