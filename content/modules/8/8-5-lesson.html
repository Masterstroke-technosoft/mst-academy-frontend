02   Real-World Introduction Story

The Staking Contract That Cost Users ₹500 per Claim

📖 Story

In 2022, a DeFi protocol on Ethereum launched a staking pool. Users could deposit tokens and claim rewards daily. The contract stored every user's reward history in a mapping(address => uint256[]) - a dynamic array per user, stored in storage. The claim function looped over the entire history array each time, reading from storage repeatedly. Storage reads (SLOAD) cost 2,100 gas each (cold). For a user with 100 entries, a single claim cost over 500,000 gas - about ₹1,000 at the time. Users were furious.


The developers had not understood Solidity's storage model. They could have stored only the last claim timestamp and a running total, both in storage, and computed rewards mathematically without any loop. Or they could have used a mapping directly for each day's reward, eliminating the array. A simple data structure change reduced the claim cost from 500,000 gas to 65,000 gas - an 87% savings. On MST, the same mistake would have cost only ₹0.03 instead of ₹0.004, but the lesson is the same: storage is expensive, and choosing the right data type and location is the most impactful gas optimization you can make.

This sub-module equips you with a deep understanding of every Solidity data type and the three storage locations - storage, memory, and calldata. You'll learn when to use each, what gas they consume, and how to design contracts that are both functional and cost-efficient on MST.

03   Background & History

Why the EVM Has Three Data Locations

When the Ethereum Virtual Machine (EVM) was designed, its creators faced a fundamental challenge: a blockchain is a permanent, replicated database. Every byte stored on-chain must be stored by every full node forever. This makes storage extremely expensive - it's the scarcest resource on the network. In contrast, computations are relatively cheap because they are ephemeral: they run once and disappear. The EVM's gas model reflects this reality.

To balance permanence with cost, the EVM introduced three distinct data locations:

  • Storage: Persistent key-value store, written to the blockchain's state trie. Data survives forever. Costs 20,000 gas for a new write (SSTORE), 5,000 gas for an update, and 2,100 gas for a cold read (SLOAD). This is where your contract's state lives.
  • Memory: Temporary workspace, cleared after external function execution. Much cheaper than storage, but costs scale quadratically with size. Used for complex data manipulation within a function.
  • Calldata: Read-only input data attached to a transaction. The cheapest location - you pay only for the bytes you include. Perfect for function parameters that don't need to be modified.

Solidity, as a high-level language, abstracts these locations but requires the developer to explicitly specify them for reference types (arrays, structs, mappings). This is one of the most important skills in smart contract development: knowing where your data lives.

MST, as an EVM-compatible chain, inherits the exact same storage model. The gas costs are identical per opcode, but because MST's gas price is ~12 Gwei (vs Ethereum's 30-500 Gwei), the rupee cost is orders of magnitude lower. However, the architectural principles remain unchanged: you should still write efficient Solidity to keep your contracts scalable and your users happy.

04   Beginner-Friendly Explanation

Stone Tablet, Whiteboard, and Sticky Note - The Storage Analogy

🪨 The Analogy

Think of storage as carving words into a stone tablet - it's permanent, visible to everyone forever, and extremely laborious. Memory is like writing on a whiteboard in your office - you can scribble calculations, but it gets erased when you leave the room (end of function). Calldata is a sticky note someone handed you - you can read it, but you can't write on it, and it's cheap because it's just paper. Smart contract developers spend most of their optimization time moving data from the stone tablet to the whiteboard or sticky note.

Let's explore each data type and its typical location, starting with the simplest value types.

V
VALUE
What are value types?
Value types are simple data that fit into 32 bytes. They are always passed by value (copied). This includes uint256, int256, bool, address, bytes32, and enum. When you assign a value type, the EVM copies the data. They can be stored in storage (state variables), memory, or calldata, but since they are small, the location matters less for gas - though using calldata for function parameters still saves a few hundred gas.
R
REF
What are reference types?
Reference types can be larger than 32 bytes and include array, struct, mapping, and bytes (dynamic). They are stored as pointers to a memory or storage location. When you assign a reference type, you copy the pointer, not the data - unless you explicitly create a copy. You must specify the data location (memory, storage, or calldata) for reference types in function parameters and local variables. Getting this wrong is the #1 compiler error for new Solidity developers.
05   Technical Deep Dive

Every Data Type, Every Size, Every Default - Complete Reference

Integer Types: uint and int

Solidity provides unsigned (uint) and signed (int) integers, from 8 bits to 256 bits in steps of 8. uint is an alias for uint256. The default value is 0. Arithmetic overflow/underflow was historically a major vulnerability, but since Solidity 0.8.0, the compiler automatically inserts checks and reverts on overflow - unless you use an unchecked { ... } block, which saves gas but must be used with extreme caution.

Why choose smaller sizes like uint8? The EVM operates on 256-bit words. A uint8 still takes up a full 32-byte storage slot, but multiple uint8 variables can be packed into a single slot by the compiler if they are declared consecutively. This is a crucial gas optimization: reading one slot costs 2,100 gas (cold) or 100 gas (warm). Reading two packed variables from the same slot costs the same as reading one. Declare small variables together to maximize packing.

Bool and Address

bool is either true or false, default false. It occupies 1 byte but, like other small types, can be packed into a slot. address is a 20-byte (160-bit) Ethereum address, default 0x0000...0000. Addresses have special members like .balance (returns the native coin balance in wei) and .transfer(uint256 amount) (sends native coin). address payable is a subtype that allows .send() and .transfer().

Bytes and Strings

Fixed-size byte arrays (bytes1 to bytes32) hold exactly that many bytes. They are value types and are cheap because they fit in a single EVM word. bytes32 is often used for hashes and identifiers. Dynamic bytes and string are reference types. They are stored in storage as a length prefix and data. string is essentially bytes with UTF-8 encoding but with no length operator (you can't do myString.length without converting to bytes). Use bytes for raw data, string for human-readable text.

Mapping: The Workhorse of Solidity

A mapping(KeyType => ValueType) is a hash table that maps keys to values. All possible keys exist and return the default value of the ValueType (0, false, zero address) - there is no "does this key exist" check. Mappings are stored in storage only; you cannot have a memory mapping. They are ideal for keeping track of balances (mapping(address => uint256)), allowances, or any associative data.

Mappings cannot be iterated over. To track which keys have been set, use a separate array of keys or an enumerable mapping pattern (using OpenZeppelin's EnumerableMap). On MST, where gas is cheap, even storage-heavy patterns like enumerable mappings are practical.

Structs: Custom Data Bundles

A struct groups multiple variables under one name. Structs can be stored in storage or memory. When stored in storage, they are written to consecutive storage slots, with potential packing. You can create arrays of structs and mappings to structs. Example:

struct User {
    address addr;
    uint256 balance;
    bool isActive;
}
mapping(address => User) public users;

This mapping stores a User struct per address. Accessing users[addr].balance reads from a storage slot determined by keccak256(key, slot).

Enums: Finite State Machines

enum Status { Pending, Shipped, Delivered } creates a custom type with named constants. Internally, they are stored as uint8 starting from 0. Use them to make contract states explicit and readable.

The Gas Cost of Ignoring Calldata

When you write an external function like function setData(string memory _data), the EVM first copies the entire string from calldata into memory - every character. For a 1 KB string, that's an extra ~3,000 gas, plus memory expansion costs. Change memory to calldata and the data stays in the read-only input area, with no copy. The function can read it directly. Rule of thumb: if a parameter is not being modified, declare it as calldata. This applies to string, bytes, and uint256[] arrays.

Storage Packing - Fit More in Less Gas

The EVM stores data in 32-byte slots. If you declare a uint128 a; uint128 b; consecutively, the compiler packs them into a single slot. If you interleave them with a uint256, they each take their own slot. Packing reduces the number of SSTORE/SLOAD operations, directly lowering gas costs. On MST, a packed storage layout might save only ₹0.001 per read, but for a contract called thousands of times, those savings add up - and it's excellent practice for any EVM chain.

The Solidity compiler automatically packs small variables that are adjacent in the contract declaration. Struct fields are also packed. Check your storage layout with solc --storage-layout.
🗝️ The Golden Rule of Solidity Data

Use storage for state that must persist. Use memory for temporary computations. Use calldata for read-only external function parameters. Always ask: "Does this variable need to survive after this function call?" If no, keep it in memory or calldata. This mindset alone will make you a better smart contract developer.

07   Interactive Code Examples

Common Patterns - Click to Reveal

📋 Using mapping for balances (storage) 
contract Token {
    mapping(address => uint256) public balanceOf;

    function transfer(address to, uint256 amount) external {
        require(balanceOf[msg.sender] >= amount, "insufficient");
        balanceOf[msg.sender] -= amount;  // SSTORE (5,000 gas)
        balanceOf[to] += amount;           // SSTORE (5,000 or 20,000)
        emit Transfer(msg.sender, to, amount);
    }
}

The mapping is stored in storage. Each write costs 5,000-20,000 gas. MST makes this affordable at ~₹0.006 per update.

🧠 Building a struct in memory 
function createUser(address addr, uint256 bal) external pure returns (User memory) {
    User memory newUser;
    newUser.addr = addr;
    newUser.balance = bal;
    newUser.isActive = true;
    return newUser;  // copies memory to calldata (return data)
}

All struct fields are in memory, so no permanent storage cost. The returned struct is copied to calldata automatically.

📨 Using calldata for read-only array 
function sumArray(uint256[] calldata arr) external pure returns (uint256) {
    uint256 total = 0;
    for (uint256 i = 0; i < arr.length; i++) {
        total += arr[i];  // reads directly from calldata, no copy
    }
    return total;
}

If arr were declared memory, the EVM would first copy the entire array from calldata to memory, costing ~3 gas per byte plus memory expansion. Calldata avoids that overhead entirely.

08   Real-World Case Studies

Storage Choices That Saved (and Cost) Millions

💰
Uniswap v2 - Mapping Over Array for Pair Tracking
Uniswap v2 tracks every trading pair in a mapping(address => mapping(address => address)) - a nested mapping from token0 to token1 to the pair contract address. This O(1) lookup costs one SLOAD (100-2,100 gas) per query. If they had used an array and looped through it for every swap, the cost would have scaled linearly with the number of pairs, eventually making swaps prohibitively expensive. The choice of mapping over array was a deliberate storage optimization that keeps the protocol scalable.
🔴
The "String in Storage" Anti-Pattern
A common beginner mistake is storing user names or descriptions as string state variables. A string in storage is a dynamic array of bytes - it occupies at least one slot for the length, plus slots for the data. For a 50-character name, that's 3+ slots (over 60,000 gas to write). The better pattern is to store an IPFS hash (bytes32) that points to the actual data off-chain. This reduces the storage cost to a single 32-byte slot (20,000 gas). On MST, the gas savings are modest, but the principle of storing only essential on-chain data remains critical for scalability.
09   Assessment Preparation

Top Facts - Know These Cold for the Exam

01
Storage is persistent and expensive; memory is temporary and cheap
SSTORE costs 20,000 gas for a new write. MLOAD costs 3 gas. Choose memory over storage whenever possible. For read-only external parameters, use calldata.
02
Mapping cannot be iterated; use an array of keys if you need enumeration
mapping(address => uint) gives O(1) lookup but no way to list all keys. Pair it with an address[] if you need to iterate.
03
Use calldata for external function parameters that are not modified
It avoids an expensive memory copy. Example: function foo(uint[] calldata arr) external { ... }
04
Pack consecutive small variables into storage slots
Declare uint128, uint64, bool together. The compiler packs them. Verify with solc --storage-layout. On MST, gas savings are smaller but still best practice.
05
All value types have a default zero value (0, false, 0x0)
Mappings return the default for nonexistent keys. No "key exists" check. Use an isActive boolean or a separate Set contract if needed.
10   Glossary

Key Terms - Sub-Module 8.5

Value Type
A Solidity type that fits in 32 bytes and is always copied when assigned. Includes uint, int, bool, address, bytes32, enum.
Reference Type
A type that can be larger than 32 bytes, stored as a reference to a location. Includes arrays, structs, mappings, strings, dynamic bytes. Must specify data location.
Storage
Persistent key-value store on the blockchain. Costs 20,000 gas for a new write (SSTORE). Used for contract state variables. Survives forever.
Memory
Temporary data area cleared after each external function call. Cheaper than storage (MLOAD = 3 gas). Used for local variables and computations.
Calldata
Read-only input data attached to a transaction. The cheapest data location; use for external function parameters that aren't modified.
Mapping
A hash table mapping(Key => Value) stored in storage. O(1) lookup. Cannot be iterated; all keys return the default value if unset.
Struct
A custom type that groups multiple variables. Can be stored in storage or memory. Use for complex data like user profiles or order details.
Storage Slot Packing
Compiler optimization that places multiple small variables into a single 32-byte storage slot. Reduces gas costs by lowering SSTORE/SLOAD count.
SLOAD / SSTORE
EVM opcodes for reading and writing storage. SLOAD: 100-2,100 gas. SSTORE: 5,000-20,000 gas. Most expensive operations; minimize them.
Gas Optimization
The practice of reducing computational cost by choosing efficient data types, minimizing storage access, and using calldata over memory. Even on low-fee chains like MST, good practice is essential.
