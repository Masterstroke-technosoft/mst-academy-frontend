02   Real-World Story

The Developer Who Confused a Contract with a Class

📖 Story

Meera was a brilliant Java developer from Pune. She had built banking applications for five years. When her company pivoted to Web3, she dove into Solidity with confidence. "It's just like Java," she thought. "Contracts are classes. Functions are methods. Simple."


Within her first week, she wrote a token contract, deployed it to the testnet, and sent 500 tokens to her colleague. The transaction went through. But when her colleague tried to send those tokens back, the transaction failed. Meera had declared the transfer function as private - thinking "private" was a security feature that prevented unauthorized access. She didn't realize that private in Solidity means the function cannot be called from outside the contract at all - not even by its legitimate users. The tokens were stuck forever.


This story illustrates the single most important lesson of Solidity: it is not Java, JavaScript, Python, or C++. It borrows syntax from these languages, but it is fundamentally different. It runs on a blockchain, where code is immutable once deployed, where every operation costs real money (gas), and where mistakes cannot be patched with a quick update. Solidity demands precision, deep understanding of its unique concepts, and a completely different mental model from traditional programming. This sub-module is where you build that foundation.

03   What Is Solidity?

Solidity - The Language of the EVM

Solidity is a high-level, statically typed, contract-oriented programming language designed specifically for writing smart contracts that run on the Ethereum Virtual Machine (EVM). It was created by Gavin Wood (Ethereum's first CTO) and developed by Christian Reitwiessner and the Ethereum Foundation's Solidity team. The first version, Solidity 0.1.0, was released in July 2015.

Solidity is statically typed, meaning every variable's type must be known at compile time - you cannot assign a string to an integer variable. It is contract-oriented, meaning the primary building block is the contract, which encapsulates state (data stored on the blockchain) and behavior (functions that modify that state). It is curly-brace syntax, making it visually familiar to developers who know JavaScript, C++, or Java.

🔑 The Critical Distinction

Solidity is not a general-purpose programming language. You don't use it to build websites, mobile apps, or desktop software. You use it exclusively for smart contracts: self-executing programs that run on a blockchain, control digital assets, and cannot be modified after deployment. Every line of Solidity you write will be deployed to a blockchain, will cost gas to execute, and will be publicly visible forever. This changes everything about how you think about code.

🔬 Click to expand: The Journey of Solidity Code - From .sol File to EVM Bytecode

When you run npx hardhat compile, the Solidity compiler (solc) performs several transformations:

1. Lexical Analysis: The source code is broken into tokens (keywords, identifiers, operators, literals).

2. Parsing: Tokens are organized into an Abstract Syntax Tree (AST) representing the grammatical structure of the code.

3. Semantic Analysis: Type checking, scope resolution, and other semantic validations are performed. This is where errors like type mismatches are caught.

4. Optimization: The compiler applies optimizations to reduce gas costs, such as removing unused code and simplifying expressions.

5. Code Generation: The AST is translated into EVM bytecode - a sequence of hexadecimal opcodes that the EVM can execute. This bytecode, along with the ABI (Application Binary Interface), is stored in the artifacts/ folder.

The ABI is a JSON file that describes the contract's functions, parameters, and return types. Frontend applications and other contracts use the ABI to know how to interact with your deployed contract.

04   File Structure

Anatomy of a Solidity File - Line by Line

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract SimpleStorage {
    // State variable – stored on the blockchain forever
    uint256 public storedData;

    // Constructor – runs once, when the contract is deployed
    constructor(uint256 initialValue) {
        storedData = initialValue;
    }

    // Function – can read and modify state
    function set(uint256 newValue) public {
        storedData = newValue;
    }

    // Function – reads state but does not modify it (free to call!)
    function get() public view returns (uint256) {
        return storedData;
    }
}

Let's dissect every line of this contract to understand exactly what it does. This contract is small, but it contains every fundamental concept you need to understand before building more complex applications.

05   Pragma & License

The SPDX License and Pragma Directive

SPDX License Identifier. The first line of every Solidity file should be a comment specifying the license under which the code is distributed: // SPDX‑License‑Identifier: MIT. SPDX stands for Software Package Data Exchange. This is a standardized way to declare the license. Solidity ≥0.6.8 generates a warning if this line is missing. Common licenses include MIT (permissive, widely used in open-source blockchain projects), GPL-3.0 (copyleft, requires derivative works to also be open source), and UNLICENSED (all rights reserved, no permission to use the code).

The Pragma Directive. The line pragma solidity ^0.8.20; is the version pragma. It tells the Solidity compiler (solc) which version of the language this code was written for. The caret (^) means "any version from 0.8.20 up to, but not including, 0.9.0." This is important because Solidity is a rapidly evolving language. Breaking changes are introduced in major version increments (0.7.x to 0.8.x, and eventually 1.0.0). The pragma ensures your contract compiles with a compatible compiler version.

⚠️ Why Pragma Matters

If you write code for Solidity 0.8.20 and someone tries to compile it with Solidity 0.7.0, it will fail because of syntax and semantic differences. Similarly, if Solidity 0.9.0 introduces breaking changes, your code protected by ^0.8.20 will not compile with it. This protects your contract from being compiled with an incompatible compiler that could introduce vulnerabilities or unexpected behavior.

06   The Contract

The Contract - Your On-Chain Object

A contract in Solidity is the fundamental building block. It is similar to a class in object-oriented languages like Java or C++, but with critical differences. A contract encapsulates:

State Variables: Data that is permanently stored on the blockchain. These are the contract's memory. Every state variable consumes storage space, which costs gas to write and update.

Functions: Executable code that can read and modify state variables. Functions are the contract's behavior.

Events: Logging mechanisms that emit data to the blockchain logs, which frontend applications can listen to.

Modifiers: Reusable condition checks that can be attached to functions (covered in depth in Sub-Module 8.6).

Constructor: A special function that runs only once, when the contract is deployed. It is used to set initial state - like assigning the contract owner or setting initial parameter values.

🏠 The House Analogy

Think of a contract like a house blueprint. When you deploy the contract, you build one specific house from that blueprint. The state variables are the furniture and belongings inside the house - they persist and can be rearranged. The functions are the doors and windows - they let people interact with the house. The constructor is the construction crew that builds the house and sets the initial furniture arrangement. Once the house is built (deployed), you cannot change the blueprint - but you can interact with the house through its doors (functions).

07   State Variables

State Variables - Permanent Data on the Blockchain

A state variable is a value that is stored directly in the contract's storage on the blockchain. It persists permanently between function calls and between transactions. State variables are the most expensive thing in Solidity - writing to storage costs 20,000 gas for a 32-byte slot (or 5,000 gas for a non-zero to non-zero update). Reading from storage costs 2,100 gas (cold access) or 100 gas (warm access). This is why efficient Solidity developers carefully consider what truly needs to be stored on-chain.

contract StateDemo {
    // Simple value types
    uint256 public count;          // Unsigned integer – 0 by default
    address public owner;          // Ethereum address – 0x0 by default
    bool public isActive;          // Boolean – false by default
    string public name;            // String – "" by default

    // The 'public' keyword auto‑generates a getter function:
    // function count() public view returns (uint256) { return count; }
}

Default Values. Every state variable has a default value - the zero value for its type. This is different from many other languages where uninitialized variables contain garbage or cause errors. In Solidity, uint256 defaults to 0, bool to false, address to 0x0000...0000, string to "". This is guaranteed by the EVM. You never read uninitialized memory in Solidity.

08   Data Types

The Fundamental Data Types of Solidity

Solidity is statically typed - every variable has a type, and that type is fixed at compile time. Understanding the types and when to use each one is essential for writing efficient, secure contracts. Here are the core types you will use in every contract.

1
uint - Unsigned Integers (0 and above)
uint256 is the most common type in Solidity. It stores whole numbers from 0 to 2²⁵⁶-1 (approximately 1.15 × 10⁷⁷). Solidity supports sizes from uint8 (0 to 255) to uint256, in steps of 8. Always prefer uint256 unless you have a specific gas-saving reason to use a smaller size. The EVM operates on 256-bit words, so smaller uints require extra masking operations that can actually cost more gas. The primary use case for smaller uints is when packing multiple variables into a single storage slot.
2
int - Signed Integers (negative, zero, positive)
int256 stores values from -2²⁵⁵ to 2²⁵⁵-1. Use int when you need negative numbers, which is rare in most smart contracts. Most blockchain values (token balances, timestamps, counters) are naturally non-negative, so uint is the default choice.
3
bool - True or False
Stores true or false. Used for flags, conditions, and status indicators. Defaults to false. Example: bool public isPaused = false;
4
address - The 20-Byte Blockchain Address
address is a 20-byte (160-bit) value that uniquely identifies an account or contract on the blockchain. It comes with built-in members: .balance returns the address's native coin balance, .transfer(amount) sends native coins, and .call{value: amount}("") is the modern, recommended way to send native coins. address payable is a subtype that allows all these operations plus transfer and send.
5
string - Text Data
string stores UTF-8 encoded text. Strings are expensive in Solidity because they are dynamic-sized arrays of bytes. Each character costs storage. For short, fixed-length text (like names or symbols), consider using bytes32 instead, which fits in a single storage slot.
6
bytes - Raw Byte Data
bytes is a dynamic array of bytes. bytes32 is a fixed-size 32-byte value, useful for storing hashes, identifiers, and short text efficiently. bytes1 through bytes32 are fixed-size variants. When in doubt between string and bytes32, use bytes32 for fixed-length data.
🔬 Click to expand: Why uint256 Is the Default - The EVM's 256-Bit Architecture

The EVM's stack operates on 256-bit words. When you use a smaller type like uint8, the EVM must perform additional operations to mask off the unused bits after every arithmetic operation. This masking costs gas. For a single uint8, the overhead is small, but if you use uint8 everywhere thinking it saves gas, you may actually increase costs. The correct optimization strategy is to use uint256 for most variables and pack multiple small variables into a single 256-bit storage slot using struct packing - a topic covered in Sub-Module 8.5.

09   Functions & Visibility

Functions - The Actions Your Contract Can Perform

Functions are the executable units of a Solidity contract. They can read state, modify state, emit events, and interact with other contracts. Every function has a visibility specifier that determines who can call it, and optional mutability modifiers that tell the compiler (and the caller) whether the function modifies state.

Function Mutability Modifiers. Beyond visibility, functions can be marked with modifiers that describe their relationship to state:

view - The function reads state but does not modify it. When called externally, view functions cost zero gas (they are executed locally by the node, not as a transaction). Use view for getters, balance checks, and any function that returns information without changing anything.

pure - The function neither reads nor modifies state. It is a mathematical function that operates only on its inputs. Like view, pure functions cost zero gas when called externally. Use pure for calculations, string manipulation, and utility functions.

payable - The function can receive native coins (MSTC, ETH). Without this modifier, any attempt to send coins to the function will cause the transaction to revert. Use payable for deposit functions, purchase functions, and any function that should accept payment.

(no modifier) - The function can read and modify state. It costs gas and must be executed as a transaction.

10   The Constructor

The Constructor - Your Contract's Birth Certificate

The constructor is a special function that runs exactly once, at the moment the contract is deployed to the blockchain. It cannot be called again. It is defined using the constructor keyword (not the contract name - that was the old syntax, pre-Solidity 0.5.0). The constructor is used for one-time setup: setting the contract owner, initializing state variables with deployment-specific values, and performing any setup that must happen before the contract is usable.

contract Token {
    address public immutable owner;
    uint256 public immutable deployTime;

    constructor(string memory name, string memory symbol) {
        owner = msg.sender;       // The address that deployed the contract
        deployTime = block.timestamp; // When the contract was deployed
        _name = name;
        _symbol = symbol;
    }
}

In this example, the constructor captures msg.sender - the address that submitted the deployment transaction. This is the most common pattern for setting the contract's owner. The immutable keyword (introduced in Solidity 0.6.5) tells the compiler that these variables can only be set in the constructor and will never change. Immutable variables are stored in the contract's bytecode, not in storage, making them much cheaper to read than regular state variables.

🔧 Click to expand: msg.sender - The Single Most Important Global Variable

msg.sender returns the address that directly called the current function. In a simple transaction from a user's wallet, msg.sender is the user's address. In a call from one contract to another, msg.sender is the calling contract's address. This is the foundation of access control in Solidity - the onlyOwner pattern compares msg.sender to the stored owner address.

Warning: Do not confuse msg.sender with tx.origin. tx.origin returns the original sender of the transaction (the wallet), even if the call passed through multiple contracts. Using tx.origin for access control is a known vulnerability - a malicious contract can trick a user into calling it, and then the malicious contract calls your contract, where tx.origin would return the user's address (not the malicious contract's). Always use msg.sender.

11   Your First Contract

Hello World on MST - Write, Compile, Deploy

Below is a complete, compilable "Hello World" contract that you will now write, compile, and deploy to the MST testnet. This contract stores a greeting message, allows the owner to update it, and lets anyone read the current message. It combines every concept from this sub-module into a working example.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

/**
 * @title HelloWorld
 * @notice A simple contract that stores and retrieves a greeting message.
 *         Only the contract owner can update the greeting.
 */
contract HelloWorld {
    // The owner of the contract (set during deployment)
    address public immutable owner;

    // The greeting message stored on the blockchain
    string private greeting;

    // Event emitted when the greeting is updated
    event GreetingUpdated(
        address indexed updater,
        string oldGreeting,
        string newGreeting
    );

    /**
     * @notice Constructor sets the initial greeting and records the owner.
     * @param initialGreeting The first greeting message.
     */
    constructor(string memory initialGreeting) {
        owner = msg.sender;
        greeting = initialGreeting;
    }

    /**
     * @notice Updates the greeting message. Only the owner can call this.
     * @param newGreeting The new greeting message.
     */
    function updateGreeting(string memory newGreeting) public {
        require(msg.sender == owner, "Only the owner can update the greeting");
        string memory oldGreeting = greeting;
        greeting = newGreeting;
        emit GreetingUpdated(msg.sender, oldGreeting, newGreeting);
    }

    /**
     * @notice Returns the current greeting. Anyone can call this (no cost).
     * @return The current greeting message.
     */
    function getGreeting() public view returns (string memory) {
        return greeting;
    }
}

13   Glossary

Key Terms - Sub-Module 8.4

Solidity
A high-level, statically typed, contract-oriented programming language for writing smart contracts on the Ethereum Virtual Machine (EVM).
Pragma
A compiler directive that specifies the Solidity version the code is compatible with. Prevents compilation with incompatible compiler versions.
Contract
The fundamental building block in Solidity, similar to a class. Encapsulates state variables, functions, events, and modifiers.
State Variable
A variable stored permanently on the blockchain as part of the contract's storage. Expensive to write, cheaper to read.
Constructor
A special function that executes once when the contract is deployed. Used to set initial state and capture deployment-time values.
Visibility
Controls who can call a function: public (anyone), external (outside only), internal (this contract + children), private (this contract only).
view
A function modifier indicating the function reads state but does not modify it. Free to call externally.
pure
A function modifier indicating the function neither reads nor modifies state. Operates only on inputs. Free to call externally.
msg.sender
A global variable that returns the address that directly called the current function. Foundation of access control.
immutable
A keyword for variables set only in the constructor and stored in bytecode, saving gas on every read compared to regular storage variables.
