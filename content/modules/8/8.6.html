02   Real-World Introduction Story

The DeFi Protocol That Let Anyone Withdraw - Because of One Missing Modifier

📖 Story

In early 2023, a newly launched DeFi vault on a popular EVM chain held over $2 million in user deposits. The contract had a withdrawFees() function intended solely for the protocol owner to collect performance fees. However, the developer - working late, tired, and rushing to meet a deadline - forgot to add an access control modifier. The function was left as public with no restrictions whatsoever.


Within hours of deployment, an automated bot scanning the blockchain detected the unprotected function. It called withdrawFees() in a single transaction and drained the entire fee pool - over $80,000 worth of tokens - into an anonymous wallet. The community was outraged. The protocol's reputation never recovered, and the developer's career took a serious hit.


The fix would have been trivial: a single line adding modifier onlyOwner to the function signature, and a one-line modifier definition checking require(msg.sender == owner). That oversight - essentially forgetting to lock the door - cost real money and trust. This sub-module ensures you never make that mistake. By the end, you'll be able to write contracts that are secure by design, using functions, modifiers, and events as your primary tools.

03   Background & History

The Evolution of Solidity Functions, Modifiers, and Events

Solidity's design has evolved dramatically since its inception in 2014. Understanding that evolution helps you appreciate the "why" behind every syntax rule.

The Early Wild West: All Functions Public by Default

In Solidity versions before 0.4.0, every function was implicitly public. If you wrote function withdraw(uint amount) { ... }, it was callable by anyone - from outside the contract, from inside, from anywhere. Developers who were new to blockchain often didn't realize that public meant globally accessible. Countless contracts were exploited because a helper function, intended only for internal use, was accidentally exposed to the world.

The introduction of explicit visibility specifiers (public, private, internal, and later external) forced developers to consciously declare who could call each function. This single language change prevented an entire class of security vulnerabilities.

Modifiers: From Copy-Paste to Centralized Guards

Before modifiers, developers duplicated the same require(msg.sender == owner) check at the top of every admin function. A contract with ten admin functions had ten identical checks. If one was missed or written incorrectly, the contract was vulnerable. Modifiers were introduced as a way to encapsulate access control logic in a single, reusable block. Now you define the check once and apply it everywhere. This not only made contracts more secure but also made them easier to audit - an auditor could verify one modifier and know that every function using it was protected.

Events: Cheap, Permanent Logs

Events were part of Ethereum from the very beginning, inspired by the logging facilities in the underlying Yellow Paper. They are emitted via the LOG0 through LOG4 EVM opcodes. Unlike storage, event data is not stored in the state trie - it's recorded in the transaction receipt, which is accessible but not part of the state that smart contracts can read. This makes events extremely cheap (375 gas base + 8 gas per byte) compared to storage (20,000 gas for a new write). The indexed keyword was added to enable efficient filtering of events by specific parameter values, which is how every wallet displays your transaction history.

The ERC-20 standard's Transfer event became the most emitted event in blockchain history. Every token transfer, every swap, every mint emits a Transfer event. Frontends listen for these events to update balances in real-time. Without events, dApps would need to poll storage constantly, which is slow and expensive.

04   Beginner-Friendly Explanation

Functions Are the Buttons, Modifiers Are the Locks, Events Are the Receipts

🔒 The Analogy

Think of a smart contract as a vending machine in a public place. Each function is a button on the machine - press it, and something happens: a snack drops, a refund processes, or the machine reports its inventory. Modifiers are the locks on those buttons. The onlyOwner modifier is like a keyhole - only the owner's key can press the "empty the cash box" button. Events are the receipts printed by the machine, recording what happened, when, and for how much, so anyone can audit the machine's activity later. Together, they make the machine usable, secure, and accountable.

Let's explore each concept with the precision you'll need for writing and debugging real contracts on MST:

F
FUNC
Function anatomy: visibility, mutability, parameters, returns
A Solidity function has a visibility (who can call it), an optional mutability (view, pure, payable), input parameters, return values, and a body. Visibility must be declared explicitly. Mutability specifiers tell the compiler (and the user) whether the function will modify state or receive coins. The function signature - its name and parameter types - determines the 4-byte selector used to call it.
M
MODIFIER
Modifiers as reusable guard clauses
A modifier is a special function that runs before (and potentially after) the function it decorates. It can check conditions, modify arguments, or even prevent execution. The placeholder _; marks where the original function body will execute. Modifiers can be stacked, and the order of execution is from left to right. They are the primary tool for access control, reentrancy protection, and validation in Solidity.
E
EVENT
Events as a permanent, cheap, off-chain log
An event is a structured log entry emitted by a contract. It is stored in the transaction receipt and can be read by any off-chain application. Events are extremely cheap - around 1,200-1,500 gas for a typical Transfer event, compared to 20,000 gas for a storage write. Up to three parameters can be marked indexed, enabling efficient filtering. Events are the backbone of dApp frontends and blockchain explorers.
05   Technical Deep Dive - Functions

Function Visibility: The Gatekeeper of Smart Contract Logic

Every function must declare exactly one of four visibility specifiers. The choice affects security, gas costs, and the contract's public API. Here's the detailed breakdown with gas implications and common pitfalls:

VisibilityCallable from this contract?Callable from derived contracts?Callable externally (via transaction)?Gas efficiencyTypical use
publicYesYesYes (auto-generates getter for state vars)Slightly higher overhead for external calls because parameters are copied to memoryMain API functions, state variable getters
externalNo (only via this.f())NoYesCheapest for external calls: parameters read directly from calldata without copying to memoryFunctions intended for external callers only
internalYesYesNoCheaper than public for internal calls (no external dispatch overhead)Helper functions shared with child contracts
privateYesNoNoCheapest for purely internal logic; keeps bytecode cleanerFunctions strictly for this contract's internal logic

Why external saves gas: When an externally called public function receives an array or string parameter, the EVM first copies the data from calldata into memory - an expensive operation for large data. An external function reads the parameter directly from calldata, avoiding the copy. For small value types (uint256, address), the difference is negligible. But for dynamic arrays or long strings, the savings can be thousands of gas.

Real-world example: In the ERC-20 standard, the transfer and approve functions are usually declared public because they may be called internally by the contract itself (e.g., in a transferFrom implementation). But the transferFrom function is often external because it's only called from outside (by the spender). Understanding this nuance helps you design efficient contracts.

Mutability Modifiers: view, pure, payable

Beyond visibility, you can (and often should) declare a function's relationship to the contract state:

  • view: Promises not to modify the blockchain state. It can read storage variables, call other view functions, and use staticcall. When called externally (via a transaction), a view function costs zero gas because the node executes it locally and returns the result without broadcasting a transaction. This is perfect for getter functions.
  • pure: Even more restrictive - it cannot read or modify state. It can only operate on its input parameters and local variables. Think of it as a mathematical function: function add(uint a, uint b) pure returns (uint) { return a + b; }. Also costs zero gas when called externally.
  • payable: Marks a function that can receive the native coin (MSTC on MST, ETH on Ethereum) via msg.value. If a transaction sends coins to a non-payable function, the transaction reverts. This is a critical safety feature - it prevents accidental loss of funds. The receive() and fallback() functions must also be payable if they are to accept coins.

A state-changing function (one that writes to storage) does not need any special keyword - the absence of view or pure tells the compiler it will modify state. The compiler enforces these rules: if you try to write to storage inside a view function, compilation fails.

Function Overloading and the Fallback

Solidity supports function overloading - multiple functions with the same name but different parameter types. The compiler distinguishes them by their function selector (the first 4 bytes of the Keccak-256 hash of the signature). For example, transfer(address,uint256) and transfer(address,uint256,bytes) are different functions. Overloading is resolved at compile time.

Every contract can also have special receive() and fallback() functions:

  • receive() external payable - called when the transaction has no calldata and sends coins. Used for plain MSTC transfers.
  • fallback() external payable - called when no other function matches the selector, or when the receive function is not defined and coins are sent. Can also be non-payable.

06   Technical Deep Dive - Modifiers

Modifiers: The Swiss Army Knife of Function Guards

Modifiers are one of Solidity's most elegant features. They let you inject reusable logic before and after a function execution, without polluting the function body with repeated checks. Let's break down exactly how they work, from simple access control to sophisticated patterns.

The Anatomy of a Modifier

modifier onlyOwner() {
    require(msg.sender == owner, "Not the owner");
    _; // The function body will be inserted here
}

function withdraw(uint amount) external onlyOwner {
    // ... function logic
}

When withdraw is called, the modifier's code runs first. It checks require(msg.sender == owner). If the check fails, the entire transaction reverts with the error message "Not the owner". If it passes, the modifier reaches the _; placeholder, at which point the function body executes. After the function body finishes, execution returns to the modifier, which can run additional code after the placeholder (if any).

Common Modifier Patterns

1
onlyOwner - Basic Ownership
The simplest and most common modifier. Checks that the caller is the contract owner. Used for admin functions like pausing, withdrawing fees, or upgrading contracts.
2
nonReentrant - Blocking Recursive Calls
Sets a boolean lock locked = true before the function and clears it after. If the function is called again while locked, the modifier's require(!locked) reverts. This is the standard defense against reentrancy attacks. Used in OpenZeppelin's ReentrancyGuard.
3
onlyBefore / onlyAfter - Time-Based Access
Accepts a timestamp parameter and checks block.timestamp. Used for auctions (only accept bids before deadline) or token vesting (only release tokens after cliff).
4
onlyRole - Role-Based Access Control (RBAC)
Instead of a single owner, the contract maintains a mapping of roles (e.g., keccak256("MINTER_ROLE") → addresses). The modifier checks if the caller has the required role. Used in complex dApps with multiple admin tiers.

Stacking Modifiers

Modifiers execute in the order they are listed, from left to right. Each modifier can independently pass or revert. Consider a function that must be called only by the owner, only before a deadline, and must not be reentrant:

function emergencyWithdraw() external onlyOwner onlyBefore(deadline) nonReentrant {
    // ... protected logic
}

Execution flow:

  1. onlyOwner runs → checks msg.sender == owner.
  2. If passed, onlyBefore(deadline) runs → checks block.timestamp < deadline.
  3. If passed, nonReentrant runs → sets locked = true, then _;.
  4. Function body executes.
  5. After body, nonReentrant clears locked = false.

This composability is what makes modifiers so powerful. Each is a self-contained guard that you can mix and match.

07   Technical Deep Dive - Events

Events: Permanent, Off-Chain, Filterable Logs

Events are the bridge between the blockchain and the outside world. Smart contracts are deterministic, isolated environments - they cannot make HTTP requests or push notifications. Events solve this by emitting structured data into the transaction log, where off-chain systems (indexers, frontends, alert bots) can read and react to them.

The Anatomy of an Event

event Transfer(address indexed from, address indexed to, uint256 value);
event Deposit(address indexed user, uint256 amount, uint256 timestamp);

function deposit() external payable {
    emit Deposit(msg.sender, msg.value, block.timestamp);
}

Each event declaration creates a signature - the Keccak-256 hash of its name and parameter types - which becomes the first topic of the log entry. Indexed parameters become additional topics (up to three, plus the signature). Non-indexed parameters are ABI-encoded and stored in the log's data section.

Why Indexed Matters for Gas and Efficiency

Indexed parameters are searchable - you can ask a node "give me all Transfer events where from equals this address" without scanning every log. Non-indexed parameters require downloading all logs and filtering client-side. The cost of an indexed parameter is an additional topic (32 bytes of storage in the log), which adds a small gas overhead (375 gas per additional topic).

Golden rule: Index the parameters you'll query by. For ERC-20, from and to are indexed so wallets can display your transaction history. For a staking contract, index the user address so frontends can show "My Staking History". Don't index parameters you'll never filter on - it's wasted gas.

Gas Comparison: Events vs Storage

ActionGas CostPermanenceReadable by contracts?

Events are 30x cheaper than storage for permanent records. For historical data that doesn't need to be read by the contract itself, events are the clear winner. On MST, where gas is already extremely cheap, the rupee cost of emitting an event is about ₹0.00001 - effectively free - but the practice of using events over storage is still critical for scalability and clean architecture.

MST Advantage: With MST's gas price around 12 Gwei and MSTC at ₹8, emitting a typical Transfer event costs approximately ₹0.00014. That's less than 1/60th of a paisa. You can log millions of events for less than the cost of a single Ethereum storage write. This makes event-driven architectures extremely practical on MST.
08   Interactive Code Examples

Build Along: A Secure Vault with Modifiers and Events

Let's build a complete vault contract step by step. Click each section to reveal the code and explanation.

🏗️ Step 1: Base contract with state variables

We'll create a Vault that stores user balances and has an owner.

contract Vault {
    address public owner;
    mapping(address => uint256) public balances;
    
    constructor() {
        owner = msg.sender;
    }
}

The owner is set to the deployer. Balances are stored in a mapping.

🔐 Step 2: Add an onlyOwner modifier 
modifier onlyOwner() {
    require(msg.sender == owner, "Not owner");
    _;
}

Now any function can be protected by adding onlyOwner. The _; is where the function body goes.

📥 Step 3: Deposit function (anyone can call) 
event Deposited(address indexed user, uint256 amount);

function deposit() external payable {
    require(msg.value > 0, "No MSTC sent");
    balances[msg.sender] += msg.value;
    emit Deposited(msg.sender, msg.value);
}

This function is payable and external. Anyone can send MSTC to the contract. The deposit is recorded in storage and an event is emitted so frontends can update instantly.

🚀 Step 4: Withdraw function with onlyOwner, Checks-Effects-Interactions, and event 
event Withdrawn(address indexed user, uint256 amount);

function withdraw(uint256 amount) external onlyOwner {
    // Checks
    require(balances[owner] >= amount, "Insufficient balance");
    // Effects
    balances[owner] -= amount;
    // Interactions
    payable(owner).transfer(amount);
    // Log
    emit Withdrawn(owner, amount);
}

Notice the order: state is updated before the external call. This prevents reentrancy. The event is emitted after the transfer, though it could be before - the important thing is that the state change precedes the external call.

🔄 Step 5: Adding nonReentrant for extra safety 
bool private locked;
modifier nonReentrant() {
    require(!locked, "Reentrant call");
    locked = true;
    _;
    locked = false;
}

function withdraw(uint256 amount) external onlyOwner nonReentrant {
    // ... same body
}

Even if someone finds a way to re-enter, the lock will prevent it. This is belt-and-suspenders with C-E-I.

09   Common Pitfalls and Best Practices

Don't Learn These the Hard Way

⚠️
Forgetting payable on a function that must receive MSTC
If a user sends MSTC to a non-payable function, the transaction reverts. The user loses gas and gets nothing. Always mark functions payable if they need to accept coins. Use receive() for plain transfers.
⚠️
Using public instead of external for large calldata arrays
For functions that accept dynamic arrays or long strings, public will copy the data to memory, costing thousands of extra gas. Use external and read directly from calldata.
⚠️
Emitting events after external calls
Always emit events before external interactions, or at least ensure the state changes are committed first. If an external call fails, the entire transaction reverts, but the event will not be emitted. However, for reentrancy safety, the Checks-Effects-Interactions pattern should have Effects (state changes) before Interactions (external calls). Events are not state changes per se, but they should be positioned logically.
💡
Over-indexing events
Each indexed parameter costs extra gas and occupies a topic slot. Only index the fields you'll actually filter on. For a Transfer event, from and to are essential; for a Deposit event, user is essential.

Top Facts - Know These Cold for the Exam

public, external, internal, private define the caller scope. view, pure, payable define what the function can do. A function without view/pure can modify state.
02

03

Index up to 3 parameters for filtering. The event signature is always the first topic.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

contract SimpleVault {
    address public owner;
    mapping(address => uint256) public balances;

    event Withdrawal(address indexed user, uint256 amount);

    constructor() {
        owner = msg.sender;
    }

    modifier onlyOwner() {
        require(msg.sender == owner, "Not owner");
        _;

    function deposit() external payable {
        balances[msg.sender] += msg.value;

        payable(owner).transfer(amount);

    }

11   Glossary

Key Terms - Sub-Module 8.6

Visibility
Specifies who can call a function: public, external, internal, private. Controls access and gas efficiency.
Modifier
A reusable code block that runs before a function. Used for access control (onlyOwner), reentrancy guards, and validation.
Event
A log record emitted by a contract, stored in transaction receipts. Used for off-chain communication. Cheaper than storage.
Indexed
Keyword for event parameters that enables filtering. Up to 3 parameters can be indexed. Indexed parameters are stored as topics.
Checks-Effects-Interactions
A security pattern: check conditions, update state, then interact with external contracts. Prevents reentrancy.
payable
Function modifier that allows the function to receive native coins (MSTC). Required for any function that uses msg.value.
view / pure
Function modifiers that promise not to modify state (view) or not to read or modify state (pure). Cost zero gas externally.
receive()
Special fallback function that is called when a contract receives plain native coin transfers without calldata. Must be payable.
nonReentrant
A modifier pattern that prevents a function from being called while it's still executing, blocking reentrancy attacks.
msg.sender
Global variable containing the address that called the current function. Often used in access control checks within modifiers.
