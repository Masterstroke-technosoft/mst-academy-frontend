03   Background & History

From Truffle to Hardhat - The Evolution of EVM Tooling

In Ethereum's early days (2015 - 2017), smart contract development was primitive. Developers used the Remix IDE in a browser, compiled contracts with the solc compiler directly, and deployed manually via the Mist browser or command-line tools. There was no standardized project structure, no automated testing framework, and no way to reproduce builds across machines.

Truffle (launched 2015) was the first attempt at a comprehensive development framework. It introduced a project structure with contracts/, migrations/, and test/ folders, and its own testing environment based on Mocha. Truffle also included a built-in local blockchain (Ganache). For years, Truffle was the de facto standard. However, it had drawbacks: a rigid migration system, limited configurability, and a disconnect between the local testnet and the real Ethereum network behavior.

In 2018, a new project called Buidler (later renamed Hardhat in 2020) was launched by the team at Nomic Foundation. Hardhat took a different approach: instead of a monolithic framework, it provided a modular, plugin-based architecture. The core was lightweight - just a task runner and a local Ethereum network. Everything else (compilation, testing, gas reporting, contract verification) was a plugin. This made Hardhat incredibly flexible and easy to extend.

Hardhat's built-in local network (Hardhat Network) was a game-changer. It wasn't a separate process like Ganache; it was integrated directly into the Node.js runtime, allowing for console.log from Solidity, stack traces on revert, and the ability to fork mainnet state - simulating transactions against real deployed contracts. This enabled testing against Uniswap, Aave, or any on-chain protocol with zero setup.

By 2022, Hardhat had surpassed Truffle in popularity and became the recommended framework for most EVM chains, including MST. MST's full EVM compatibility means that every Hardhat plugin, every Solidity compiler version, and every deployment script works identically on MST - you simply point the RPC URL to an MST node.

2015
Truffle launched. First Ethereum development framework. Introduced contract compilation, migration scripts, and Ganache local blockchain. Established the contracts/ and migrations/ folder convention.
2018
Buidler (later Hardhat) launched. Modular, plugin-based architecture. Integrated local Ethereum network with Solidity console.log. Built on Node.js and designed to be extended.
2020
Hardhat rebranded and gained traction. Added Hardhat Network forking, stack traces, and a growing plugin ecosystem. Adopted by major projects like Uniswap, Aave, and OpenZeppelin.
2023
Hardhat Ignition released. New declarative deployment system. Replaces the legacy scripts-based deployment with a more robust, type-safe approach.
Today
MST + Hardhat. Full compatibility. Deploy to MST Testnet or mainnet by adding a network entry in hardhat.config.js. All plugins work without modification. Use Hardhat's local network to simulate MST's environment.
04   Beginner-Friendly Explanation

Hardhat as a Workshop for Smart Contracts

🔧 The Analogy

Think of Hardhat as a fully equipped automobile workshop. You bring in your Solidity source code (the car blueprint). Hardhat's compiler (the assembly line) turns it into EVM bytecode (the car). The testing framework (quality control) runs the car on a closed track (Hardhat Network) to check for faults. The deployment script (the delivery truck) ships the finished car to the blockchain (the customer). And if something breaks, the debugger (the mechanic's diagnostic tool) tells you exactly which part failed. You can even test your car on real roads by forking the mainnet - driving alongside existing vehicles to ensure compatibility.

Let's break down what Hardhat actually does under the hood:

C
COMPILE
What does Hardhat compile?
Hardhat uses the Solidity compiler (solc) to convert your .sol files into two outputs: bytecode (the machine code that the EVM executes) and an ABI (Application Binary Interface - a JSON description of your contract's functions). These are stored in the artifacts/ folder. Hardhat caches compilation results, so only changed files are recompiled. It supports multiple Solidity versions, even within the same project.
T
TEST
How does Hardhat run tests?
Hardhat integrates with Mocha (test framework) and Chai (assertion library). You write tests in JavaScript (or TypeScript) that deploy your contracts to Hardhat's local network, call functions, and verify results. Each test file runs in isolation with a fresh blockchain state. Hardhat also supports console.log inside Solidity - you can print variable values from your contract during tests, which is invaluable for debugging.
D
DEPLOY
How does Hardhat deploy contracts?
You write deployment scripts in JavaScript (or TypeScript) that use ethers.js to deploy your contracts to any network defined in hardhat.config.js. A typical script creates a contract factory, calls .deploy() with constructor arguments, and waits for confirmation. Hardhat also supports Ignition, a declarative deployment system where you define deployment modules that handle dependencies and upgrades.
05   Interactive Walkthrough

Set Up a Hardhat Project for MST - Step by Step

Follow along by clicking each step. You'll have a fully functional Hardhat project connected to MST Testnet by the end.

📁 Step 1: Initialize a Hardhat project

In your terminal (inside an empty folder), run:

npx hardhat init

Hardhat will ask you to choose a project type. Select "Create a JavaScript project". This sets up the basic folder structure:

my-hardhat-project/
├── contracts/          ← Your Solidity files go here
│   └── Lock.sol        ← Sample contract
├── test/               ← Your test files
│   └── Lock.js
├── scripts/            ← Deployment and utility scripts
│   └── deploy.js
├── hardhat.config.js   ← Main configuration file
├── package.json
└── node_modules/

The sample project includes a Lock.sol contract and a test file to get you started. You can delete them and add your own contracts.

⚙️ Step 2: Configure networks for MST

Open hardhat.config.js. You'll see a basic configuration. We'll add MST Testnet and mainnet networks. The configuration file is JavaScript, so you can use environment variables for private keys.

First, install dotenv to manage secrets securely:

npm install --save-dev dotenv

Then modify hardhat.config.js:

require("@nomicfoundation/hardhat-toolbox");
require("dotenv").config();

/** @type import('hardhat/config').HardhatUserConfig */
module.exports = {
  solidity: "0.8.24",
  networks: {
    mst_testnet: {
      url: "https://testnet-rpc.mstblockchain.com",
      chainId: 5000,
      accounts: [process.env.PRIVATE_KEY]
    },
    mst_mainnet: {
      url: "https://mainnet-rpc.mstblockchain.com",
      chainId: 5001,
      accounts: [process.env.PRIVATE_KEY]
    }
  }
};

Create a .env file in your project root:

PRIVATE_KEY=your_wallet_private_key_here

Never commit .env to Git! Add it to .gitignore.

🔨 Step 3: Compile your contracts

Place your Solidity contracts in the contracts/ folder. Then run:

npx hardhat compile

Hardhat will download the correct Solidity compiler version (if not already cached), compile all .sol files, and generate artifacts in the artifacts/ folder. Successful compilation looks like:

Compiled 2 Solidity files successfully (with warnings)

If there are errors, Hardhat will show the exact file, line number, and reason - with a clickable link in VS Code's terminal.

🧪 Step 4: Run tests

Write tests in the test/ folder using Mocha and Chai. A typical test loads a contract, deploys it, and checks behavior:

const { expect } = require("chai");

describe("Lock", function () {
  it("Should set the right unlock time", async function () {
    const Lock = await ethers.getContractFactory("Lock");
    const lock = await Lock.deploy(ethers.parseEther("0.1"), 1893456000);
    expect(await lock.unlockTime()).to.equal(1893456000);
  });
});

Run the tests with:

npx hardhat test

Hardhat spins up a local blockchain, deploys the contract, runs the test, and destroys the network - all in seconds. You'll see a report of passing and failing tests.

🚀 Step 5: Deploy to MST Testnet

Create a deployment script in scripts/deploy.js:

const hre = require("hardhat");

async function main() {
  const Lock = await hre.ethers.getContractFactory("Lock");
  const lock = await Lock.deploy(hre.ethers.parseEther("0.1"), 1893456000);
  await lock.waitForDeployment();
  console.log("Lock deployed to:", lock.target);
}

main().catch(console.error);

Run the deployment on MST Testnet:

npx hardhat run scripts/deploy.js --network mst_testnet

Hardhat will connect to the MST Testnet RPC, send the deployment transaction, and wait for confirmation. You'll see the deployed contract address in the console.

06   Technical Deep Dive

Anatomy of hardhat.config.js - Every Field Explained

The hardhat.config.js file is the brain of your project. Let's dissect every major configuration option you'll use in production.

The Solidity Compiler Configuration

The solidity field can be a simple string (e.g., "0.8.24") or an object with multiple compiler versions and optimizer settings. For production contracts, you'll use the optimizer:

solidity: {
  compilers: [
    {
      version: "0.8.24",
      settings: {
        optimizer: {
          enabled: true,
          runs: 200
        }
      }
    }
  ]
}

The optimizer reduces bytecode size and gas costs by applying optimizations like function inlining and dead code elimination. The runs parameter (200 is typical) estimates how many times the contract will be called over its lifetime - higher values optimize for cheaper execution, lower values optimize for cheaper deployment. For MST, where gas is already cheap, you may not need aggressive optimization, but enabling it is good practice for complex contracts.

Network Configuration

Each network is defined with:

  • url: The JSON-RPC endpoint of an MST node. You can use public RPCs, your own node, or a service like Alchemy or Infura.
  • chainId: The network's chain ID - used to prevent transaction replay across chains. MST Testnet uses 5000; mainnet uses 5001 (example).
  • accounts: An array of private keys that can sign transactions. Use process.env.PRIVATE_KEY to avoid hardcoding secrets.
  • gasPrice: (optional) The gas price in wei to use for transactions. On MST, gas is cheap, so you can leave this auto or set a low value.
  • gas: (optional) The gas limit for transactions. Hardhat estimates gas automatically, but you can override if needed.

The Hardhat Network (Local Blockchain)

The built-in hardhat network is configured separately:

networks: {
  hardhat: {
    forking: {
      url: "https://mainnet-rpc.mstblockchain.com",
      blockNumber: 14390000  // optional: pin to a specific block
    },
    chainId: 31337,
    mining: {
      auto: true,
      interval: 0  // mine a new block on every transaction
    }
  }
}

Forking is one of Hardhat's most powerful features. It copies the entire state of the MST mainnet (or any EVM chain) at a specific block into your local network. This lets you test your contract against real deployed contracts (like DEXs, lending pools, or any protocol) without needing to deploy them yourself. For example, you can test a flash loan attack against a live Uniswap pool without risking real funds.

Plugins and Toolbox

Hardhat's power comes from plugins. The Hardhat Toolbox (@nomicfoundation/hardhat-toolbox) bundles the most essential plugins:

  • @nomicfoundation/hardhat-ethers - Integrates ethers.js with Hardhat.
  • @nomicfoundation/hardhat-verify - Verifies contracts on block explorers (like MST Explorer).
  • hardhat-gas-reporter - Generates gas usage reports for your contracts.
  • solidity-coverage - Measures test coverage of your Solidity code.
  • @typechain/hardhat - Generates TypeScript type bindings for your contracts.

To use the toolbox, simply require it in your config:

require("@nomicfoundation/hardhat-toolbox");

This single line unlocks all the bundled features. For MST-specific verification, you'll need to add the MST Explorer API URL and key. More on that in Module 11.

07   Hardhat Network

Your Personal Blockchain for Development and Testing

Hardhat Network is an in-process Ethereum node written in TypeScript. Unlike Ganache (which runs as a separate process), Hardhat Network runs within the same Node.js process as your tests and scripts. This tight integration provides unique capabilities:

console.log in Solidity

One of Hardhat's most celebrated features is the ability to use console.log directly in Solidity. Simply import Hardhat's console contract:

import "hardhat/console.sol";

contract MyContract {
    function doSomething(uint256 x) public {
        console.log("The value of x is:", x);
        // ... rest of function
    }
}

When you run a test or script on Hardhat Network, the log output appears in your terminal. This works because Hardhat Network intercepts the console calls at the EVM level. It's only available on Hardhat Network - not on live networks - but it's invaluable for debugging.

Stack Traces and Error Messages

When a transaction reverts, Hardhat provides a full Solidity stack trace showing the exact function call chain, the line number where the error occurred, and the revert reason. This is a massive improvement over the generic "execution reverted" message you'd get from a live node.

Mainnet Forking

By adding forking: { url: "..." } to the hardhat network config, you get a copy of the entire blockchain state at your chosen block. All accounts, balances, and contract storage are available locally. You can impersonate any account (using hre.network.provider.request({ method: "hardhat_impersonateAccount", params: [address] })) and send transactions as if you were that address. This is perfect for testing integrations without spending real gas.

How Top Projects Use Hardhat

🦄
Uniswap v4 - Testing Across Chains with Forking
Uniswap's development team uses Hardhat with mainnet forking to test new pool contracts against real-world liquidity. By forking Ethereum mainnet, they can simulate swaps against actual Uniswap v3 pools, verify pricing logic, and test edge cases - all without spending gas. When deploying to new chains like MST, the same Hardhat test suite runs against a forked MST mainnet state, ensuring identical behavior. This is a perfect example of how Hardhat's forking enables cross-chain development without duplicating test infrastructure.
🏦
Aave - Gas Optimisation with hardhat-gas-reporter
Aave's lending protocol is gas-sensitive - every function call costs user funds. The team uses the hardhat-gas-reporter plugin to generate detailed reports of gas consumption for each function. During optimization, they run the test suite, review the gas report, tweak the Solidity code (e.g., replacing storage variables with memory, using unchecked blocks for arithmetic), and re-run to measure improvements. On MST, where gas is cheap, gas optimization is still important for high-frequency transactions like liquidation bots, and hardhat-gas-reporter makes it measurable.

11   Glossary

Key Terms - Sub-Module 8.3

Hardhat
An EVM development framework for compiling, testing, debugging, and deploying smart contracts. Built on Node.js, fully compatible with MST.
Hardhat Runtime Environment (HRE)
The object injected into scripts and tests, providing access to ethers.js, network configuration, and plugins.
Hardhat Network
Built-in local Ethereum network that runs in-process. Supports console.log from Solidity, stack traces, mainnet forking, and account impersonation.
Forking
Copying the state of a live blockchain (e.g., MST mainnet) into your local Hardhat Network, allowing tests against real deployed contracts.
Hardhat Toolbox
A bundled plugin (@nomicfoundation/hardhat-toolbox) that includes ethers, verify, gas reporter, and other essential plugins.
hardhat.config.js
The main configuration file. Defines Solidity compiler version, networks (RPC URLs, chain IDs, accounts), and plugins.
Artifacts
The compiled output of Solidity contracts - bytecode and ABI - stored in the artifacts/ folder.
Chain ID
A unique identifier for a blockchain network, used to prevent replay attacks. MST Testnet chain ID is 5000 (example).
gas reporter
A Hardhat plugin (hardhat-gas-reporter) that outputs a table of gas consumption per function, helping optimize contracts.
Ignition
Hardhat's new declarative deployment system, replacing legacy scripts with type-safe deployment modules.
