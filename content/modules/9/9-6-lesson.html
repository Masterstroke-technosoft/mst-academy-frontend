03   Background & History

From Linus's Law to OpenZeppelin Audits - Why Peer Review Is Non-Negotiable

In software engineering, the value of peer review has been enshrined for decades. Linus's Law, coined by Eric Raymond in his essay "The Cathedral and the Bazaar," states: "Given enough eyeballs, all bugs are shallow." The idea is simple: while one developer might miss a flaw, a group of reviewers with diverse perspectives will catch it quickly. This principle is the foundation of open-source software quality.

In the blockchain space, peer review is not just a best practice - it's an existential necessity. Smart contracts handle real money, are immutable once deployed, and have no "undo" button. A single overlooked vulnerability can drain millions of dollars. The industry learned this the hard way through a series of catastrophic hacks: The DAO (2016, $60M), Parity Wallet (2017, $150M), Wormhole Bridge (2022, $325M). In each case, a fresh set of eyes - or a formal audit - could have caught the bug before deployment.

Today, professional blockchain development follows a multi-layered review process:

  • Peer code review: Every pull request is reviewed by at least one other developer before merging. Tools like GitHub's pull request system and formal review checklists are standard.
  • Automated analysis: Static analysis tools (Slither, Mythril) and fuzzing (Echidna, Foundry) run automatically on every commit to catch common vulnerabilities.
  • Internal audit: A dedicated security team manually reviews the codebase, focusing on economic exploits and logic errors that automated tools miss.
  • External audit: A third-party firm (Trail of Bits, OpenZeppelin, CertiK) performs a deep, paid audit before mainnet deployment.
  • Bug bounties: After deployment, ethical hackers are incentivized to find and report vulnerabilities before malicious actors do.

Your capstone project sits at the beginning of this pipeline. The peer feedback exercise you'll complete mirrors the code review process used by Uniswap, Aave, and every serious protocol. Approach it with the same rigor, and your project will be stronger for it.

Key insight: On MST, low gas fees mean you can iterate faster - deploy a fix, test it, redeploy - without spending a fortune. This makes the "review → fix → retest" loop dramatically tighter and more practical than on Ethereum.
04   Beginner-Friendly Explanation

Feedback Is Like a Building Inspection - It Finds Cracks Before the Earthquake

🏗️ The Analogy

Imagine you're building a house. You've framed the walls, installed the plumbing, and wired the electricity. Before you put up drywall, you call a building inspector. The inspector walks through, clipboard in hand, and points out: "This beam is undersized. That wire is the wrong gauge. The bathroom vent doesn't meet code." You could get defensive - "I spent weeks on this!" - or you could be grateful: they just saved you from a collapsed roof and an electrical fire. Peer review is exactly that inspection for your code. Your reviewer isn't attacking you; they're protecting your users from bugs you couldn't see because you're too close to the code.

Let's break down the two sides of peer feedback - giving and receiving - and then see how they combine into an iteration cycle.

G
GIVE
How do you give feedback that actually helps?
Effective feedback is specific, actionable, and kind. Instead of "your code is messy," say "the withdraw function lacks a reentrancy guard - consider adding OpenZeppelin's nonReentrant modifier." Instead of "the UI is ugly," say "the transaction button doesn't show a loading state, so users might click it twice. Try adding a spinner and disabling the button during pending." Always tie feedback to a concrete observation and a suggested improvement. Use the SBI model: Situation (where in the code), Behavior (what the code does or doesn't do), Impact (why it matters - user confusion, security risk, gas waste).
R
RECEIVE
How do you receive feedback without getting defensive?
The most successful developers cultivate a growth mindset: they see their code as a work in progress, not a reflection of their worth. When a peer points out a bug, they say "Thank you for catching that - let me fix it" instead of "But it worked in my tests!" Separate your identity from your work. Remember: the goal is a better product, not ego protection. Before responding, take a breath. Read the feedback twice. Ask clarifying questions if needed. Then create a prioritized list of fixes. Not all feedback must be implemented - some may be matters of style or preference - but all should be considered.
I
ITERATE
How do you turn feedback into improvements?
Iteration is the engine of quality. After receiving feedback, you follow a cycle: Prioritize (fix critical security issues first, then UX, then polish), Implement (make the changes in small, testable commits), Redeploy (push to MST Testnet and verify), Re-review (ask your peer to check the fixes). This loop repeats until both the reviewer and you are satisfied. In professional teams, code goes through multiple review rounds before merging. On MST, redeploying a contract costs ~₹0.02, so you can afford to iterate aggressively.

06   How to Give Constructive Feedback

The Art of Reviewing Code Without Crushing Souls

Giving feedback is a skill that improves with practice. A bad review can demoralize a peer; a good review can accelerate their learning. Here's how to do it right.

The SBI Model: Situation, Behavior, Impact

Structure every piece of feedback using this framework:

  • Situation: Where exactly is the issue? Reference the file name, function, or line number. "In Escrow.sol, line 42, in the releaseFunds function…"
  • Behavior: What is the code doing (or failing to do)? "…the function sends funds before updating the escrow status…"
  • Impact: Why does this matter? "…which creates a reentrancy vulnerability where an attacker could drain the escrow by calling releaseFunds recursively."

Always end with a suggestion: "Consider moving the status update to before the external call, and adding a nonReentrant modifier."

The Review Checklist

When reviewing a peer's capstone, examine these areas systematically:

🔒 Smart Contract Security Checklist
  • Are all external functions protected with appropriate access control (modifiers, require)?
  • Is the Checks-Effects-Interactions pattern followed in every function that transfers funds?
  • Are there any reentrancy risks? (Look for external calls before state updates.)
  • Are integer overflows/underflows possible? (Pre-Solidity 0.8, or in unchecked blocks.)
  • Are there any unbounded loops that could exceed block gas limits?
  • Are all external contract calls using a safe pattern (e.g., checking return values)?
  • Is the contract using tx.origin instead of msg.sender? (That's a red flag.)
  • Are there any hardcoded addresses that should be constructor parameters?
⚡ Gas Optimization Checklist
  • Are storage variables packed efficiently? (Small types together.)
  • Are read-only function parameters marked calldata instead of memory?
  • Are there redundant storage reads that could be cached in memory?
  • Are events emitted efficiently? (Unnecessary indexed parameters?)
  • Could unchecked blocks be used safely for arithmetic?
🎨 UI/UX Checklist
  • Does the wallet connection flow handle all states (no wallet, wrong network, rejected)?
  • Do transaction buttons show loading, success, and error states?
  • Is the connected address displayed in a readable, truncated format?
  • Are error messages user-friendly, not raw JSON or console logs?
  • Is the demo link live and functional?
📝 Documentation Checklist
  • Does the README explain the project's purpose and how to run it?
  • Is the contract architecture described (even briefly)?
  • Are function purposes explained via NatSpec comments?
  • Is the deployed contract address listed?
  • Are setup instructions clear for a new developer?

08   Iterating on Your Capstone

Turning Feedback Into a Better dApp - A Step-by-Step Guide

After receiving feedback, you need a systematic approach to act on it. Here's your iteration playbook:

📋 Step 1: Categorize and Prioritize

Sort feedback into four buckets:

  • 🔴 Critical / Security: Reentrancy risks, missing access controls, integer overflows. Fix immediately.
  • 🟠 Functional: Logic errors, broken features, incorrect state transitions. Fix next.
  • 🟡 UX / Polish: Missing loading states, unclear error messages, confusing UI. Fix after functionality.
  • 🟢 Nice-to-Have: Suggested refactors, alternative implementations, documentation enhancements. Consider, but don't block submission.
🔧 Step 2: Implement Fixes

For each fix, create a separate Git commit with a clear message:

git commit -m "fix: add nonReentrant guard to releaseFunds (peer review feedback)"
git commit -m "fix: add loading spinner to vote button"

Write a test that would have caught the bug, if one doesn't already exist. This prevents regression.

🚀 Step 3: Redeploy to MST Testnet

After fixing, redeploy your contract using Hardhat:

npx hardhat run scripts/deploy.js --network mst_testnet

Verify the contract on MST Explorer. Update the contract address in your frontend. Deploy the updated frontend to your demo URL.

🔄 Step 4: Re-review (Optional)

Ask your peer to check the fixes. A quick "I fixed the reentrancy issue - can you take a look?" goes a long way. This closes the feedback loop and shows professionalism.

09   Feedback-Driven Improvements

Common Peer Review Findings - And How to Fix Them

Based on hundreds of capstone reviews, here are the most frequent issues and their solutions:

🔴
Missing access control on admin functions
Feedback: "Your withdrawFees() function is public with no modifier. Anyone can call it." Fix: Add an onlyOwner modifier. If using OpenZeppelin, inherit Ownable. Always check: does this function need to be callable by everyone? If not, protect it.
🟠
Reentrancy vulnerability in fund transfer
Feedback: "You transfer funds before updating state in releaseFunds. This is a reentrancy risk." Fix: Reorder: update state first (Checks → Effects), then transfer (Interactions). Add OpenZeppelin's nonReentrant modifier for defense in depth.
🟡
No loading state on transaction buttons
Feedback: "I clicked 'Vote' twice because nothing happened. I didn't know the first transaction was pending." Fix: Add a isProcessing state. Show a spinner and disable the button during the transaction. Re-enable on success or error.
🟢
Missing NatSpec documentation
Feedback: "Your functions don't have comments. I had to read the code to understand what they do." Fix: Add /// @notice and /// @param tags above every public function. This is a professional standard that also helps during audits.

12   Common Pitfalls in Peer Feedback

Don't Fall Into These Traps

⚠️
Giving vague feedback
"Your code is messy" is useless. Be specific: "The createEscrow function is 80 lines long and does validation, state updates, and event emission. Consider extracting the validation into a modifier and the state update into an internal function."
⚠️
Receiving feedback as a personal attack
If you find yourself thinking "they just don't understand my genius," pause. Your peer is trying to help. Listen first, defend later - or better, don't defend at all. Ask questions instead.
⚠️
Fixing feedback without testing
You fix a bug and redeploy, but you don't test it. The fix introduces a new bug. Always run your test suite after changes, and manually verify the fix on the live demo.

Top Facts - Know These for Your Peer Review Submission

01
Use the SBI model: Situation, Behavior, Impact
Every piece of feedback should reference the specific location (file, function), describe what the code does, and explain why it matters. End with a suggestion.
02

14   Glossary

Key Terms - Sub-Module 9.6

Peer Review
The process of having one's code, design, or documentation evaluated by fellow developers to identify issues, improve quality, and share knowledge.
SBI Model
Situation-Behavior-Impact - a structured feedback framework. Describe the situation (where), the behavior (what the code does), and the impact (why it matters).
Growth Mindset
The belief that abilities can be developed through effort and feedback. In coding, it means viewing bugs as learning opportunities rather than personal failures.
Iteration
The cycle of building, reviewing, and refining a product. Each cycle improves the product based on feedback and new insights.
Checks-Effects-Interactions
A security pattern that prevents reentrancy by ordering code: check conditions, update state, then interact with external contracts.
NatSpec
Solidity's natural language specification format for documenting functions using /// @notice and /// @param tags.
