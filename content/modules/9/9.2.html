02   Real-World Introduction Story

The Startup That Built Without a Blueprint

📖 Story

In 2022, a team of three developers set out to build a decentralized freelance platform on Ethereum. They started coding immediately - a single large FreelanceMarket.sol contract that handled user profiles, job postings, escrow, disputes, and payments. No diagram, no component separation, no modular design.


Three months in, they needed to add arbitration by a DAO. The single contract was already near the 24 KB size limit. Adding more code was impossible without splitting, but the logic was so intertwined that refactoring took weeks. The gas costs were astronomical because every user action looped through arrays. When they applied for a grant, the evaluators asked for an architecture diagram - and they had none. The grant was rejected.


A mentor stepped in: "Draw the system first. Separate concerns. Profile, escrow, dispute - each its own contract. Link them through interfaces. Use events for off-chain data. Only then code." In one week, they redesigned the entire architecture. The new system was modular, upgradeable, and gas-efficient. They received the grant on the second attempt and launched successfully.

This sub-module gives you the mindset and tools to be that mentor for your own capstone project. Before you write a single line of Solidity, you'll learn to think like a system architect - designing components, defining interfaces, and anticipating future needs. The result: a project that is robust, maintainable, and impressive to evaluators.

03   Background & History

From Monolithic to Modular - The Evolution of dApp Architecture

The earliest dApps (2015 - 2017) were monolithic. A single smart contract contained all the logic: storage, business rules, and access control. This was simple but quickly became unmanageable. Contracts hit the size limit. Bugs in one feature threatened the entire system. Upgrading meant deploying a new contract and migrating all state - a nightmare.

The breakthrough came with modular design and proxy patterns. Developers started separating concerns: a token contract, a governance contract, a treasury contract, each talking through defined interfaces. This made audits easier (each contract could be audited independently) and upgrades possible (only one module needed replacement).

ERC-20 and ERC-721 are themselves examples of successful architecture: the standard defines a minimal interface, and thousands of projects implement it. The Uniswap architecture - separating the factory, pair, and router contracts - became a reference design for DeFi protocols. The Diamond Standard (EIP-2535) pushed modularity further by allowing a single contract to delegate function calls to multiple facet contracts, enabling unlimited contract size.

On MST, you have additional architectural options: WASMify enables off-chain computation bridged on-chain, SARAL provides non-custodial wallet abstraction, and the hybrid L1 allows permissioned zones for private data. Understanding how to integrate these into your architecture is key to building next-generation dApps.

The architectural evolution timeline:

2015-2016
Monolithic era. Single contracts with all logic. Size limits, no upgradeability. Examples: early token ICOs.
2017-2018
Interface standardization. ERC-20, ERC-721 established. Contracts started using interfaces for composability. Factory pattern emerges.
2019-2020
Proxy patterns and upgradeability. OpenZeppelin's Transparent Proxy and UUPS pattern enable logic upgrades without state migration. DeFi summer explodes with modular designs.
2021-2022
Diamond Standard (EIP-2535). Multi-facet proxy allows unlimited contract size. Layer 2 architectures separate execution from data availability.
Today
MST-native architectures. Hybrid L1 with public + permissioned zones, WASMify for verifiable off-chain compute, SARAL for wallet abstraction. Architects can design systems that blend on-chain security with off-chain efficiency.
04   Beginner-Friendly Explanation

The Blueprint Analogy - Why Architects Don't Build Without Plans

🏛️ The Analogy

When constructing a building, you don't start by mixing cement. You draw a blueprint first: the foundation, the pillars, the electrical wiring, the plumbing - each a separate system that connects at defined points. A smart contract system is no different. The contracts are the pillars, the frontend is the facade, the events are the communication pipes, and the oracles are the external utility connections. If you pour concrete without a plan, you'll regret it when you need to add a window.

At its core, every dApp architecture answers five questions:

W
WHAT
What data must live on-chain, and what can be off-chain?
Critical state (ownership, balances, governance) must be on-chain for trustlessness. Large blobs (images, documents) belong on IPFS or Arweave. Derived data (analytics, leaderboards) can be indexed off-chain via events. Choosing the boundary is the most impactful architectural decision - it determines gas costs and scalability. On MST, the low gas costs mean you can store more on-chain, but you should still be disciplined. Every byte on-chain is replicated forever on every node. Use events for historical logs.
H
HOW
How will contracts interact and be upgraded?
Monolithic vs modular. If modular, through interfaces, inheritance, or proxy delegation? Will you use the Diamond Standard for unlimited size? Will you use a factory pattern for user-deployed contracts? The interaction pattern dictates security, gas efficiency, and upgradeability. For a simple capstone, a modular set of contracts with clear interfaces is often best. For a long-lived protocol, proxy patterns are essential. Think about how you'll add features in v2 without breaking v1 users.
A
ACCESS
Who can do what - access control architecture
A single onlyOwner modifier, or role-based access control (RBAC) with multiple roles (minter, pauser, upgrader)? How will governance (DAO) interact with admin functions? This is the security foundation - misdesigning it leads to exploits. Production contracts almost always use OpenZeppelin's AccessControl or a custom role system. Document who holds each role and why. For a DAO-based project, define how proposals can change roles or parameters.
O
OFF-CHAIN
What off-chain services does the dApp need?
Oracles for price feeds, WASMify for verifiable computation, IPFS for metadata, The Graph for indexing, SARAL for wallet abstraction. Identify these early - they shape the contract interface and the frontend integration. For example, if you're using a subgraph, your events must be designed for efficient indexing (indexed parameters). If you're using WASMify, your contract needs a callback or verification function.
S
SCALE
How will the architecture handle growth?
Can you migrate state? Can you add new features without breaking existing ones? Are you using proxy patterns? Does your event schema support new data fields? Planning for growth from day one avoids costly rewrites. Design your storage layout with extra padding or use a flexible struct. Ensure your events include a version or extra data field that can be extended. For MST, the hybrid architecture lets you expand from a public dApp into permissioned enterprise zones later.
05   Architectural Components

The Building Blocks of a dApp System - A Deep Dive

1. Frontend Layer (React / Next.js + ethers.js / MST SDK)

The frontend is the user's window into your smart contracts. It connects to the wallet (BridgeKey or MetaMask), reads contract state via view functions or event indexing, and sends transactions. A well-architected frontend separates UI components, data fetching hooks, and contract interaction services. On MST, you can use the same React components with the MST SDK (which is ethers-compatible) - just configure the RPC URL and chain ID.

Key architectural decisions for the frontend:

  • State management: Use React context or a lightweight state library (Zustand, Jotai) for wallet connection state. Avoid storing sensitive data in local storage.
  • Data fetching: Use ethers.Contract for on-chain calls. For complex queries, set up a subgraph and use Apollo Client for GraphQL. Polling is simpler but less scalable.
  • Transaction handling: Provide clear UI feedback during transaction lifecycle: pending (spinner), confirmed (checkmark), failed (error message). Use the MST SDK's gas estimation for predictable fees.

2. Smart Contract Layer (Solidity on MST)

This is where business logic lives. A typical modular dApp might have:

  • Core Logic Contract: The main entry point, delegating to other modules.
  • Token Contract(s): ERC-20 for utility/governance, ERC-721 or ERC-1155 for assets.
  • Storage Contract: Holds all mutable state, separated from logic (Eternal Storage pattern) to enable upgrades.
  • Access Control Contract: RBAC or Ownable, managing roles.
  • Upgrade Proxy: OpenZeppelin's Transparent Proxy or UUPS pattern.

On MST, because of low gas costs, you can afford more on-chain logic, but the separation of concerns remains critical for security and maintainability. Consider how these contracts will be deployed - should the factory contract be owned by a multisig? What addresses get initial admin rights?

3. Storage Layer (IPFS, Arweave, Subgraph)

Smart contracts store only what must be on-chain. Everything else goes to decentralized storage (IPFS for NFT metadata, Arweave for permanent archives) or is indexed from events into a queryable database via The Graph (subgraph). A subgraph listens to contract events, transforms them, and provides a GraphQL API for the frontend - far more efficient than reading storage directly.

When designing your event schema, think about what the frontend will query. Index the parameters needed for filtering. For a marketplace, index seller, buyer, tokenId. For a DAO, index proposalId and voter. Good event design makes your subgraph queries fast and simple.

4. Oracle / External Data Layer (WASMify, Chainlink)

Smart contracts can't natively fetch external data. Oracles bridge this gap. On MST, WASMify allows you to run verifiable off-chain computation and bring the result on-chain with a ZK proof. This is ideal for complex computations, IoT data, or custom APIs. Chainlink can also be used on MST for price feeds.

Architecturally, an oracle integration involves: a user or keeper calls an off-chain service, the service produces a signed result, and the contract verifies the signature and uses the data. With WASMify, the computation itself produces a ZK proof that the contract verifies in a single transaction. This reduces trust assumptions.

5. Wallet / Authentication Layer (SARAL, BridgeKey)

User onboarding is critical. MST's SARAL protocol allows users to create wallets with Google login or biometrics, without seed phrases. Architecturally, your dApp interacts with the wallet through the standard EIP-1193 provider interface. The wallet signs transactions; the dApp never sees the private key.

For your capstone, document the wallet flow: user clicks "Connect", BridgeKey prompts for login, wallet returns accounts. The dApp uses the connected signer to send transactions. If using SARAL, the user experience is even simpler - no extension needed, just a web login.

06   Smart Contract Design Patterns

Proven Blueprints for Solidity Contracts

1. Factory Pattern

A factory contract creates instances of another contract. Example: Uniswap's factory creates pair contracts for each token pair. This keeps the factory lightweight and the pairs isolated - a bug in one pair doesn't affect others. For your capstone, if users create their own resources (like custom tokens or staking pools), use a factory.

contract PoolFactory {
    mapping(address => address) public getPool;
    event PoolCreated(address indexed pool, address token, address creator);

    function createPool(address token) external returns (address) {
        require(getPool[token] == address(0), "Pool exists");
        Pool pool = new Pool(token, msg.sender);
        getPool[token] = address(pool);
        emit PoolCreated(address(pool), token, msg.sender);
        return address(pool);
    }
}

Gas consideration: deploying a child contract costs around 32,000 gas + the deployment code. On MST, this is about ₹0.003. On Ethereum, it could be ₹10,000+. The factory pattern is far more practical on low-fee chains.

2. Proxy Upgradeability (Transparent / UUPS)

Smart contracts are immutable - once deployed, code cannot change. Proxy patterns separate storage and logic. Users interact with a proxy contract that delegates calls to an implementation contract. To upgrade, you deploy a new implementation and point the proxy to it. OpenZeppelin provides audited implementations.

Two common proxy patterns:

  • Transparent Proxy: The admin (upgrader) cannot call the implementation; users cannot call admin functions. Simpler, but slightly higher gas cost.
  • UUPS (Universal Upgradeable Proxy Standard): The upgrade logic is in the implementation contract itself. Cheaper gas, but requires careful implementation to avoid locking the upgrade mechanism.

For your capstone, if you anticipate long-term maintenance, UUPS is the modern recommendation. Document who holds the upgrade admin role and how it might transition to a DAO.

3. Diamond Standard (EIP-2535)

The Diamond pattern takes modularity to the extreme: one proxy delegates to many "facet" contracts, each implementing a subset of functions. This removes the 24 KB contract size limit and allows granular upgrades. It's complex but powerful for large dApps.

How it works: a mapping bytes4 => address routes function selectors to facet contracts. When a call comes in, the proxy looks up the selector and delegatecalls to the matching facet. Adding a new feature means deploying a new facet and updating the mapping. MST's low gas costs make the extra delegation overhead negligible.

4. Eternal Storage

Separate the data storage from the logic. The logic contract writes to a separate storage contract. When upgrading logic, only the logic contract changes; data remains intact. This is simpler than a full proxy but requires disciplined separation.

Example: a UserStorage contract holds all mappings, and a UserLogic contract reads/writes through it. The logic contract address can be updated by an admin. This pattern is great for projects that want upgradeability without the complexity of proxy patterns.

5. Role-Based Access Control (RBAC)

Instead of a single owner, define roles like MINTER_ROLE, PAUSER_ROLE, UPGRADER_ROLE. Each role is a bytes32 hash. The contract stores a mapping from role to address list. Functions check onlyRole(MINTER_ROLE). OpenZeppelin's AccessControl provides a battle-tested implementation.

For your capstone, use AccessControl for any contract with multiple admin tiers. Example: a DAO contract might have a PROPOSER_ROLE and an EXECUTOR_ROLE. The DAO governance contract itself could be granted the EXECUTOR_ROLE to execute passed proposals automatically.

07   Interactive Design Walkthrough

Design Your Capstone Architecture - Step by Step

Follow the steps below to build your own architecture diagram. Click each step to expand. By the end, you'll have a complete blueprint.

📝 Step 1: Define user stories and on-chain state

Write down 3-5 core user stories for your dApp. Example for a "Decentralized Swiggy":

  • Customer places order with MSTC payment held in escrow.
  • Restaurant confirms order and marks it ready for pickup.
  • Delivery partner picks up and delivers.
  • Upon delivery confirmation, escrow releases payment to restaurant and delivery partner.

Identify what state must be on-chain: order details, escrow balances, delivery status, participant addresses. What can be off-chain: restaurant menus, images, reviews (these can be indexed from events or stored on IPFS).

Pro tip: For each piece of state, ask: "If this data were wrong, could someone steal funds?" If yes, it must be on-chain with appropriate validation.

🧩 Step 2: Decompose into smart contract modules

Based on the user stories, list the necessary contracts:

  • OrderContract: Manages the lifecycle of an order - placed, accepted, picked, delivered, disputed.
  • EscrowContract: Holds MSTC until delivery is confirmed, with rules for release and refund.
  • UserProfileContract: Stores reputation, delivery stats, and preferences.

Decide how they interact. Should OrderContract hold the escrow logic internally, or delegate to a separate EscrowContract? A separate contract is more modular and upgradable. Define the interface between them. For each contract, write a one-paragraph description of its responsibility.

🔗 Step 3: Define interfaces and data flow

For each contract, specify:

  • State variables: mappings, structs, arrays.
  • Public functions: their visibility, parameters, and effects.
  • Events: what events are emitted and when.
  • Modifiers: access control rules.

Draw a sequence diagram for a key flow, e.g., "Place Order".

User → OrderContract.placeOrder(restaurant, deliveryPartner, amount)
    → creates new Order struct, calls EscrowContract.deposit{value: amount}(orderId)
    → EscrowContract holds MSTC, emits EscrowDeposited(orderId, amount)
OrderContract emits OrderPlaced(orderId, user, restaurant, ...)

Define the interface IEscrow with deposit(uint orderId) payable and release(uint orderId). OrderContract calls IEscrow(escrowAddress).deposit{value: amount}(orderId). This decoupling lets you swap the escrow implementation later.

🔄 Step 4: Choose off-chain components

Determine what data the frontend needs in real time. For a delivery app, the user needs to see order status updates. You could poll the contract (gas-free via view functions) or set up a subgraph to index OrderStatusChanged events. A subgraph is more efficient for complex queries (e.g., "all orders by user X in the last week").

If you need external data (e.g., GPS proof of delivery), consider WASMify to verify location data and submit a ZK proof on-chain, triggering the escrow release. Document how the off-chain worker will be triggered (cron, event listener) and how the contract verifies the proof.

📐 Step 5: Draw the architecture diagram

Using any drawing tool (draw.io, Excalidraw, Figma, or even pen and paper), create a component diagram with:

  • Boxes for each contract, frontend, and off-chain service.
  • Arrows showing function calls and event flow.
  • Labels indicating the interface or data passed.

Keep it clean. Use different colors for on-chain vs off-chain. Include a legend. This diagram will be part of your assessment submission. Example:

[User Browser] → [React Frontend] → [BridgeKey Wallet]
[React Frontend] → ethers.js → [OrderContract] → [EscrowContract]
[OrderContract] emits OrderPlaced → [Subgraph] → [Frontend API]
08   Real-World Case Studies

How the Best dApps Are Architectured

🦄
Uniswap v2 - The Factory + Pair Model
Uniswap's architecture is a masterclass in modularity. The UniswapV2Factory creates UniswapV2Pair contracts for each trading pair. The UniswapV2Router provides a user-friendly interface to interact with pairs. This separation means each pair is isolated - a bug in one doesn't drain the entire exchange. The factory stores an array of all pairs, making them discoverable. This pattern is so effective that it's replicated across hundreds of DeFi protocols. For your capstone, if you need multiple instances of a contract (like staking pools), a factory is the way to go.
🏦
Aave - Modular Lending with Upgradeable Proxy
Aave uses a proxy-based architecture. The LendingPool is a proxy that delegates to upgradeable implementation contracts. The aToken contracts (representing deposits) are also upgradeable. This has allowed Aave to evolve from v1 to v3 while preserving user funds - the storage remained intact across upgrades. They also use an AddressesProvider contract as a central registry for all protocol addresses, so other contracts can look up the latest versions without hardcoding. This pattern is ideal for projects that expect long-term evolution.
🎮
Axie Infinity (Ronin) - ERC-1155 + Off-Chain Indexing
Axie Infinity's architecture uses ERC-1155 for all in-game assets (Axies, land, items) to allow batch transfers and multiple asset types in one contract. The game logic (breeding, battling) happens largely off-chain on the game server, with only final state changes (ownership transfers, rewards) settled on-chain. This hybrid architecture balances decentralization with performance. Events are indexed by a custom backend to power the marketplace and user inventories. MST's low fees make such hybrid models even more practical, allowing more logic on-chain if desired.
09   Access Control & Security Architecture

Locking the Doors - Designing Permissions That Don't Leak

Access control is the single most important security decision in your architecture. A well-designed permission system prevents unauthorized actions while remaining flexible for future governance. Here's a comprehensive guide:

Simple Ownership (Ownable)

Best for: simple projects, single-admin dApps. An owner address is set at deployment. A modifier onlyOwner protects admin functions. Ownership can be transferred to a new address. Pros: simple, well-understood. Cons: single point of failure; if the owner key is lost or compromised, the entire contract is compromised.

Role-Based Access Control (AccessControl)

Best for: projects with multiple admin tiers, DAO-controlled parameters. OpenZeppelin's AccessControl uses a mapping from bytes32 role => address[]. Roles are granted and revoked by an admin. Functions check onlyRole(role). Example roles: MINTER_ROLE, PAUSER_ROLE, UPGRADER_ROLE. This is the industry standard for production contracts.

Timelocks and Governance

For critical functions (like upgrading a proxy or draining fees), add a timelock between the proposal and execution. This gives users time to exit if they disagree with a change. A timelock contract holds the admin role; the DAO or multisig proposes transactions through it. OpenZeppelin's TimelockController provides this.

Multi-Signature (Multisig) Wallets

For the initial admin of your contracts, use a multisig wallet like Gnosis Safe. A 3-of-5 multisig requires multiple team members to sign any transaction. This prevents a single compromised key from draining the treasury. The multisig address becomes the owner or is granted the DEFAULT_ADMIN_ROLE.

Architecture rule: Never give a single EOA (externally owned account) unchecked power over a production contract. Use a multisig for admin, and timelocks for irreversible changes. Your architecture diagram should show the admin structure.
10   Gas-Aware Architecture Design

Structuring Contracts to Save Gas - Even on MST

MST's gas fees are a fraction of Ethereum's, but gas optimization still matters for scale. A contract called 1,000,000 times with a 50,000 gas difference will save 50 MSTC overall. Here's how to design for gas efficiency:

Minimize On-Chain Storage

Every storage write costs 5,000-20,000 gas. Design your contract to write as little as possible. Use events for historical data. Use calldata for read-only parameters. Pack small variables into structs. If you only need to track "last updated timestamp," don't store an entire history array.

Use External Calls Wisely

Calling another contract costs gas. If multiple contracts need the same data, consider caching it locally or using a central registry. However, avoid redundant calls. Batch multiple operations into a single transaction when possible (e.g., use ERC-1155 batch transfers instead of multiple ERC-721 transfers).

Proxy Gas Overhead

Proxy patterns add a delegatecall overhead of about 700-1,000 gas per function call. This is negligible on MST but worth noting if your dApp has millions of calls. Choose UUPS over Transparent Proxy for slightly lower gas (no admin check overhead).

Computation vs Storage Trade-off

On-chain computation is cheap (3 gas for ADD). Storage is expensive. It's often cheaper to recompute a value from other stored values than to store the derived value itself. For example, don't store a user's "total rewards" if you can calculate it from a base reward rate and the number of blocks staked. Let the frontend do the math.

Top Facts - Know These Cold for Your Architecture Submission

01
Separate concerns: logic, storage, access control, tokens
A good architecture has distinct contracts for distinct responsibilities. This enables independent testing, auditing, and upgrading.
02
Use interfaces (ISomething.sol) to define contract interactions

Key Terms - Sub-Module 9.2

Architecture Diagram
A visual representation of a system's components, their relationships, and data flows. Essential for communicating design to developers and auditors.
Modular Design
Breaking a system into independent, interchangeable modules with well-defined interfaces. Improves maintainability and testability.
Proxy Pattern
A design that separates contract logic from storage, allowing logic to be upgraded without changing the contract address or losing state.
Factory Pattern
A contract that creates instances of other contracts. Used when many independent instances of a contract are needed (e.g., trading pairs).
Diamond Standard
EIP-2535, allowing a single proxy contract to delegate calls to many facet contracts, removing the contract size limit.
Subgraph
An off-chain indexer that transforms blockchain events into a queryable GraphQL API, used to efficiently power frontends.
SARAL

Checks-Effects-Interactions
A security pattern: check conditions, update contract state, then interact with external contracts. Prevents reentrancy.
RBAC
Role-Based Access Control - an access control model where permissions are assigned to roles rather than individual addresses.
UUPS
Universal Upgradeable Proxy Standard - a proxy pattern where upgrade logic resides in the implementation contract, offering lower gas costs.
Timelock
A contract that delays execution of transactions, giving users time to react to governance changes. Enhances security for critical operations.
