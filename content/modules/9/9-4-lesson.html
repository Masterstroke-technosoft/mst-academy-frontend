02   Real-World Story

The Un-tested Withdraw Function That Drained ₹50 Lakh

📖 Story

A DeFi startup in Bengaluru had built a staking protocol with a withdrawal function. The team had tested the happy path: stake tokens, earn rewards, withdraw. All tests passed. They deployed to mainnet. Within three weeks, an attacker found a subtle bug: if you called withdraw with an amount just slightly above your stake, the require statement failed to trigger because of a rounding error in the Solidity math. The attacker drained over ₹50 lakh in tokens from the pool before anyone noticed.


During the post-mortem, the developers discovered that their test suite had zero edge-case tests. They had never tested questions like: "What happens if someone withdraws 0 tokens?" "What happens if someone withdraws exactly their full balance plus 1 wei?" "What happens if two withdrawals are made in the same block?" Every single one of these edge cases would have caught the rounding bug. But they only tested the one happy path that worked perfectly.


After fixing the bug, the team rewrote their entire test suite with a new rule: for every happy path test, there must be at least two edge-case tests. They added fuzzing, invariant checks, and a strict requirement that every pull request must achieve 100% branch coverage on all modified functions. They never lost money to a preventable bug again. Testing is not optional. It is not a formality. It is the single most important thing you do before deploying a smart contract to any blockchain.

In this sub-module, you will build a test suite that protects your capstone project exactly the way that team's second test suite protected theirs. You will test not just what should happen, but what should never happen. You will run a local blockchain on your own computer where you can break things safely without consequence. And you will learn to read the tools that tell you whether your tests are actually covering every line of code that matters.

03   Why Testing Matters for Smart Contracts

Immutability + Real Money = Testing Is Absolutely Non-Negotiable

In traditional software development, if a bug is discovered after deployment, you patch the code and push an update. Users get the fix within hours or days. The cost is inconvenience and perhaps a short outage.

In blockchain development, if a bug is discovered after deployment, the code cannot be changed. The contract is immutable - it lives forever on the blockchain exactly as it was deployed. The funds controlled by that contract are permanently vulnerable to exploitation. The only "fix" is to deploy an entirely new contract at a new address, migrate all users and all funds (if the funds haven't already been stolen), and somehow convince everyone to switch to the new address. This is extremely difficult, expensive, and often impossible.

This has happened repeatedly in blockchain history, with devastating consequences:

The DAO Hack (2016): A reentrancy bug in a fundraising contract allowed an attacker to drain $60 million worth of ETH. The Ethereum community was so divided on how to respond that it caused a permanent chain split, creating Ethereum Classic.

Parity Wallet Freeze (2017): A single line of code in a library contract was deleted by a user who accidentally became the owner. This froze over $150 million worth of ETH permanently - not stolen, but completely inaccessible forever.

Wormhole Bridge Exploit (2022): An attacker exploited a validation bug in the bridge's smart contract and stole $325 million worth of wrapped tokens. The bug existed because a specific edge case in the signature verification was never tested.

Every single one of these could have been prevented or detected by better testing. A comprehensive test suite that covered edge cases, reverts, and unexpected inputs would have caught these bugs on a local machine, long before any money was at risk.

Five reasons testing is non-negotiable for smart contracts:

1. Immutability: Once deployed, the code cannot change. Every bug is a permanent vulnerability. You get exactly one chance to get it right.

2. Financial stakes: Smart contracts control real money - sometimes millions or billions of dollars. A single logical error can drain everything instantly.

3. Composability: Your contract will likely interact with other contracts - tokens, exchanges, lending protocols. A bug in your contract can cascade through the entire ecosystem and break other people's applications too.

4. No take-backs: Blockchain transactions are irreversible by design. If a mainnet transaction fails due to a bug, you cannot call a bank and reverse it. The money is gone.

5. Trust and reputation: Users will only use contracts they trust. Open-source code accompanied by a comprehensive, passing test suite builds that trust. A contract without tests signals that the developer either doesn't understand the stakes or doesn't care - and users will stay away.

04   The Testing Stack

Hardhat + Mocha + Chai + ethers.js - Your Complete Testing Toolkit

Hardhat comes with a built-in testing environment that seamlessly integrates three powerful tools. Understanding each tool's role is essential for writing effective tests.

Mocha - The Test Framework: Mocha provides the overall structure for your tests. It gives you describe blocks (which group together related tests), it blocks (which define individual test cases), and lifecycle hooks like beforeEach (which runs before every single test). Mocha is responsible for running all your tests and reporting which ones pass and which ones fail, along with timing information.

Chai - The Assertion Library: Chai provides the language you use to check whether test results are correct. Instead of writing complex if/else logic, you write expressive assertions like expect(balance).to.equal(100) or expect(transaction).to.be.revertedWith("Insufficient funds"). Chai makes your tests readable - someone unfamiliar with your code should be able to read a test and understand what it verifies.

ethers.js - The Blockchain Interaction Library: ethers.js is how your tests talk to the blockchain. It lets you deploy contracts, call functions, send transactions, query state, and listen for events - all from within JavaScript test code. Hardhat automatically provides a pre-configured ethers instance that connects to the built-in Hardhat Network, so you don't need to set up any connections manually.

Hardhat Network - The Built-In Local Blockchain: This is the secret weapon. Hardhat Network is a full Ethereum node that runs entirely in memory on your computer - no external software needed, no testnet tokens needed, no waiting for blocks. It starts automatically when you run tests, mines blocks instantly on demand, and provides detailed error messages and stack traces that real nodes cannot give you. Each test file gets its own completely fresh network state, so tests never interfere with each other. It also supports console.log directly from Solidity, which is invaluable for debugging.

05   Setting Up Your Test Environment

Project Structure and Required Dependencies

If you initialized your Hardhat project using the Hardhat Toolbox, all testing dependencies are already installed. You can verify this by checking your hardhat.config.js file and your package.json.

// hardhat.config.js — make sure this line is present:
require("@nomicfoundation/hardhat-toolbox");

module.exports = {
    solidity: "0.8.20",
    networks: {
        hardhat: { }  // The built‑in local network is always available
    }
};

Your test files live in the test/ directory at the root of your project. Hardhat automatically discovers and runs every .js or .ts file in this folder when you execute npx hardhat test. The convention is to name test files after the contract they test: for example, TodoList.test.js tests TodoList.sol, and Token.test.js tests Token.sol. You can also create subdirectories inside test/ to organize larger test suites.

06   Anatomy of a Test File

describe, it, and beforeEach - The Three Building Blocks of Every Test

Every test file you write will follow the same fundamental structure. Understanding these three building blocks is essential before you start writing your own tests.

const { expect } = require("chai");
const { ethers } = require("hardhat");

describe("MyContract", function () {
    let myContract;
    let owner;
    let addr1;

    // This runs before EVERY single test case below
    beforeEach(async function () {
        const MyContract = await ethers.getContractFactory("MyContract");
        [owner, addr1] = await ethers.getSigners();
        myContract = await MyContract.deploy();
        await myContract.waitForDeployment();
    });

    describe("Deployment", function () {
        it("Should set the right owner", async function () {
            expect(await myContract.owner()).to.equal(owner.address);
        });
    });

    describe("MyFunction", function () {
        it("Should do something specific", async function () {
            await myContract.connect(addr1).myFunction(42);
            expect(await myContract.someValue()).to.equal(42);
        });

        it("Should revert when called with invalid input", async function () {
            await expect(
                myContract.myFunction(0)
            ).to.be.revertedWith("Invalid input");
        });
    });
});

describe blocks are used to group related tests together. They can be nested - you can have a top-level describe for the entire contract, then inner describe blocks for deployment, for each function, and for specific scenarios. This creates a readable hierarchy that makes test output easy to navigate.

beforeEach is the most important hook in smart contract testing. It runs before every single it block, deploying a brand-new instance of your contract. This guarantees test isolation: Test A can modify state as much as it wants, and Test B will start with a completely fresh, untouched contract. Without beforeEach, state changes from one test would leak into the next, causing confusing failures and false positives.

it blocks define individual test cases. The string description should clearly state what the test verifies. Inside the function, you deploy, interact, and assert. If any assertion fails or the function throws an unhandled error, the test fails and Hardhat reports exactly what went wrong.

07   Deployment Tests

Testing That Your Contract Deploys Correctly

Every test suite should start with deployment tests. These verify that the constructor executed correctly, that initial state variables are set to their expected values, and that the contract exists at a valid address. A contract that deploys incorrectly can fail silently - for example, if the owner is set to address(0) because of a constructor bug, nobody will ever be able to call owner-restricted functions, and you might not discover this until after deployment.

describe("Deployment", function () {
    it("Should deploy successfully and have a valid address", async function () {
        const address = await myContract.getAddress();
        expect(address).to.be.properAddress;
    });

    it("Should set the correct initial owner", async function () {
        expect(await myContract.owner()).to.equal(owner.address);
    });

    it("Should start with zero items in the list", async function () {
        expect(await myContract.getItemCount()).to.equal(0);
    });

    it("Should record the deployment timestamp", async function () {
        const deployTime = await myContract.creationTime();
        const latestBlock = await ethers.provider.getBlock("latest");
        expect(deployTime).to.be.closeTo(latestBlock.timestamp, 2);
    });
});

These four tests cover the essential deployment checks. The first verifies the contract actually exists on the blockchain (it has a valid Ethereum address). The second verifies the constructor correctly assigned the owner. The third checks that internal counters and collections start empty. The fourth verifies that any timestamp or block-dependent initialization works correctly. If any of these fail, stop and fix the constructor before testing individual functions.

08   Function Tests - Happy Path

Testing the Normal, Expected Behavior of Every Function

Happy path tests verify that functions work correctly when given valid, expected inputs by legitimate users. These tests prove that your contract does what it claims to do under normal operating conditions. For every state-changing function, you should test that it correctly modifies state, returns the right values, emits the expected events, and handles multiple consecutive calls properly.

describe("createItem", function () {
    it("Should create an item and increment the count", async function () {
        await myContract.createItem("Buy groceries");
        expect(await myContract.getItemCount()).to.equal(1);
    });

    it("Should store the correct content for the created item", async function () {
        await myContract.createItem("Learn Solidity");
        const item = await myContract.items(0);
        expect(item.content).to.equal("Learn Solidity");
        expect(item.completed).to.equal(false);
    });

    it("Should emit the ItemCreated event with correct arguments", async function () {
        await expect(myContract.createItem("Second item"))
            .to.emit(myContract, "ItemCreated")
            .withArgs(1, "Second item");
    });

    it("Should handle multiple items being created in sequence", async function () {
        await myContract.createItem("First");
        await myContract.createItem("Second");
        await myContract.createItem("Third");
        expect(await myContract.getItemCount()).to.equal(3);
    });
});

The happy path testing pattern: For every state-changing function in your contract, test at least these four things: (a) the state variable changes correctly, (b) the function returns the expected value if it has a return, (c) the correct event is emitted with the correct arguments, and (d) calling the function multiple times produces the expected cumulative result. This pattern ensures comprehensive coverage of normal usage.

09   Edge Cases and Revert Tests

Testing What Should NOT Happen - The Most Important Tests

This section is where most developers stop testing - and where most catastrophic bugs originate. Edge case tests verify that your contract correctly rejects invalid, malicious, or unexpected inputs rather than silently corrupting its own state or losing funds. These tests are often more important than happy path tests because they catch the scenarios that attackers actively search for.

describe("createItem — Edge Cases and Invalid Inputs", function () {
    it("Should revert if the content string is empty", async function () {
        await expect(
            myContract.createItem("")
        ).to.be.revertedWith("Content cannot be empty");
    });

    it("Should revert if content exceeds the maximum allowed length", async function () {
        const longString = "a".repeat(281);
        await expect(
            myContract.createItem(longString)
        ).to.be.revertedWith("Content too long");
    });

    it("Should revert when trying to access a non‑existent item ID", async function () {
        await expect(
            myContract.toggleItem(999)
        ).to.be.revertedWith("Item does not exist");
    });

    it("Should revert when a non‑owner tries an owner‑only function", async function () {
        await expect(
            myContract.connect(addr1).adminFunction()
        ).to.be.revertedWith("Not authorized");
    });

    it("Should revert on zero‑value transfers", async function () {
        await expect(
            myContract.transfer(addr1.address, 0)
        ).to.be.revertedWith("Amount must be greater than zero");
    });

    it("Should revert when transferring more than the available balance", async function () {
        await expect(
            myContract.transfer(addr1.address, ethers.parseEther("1000"))
        ).to.be.revertedWith("Insufficient balance");
    });
});

The complete edge-case checklist for every function you write:

• Empty inputs - empty strings, zero values, and address(0) should all be rejected with clear error messages.

• Boundary values - test the maximum allowed value, the minimum allowed value, and values just outside those boundaries (max + 1, min - 1).

• Authorization checks - test that every restricted function reverts when called by an unauthorized account. Test with a completely unrelated address, not just the owner.

• State-dependent behavior - test what happens when functions are called while the contract is paused, before initialization, or after finalization.

• Insufficient resources - test transfers and withdrawals when the balance is too low, when the allowance is insufficient, or when a time lock hasn't expired.

• Re-entrancy protection - if your contract has a reentrancy guard, explicitly test that nested calls are rejected.

10   Event Assertions

Verifying That Events Fire Correctly

Events are the primary way that frontend applications and off-chain services detect changes in your contract's state. If your contract fails to emit the correct event with the correct parameters, your frontend will display stale or incorrect data to users. Testing events is not optional - it is a critical part of full-stack dApp correctness.

it("Should emit ItemCreated with the correct ID and content", async function () {
    await expect(myContract.createItem("Test item"))
        .to.emit(myContract, "ItemCreated")
        .withArgs(0, "Test item");
});

it("Should emit ItemToggled when toggling completion status", async function () {
    await myContract.createItem("Toggle me");
    await expect(myContract.toggleItem(0))
        .to.emit(myContract, "ItemToggled")
        .withArgs(0, true);
});

it("Should NOT emit any event when the transaction reverts", async function () {
    await expect(
        myContract.createItem("")
    ).to.be.reverted;
});

Important details about event testing: The event arguments must match exactly - both the values and the order. If your event has indexed parameters, those can be used for filtering. The Chai matcher checks the event name and all arguments. If you expect an event not to be emitted (because the transaction should revert), simply check that the transaction reverts - a reverted transaction automatically means no events were permanently recorded.

11   Testing with Multiple Accounts

Simulating Real Users with Different Wallets

Hardhat provides ethers.getSigners(), which returns an array of pre-funded Ethereum accounts. Each account has 10,000 test ETH and can be used to simulate different real-world users interacting with your contract simultaneously. This is essential for testing multi-user scenarios like deposits, withdrawals, transfers, and access control.

it("Should allow different users to maintain independent balances", async function () {
    const [owner, user1, user2] = await ethers.getSigners();

    await myContract.connect(user1).deposit({ value: ethers.parseEther("1") });
    await myContract.connect(user2).deposit({ value: ethers.parseEther("2") });

    expect(await myContract.balances(user1.address))
        .to.equal(ethers.parseEther("1"));
    expect(await myContract.balances(user2.address))
        .to.equal(ethers.parseEther("2"));
});

it("Should prevent one user from spending another user's funds", async function () {
    const [owner, user1, user2] = await ethers.getSigners();
    await myContract.connect(user1).deposit({ value: ethers.parseEther("1") });

    await expect(
        myContract.connect(user2).withdraw(ethers.parseEther("0.5"))
    ).to.be.revertedWith("Insufficient balance");
});

How to use different accounts: Use .connect(signer) to execute a function call from a specific account's perspective. Without .connect(), the call originates from the first signer (usually the deployer). This simple pattern lets you verify that your contract correctly enforces ownership, tracks per-user balances, and prevents unauthorized access to other users' funds.

12   Using Fixtures for Complex Setup

Reusing Setup Code Across Multiple Test Files

As your test suite grows, you will find yourself repeating the same deployment and initialization code across multiple test files. Hardhat supports fixtures - reusable setup functions that deploy contracts, configure initial state, and return everything in a consistent, ready-to-use format.

// test/fixtures.js — Create this file to hold your shared setup logic

const { ethers } = require("hardhat");

async function deployTodoList() {
    const TodoList = await ethers.getContractFactory("TodoList");
    const [owner, user1, user2] = await ethers.getSigners();
    const todoList = await TodoList.deploy();
    await todoList.waitForDeployment();
    return { todoList, owner, user1, user2 };
}

async function deployTodoListWithTasks() {
    const { todoList, owner, user1, user2 } = await deployTodoList();
    await todoList.createTask("Existing task 1");
    await todoList.createTask("Existing task 2");
    return { todoList, owner, user1, user2 };
}

module.exports = { deployTodoList, deployTodoListWithTasks };

Then import and use these fixtures in any test file:

const { deployTodoListWithTasks } = require("./fixtures");

beforeEach(async function () {
    const fixtures = await deployTodoListWithTasks();
    this.todoList = fixtures.todoList;
    this.owner = fixtures.owner;
    this.user1 = fixtures.user1;
});

This approach keeps your tests DRY (Don't Repeat Yourself) and ensures that every test file uses exactly the same deployment logic. If you change your constructor parameters or initialization sequence, you only need to update the fixture - not every individual test file.

13   Running Tests

The npx hardhat test Command and Its Options

To execute all tests in your project: npx hardhat test

Hardhat will automatically compile your contracts if they have changed, start the built-in Hardhat Network, execute every test file in the test/ directory, and display a clean summary showing which tests passed and which failed.

Useful command-line flags for different situations:

npx hardhat test --grep "Deployment" - Run only the tests whose description contains the word "Deployment". Perfect for focusing on a specific section when debugging.

npx hardhat test --parallel - Run test files simultaneously across multiple threads. Each file gets its own independent Hardhat Network instance, so there are no conflicts. This dramatically speeds up large test suites.

npx hardhat test --verbose - Show the full name of every test, including nested describe blocks, plus which hooks ran. Useful for understanding exactly what executed.

When a test fails, Hardhat displays: the exact line in your test file where the assertion failed, the expected value versus the actual value, and if the transaction reverted, the complete Solidity stack trace showing exactly which line of your contract caused the revert.

15   Deploying to the Local Node

Connecting Your Deployment Script to the Local Network

To deploy your contracts to the persistent local node, first add the localhost network to your hardhat.config.js:

networks: {
    hardhat: { },
    localhost: {
        url: "http://127.0.0.1:8545",
    }
}

Then deploy with: npx hardhat run scripts/deploy.js --network localhost

Your contract will be deployed to the local node. You will see the contract address printed in the terminal. You can now interact with this contract through the Hardhat console, your frontend, or any tool that connects to the RPC endpoint at 127.0.0.1:8545.

16   Code Coverage

Measuring Exactly How Much of Your Solidity Code Is Tested

solidity-coverage is a Hardhat plugin that generates a detailed coverage report showing which lines of your Solidity source code are executed during your tests - and, more importantly, which lines are never touched. Untested code is where bugs hide.

Installation: npm install --save-dev solidity-coverage

Configuration: Add require("solidity-coverage"); at the top of your hardhat.config.js.

Execution: npx hardhat coverage

This command runs your entire test suite with instrumentation enabled and produces a coverage/ folder containing an HTML report. Open coverage/index.html in your browser to see a line-by-line view of every contract, with green highlighting for tested lines and red highlighting for untested lines.

Aim for at least 90% branch coverage on your capstone contract. Branch coverage is the strictest metric - it checks that every if/else, every require condition, and every ternary operator has been tested with both true and false outcomes. It is significantly harder to achieve than line coverage, but it catches far more bugs. A contract with 100% line coverage but only 60% branch coverage still has many untested logical paths where vulnerabilities can hide.

17   Debugging Failing Tests

Console.log in Solidity, Stack Traces, and Common Fixes

When a test fails, your first debugging tool should be console.log inside Solidity. Hardhat supports this natively. Import import "hardhat/console.sol"; at the top of your contract, then add console.log(variableName); wherever you need to inspect values. The output appears directly in your test terminal.

Reading Hardhat stack traces: When a transaction reverts, Hardhat shows the complete call trace, starting from the test function, through any contract calls, down to the exact Solidity line that caused the revert. Find the first frame in the trace that is inside your contract - that is where the error originated. The frames above it are just the call chain leading there.

Common test failures and their solutions:

SymptomLikely CauseHow to Fix It
Transaction reverted without a reason stringA require condition failed but has no error messageAdd a descriptive reason string to every require: require(condition, "descriptive reason")
Out of gas errorThe gas limit is too low for the complexity of the operationIncrease the gas limit in the test transaction: { gasLimit: 300000 }
Wrong address or signer mismatchUsed owner when you meant addr1, or forgot .connect()Double-check all .connect() calls and which address you expect to be msg.sender
Unexpected event argument valuesEvent arguments are in the wrong order or have incorrect valuesReview the event definition in your Solidity code and verify the argument order matches
Test passes alone but fails when run with othersState is leaking between tests because beforeEach isn't resetting everythingEnsure beforeEach deploys a completely fresh contract and resets all variables

19   Assessment Preparation

Top Facts - Know These Cold Before Submitting

01
beforeEach deploys a completely fresh contract for every single test case. This is the golden rule of smart contract testing. Never share state between tests. Every test starts with a pristine, untouched contract deployed moments ago.
02
For every happy path test, write at least two edge-case tests. Test empty inputs, boundary values, unauthorized callers, and unexpected contract states. Most catastrophic bugs are found in the edge cases, not the happy paths.
03
Use .to.be.revertedWith("exact reason") to verify that transactions fail correctly. A reverted transaction that produces a different error message will fail the test, alerting you that your validation logic needs attention.
04
solidity-coverage shows exactly which lines of your Solidity code are tested and which are not. Aim for at least 90% branch coverage. Red lines in the report represent untested code paths that could contain hidden vulnerabilities.
05
Hardhat's built-in network is significantly faster and better for automated tests than a persistent local node. Use the persistent local node (npx hardhat node) only for manual testing, frontend integration testing, or demonstrations.
20   Glossary

Key Terms - Sub-Module 9.4

Mocha
A JavaScript test framework that provides the describe, it, and lifecycle hooks (beforeEach, afterEach) for organizing and executing tests in a structured, readable way.
Chai
An assertion library that provides expressive matchers like .to.equal(), .to.be.revertedWith(), and .to.emit() for validating test outcomes in readable English.
beforeEach
A Mocha lifecycle hook that executes before every single test case. In smart contract testing, it deploys a fresh contract instance to guarantee complete isolation between tests.
Fixture
A reusable setup function that deploys contracts and returns initialized objects. Shared across multiple test files to avoid duplicating deployment logic.
Hardhat Network
A built-in, in-memory Ethereum network that Hardhat automatically starts for tests. Mines blocks instantly, provides detailed stack traces, and resets between test files.
solidity-coverage
A Hardhat plugin that instruments Solidity code and generates HTML reports showing exactly which lines, branches, and functions are covered by tests.
Branch Coverage
The percentage of conditional branches - if/else statements, require conditions, and ternary operators - that have been tested with both true and false outcomes. The hardest and most important coverage metric to achieve.
Local Node
A persistent Hardhat JSON-RPC server started with npx hardhat node that provides a full local blockchain with pre-funded accounts for manual testing and frontend development.
