02   Real-World Introduction Story

The Escrow That Escaped - A Coding Horror Story

📖 Story

In late 2023, a freelance developer named Ankit built a simple escrow dApp for a client. The design was straightforward: a buyer deposits MSTC, the seller delivers work, the buyer releases payment. Ankit coded the Escrow.sol contract in a single afternoon. He tested it once, saw it worked, and handed it over.


Two weeks later, the client called in a panic. A buyer had found a loophole: after the seller delivered, the buyer could call releaseFunds() multiple times, each time draining the escrow balance. Ankit had forgotten to update the escrow state before sending the funds - the classic reentrancy vulnerability. Worse, the contract had no nonReentrant modifier, no event emission, and no test suite. The buyer drained 5,000 MSTC before anyone noticed.


Ankit rewrote the contract from scratch, this time following the Checks-Effects-Interactions pattern, adding events, a nonReentrant guard, and a full test suite with Hardhat. The new contract was secure, auditable, and unbreakable. The lesson: coding a smart contract isn't just about making it work - it's about making it impossible to break. This sub-module gives you the discipline to do it right the first time.

03   Background & History

The Evolution of Solidity Coding Practices

In Ethereum's early days, Solidity was young and dangerous. Developers deployed contracts without testing frameworks, without access control patterns, and often without understanding reentrancy. The infamous DAO hack of 2016 - where $60 million worth of ETH was drained from a smart contract - was caused by a simple reentrancy bug in a splitDAO function. The attacker repeatedly called back into the contract before it could update its balance, draining funds over and over.

That single event changed the industry. Security became the top priority. New patterns emerged: Checks-Effects-Interactions (C-E-I), OpenZeppelin libraries for reusable, audited code, and automated testing frameworks like Hardhat and Foundry. Today, no professional developer would deploy a contract without tests and security audits.

MST, being fully EVM-compatible, inherits these same best practices. The contracts you write for MST follow the same Solidity standards, but with the added benefit of extremely low gas fees - meaning you can afford more on-chain logic, richer events, and comprehensive testing without worrying about cost.

Modern Solidity development rule: Never deploy a contract without tests. Never write a function that transfers funds without C-E-I. Never skip OpenZeppelin for standard patterns. These aren't suggestions - they're the difference between a successful project and a headline about another hack.
04   Beginner-Friendly Explanation

Coding a Smart Contract Is Like Building a Vending Machine

🥤 The Analogy

Think of your smart contract as a vending machine. The machine has state - how many snacks are left, how much money has been deposited. It has functions - select snack, insert coin, dispense change. It has security - a lock on the cash box, a coin validator that rejects fakes. And it prints a receipt (event) for every transaction. If you build the machine wrong - say, the cash box doesn't lock, or the change dispenser can be triggered without paying - people will exploit it. Coding a smart contract is the same: you define the state, the functions, the guards, and the logs, and you test everything before letting anyone use it.

Every core smart contract has four essential parts. Click on any to expand:

S
STATE
What is the contract's persistent memory?
State variables are the permanent data stored on the blockchain. They include mappings (like balances), structs (like Order), and simple values (like owner). Every write to state costs gas, so you design them carefully - storing only what's essential and packing small types together.
F
FUNCTIONS
What actions can users take?
Functions are the executable logic. They read and modify state, enforce rules, and emit events. Each function must have a clear visibility (public, external) and may be marked view, pure, or payable. A well-designed contract has small, focused functions that do one thing well.
G
GUARDS
How are sensitive actions protected?
Modifiers and require statements act as guards. onlyOwner ensures only the admin can withdraw fees. nonReentrant prevents recursive calls. require checks validate inputs and state conditions. Guards are your first line of defense - if they fail, the entire transaction reverts.
E
EVENTS
How does the contract communicate with the outside?
Events are logged into the transaction receipt. They are cheap, permanent, and indexable. Every state change should emit an event so that frontends, subgraphs, and auditors can track the contract's activity without reading storage directly.
05   Technical Deep Dive

Write a Complete Escrow Contract - Step by Step with Visual Guides

Let's build a real contract together - a secure escrow for a freelance platform. A client deposits MSTC, a freelancer delivers work, and when both confirm, the funds are released. If there's a dispute, an admin can refund or release.

The Data Structure: What State Do We Need?

enum EscrowStatus { AWAITING_PAYMENT, FUNDED, DELIVERED, COMPLETED, DISPUTED }

struct Escrow {
    address client;
    address freelancer;
    uint256 amount;
    EscrowStatus status;
    bool clientConfirmed;
    bool freelancerConfirmed;
}

mapping(uint256 => Escrow) public escrows;
uint256 public nextEscrowId;
address public admin;

We use an enum for status clarity. The struct groups all the data for one escrow. The mapping gives O(1) access by ID. We store an admin address for dispute resolution.

Access Control Modifiers

modifier onlyAdmin() {
    require(msg.sender == admin, "Not admin");
    _;
}
modifier onlyClient(uint256 escrowId) {
    require(msg.sender == escrows[escrowId].client, "Not client");
    _;
}
modifier onlyFreelancer(uint256 escrowId) {
    require(msg.sender == escrows[escrowId].freelancer, "Not freelancer");
    _;
}

Funding an Escrow (Function Flow)

function createEscrow(address freelancer) external payable returns (uint256) {
    require(msg.value > 0, "No MSTC sent");
    uint256 id = nextEscrowId++;
    escrows[id] = Escrow({
        client: msg.sender,
        freelancer: freelancer,
        amount: msg.value,
        status: EscrowStatus.FUNDED,
        clientConfirmed: false,
        freelancerConfirmed: false
    });
    emit EscrowCreated(id, msg.sender, freelancer, msg.value);
    return id;
}

Confirming Delivery and Releasing Funds (with Security Flow)

The delivery confirmation and fund release involve the most critical security pattern: Checks-Effects-Interactions.

function confirmDelivery(uint256 escrowId) external onlyFreelancer(escrowId) {
    Escrow storage escrow = escrows[escrowId];
    require(escrow.status == EscrowStatus.FUNDED, "Not funded");
    escrow.freelancerConfirmed = true;
    escrow.status = EscrowStatus.DELIVERED;
    emit DeliveryConfirmed(escrowId);
}

function releaseFunds(uint256 escrowId) external onlyClient(escrowId) nonReentrant {
    Escrow storage escrow = escrows[escrowId];
    require(escrow.status == EscrowStatus.DELIVERED, "Not delivered");
    require(escrow.clientConfirmed == false, "Already confirmed");
    // Effects first
    escrow.clientConfirmed = true;
    escrow.status = EscrowStatus.COMPLETED;
    uint256 amount = escrow.amount;
    // Interactions last (Checks‑Effects‑Interactions)
    (bool success, ) = payable(escrow.freelancer).call{value: amount}("");
    require(success, "Transfer failed");
    emit FundsReleased(escrowId, escrow.freelancer, amount);
}

The key is that the contract's state is updated before the external call. If the freelancer's fallback attempts to re-enter, it will see that clientConfirmed is already true and revert.

Dispute Resolution

function raiseDispute(uint256 escrowId) external onlyClient(escrowId) {
    Escrow storage escrow = escrows[escrowId];
    require(escrow.status == EscrowStatus.DELIVERED, "Not delivered");
    escrow.status = EscrowStatus.DISPUTED;
    emit DisputeRaised(escrowId);
}

function resolveDispute(uint256 escrowId, bool refundClient) external onlyAdmin nonReentrant {
    Escrow storage escrow = escrows[escrowId];
    require(escrow.status == EscrowStatus.DISPUTED, "Not disputed");
    escrow.status = EscrowStatus.COMPLETED;
    uint256 amount = escrow.amount;
    address receiver = refundClient ? escrow.client : escrow.freelancer;
    (bool success, ) = payable(receiver).call{value: amount}("");
    require(success, "Transfer failed");
    emit DisputeResolved(escrowId, receiver, amount);
}

The escrow contract follows a strict state machine. Transitions are controlled by modifiers and status checks, preventing unauthorized or out-of-order actions.
🗝️ The Golden Rule of Smart Contract Coding

Every function that transfers native coins or tokens must follow Checks-Effects-Interactions. Check conditions first, update contract state second, interact with external contracts last. Combined with OpenZeppelin's nonReentrant modifier, this is the industry-standard defense against the most common attack vector in blockchain history.

contract Staking {
    IERC20 public stakingToken;
    uint256 public rewardRate = 100; // reward per block per token staked
    mapping(address => uint256) public stakedBalance;
    mapping(address => uint256) public lastUpdateBlock;
    mapping(address => uint256) public rewards;

    constructor(address _stakingToken) {
        stakingToken = IERC20(_stakingToken);
    }
}
📥 Step 2: Stake and unstake functions 
function stake(uint256 amount) external {
    updateReward(msg.sender);
    stakingToken.transferFrom(msg.sender, address(this), amount);
    stakedBalance[msg.sender] += amount;
    emit Staked(msg.sender, amount);
}

function unstake(uint256 amount) external {
    require(stakedBalance[msg.sender] >= amount, "Insufficient stake");
    updateReward(msg.sender);
    stakedBalance[msg.sender] -= amount;
    stakingToken.transfer(msg.sender, amount);
    emit Unstaked(msg.sender, amount);
}
🏆 Step 3: Reward calculation and claim 
function updateReward(address user) internal {
    if (lastUpdateBlock[user] > 0) {
        uint256 blocks = block.number - lastUpdateBlock[user];
        rewards[user] += stakedBalance[user] * rewardRate * blocks;
    }
    lastUpdateBlock[user] = block.number;
}

function claimReward() external {
    updateReward(msg.sender);
    uint256 reward = rewards[msg.sender];
    require(reward > 0, "No rewards");
    rewards[msg.sender] = 0;
    stakingToken.transfer(msg.sender, reward);
    emit RewardClaimed(msg.sender, reward);
}
📊 Step 4: Event declarations and modifiers 
event Staked(address indexed user, uint256 amount);

event RewardClaimed(address indexed user, uint256 amount);

modifier updateRewardModifier(address user) {
    updateReward(user);
    _;

09   Gas Optimization in Core Contracts

Write Efficient Code - Even Cheap Gas Deserves Respect

1
Pack storage variables
Place small uint types, bool, and address together in your contract declarations. The compiler will pack them into a single 32-byte slot, reducing SSTORE and SLOAD gas.
2
Use calldata for read-only parameters
In external functions, mark array and string parameters as calldata. This avoids copying them into memory, saving thousands of gas for large inputs.
3
Cache storage reads
If you read the same storage variable multiple times in a function, cache it in a local memory variable first. Each SLOAD costs 100-2,100 gas; a memory copy is essentially free.
4
Use unchecked for arithmetic where safe
Solidity 0.8+ adds overflow checks. If you're certain a loop index won't overflow, wrap it in unchecked { i++ } to save a small amount of gas. Use cautiously.

12   Glossary

Key Terms - Sub-Module 9.3

Core Contract
The main smart contract that contains the essential business logic of a dApp. Typically handles state transitions, fund transfers, and access control.
Checks-Effects-Interactions
A security pattern: check conditions, update contract state, then interact with external contracts. Prevents reentrancy.
OpenZeppelin
A library of audited, reusable Solidity contracts for access control, tokens, proxies, and security. Industry standard for production contracts.
Hardhat Test Suite
A JavaScript testing environment using Mocha and Chai that runs against a local Hardhat Network. Enables automated, repeatable tests for smart contracts.
NatSpec
Solidity's natural language specification format. Uses /// @notice, @dev, @param tags to document code for users and developers.
Event
A log record emitted by a contract, stored in transaction receipts. Used for off-chain communication and historical audit trails.
