03   Background & History

The Evolution of Web3 Frontends - From Web3.js to Modern React Hooks

The journey of Web3 frontend development is a story of progressive simplification. In Ethereum's earliest days (2015 - 2017), building a dApp frontend meant wrestling with web3.js, a massive monolithic JavaScript library that injected a global web3 object into the browser. Frontends were often vanilla HTML files with script tags - no React, no state management, no component architecture. Wallet connection meant "install MetaMask or go home." There were no standards for how wallets should communicate with dApps. Every wallet had its own API. Developers spent more time fighting the tooling than building features.

The release of ethers.js v4 in 2018, and especially v5 in 2019, was a watershed moment. ethers.js was lightweight, meticulously documented, and designed with modern JavaScript in mind. It introduced clean abstractions like Provider, Signer, and Contract that made interacting with the blockchain feel natural. Developers started creating custom React hooks - useContract, useProvider, useSigner - that encapsulated the complexity of wallet connection and contract interaction into reusable pieces of logic. The React + ethers.js combination became the de facto standard for Web3 frontends.

The EIP-1193 standard (Ethereum Provider JavaScript API), finalized in 2020, was the glue that unified the ecosystem. It defined a standard interface that every wallet must expose via window.ethereum. Whether the user has MetaMask, BridgeKey, Coinbase Wallet, or any other EIP-1193-compliant wallet, the dApp interacts with it through the same API. This was revolutionary: developers could write one wallet integration and support dozens of wallets automatically.

Today, building a Web3 frontend is a mature discipline. The typical stack is React + Vite + ethers.js + a wallet provider. Vite provides lightning-fast development builds and hot module replacement. React provides a component-based architecture with hooks for state management. ethers.js handles all blockchain communication. And the wallet (BridgeKey, MetaMask) handles key management and transaction signing - the dApp never sees the private key.

MST's position in this ecosystem is uniquely advantageous. Because MST is fully EVM-compatible, every React-ethers tutorial, every Stack Overflow answer, and every open-source dApp template works on MST with minimal changes. You only need to point the provider to an MST RPC URL and set the correct chain ID. BridgeKey implements EIP-1193 natively, so any code that works with MetaMask works with BridgeKey. And MST's sub-paisa gas fees and ~3-second block times mean that the frontend experience is dramatically better than on Ethereum - users don't wait minutes for confirmations, and they don't pay hundreds of rupees per interaction.

Key insight: MST is fully EVM-compatible, so every React-ethers tutorial you find online works on MST. You just need to point the provider to an MST RPC URL and set the correct chain ID. The entire Web3 frontend ecosystem is at your disposal.
04   Beginner-Friendly Explanation

The Frontend Is the Bridge Between Humans and Code

🌉 The Analogy

Think of your smart contract as a bank vault. It holds value and enforces rules with mathematical precision. But nobody can use the vault directly - they need an ATM (the frontend). The ATM has a screen (the UI), buttons (wallet connection and transaction triggers), and a card reader (the wallet). It shows you your balance by reading from the vault's records (view functions), lets you withdraw cash by sending signed instructions (transactions), and prints receipts (transaction confirmations). Building the frontend is like designing that ATM: it must be clear enough for a first-time user, safe enough to protect their funds, and robust enough to handle every edge case - network errors, rejected transactions, insufficient funds - without crashing or confusing the user.

Every dApp frontend has three core layers that work together in a precise sequence. Let's understand each deeply:

C
CONNECT
How does the user prove identity without a password?
The frontend requests wallet connection via window.ethereum.request({ method: 'eth_requestAccounts' }). This is the EIP-1193 standard call that every modern wallet implements. The wallet (BridgeKey, MetaMask) prompts the user to approve the connection. After approval, the frontend receives the user's Ethereum address - their public identity on the blockchain - and a Signer object that can cryptographically sign transactions. Critically, the frontend never sees the private key. The private key remains inside the wallet's secure enclave. All the frontend can do is request that the wallet sign a transaction and return the signature. This is the fundamental security model of Web3: not your keys, not your coins - and not your dApp's keys either.
R
READ
How does the frontend display on-chain data instantly and for free?
Using ethers.js, the frontend creates a Contract instance connected to a read-only Provider (typically a JSON-RPC connection to an MST node). It then calls view functions on the contract - functions marked with the view or pure modifier in Solidity. These functions are executed locally by the RPC node against the current blockchain state. They do not create a transaction, do not consume gas, and return results immediately. Examples include balanceOf(address), totalSupply(), getEscrow(id). The frontend can call these as often as it wants without any cost to the user. This is how wallets display your token balances and how dApps show you your staked amount or voting power.
W
WRITE
How does the frontend change the blockchain state?
To modify state, the frontend creates a Contract instance connected with a Signer (the wallet). It then calls a state-changing function - any function not marked view or pure. This creates a transaction that is sent to the wallet for signing, then broadcast to the MST network. The frontend tracks the transaction through its lifecycle: submitted (waiting for wallet approval), pending (broadcast to the network, waiting for inclusion in a block), confirmed (included in a block), and finalized (enough blocks built on top). On MST, with ~3-second block times, this entire cycle typically completes in under 10 seconds. The frontend must provide visual feedback at each stage - a spinner during pending, a success message with a link to the transaction on MST Explorer, or a clear error message if the transaction reverts.

🔗 Step 3: Create a wallet connection hook - The Heart of Your dApp

Create the folder src/hooks/ and add useWallet.js. This custom React hook encapsulates all wallet state - connection status, address, provider, signer, and the connection function. It's the single source of truth for wallet state in your app:

import { useState, useCallback, useEffect } from 'react';
import { ethers } from 'ethers';

const MST_TESTNET_CHAIN_ID = '0x1388'; // 5000 in hexadecimal

export function useWallet() {
  const [address, setAddress] = useState(null);
  const [provider, setProvider] = useState(null);
  const [signer, setSigner] = useState(null);
  const [chainId, setChainId] = useState(null);
  const [connecting, setConnecting] = useState(false);
  const [error, setError] = useState(null);

  // Listen for account changes (user switches wallet account)
  useEffect(() => {
    if (!window.ethereum) return;
    const handleAccountsChanged = (accounts) => {
      if (accounts.length === 0) {
        // User disconnected
        setAddress(null);
        setSigner(null);
      } else {
        setAddress(accounts[0]);
        reconnect(accounts[0]);
      }
    };
    const handleChainChanged = (chainId) => {
      setChainId(chainId);
      // Refresh the page on chain change (simplest approach)
      window.location.reload();
    };
    window.ethereum.on('accountsChanged', handleAccountsChanged);
    window.ethereum.on('chainChanged', handleChainChanged);
    return () => {
      window.ethereum.removeListener('accountsChanged', handleAccountsChanged);
      window.ethereum.removeListener('chainChanged', handleChainChanged);
    };
  }, []);

  const reconnect = async (addr) => {
    const provider = new ethers.BrowserProvider(window.ethereum);
    const signer = await provider.getSigner();
    const chainId = await window.ethereum.request({ method: 'eth_chainId' });
    setProvider(provider);
    setSigner(signer);
    setChainId(chainId);
  };

  const connect = useCallback(async () => {
    setConnecting(true);
    setError(null);
    try {
      // Check if wallet is installed
      if (!window.ethereum) {
        throw new Error(
          "No wallet detected! Please install BridgeKey or MetaMask to use this dApp."
        );
      }

      // Request accounts (prompts user to approve connection)
      const accounts = await window.ethereum.request({
        method: 'eth_requestAccounts',
      });

      if (accounts.length === 0) {
        throw new Error("No accounts found. Please create or unlock your wallet.");
      }

      // Switch to MST Testnet
      await switchToMSTTestnet();

      // Create provider and signer
      const provider = new ethers.BrowserProvider(window.ethereum);
      const signer = await provider.getSigner();
      const chainId = await window.ethereum.request({ method: 'eth_chainId' });

      setAddress(accounts[0]);
      setProvider(provider);
      setSigner(signer);
      setChainId(chainId);
    } catch (err) {
      console.error('Connection error:', err);
      setError(err.message || 'Failed to connect wallet');
    } finally {
      setConnecting(false);
    }
  }, []);

  const disconnect = useCallback(() => {
    setAddress(null);
    setProvider(null);
    setSigner(null);
    setChainId(null);
    setError(null);
  }, []);

  return {
    address,
    provider,
    signer,
    chainId,
    connecting,
    error,
    connect,
    disconnect,
    isConnected: !!address,
    isMSTNetwork: chainId === MST_TESTNET_CHAIN_ID,
  };
}

// Helper function to switch network to MST Testnet
async function switchToMSTTestnet() {
  try {
    await window.ethereum.request({
      method: 'wallet_switchEthereumChain',
      params: [{ chainId: '0x1388' }], // 5000 in hex
    });
  } catch (switchError) {
    // This error code indicates the chain has not been added to the wallet
    if (switchError.code === 4902) {
      try {
        await window.ethereum.request({
          method: 'wallet_addEthereumChain',
          params: [{
            chainId: '0x1388',
            chainName: 'MST Testnet',
            rpcUrls: ['https://testnet-rpc.mstblockchain.com'],
            nativeCurrency: {
              name: 'MSTC',
              symbol: 'MSTC',
              decimals: 18,
            },
            blockExplorerUrls: ['https://testnet-explorer.mstblockchain.com'],
          }],
        });
      } catch (addError) {
        throw new Error('Failed to add MST Testnet to your wallet');
      }
    } else {
      throw new Error('Failed to switch to MST Testnet');
    }
  }
}

This hook handles the complete wallet lifecycle: installation check, connection request, network switching, account change detection, and disconnect. It exposes a clean API: address, isConnected, connect, disconnect, and status flags.

📖 Step 4: Create a contract interaction hook - Reading and Writing

Create src/hooks/useEscrow.js (or name it for your specific contract). This hook provides functions to read data and send transactions to your deployed contract:

import { useState, useCallback } from 'react';
import { ethers } from 'ethers';

// Your deployed contract address on MST Testnet
const CONTRACT_ADDRESS = "0xYourDeployedContractAddress";

// Import your contract's ABI from the Hardhat artifacts
import EscrowABI from '../abis/Escrow.json';

// Read‑only provider (no signer needed for view calls)
const readProvider = new ethers.JsonRpcProvider(
  "https://testnet-rpc.mstblockchain.com"
);

export function useEscrow(signer, address) {
  const [loading, setLoading] = useState(false);
  const [txHash, setTxHash] = useState(null);

  // Get a read‑only contract instance (for view functions)
  const getReadContract = useCallback(() => {
    return new ethers.Contract(CONTRACT_ADDRESS, EscrowABI, readProvider);
  }, []);

  // Get a signer‑connected contract instance (for transactions)
  const getWriteContract = useCallback(() => {
    if (!signer) throw new Error("Wallet not connected");
    return new ethers.Contract(CONTRACT_ADDRESS, EscrowABI, signer);
  }, [signer]);

  // READ: Fetch a specific escrow by ID
  const fetchEscrow = useCallback(async (escrowId) => {
    try {
      const contract = getReadContract();
      const escrow = await contract.escrows(escrowId);
      return {
        client: escrow.client,
        freelancer: escrow.freelancer,
        amount: ethers.formatEther(escrow.amount),
        status: getStatusText(escrow.status),
        clientConfirmed: escrow.clientConfirmed,
        freelancerConfirmed: escrow.freelancerConfirmed,
      };
    } catch (err) {
      console.error("Failed to fetch escrow:", err);
      throw err;
    }
  }, [getReadContract]);

  // READ: Fetch total number of escrows
  const fetchNextEscrowId = useCallback(async () => {
    const contract = getReadContract();
    const id = await contract.nextEscrowId();
    return Number(id);
  }, [getReadContract]);

  // WRITE: Create a new escrow (requires signer)
  const createEscrow = useCallback(async (freelancerAddress, amountInMSTC) => {
    setLoading(true);
    setTxHash(null);
    try {
      const contract = getWriteContract();
      const amountInWei = ethers.parseEther(amountInMSTC);
      const tx = await contract.createEscrow(freelancerAddress, {
        value: amountInWei,
      });
      setTxHash(tx.hash);
      const receipt = await tx.wait();
      setLoading(false);
      return receipt;
    } catch (err) {
      setLoading(false);
      throw err;
    }
  }, [getWriteContract]);

  // WRITE: Confirm delivery (freelancer)
  const confirmDelivery = useCallback(async (escrowId) => {
    setLoading(true);
    setTxHash(null);
    try {
      const contract = getWriteContract();
      const tx = await contract.confirmDelivery(escrowId);
      setTxHash(tx.hash);
      const receipt = await tx.wait();
      setLoading(false);
      return receipt;
    } catch (err) {
      setLoading(false);
      throw err;
    }
  }, [getWriteContract]);

  // WRITE: Release funds (client)
  const releaseFunds = useCallback(async (escrowId) => {
    setLoading(true);
    setTxHash(null);
    try {
      const contract = getWriteContract();
      const tx = await contract.releaseFunds(escrowId);
      setTxHash(tx.hash);
      const receipt = await tx.wait();
      setLoading(false);
      return receipt;
    } catch (err) {
      setLoading(false);
      throw err;
    }
  }, [getWriteContract]);

  return {
    fetchEscrow,
    fetchNextEscrowId,
    createEscrow,
    confirmDelivery,
    releaseFunds,
    loading,
    txHash,
  };
}

// Helper: Convert enum status number to readable text
function getStatusText(status) {
  const statuses = ['AWAITING_PAYMENT', 'FUNDED', 'DELIVERED', 'COMPLETED', 'DISPUTED'];
  return statuses[Number(status)] || 'UNKNOWN';
}

This hook cleanly separates read operations (free, no wallet needed) from write operations (require signer). Each write function tracks loading state and the transaction hash for UI feedback.

07   Reading and Writing Data

The Complete Transaction Lifecycle - From Click to Confirmation

Once the wallet is connected, the frontend needs to interact with the deployed contract. This section covers both reading (free, instant) and writing (gas-costing, asynchronous) operations in complete detail, including the UI states that make the experience professional.

Reading Data: View Functions

View functions are the simplest interaction. The frontend calls a function on the contract using a read-only provider. No transaction is created, no gas is spent, and the result is returned immediately. This is ideal for displaying balances, escrow details, or any data that the user needs to see before taking action.

// Read‑only provider — no signer needed
const readProvider = new ethers.JsonRpcProvider("https://testnet-rpc.mstblockchain.com");
const contract = new ethers.Contract(ADDRESS, ABI, readProvider);

// This call is free and instant
const escrow = await contract.escrows(escrowId);
console.log("Status:", escrow.status);
console.log("Amount:", ethers.formatEther(escrow.amount));

                                             │ SUCCESS   │   │ REJECTED  │   │  FAILED   │
                                             │ (refresh) │   │ (message) │   │ (retry?)  │
                                             └───────────┘   └───────────┘   └───────────┘
A transaction has four possible end states. A well-designed UI communicates each clearly. On MST, the mining phase typically lasts 3 - 5 seconds, meaning the user gets near-instant feedback compared to Ethereum's 15 - 60 second wait.

Here's the complete transaction handler with all states:

async function handleCreateEscrow(freelancerAddress, amountInMSTC) {
  // 1. IDLE → PENDING: Update UI immediately
  setTxState({ status: 'pending', message: 'Waiting for wallet approval...' });

  try {
    const contract = new ethers.Contract(ADDRESS, ABI, signer);
    const amountWei = ethers.parseEther(amountInMSTC);

    // 2. Send transaction (wallet popup appears)
    const tx = await contract.createEscrow(freelancerAddress, { value: amountWei });

    // 3. MINING: Transaction submitted, waiting for block
    setTxState({
      status: 'mining',
      message: 'Transaction submitted. Waiting for confirmation...',
      hash: tx.hash,
    });

    // 4. Wait for confirmation (block inclusion)
    const receipt = await tx.wait();

    // 5. SUCCESS: Transaction confirmed
    setTxState({
      status: 'success',
      message: 'Escrow created successfully!',
      hash: tx.hash,
      receipt: receipt,
    });

    // Refresh data
    await refreshEscrowList();
  } catch (err) {
    // 6. ERROR: Determine the type of failure
    if (err.code === 'ACTION_REJECTED') {
      setTxState({
        status: 'rejected',
        message: 'Transaction was rejected in your wallet.',
      });
    } else if (err.reason) {
      setTxState({
        status: 'failed',
        message: `Transaction reverted: ${err.reason}`,
      });
    } else {
      setTxState({
        status: 'failed',
        message: `Transaction failed: ${err.message}`,
      });
    }
  }
}

09   MST-Specific Optimizations

Leverage MST's Speed and Low Fees for a Superior UX

1
Near-instant confirmations (3-second blocks)
After calling tx.wait(), the transaction is typically confirmed within one block - about 3 seconds on MST. Compare this to Ethereum's 15 seconds or more during congestion. Your UI should reflect this speed: use a brief spinner, then immediately show success and refresh data. No need for "check back later" messages or long polling intervals. The user experience feels like a traditional web app, not a slow blockchain interaction.
2
Sub-paisa gas fees = more interactions are viable
Because transactions cost approximately ₹0.006 on MST, you can design flows with multiple on-chain steps without worrying about user cost. A "multi-step confirm" pattern where each step is a separate transaction (e.g., create escrow → confirm delivery → release funds) costs less than ₹0.02 total. On Ethereum, the same flow could cost ₹2,000+. This enables UX patterns that are simply uneconomical on expensive chains.
3
BridgeKey's auto-network switching
MST's BridgeKey wallet automatically detects when a dApp requests the MST chain and can switch networks for the user without manual intervention. This eliminates the common "wrong network" error. If the user is on Ethereum mainnet and visits your MST dApp, BridgeKey prompts them to switch with a single click.
4
Faucet integration for testnet users
For your capstone demo on MST Testnet, users need MSTC tokens to pay gas. Integrate a "Get Testnet MSTC" button that links to the MST Faucet. This removes the biggest friction point for new users trying your dApp. A simple link or embedded iframe can make the onboarding seamless.
10   Common Pitfalls and Debugging

The Bugs You Will Encounter (And How to Fix Them Immediately)

⚠️
"Unsupported network" when trying to interact
The user's wallet is connected to the wrong chain (e.g., Ethereum mainnet instead of MST Testnet). Check chainId before attempting transactions. Use wallet_switchEthereumChain to prompt the user to switch. Display a clear message: "Please switch to MST Testnet (Chain ID 5000) in your wallet."
⚠️
"Insufficient funds for gas" - even on testnet
Even though gas costs only ₹0.006 per transaction, the user must have MSTC in their wallet. For testnet users, provide a prominent link to the MST Faucet. For mainnet users, ensure your UI explains how to acquire MSTC. Never assume users already have the native coin.
⚠️
"Nonce too high" or "Replacement transaction underpriced"
This happens when a previous transaction is still pending in the mempool and the user tries to send another. Implement a transaction queue, or wait for tx.wait() to resolve before enabling the next transaction button. A simple isProcessing flag that disables all transaction buttons during a pending transaction solves this.
⚠️
"User rejected the request" - ACTION_REJECTED error
This is normal user behavior (changing their mind, canceling). Handle it gracefully - show a subtle message like "Transaction cancelled" rather than a red error. Don't punish users for changing their mind. Clear the pending state and re-enable the button.
11   Assessment Preparation

Top Facts - Know These Cold for Your Submission

01
The frontend communicates with the blockchain via a wallet provider (EIP-1193)
Never store private keys in the frontend. Never hardcode mnemonics. The wallet signs transactions; the dApp only receives the signed transaction to broadcast. This is the bedrock of Web3 security.
02
Use ethers.Contract with a read-only provider for view functions, and with a signer for transactions
View functions cost no gas and execute locally on the RPC node. Transactions require a signer, cost gas, and must be mined. Never mix the two - always create separate contract instances for reads and writes.
03
Always handle loading, success, and error states for every transaction
Use spinners for pending transactions, success messages with links to the block explorer, and clear error messages for common failures. The UI should never leave the user guessing.
04
MST's fast block times (~3s) mean instant UI updates after tx.wait()
After the transaction is mined, immediately refetch contract state and update the UI. The user should see their action reflected within seconds, not minutes.
05
Export your contract's ABI from Hardhat artifacts into a .json file in your frontend
After deploying with Hardhat, copy the ABI from artifacts/contracts/YourContract.sol/YourContract.json into src/abis/. The ABI is the interface that tells ethers.js what functions your contract has.

12   Glossary

Key Terms - Sub-Module 9.5

EIP-1193
Ethereum Provider JavaScript API - the standard interface that every wallet (BridgeKey, MetaMask, Coinbase Wallet) must implement. Defines window.ethereum and methods like eth_requestAccounts and wallet_switchEthereumChain. Ensures one dApp works with all wallets.
BridgeKey
MST's native wallet, available as a browser extension and mobile app. Fully EIP-1193 compatible. Supports auto-network switching to MST and integrates with MST's ecosystem features.
BrowserProvider
An ethers.js v6 class that wraps window.ethereum. Provides a high-level API for querying the blockchain and sending transactions. Automatically manages the connection to the user's wallet.
Signer
An ethers.js object tied to a specific account that can cryptographically sign and send transactions. Obtained via provider.getSigner(). The signer holds the private key inside the wallet - the dApp only receives the ability to request signatures.
tx.wait()
An ethers.js method that waits for a transaction to be mined (included in a block) and returns the transaction receipt. Essential for updating the UI after a transaction confirms. On MST, this typically resolves in ~3 seconds.
Vite
A fast, modern build tool and development server for JavaScript projects. Provides instant hot module replacement and optimized production builds. The recommended way to create React apps for Web3.
ABI
Application Binary Interface - a JSON description of a smart contract's functions, parameters, and events. Generated by the Solidity compiler. ethers.js uses the ABI to know how to encode function calls and decode return values.
JSON-RPC Provider
A connection to a blockchain node that accepts JSON-RPC requests. The read-only provider uses this to query the blockchain without a wallet. MST's public RPC URL is https://testnet-rpc.mstblockchain.com for testnet.
Custom Hook
A React pattern that encapsulates reusable stateful logic. useWallet and useEscrow are custom hooks that keep the component code clean and testable.
Chain ID
A unique identifier for a blockchain network, used to prevent transaction replay across chains. MST Testnet uses chain ID 5000 (hex: 0x1388). Always check the chain ID before sending transactions.
