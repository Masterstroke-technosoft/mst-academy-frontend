02   Real-World Introduction Story

The UPI Payment vs. The Bitcoin UTXO - Two Ways to Send Money

📖 Story

You want to send ₹500 to your friend Priya. You open Google Pay, type her number, enter ₹500, and hit send. Done in 3 seconds. Behind the scenes, the NPCI (National Payments Corporation of India) server simply looked up two numbers: your account balance went from ₹5,000 to ₹4,500. Priya's went from ₹2,000 to ₹2,500. Two numbers changed. That's it. This is the account model.


Now imagine you're paying with physical cash. You have a ₹500 note and a ₹200 note in your wallet. To pay Priya ₹500, you hand over the ₹500 note. You get no change because it's exact. But to pay Rahul ₹300, you hand over your ₹200 note - that's not enough, so you also use the ₹500 note - get ₹400 change back. Each note has a unique identity. You consume some notes, create new ones. This is the UTXO model - and this is exactly how Bitcoin works.


But here's the question nobody asks: where do all these transactions actually live? They don't float in the air. They are packaged into containers - called blocks - that are sealed, stamped, and linked into a permanent chain. This sub-module opens up those containers and shows you everything inside.

03   Technical Deep Dive - The Anatomy of a Block

What Is Actually Inside a Block?

A block is a container with two distinct parts: a Header (the metadata - like the label on a postal package) and a Body (the actual contents - the transactions). Let's dissect each field precisely.

⬡ BLOCK #847,293  ·  Bitcoin Network  ·  Mined: 2024-01-15 10:42:07 UTC
BLOCK HEADER - Metadata
Previous Hash
0x0000000000000000000a7f3c... 🔗 The Chain
The cryptographic fingerprint (SHA-256 hash) of the previous block. This is what creates the "chain" - every block points backwards to its parent. Change any block and this reference breaks.
Merkle Root
0xab12f3e9c7d8... 📋 TX Summary
A single hash that summarises ALL transactions in this block. If even one transaction is altered, the Merkle Root changes. Allows light clients (mobile wallets) to verify transactions without downloading the full block. (Covered deeply in 3.8.)
Timestamp
1705315327 (Unix epoch = 2024-01-15 10:42:07 UTC)
When the miner/validator produced this block. Used to adjust mining difficulty and prevent future-dating attacks.
Nonce
3,847,291 ⛏️ PoW Answer
In Proof of Work, miners try billions of nonce values until the resulting block hash falls below the difficulty target. This is the "answer" the miner found. In PoS chains (Ethereum, MST), the nonce is repurposed as a transaction counter.
Difficulty / Target
0x00000000000000000003A30C...
The current mining difficulty target. The block hash must be numerically smaller than this target to be valid. Difficulty adjusts every 2,016 blocks (~2 weeks) in Bitcoin to maintain the 10-minute average block time.
Version
0x20000000
Protocol version. Used to signal support for new features or soft fork activations. Validators use this to know which rule set applies to this block.
Block Hash (own)
0x00000000000000000001a2b3... 🆔 This Block's ID
Computed from all the above header fields. Not stored in the header itself - it is computed by every node independently. The next block will include this as its PrevHash.
BLOCK BODY - Transaction List
Coinbase TX
Block reward: 3.125 BTC → Miner's address
The first transaction in every block. Created by the miner - no sender, only a recipient. This is how new Bitcoin enters circulation. Bitcoin's current reward (post-2024 halving) is 3.125 BTC per block.
TX #2…#3,500
User transactions - transfers of value, smart contract calls, token swaps
Each transaction contains: Sender address, Recipient address, Amount, Gas fee, Digital signature proving authorisation, Nonce (replay protection). Average Bitcoin block: ~2,000 - 3,500 transactions. Ethereum: ~100 - 300 (heavier TXs). MST: 4,000+ TPS capacity.
🗝️ The Chain Is Made of One Field

The entire "chain" property of blockchain comes from one field: PrevHash. Each block header stores the hash of the previous block. This creates an unbreakable sequence - like a DNA chain where each link carries genetic code from its parent. Alter any block, its hash changes, and every subsequent block's PrevHash reference becomes invalid - instantly detectable by all nodes.

04   Transaction Lifecycle

From Your Wallet to the Blockchain - A Transaction's Journey

A common misconception is that a transaction is "sent" directly to the blockchain. In reality, it passes through several stages. Understanding this is critical for building dApps - many bugs come from not understanding transaction finality.

  1. You create and sign the transaction. In your BridgeKey wallet (or MetaMask), you specify: recipient address, amount, gas limit, and gas price. Your wallet uses your private key to create a digital signature - mathematical proof that you authorised this and nobody else could have. The private key never leaves your device. (Signatures are covered in Module 4.)
  2. Broadcast to the peer-to-peer network. Your signed transaction is sent to one or more nearby nodes (like sending a WhatsApp message to a group - it propagates outward). Each node that receives it checks: Is the signature valid? Does the sender have sufficient balance? Is the nonce correct? Valid transactions are stored and forwarded. Invalid ones are dropped silently.
  3. Waiting in the Mempool (Memory Pool). Every node maintains a mempool - a waiting room of valid but unconfirmed transactions. Think of it as the queue at a government office. All transactions sit here until a validator or miner picks them up. Higher gas fees get priority - like paying for fast-track processing.
  4. Selected by a validator/miner. The node producing the next block selects transactions from the mempool - typically choosing the highest-fee transactions first (maximising their revenue). On MST (Trika POSA), the Cabinet validator whose turn it is selects transactions and assembles the block.
  5. Included in a block and propagated. The new block containing your transaction is assembled: header fields are filled, the Merkle Root is computed from all included transactions, and (in PoW) the nonce is found. The completed block is broadcast to the entire network.
  6. Confirmed - the transaction is final. Other nodes verify the block (check all transactions are valid, block hash is correct, PrevHash matches) and add it to their chain. Your transaction is now permanently recorded. On MST: ~3 seconds. On Ethereum: ~12 seconds. On Bitcoin: ~10 minutes. More blocks built on top = higher "confirmations" = lower risk of reversal.

💡 Why Gas Fees Exist - The IndiGo Priority Boarding Analogy

The mempool is like an airport check-in queue. Everyone wants to board (get their transaction confirmed). The plane (block) has limited seats (~3,500 transactions for Bitcoin, more for MST). Validators are the airlines - they pick passengers who paid for priority boarding (higher gas fee) first. When demand is high (e.g., during a popular NFT mint), gas fees spike because everyone is competing for the same limited seats. On MST, with 4,000+ TPS capacity, this congestion problem is dramatically reduced.

05   UTXO vs Account Model - The Two Philosophies of Money

How Bitcoin Tracks Money vs How Ethereum (and MST) Track Money

This is one of the most important conceptual distinctions in blockchain. The two models represent fundamentally different answers to the question: "What does it mean to 'have' digital money?"

🔑 The Core Analogy

UTXO (Bitcoin): Think of physical cash. You have a ₹500 note, a ₹200 note, and a ₹100 note. Total: ₹800. Each note has a unique identity. To pay ₹300, you hand over the ₹200 + ₹100 notes - those notes are "consumed" (spent) and a new "note" of change is created.

Account Model (Ethereum, MST): Think of a Paytm or bank account. You have a balance: ₹800. To pay ₹300, the system subtracts ₹300 from your balance and adds ₹300 to the recipient's balance. No individual "notes" - just a number that changes.

UTXO (Unspent Transaction Output) - Bitcoin's Model

UTXO stands for Unspent Transaction Output. In Bitcoin, your "wallet balance" is not actually stored anywhere as a single number. Instead, Bitcoin tracks discrete "coins" - outputs of previous transactions that have not yet been spent.

When Priya received 0.3 BTC from Ravi last Tuesday, that creates a UTXO: a specific output worth 0.3 BTC, associated with Priya's address. When she wants to spend it, she must use the entire UTXO as an input to a new transaction - and create new UTXOs as outputs (payment to recipient + change back to herself).

In UTXO model, "spending" means consuming existing UTXOs as inputs and creating new UTXOs as outputs. Old UTXOs are destroyed; new ones are created. The sum of inputs must always equal the sum of outputs (including fees).

Account Model - Ethereum & MST's Approach

Ethereum, and MST (as an EVM-compatible chain), use the account model. This is conceptually simpler: every address has a stored balance. A transaction simply decrements the sender's balance and increments the recipient's balance by the specified amount.

The account model maintains a global state - a giant ledger of every address and its current balance. Transactions modify this state by adjusting balances. The "Nonce" prevents replay attacks - each transaction must have a nonce exactly one higher than the previous one from that address.

🗝️ Why MST Uses the Account Model

MST is EVM-compatible and uses the account model - specifically because it's designed for smart contracts, DeFi, gaming, and enterprise applications. The account model makes it trivial to write code like: "IF delivery confirmed THEN transfer ₹500." With UTXOs, this kind of programmable logic is extremely difficult to implement. The account model's simplicity is precisely what enabled Ethereum's smart contract ecosystem - and MST inherits all of it.

07   The Mempool - The Blockchain's Waiting Room

Why Transactions Don't Confirm Instantly

The mempool (Memory Pool) is a temporary data store on each node for transactions that have been validated but not yet included in a block. It is the most misunderstood part of blockchain by beginners - and the source of the most user confusion.

🏊 MEMPOOL - Current Unconfirmed Transactions (Illustrative)
Ravi → Priya: 0.5 BTC | Fee: 50 sat/vB 🟢
Anita → Contract: Swap USDT | Fee: 45 sat/vB 🟢
Rahul → Exchange: 0.2 ETH | Fee: 25 sat/vB 🟡
Ali → Priya: 0.1 BTC | Fee: 20 sat/vB 🟡
John → Mary: 0.05 BTC | Fee: 5 sat/vB 🔴
Sita → Exchange: Tiny TX | Fee: 2 sat/vB 🔴

🟢 = High fee = Gets into next block  ·  🟡 = Medium = Waits 1-2 blocks  ·  🔴 = Low = May wait hours

During times of high network activity (a popular NFT launch, a market crash triggering mass trading), the mempool fills up with thousands of pending transactions. Miners/validators only include the highest-fee transactions in each block. Low-fee transactions can wait hours or even days. This is why gas fee estimators exist - they help you bid enough to get confirmed quickly without overpaying.

💡 MST's Solution to the Mempool Problem

Because MST uses Trika POSA with 4,000+ TPS capacity, the mempool rarely fills up. A standard block produces in 3 seconds, and with high throughput, most transactions are confirmed within one block. The gas fees on MST are deliberately kept low to enable retail and gaming use cases - making it practical for everyday transactions like a Swiggy delivery payment confirmation triggering on-chain.

08   Real-World Case Studies

Blocks and Transactions in the Wild

₿
Bitcoin Block #170 - The First Ever Bitcoin Transaction
On 12 January 2009, Satoshi Nakamoto sent 10 BTC to Hal Finney - the first person-to-person Bitcoin transaction in history. This transaction exists in Block #170. You can look it up on blockchain.com right now - the exact inputs, outputs, amounts, and timestamp, permanently preserved. This is the power of the append-only ledger: 15 years later, that transaction is still there, unchanged, freely verifiable by anyone on Earth.
🍕
The ₹52 Crore Pizza - UTXO in Action
On 22 May 2010, programmer Laszlo Hanyecz paid 10,000 BTC for two pizzas - the first known commercial Bitcoin transaction. He consumed UTXOs worth 10,000 BTC as inputs. New UTXOs were created: the pizza seller received 10,000 BTC. At Bitcoin's peak (November 2021, ~$69,000/BTC), those pizzas were worth $690 million (~₹57 crore). This transaction is permanently recorded in the Bitcoin blockchain and is now called "Bitcoin Pizza Day" (22 May). Every year it is verifiable.
🎮
MST Gaming - Every Card Deal in a Block
On MST, when a poker game begins using the SatvaShuffle protocol, the deck commitment (a hash of the shuffled deck) is submitted as a transaction and included in a block. Every player can see the commitment before any cards are dealt. After the game, the actual deck is revealed - players can verify the commitment matches. This entire verification happens through transactions in blocks. The block timestamp proves the commitment was made before the game started - making post-hoc cheating impossible.
09   Assessment Preparation

Top Facts - Know These Cold

01
Block Header Fields (Must Memorise)
PrevHash (creates the chain), Merkle Root (summarises all TXs), Timestamp, Nonce (PoW answer), Version, Difficulty/Target. The block's own hash is computed FROM these fields - it is not stored in the header.
02
UTXO = Unspent Transaction Output
Bitcoin's model. Your balance = sum of all UTXOs you control. To spend, you consume UTXOs as inputs and create new UTXOs as outputs. Inputs must equal outputs (plus fee). Each UTXO can only be spent once.
03
Account Model = Ethereum / MST
Each address has a stored balance (like a Paytm wallet). Transactions decrement sender and increment recipient. A "nonce" (transaction counter) prevents replay attacks. The nonce must increment by exactly 1 for each transaction from the same address.
04
The Mempool
The temporary pool of valid, unconfirmed transactions on each node. Validators/miners select transactions from the mempool - prioritising by gas fee. High demand = high fees. Low demand = cheap and fast.
05
Coinbase Transaction
The FIRST transaction in every block. Has no inputs (no sender). Creates new currency out of thin air - the block reward. This is the only way new BTC (or MSTC) enters circulation.
06
Why UTXO Is Better for Privacy
In UTXO model, you can use a fresh address for every transaction's change output - making it harder to link your complete transaction history. In the account model, your address is persistent and your entire history is permanently visible.

Q1
MCQEasy1 mark
Which field in a block header creates the "chain" by linking each block to the one before it?
  • A Timestamp
  • B Merkle Root
  • C Previous Hash (PrevHash) ✅
  • D Nonce
✅ Correct Answer: C - Explanation

The Previous Hash field stores the cryptographic hash of the preceding block. This is what makes it a chain - each block provably and mathematically references its parent. Changing any block changes its hash, which breaks this reference in the next block, cascading an invalidation through every subsequent block. The Merkle Root summarises the transactions within the block, the Timestamp records when it was mined, and the Nonce is the PoW puzzle answer - none of these create the chain linking.

Q2
MCQEasy1 mark
UTXO stands for:
  • A Universal Token eXchange Output
  • B Unspent Transaction Output ✅
  • C Unified Transfer eXchange Operation
  • D Unverified Transaction Origin
✅ Correct Answer: B - Explanation

Unspent Transaction Output. In Bitcoin's model, when you receive Bitcoin, you don't receive a "balance update" - you receive a specific output from a previous transaction that hasn't been spent yet. This is a UTXO. Your total wallet "balance" is simply the sum of all UTXOs your private key can spend. Options A, C, and D are fabricated acronyms designed as distractors.

Q3
MCQMedium1 mark
Priya has 3 UTXOs: 0.3 BTC, 0.2 BTC, and 0.1 BTC. She wants to send 0.4 BTC to Rahul (network fee: 0.01 BTC). Which combination of UTXOs should she use as inputs, and what change does she receive?
  • A Use only 0.3 BTC UTXO; receive 0.1 BTC change
  • B Use 0.3 BTC + 0.2 BTC UTXOs; receive 0.09 BTC change ✅
  • C Use all three UTXOs; receive 0.19 BTC change
  • D Use only the 0.1 BTC UTXO repeated 5 times
✅ Correct Answer: B - Explanation

Total required = 0.40 BTC (payment) + 0.01 BTC (fee) = 0.41 BTC. Only the 0.3 BTC UTXO (Option A) is insufficient (only 0.30 BTC). Using 0.3 + 0.2 = 0.50 BTC: send 0.40 to Rahul, 0.01 to miner = 0.50 − 0.41 = 0.09 BTC change back to Priya. Option C (all three = 0.60 BTC) works but is wasteful - creates a larger UTXO change. Option D is impossible - UTXOs are discrete units, you cannot "use the same UTXO multiple times."

Examiner note: This is the classic UTXO arithmetic question. Always check: inputs ≥ payment + fee. Change = inputs − payment − fee.
Q4
MCQEasy1 mark
Where do valid, signed transactions wait before being included in a block?
  • A Genesis Pool
  • B State Trie
  • C Mempool (Memory Pool) ✅
  • D Block Header
✅ Correct Answer: C - Explanation

The Mempool (Memory Pool) is the temporary holding area on each node for transactions that have been validated (signature correct, balance sufficient, nonce correct) but not yet included in a confirmed block. Validators/miners select from the mempool - usually prioritising by gas fee - to assemble each new block. The State Trie is Ethereum's data structure for account balances. Block Headers contain metadata, not pending transactions. "Genesis Pool" does not exist.

Q5
MCQEasy1 mark
MST Blockchain uses which transaction model?
  • A Account model (like Ethereum) - each address has a stored balance ✅
  • B UTXO model (like Bitcoin) - discrete unspent outputs
  • C A hybrid of UTXO and Account model
  • D Neither - MST uses its own proprietary model
✅ Correct Answer: A - Explanation

MST is EVM-compatible (Ethereum Virtual Machine compatible), meaning it inherits Ethereum's architecture - including the account model. Every address on MST has a balance field in the state trie, and transactions adjust balances through arithmetic operations. This is what makes it possible for Solidity smart contracts (written for Ethereum) to run unchanged on MST. The UTXO model is used by Bitcoin-family coins.

Q6
MCQMedium1 mark
The Merkle Root in a block header serves what primary purpose?
  • A It stores the hash of the previous block to create the chain
  • B It is a single cryptographic summary of all transactions in the block, enabling fraud detection and light-client verification ✅
  • C It records the wallet address of the block's creator (validator/miner)
  • D It stores the solution to the Proof-of-Work puzzle
✅ Correct Answer: B - Explanation

The Merkle Root is a single 32-byte hash that summarises ALL transactions in the block. It is computed by hashing pairs of transaction hashes repeatedly (a Merkle tree) until one root hash remains. Two critical functions: (1) If any transaction in the block is altered, the Merkle Root changes - tamper detection. (2) A mobile wallet (light client) can verify that a specific transaction is in a block by only downloading the Merkle Root + a short "proof path" - without downloading all 3,000+ transactions. Option A describes PrevHash. Option D describes the Nonce (PoW answer).

Q7

Arjun has the following UTXOs in his Bitcoin wallet:

  • UTXO-X: 0.50 BTC
  • UTXO-Y: 0.30 BTC
  • UTXO-Z: 0.15 BTC

He wants to send 0.60 BTC to Kavya. The network fee is 0.01 BTC.

(a) What is the total amount required for this transaction? (1 mark)
(b) Which UTXOs must Arjun use as inputs? Justify why he cannot use UTXO-X alone. (1 mark)
(c) List all the outputs this transaction creates and their amounts. (1 mark)
(d) After this transaction completes, list Arjun's remaining UTXO set. (1 mark)
✅ Model Answer - 4 Marks

(a) Total amount required:
Payment to Kavya: 0.60 BTC + Network fee: 0.01 BTC = 0.61 BTC total required as inputs.


(b) UTXOs to use and justification:
Arjun must use UTXO-X (0.50 BTC) + UTXO-Y (0.30 BTC) = 0.80 BTC total inputs.
He cannot use UTXO-X alone because 0.50 BTC < 0.61 BTC required — insufficient to cover both the payment and fee. In the UTXO model, each UTXO must be used entirely (you cannot "partially" spend a UTXO), so he must combine UTXOs to meet or exceed the total requirement. UTXO-X + UTXO-Z = 0.65 BTC would also work — accept either valid combination with correct reasoning.


(c) Transaction outputs:
Using UTXO-X + UTXO-Y (0.80 BTC input):
• Output 1: 0.60 BTC → Kavya's address (the payment)
• Output 2: 0.19 BTC → Arjun's own address (change: 0.80 − 0.60 − 0.01 = 0.19)
• Output 3: 0.01 BTC → Miner/Validator (the network fee)
Total outputs: 0.60 + 0.19 + 0.01 = 0.80 BTC ✅ (equals total inputs)


(d) Arjun's remaining UTXO set after the transaction:
• UTXO-Z: 0.15 BTC (untouched - was not used as input)
• New change UTXO: 0.19 BTC (created as output from this transaction)
Total remaining balance: 0.15 + 0.19 = 0.34 BTC

1 mark each for: correct total (a), correct input selection + justification (b), correct three outputs with amounts (c), correct remaining UTXO set (d).

Key Terms - Sub-Module 3.2

Block Header
The metadata section of a block containing: PrevHash, Merkle Root, Timestamp, Nonce, Version, and Difficulty. The block's own hash is computed from these fields.
Block Body
The section of a block containing the actual list of transactions, starting with the Coinbase transaction followed by user transactions ordered by fee.
UTXO (Unspent Transaction Output)
Bitcoin's model of tracking value. Discrete "coins" output by previous transactions that have not yet been spent. Your balance = sum of all UTXOs your keys control.
Account Model
Ethereum/MST's approach: each address has a stored balance field. Transactions deduct from sender and add to recipient - like a bank account number.
Mempool (Memory Pool)
A temporary holding area on each node for valid but unconfirmed transactions. Validators select from the mempool - higher gas fees get priority.
Coinbase Transaction
The first transaction in every block. Has no inputs - it creates new currency (the block reward) out of thin air, paid to the miner/validator.
Nonce (Block)
In PoW: the number miners repeatedly adjust to find a valid block hash. In account model: a per-address transaction counter preventing replay attacks.
Merkle Root
A single hash summarising all transactions in a block. If any transaction is altered, the Merkle Root changes. Enables light-client transaction verification without downloading the full block.
Transaction Confirmation
A transaction is "confirmed" once it is included in a block. Each additional block built on top is another "confirmation" - more confirmations = lower risk of reversal.
Replay Attack
An attack where a valid signed transaction is maliciously rebroadcast. Prevented in UTXO model by nature (spent UTXOs cannot be reused) and in account model by the nonce.
