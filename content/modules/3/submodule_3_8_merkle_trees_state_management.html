02 · Real-World Introduction Story

The PAN Card Verification That Shouldn't Require the Entire CBDT Database

📖 Story

When a bank verifies your PAN card, it doesn't download the entire CBDT (Central Board of Direct Taxes) database of 700 million PAN holders and check your record locally. That would be insane - terabytes of data transferred for every verification. Instead, the bank queries a central API: "Is PAN ABCDE1234F valid?" The API returns yes or no. This works, but it requires trusting the central CBDT server.


Now imagine a blockchain version: your PAN is one of 3,500 transactions in Bitcoin Block #847,293. You want to prove to your bank that your transaction is in that block - without the bank downloading all 3,500 transactions. How?


The answer is the Merkle tree. A mathematical structure that lets you prove a specific leaf (your transaction) is part of a tree (the block) using only log₂(3500) ≈ 12 hashes instead of all 3,500 transactions. The bank only needs to know the Merkle Root (already in the block header) and your proof path - 12 hashes is a few hundred bytes, not megabytes. No central server. No trust. Just math.


This is the Merkle tree - one of the most elegant data structures in computer science, and the reason your Paytm-sized device can be a Bitcoin node.

03 · What Is a Merkle Tree?

Building from the Ground Up

A Merkle tree (named after Ralph Merkle, who described it in his 1979 PhD thesis) is a binary tree of hashes. The leaves are hashes of individual data items (transactions). Each parent node is the hash of its two children. The root - the single top hash - summarises every piece of data in the tree.

🔑 The Tournament Bracket Analogy

Imagine a cricket tournament. 8 teams play. Round 1: 4 matches. Round 2: 2 matches. Final: 1 match. The champion "represents" the entire tournament. If any match result was faked - the bracket is broken. A Merkle tree works exactly this way: transactions are teams, hashes are match results, and the Merkle Root is the champion. Change any transaction → its hash changes → its parent changes → all the way up to the Root. One wrong Root = entire block rejected.

FIGURE 3.8-A · Building a Merkle Tree - Step by Step with 4 Transactions
  STEP 1: Start with 4 transactions (leaves)
  ────────────────────────────────────────────────────────────────────
  TX_A: "Ravi → Priya: 0.5 BTC"     TX_B: "Ali → Zara: 0.2 BTC"
  TX_C: "Anita → Bank: 1.0 BTC"     TX_D: "MST → Rahul: 0.1 BTC"

  STEP 2: Hash each transaction (leaf hashes)
  ────────────────────────────────────────────────────────────────────
  H_A = SHA-256(TX_A) = "a1b2c3d4..."
  H_B = SHA-256(TX_B) = "e5f6g7h8..."
  H_C = SHA-256(TX_C) = "i9j0k1l2..."
  H_D = SHA-256(TX_D) = "m3n4o5p6..."

  STEP 3: Hash pairs of leaf hashes (parent nodes)
  ────────────────────────────────────────────────────────────────────
  H_AB = SHA-256(H_A + H_B) = "q7r8s9t0..."  (concatenate then hash)
  H_CD = SHA-256(H_C + H_D) = "u1v2w3x4..."

  STEP 4: Hash the parent nodes to get the Merkle Root
  ────────────────────────────────────────────────────────────────────
  Merkle Root = SHA-256(H_AB + H_CD) = "ROOT_HASH_FINAL"

  TREE STRUCTURE:
  ────────────────────────────────────────────────────────────────────
                     [  MERKLE ROOT  ]
                    /                 \
              [H_AB]                   [H_CD]
             /      \                 /      \
          [H_A]   [H_B]           [H_C]   [H_D]
            |       |               |       |
          TX_A    TX_B            TX_C    TX_D

  This Merkle Root is stored in the Block Header.
  It permanently and cryptographically summarises all 4 transactions.
  Alter TX_B → H_B changes → H_AB changes → Root changes → Block rejected.
Each level of the tree reduces the data by half through hashing. The Merkle Root - stored in the block header - summarises every transaction in that block with a single 32-byte hash.

For a block with 3,500 transactions, the Merkle tree has approximately 12 levels (log₂(3500) ≈ 11.8). The Root is computed from 3,500 leaf hashes through 11 rounds of pairwise hashing. The entire computational process takes milliseconds on any modern machine.

04 · Handling Odd Numbers of Transactions
🗝️ What If There's an Odd Number of Transactions?

Bitcoin's Merkle tree implementation duplicates the last transaction hash if there is an odd number at any level. If a block has 5 transactions (TX_A through TX_E), TX_E is hashed with itself: H_EE = SHA-256(H_E + H_E). This "duplication" rule is specified in the Bitcoin protocol. Ethereum handles odd numbers slightly differently - it also duplicates the last node. The key point: the algorithm always produces a valid binary tree regardless of input count.

05 · Merkle Proof - The Killer Feature

Proving TX_B Is in the Block Without Sharing All Transactions

This is where the Merkle tree's power becomes apparent. Suppose you want to prove to a light client (a mobile wallet) that TX_B is included in Block #847,293. The block has 3,500 transactions. The light client only has the block header (which includes the Merkle Root). How do you prove TX_B is included without sending all 3,500 transactions?

FIGURE 3.8-B · Merkle Proof - Proving TX_B Without Sending Everything
  FULL TREE (only the prover has this):
                     [  MERKLE ROOT  ]
                    /                 \
              [H_AB]                   [H_CD]  ← Prover sends this
             /      \                 /      \
          [H_A]   [H_B]           [H_C]   [H_D]
            |       |               |       |
          TX_A    TX_B            TX_C    TX_D
                   ↑
           This is what we're proving is included

  MERKLE PROOF for TX_B consists of just 2 hashes:
  ─────────────────────────────────────────────────────────────────
  Proof = [ H_A,   H_CD ]
           (sibling of TX_B's hash)   (sibling of H_AB)

  VERIFICATION by the light client (who only knows the Merkle Root):
  ─────────────────────────────────────────────────────────────────
  Step 1: Compute H_B = SHA-256(TX_B)                    ← light client does this
  Step 2: Compute H_AB = SHA-256(H_A + H_B)             ← using H_A from proof
  Step 3: Compute candidate_root = SHA-256(H_AB + H_CD) ← using H_CD from proof
  Step 4: Check: candidate_root == known Merkle Root?

  If YES → TX_B is provably in the block ✅
  If NO  → TX_B is NOT in the block (or proof is fraudulent) ❌

  EFFICIENCY:
  ─────────────────────────────────────────────────────────────────
  Full verification (naive): All 3,500 transactions = several MB
  Merkle proof:              Only 12 hashes = ~384 bytes (32 bytes × 12)

  The proof is O(log n) in size — dramatically more efficient.
A Merkle proof lets anyone verify a transaction is in a block using only O(log n) hashes. This is what makes lightweight clients (SPV nodes) possible - they can verify transactions without downloading the full blockchain.
06 · SPV / Light Clients - How Mobile Wallets Work

Simplified Payment Verification - Satoshi's Original Design

Satoshi described SPV (Simplified Payment Verification) in Section 8 of the Bitcoin whitepaper. The idea: a mobile device cannot store the full 500+ GB Bitcoin blockchain - but it can store just the block headers (each only 80 bytes). With the block headers, it has all the Merkle Roots. Using Merkle proofs, it can verify any transaction without downloading any full block.

1
Download Only Block Headers
Bitcoin's full blockchain (including all transactions): ~500 GB and growing. Bitcoin's block headers only: ~60 MB (80 bytes × ~850,000 blocks). An SPV client downloads only headers - a 99.99% data reduction. It connects to full nodes to request specific Merkle proofs when needed.
2
Verify Transactions via Merkle Proof
When you receive a payment, the SPV client asks a full node: "Give me a Merkle proof that TX 0xabc... is in Block #847,293." The full node returns the proof path (12 hashes). The SPV client verifies the proof against the known Merkle Root in the block header. No trust in the full node required - the math is verifiable independently.
3
Confirm the Block is Part of the Longest Chain
The SPV client also verifies that the block containing the transaction has sufficient Proof-of-Work (or PoS attestations) and is part of the main chain. This confirms the transaction hasn't been reversed. On Bitcoin, 6 block headers after the TX provide strong confirmation. On MST, BFT finality means 2 headers is equivalent security.
💡 SPV Security Trade-off

SPV nodes trust that the longest chain they can see is the honest chain - but they don't fully verify all transaction validity rules. A majority-hashrate attacker could theoretically create a valid-looking proof for a fraudulent transaction. This is the trade-off: SPV is 99.99% efficient but slightly less secure than running a full node. For everyday retail payments, SPV security is entirely sufficient. For high-value institutional transactions, running a full archival node is recommended.

07 · Patricia Merkle Trie - How Ethereum Stores the World State

From Transaction Trees to the Global State Machine

Bitcoin's Merkle tree solves one problem: summarising transactions in a block. Ethereum has a larger challenge: storing the entire global state - every account's balance, nonce, code, and storage - in a way that can be cryptographically verified, efficiently updated, and quickly queried.

Ethereum's answer is the Patricia Merkle Trie (PMT) - a combination of a Patricia trie (efficient key-value prefix tree) and a Merkle tree (cryptographic hashing). Every piece of state is indexed by a key (account address or storage slot), arranged in a prefix tree structure, and hashed bottom-up to produce a single root hash.

FIGURE 3.8-C · Ethereum's Three State Trees - Each Block Contains All Three Roots
  ETHEREUM BLOCK HEADER CONTAINS 3 TRIE ROOTS:
  ──────────────────────────────────────────────────────────────────

  1. TRANSACTIONS ROOT (transactionsRoot)
     ───────────────────────────────────
     Merkle Root of all transactions IN this block.
     Same concept as Bitcoin's Merkle Root.
     Proves: "Transaction X is in this block."

  2. RECEIPTS ROOT (receiptsRoot)
     ────────────────────────────
     Merkle Root of all transaction receipts (outcomes).
     Each receipt contains: gas used, logs/events emitted, status (success/fail).
     Proves: "Transaction X succeeded/failed and emitted these events."

  3. STATE ROOT (stateRoot) ← THE MOST IMPORTANT
     ──────────────────────────────────────────────
     Root of the Patricia Merkle Trie of the ENTIRE WORLD STATE after this block.
     World state = every account on Ethereum:
       - EOA (Externally Owned Account): balance, nonce
       - Contract account: balance, nonce, codeHash, storageRoot

     Example state trie (simplified):
     ┌──────────────────────────────────────────────────────────┐
     │            STATE ROOT HASH                               │
     │           /              \                               │
     │    [branch]              [branch]                        │
     │    /      \              /      \                        │
     │ [0xRavi]  [0xPriya]   [0xAnita] [0xContract]            │
     │  bal:5ETH  bal:3ETH    bal:1ETH  bal:100ETH             │
     │  nonce:12  nonce:7     nonce:2   code:0xabcd...         │
     │                                  storage: [...]          │
     └──────────────────────────────────────────────────────────┘

     After every transaction modifies any balance:
     → Affected leaf nodes update
     → Their parent hashes update
     → State Root changes
     → New State Root recorded in next block header

  KEY PROPERTY: Two different nodes that compute the same STATE ROOT
  are guaranteed to have identical copies of the complete world state.
  The State Root is a cryptographic commitment to the entire Ethereum state.
Ethereum maintains three separate trie roots per block - each cryptographically summarising a different aspect of the chain's state. The State Root is the most critical: it commits to every account's balance and contract state simultaneously.
Property Bitcoin Merkle Tree Ethereum Patricia Merkle Trie
What it stores Transactions within a block Entire account state (balances, contracts, storage)
Structure Simple binary tree Radix/prefix tree + Merkle hashing (more complex)
Supports efficient updates? No - must rebuild for changes Yes - only update changed branches
Supports efficient lookups? No - must scan linearly Yes - O(log n) key lookup
Cryptographic summary Merkle Root (32 bytes) State Root (32 bytes)
Used in MST? Merkle Root for transaction summary Yes - EVM-compatible state management
08 · Real-World Applications

Where Merkle Trees Appear in Practice

₿
Bitcoin SPV - Paytm-Sized Wallets on a Feature Phone
Every Bitcoin mobile wallet (BlueWallet, Muun, Phoenix) uses SPV under the hood. Your phone doesn't store the 500 GB Bitcoin blockchain - it stores 60 MB of headers and requests Merkle proofs on demand. This is why your Paytm-sized app can verify Bitcoin payments - Satoshi's Merkle tree design made it possible. Without SPV, every Bitcoin user would need a server with 500+ GB of disk space.
🌐
Git - Your Code Version Control Uses Merkle Trees
Git (the version control system used by every software project) uses a Merkle DAG (Directed Acyclic Graph) - essentially a Merkle tree - to track file history. Each commit is a hash of the repository state at that moment. If any file changes, the commit hash changes. This makes Git tamper-evident - you can verify that your codebase hasn't been silently modified between commits. GitHub, GitLab, Bitbucket all build on this structure. You use a Merkle tree every time you run git commit.
📜
IPFS - Decentralised File Storage
IPFS (InterPlanetary File System - used for storing NFT metadata, MST's off-chain data) uses a Merkle DAG to address files. Every file is content-addressed: the file's hash is its address. If the content changes, the address changes - making tampering detectable. When MST stores NFT metadata or game assets on IPFS, the IPFS content hash (CID) is recorded on-chain - creating an immutable on-chain reference to off-chain data verified by its Merkle structure.
🎮
MST - Merkle Proofs for Game State Disputes
In MST's gaming applications, the complete game state (all player positions, inventories, scores) is summarised as a Merkle Root at each game epoch. If a player disputes a game outcome - claiming "my score was X not Y" - they submit a Merkle proof showing their score in the game state trie. The smart contract verifies the proof against the published Root. No need to re-simulate the entire game - just verify the relevant leaf using O(log n) hashes. Disputes resolved in milliseconds, not minutes.
09 · Assessment Preparation

Top Facts - Know These Cold

01
Merkle Tree = Binary Tree of Hashes
Leaves = hash(transaction). Parents = hash(left_child + right_child). Root = single hash summarising all leaves. Stored in block header. Change any TX → root changes → block rejected.
02
Merkle Proof = O(log n) Hashes
To prove TX_B (one of N transactions) is in a block: provide the sibling hashes along the path from TX_B to the Root. Verifier recomputes up the tree and checks against known Root. For 3,500 TXs: ~12 hashes instead of 3,500.
03
SPV (Simplified Payment Verification)
Light clients store only block headers (~80 bytes each = ~60 MB total for all Bitcoin). Request Merkle proofs for specific transactions. Verify proofs against the Merkle Root in the header. No trust in the queried full node - the math is self-verifying.
04
Patricia Merkle Trie = Ethereum's State Storage
Three roots per block header: transactionsRoot, receiptsRoot, stateRoot. stateRoot commits to the entire world state - every account's balance and contract storage. EVM-compatible chains (including MST) use this architecture.
05
Odd-Number Duplication Rule
If any level of the Merkle tree has an odd number of nodes, the last node is duplicated and hashed with itself. This ensures the tree is always binary at every level. Bitcoin protocol specifies this behaviour explicitly.

Q1
MCQEasy1 mark
What is the Merkle Root in a blockchain block?
  • A The hash of the first transaction in the block
  • B The hash of the block header including PrevHash and Nonce
  • C A single cryptographic hash that summarises all transactions in the block through a binary tree of hashes ✅
  • D The address of the miner or validator who produced the block
✅ Correct Answer: C - Explanation

The Merkle Root is computed by hashing each transaction, then hashing pairs of hashes, repeatedly, until a single root hash remains. It is stored in the block header. This 32-byte value summarises ALL transactions in the block - altering any single transaction changes the Merkle Root, which is immediately detectable by any node that recomputes it.

Q2
MCQEasy1 mark
A block contains 1,024 transactions. How many hashes does a Merkle proof need to verify that one specific transaction is in the block?
  • A 1,024 hashes (all transactions)
  • B 512 hashes (half the transactions)
  • C 10 hashes (log₂(1,024) = 10) ✅
  • D 1 hash (just the Merkle Root)
✅ Correct Answer: C - Explanation

A Merkle proof requires exactly log₂(N) hashes where N is the number of transactions. For 1,024 transactions: log₂(1,024) = 10. These 10 hashes are the "sibling" nodes along the path from the target transaction to the Merkle Root. The verifier uses these 10 hashes to recompute the path upward and confirm the computed Root matches the known Root. This O(log n) efficiency is the Merkle tree's central advantage.

Q3
MCQMedium1 mark
SPV (Simplified Payment Verification) allows a mobile Bitcoin wallet to verify transactions without downloading the full blockchain. Which data structure makes this possible?
  • A The UTXO set - a database of all unspent transaction outputs
  • B Merkle trees - enabling transaction verification using only block headers and a short proof path ✅
  • C The mempool - a list of unconfirmed transactions on each node
  • D The Patricia Merkle Trie - Ethereum's account state storage
✅ Correct Answer: B - Explanation

Merkle trees are the direct enabler of SPV. An SPV client only downloads block headers (~80 bytes each). When it needs to verify a transaction, it requests a Merkle proof from a full node - the sibling hashes along the path from that transaction to the Root. Using the known Merkle Root (already in the header) and the proof path, the SPV client verifies the transaction without any other data. No full blocks, no full node, no trust required.

Q4
MCQMedium1 mark
In Ethereum's block header, the "stateRoot" field represents:
  • A The hash of the first smart contract deployed on Ethereum
  • B The Merkle Root of all transactions in the current block
  • C The root hash of the Patricia Merkle Trie that encodes the entire world state - every account's balance, nonce, code, and storage - after executing all transactions in this block ✅
  • D The hash of the previous block's state, used for chain linking
✅ Correct Answer: C - Explanation

The stateRoot is one of three trie roots in an Ethereum block header (along with transactionsRoot and receiptsRoot). It is the root of the full Patricia Merkle Trie of the global world state after all transactions in this block have executed. This 32-byte hash is a cryptographic commitment to every Ethereum account - two nodes with the same stateRoot have guaranteed identical world states. MST uses the same EVM-compatible architecture.

Q5
MCQMedium1 mark
Which of the following is a key advantage of the Patricia Merkle Trie over Bitcoin's simple Merkle tree for storing account state?
  • A The Patricia Merkle Trie uses less memory because it stores only recent transactions
  • B It supports efficient key-value lookups AND efficient updates - only changed branches need to be recomputed after a state change ✅
  • C It does not use hashing - making it faster than Bitcoin's Merkle tree
  • D It stores all account data in a single flat array for simple indexing
✅ Correct Answer: B - Explanation

Bitcoin's simple Merkle tree is optimal for a fixed set of transactions (read once, never updated). Ethereum's Patricia Merkle Trie is designed for a mutable state that changes with every transaction. Its prefix-tree structure allows: (1) efficient O(log n) lookups by address, (2) efficient updates - when Ravi's balance changes, only the nodes from his leaf to the root need recomputation, not the entire tree. This efficiency is critical - Ethereum's state has millions of accounts, and thousands of them change per block.

Section B - Identify-Output: Merkle Tree Computation
1 Question · 5 marks
Q6
Identify-OutputMedium5 marks
A block contains exactly 4 transactions. Their SHA-256 hash values (simplified for this exercise) are:

  • H(TX_A) = AA
  • H(TX_B) = BB
  • H(TX_C) = CC
  • H(TX_D) = DD

For hashing pairs, use the notation: H(XY) means hash(X concatenated with Y).

(a) Compute H(TX_A, TX_B) - the parent of TX_A and TX_B. (1 mark)
(b) Compute H(TX_C, TX_D) - the parent of TX_C and TX_D. (1 mark)
(c) Compute the Merkle Root. (1 mark)
(d) TX_B's data is altered (its hash is now BB_MODIFIED). Which nodes in the tree must be recomputed? List them in order from leaf to root. (1 mark)
(e) A verifier has only the Merkle Root and wants to verify TX_C is in the block. What is the minimum set of hashes the prover must provide (the Merkle proof for TX_C)? (1 mark)
✅ Model Answer - 5 Marks

(a) H(TX_A, TX_B) = H(AA + BB) = H_AB [1 mark]
The parent node is computed by concatenating the two children's hashes and hashing the result. Using simplified notation: hash("AA" + "BB") = H_AB.


(b) H(TX_C, TX_D) = H(CC + DD) = H_CD [1 mark]
Same process: hash("CC" + "DD") = H_CD.


(c) Merkle Root = H(H_AB + H_CD) = H(H(AA+BB) + H(CC+DD)) [1 mark]
The root is computed by hashing the two parent nodes together. In full form: SHA-256(SHA-256("AA"+"BB") concatenated with SHA-256("CC"+"DD")).


(d) Nodes that must be recomputed when TX_B changes to BB_MODIFIED: [1 mark]
1. H(TX_A, TX_B) - recompute as H(AA + BB_MODIFIED) = H_AB_NEW
2. Merkle Root - recompute as H(H_AB_NEW + H_CD) = NEW_ROOT
Only the path from the changed leaf to the root is affected. H_CD is unchanged (TX_C and TX_D were not modified).


(e) Merkle proof for TX_C - minimum hashes prover must send: [1 mark]
The proof path for TX_C requires its sibling at each level:
• H(TX_D) = DD - sibling of TX_C at leaf level
• H_AB = H(AA+BB) - sibling of H_CD at parent level

Verification: Verifier computes H(CC+DD) = H_CD, then H(H_AB + H_CD) = Root. If this equals the known Root → TX_C is verified. Just 2 hashes to verify 1 of 4 transactions.

1 mark each for parts (a) through (e). For (d), accept answers that list both nodes - order from leaf to root preferred but not strictly required for full marks. For (e), must correctly identify both sibling hashes (DD and H_AB) - a single hash alone earns 0.5 marks.

10 · Glossary

Key Terms - Sub-Module 3.8

Merkle Tree
A binary tree of cryptographic hashes where each parent node is the hash of its two children. The root summarises all leaf data. Change any leaf → root changes.
Merkle Root
The single top hash of a Merkle tree. Stored in each block header. Cryptographically summarises all transactions in that block in exactly 32 bytes.
Merkle Proof
A set of O(log n) sibling hashes that allows verification that a specific leaf is in a Merkle tree - without providing all leaves. Used by SPV light clients.
SPV (Simplified Payment Verification)
Satoshi's design for lightweight clients - store only block headers, verify transactions using Merkle proofs. Enables mobile Bitcoin wallets storing 60 MB vs 500 GB for full nodes.
Light Client
A blockchain node that downloads only headers (not full blocks) and verifies transactions via Merkle proofs. Lower storage and bandwidth requirements - suitable for mobile devices.
Patricia Merkle Trie (PMT)
Ethereum's data structure combining a prefix trie (for efficient key lookups) with Merkle hashing (for cryptographic summarisation). Used for stateRoot, transactionsRoot, receiptsRoot.
stateRoot
One of three trie roots in an Ethereum block header. Root hash of the Patricia Merkle Trie encoding the complete world state (all account balances, nonces, contract code, storage) after executing all block transactions.
transactionsRoot
Root hash of the Merkle tree of all transactions included in a block. Equivalent to Bitcoin's Merkle Root.
receiptsRoot
Root hash of the Merkle tree of all transaction receipts (outcomes, gas used, logs emitted) from a block's transactions.
O(log n)
Logarithmic complexity - the size/time required grows with the log of input size. A Merkle proof for 1 of 1,024 transactions requires log₂(1,024) = 10 hashes. For 1,048,576 TXs: only 20 hashes.
