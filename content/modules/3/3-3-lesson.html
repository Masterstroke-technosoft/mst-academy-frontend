02   Real-World Introduction Story

Your Aadhaar Fingerprint Was Never Stored

📖 Story

In 2013, you went to an Aadhaar enrolment centre in Pune. The operator pressed your finger onto the scanner. In that moment, you probably assumed UIDAI (Unique Identification Authority of India) was saving a photograph of your fingerprint - a digital image of your swirling ridges, stored in some massive government database.


They did not. What they stored was a number. A very specific, very long number - computed by running your fingerprint image through a mathematical function that is, for all practical purposes, irreversible. Your actual fingerprint image was discarded. Only the number remains.


Every time you use Aadhaar authentication - to open a bank account, get a SIM card, access your DigiLocker - the scanner computes the same mathematical function on your fresh scan and compares the result to the stored number. If they match: it's you. If they don't: it's not. And no hacker who steals the UIDAI database gains access to your actual fingerprint - they only get the irreversible number. This is a hash function.


Blockchain uses the exact same idea - but applied to transactions, blocks, and addresses. Every piece of data in a blockchain is fingerprinted. And the entire security model depends on one critical property: you cannot reverse the fingerprint to find the original data.

03   Background & History

The Evolution of Cryptographic Hash Functions

1979
Ralph Merkle's PhD - The Foundation
Ralph Merkle's doctoral thesis introduced the concept of one-way hash functions and Merkle trees. His work established the theoretical basis for all modern cryptographic hashing - the same Merkle trees used in every Bitcoin and Ethereum block today.
1993
SHA-1 - The First Widely Deployed Hash
The NSA designed SHA-1 for the US government. It was used everywhere - SSL certificates, digital signatures, version control (Git still uses SHA-1 internally for commit hashes). Critically broken in 2017 when Google demonstrated a "collision" - two different files with the same SHA-1 hash. This is why SHA-1 is now deprecated for security applications.
2001
SHA-256 - Bitcoin's Hash Function
SHA-256 (Secure Hash Algorithm, 256-bit output) was standardised by NIST. Satoshi Nakamoto chose it for Bitcoin in 2008 - it is used for block hashing (PoW), transaction IDs, and address derivation. As of 2024, no practical attack on SHA-256 exists. The entire Bitcoin security model rests on it.
2012
Keccak Wins the SHA-3 Competition - Ethereum's Choice
NIST ran an open competition for a new hash standard. Keccak (designed by Bertoni, Daemen, Peeters, and Van Assche) won after 5 years and 64 competing submissions. Ethereum adopted Keccak-256 (a specific Keccak variant) for all internal operations - address derivation, transaction hashing, the keccak256() Solidity opcode. MST, as an EVM-compatible chain, uses the same function.
04   The Five Properties of Cryptographic Hash Functions

What Makes a Hash Function "Cryptographic"?

Not every hash function is cryptographic. A simple hash like "sum all the ASCII values of characters" is a hash function - but it's not cryptographic. Five specific properties distinguish a cryptographic hash function from a basic one.

1. Deterministic
Must-Have
Same input → Always same output
No matter when, where, or on which computer you run SHA-256("Hello"), you always get the exact same 64-character output. This is essential for blockchain consensus - every node must independently compute the same result to agree on block validity.
🇮🇳 Your fingerprint always produces the same Aadhaar biometric code - on any scanner, any day.
2. Pre-image Resistant
One-Way
Cannot reverse the hash to find the original input
Given a hash output H, it is computationally infeasible to find any input X such that hash(X) = H. SHA-256 has 2²⁵⁶ possible outputs - approximately 10⁷⁷. Even if every atom in the universe were a computer running since the Big Bang, you could not brute-force a SHA-256 hash in the lifetime of the universe.
🇮🇳 Nobody can reconstruct your fingerprint from your Aadhaar biometric number - even UIDAI itself.
3. Avalanche Effect
Sensitivity
Tiny input change → Completely different output
Changing even a single bit in the input produces a drastically different hash output - typically about half the output bits flip. This means you cannot make subtle, undetectable changes to data. Any tampering is immediately visible through a completely changed fingerprint.
🇮🇳 A single ridge alteration on your fingerprint produces a completely unrecognisable biometric code.
4. Collision Resistant
Uniqueness
Computationally infeasible to find two different inputs with the same hash
It should be essentially impossible to find X ≠ Y where hash(X) = hash(Y). By the birthday paradox, finding a SHA-256 collision would require approximately 2¹²⁸ operations - far beyond any known or projected computational capability. SHA-1's collision resistance was broken; SHA-256's has not been.
🇮🇳 No two people in India share an Aadhaar number - the biometric hashing system is designed to be unique per individual.
5. Fixed Output Size
Consistency
Output is always exactly the same length regardless of input size
SHA-256 always produces exactly 256 bits = 32 bytes = 64 hexadecimal characters. Whether you hash the word "Hi" (2 bytes) or the entire Harry Potter series (4.1 MB), the output is always 64 hex characters. This is critical for block headers - they need a predictable-size field for PrevHash.
🇮🇳 Whether you scan 1 finger or 10 simultaneously, Aadhaar produces one fixed-length biometric template.
05   SHA-256 vs Keccak-256 - The Two Engines

Which Hash Function Powers Which Blockchain?

Both SHA-256 and Keccak-256 are cryptographic hash functions with 256-bit outputs, but they use completely different internal algorithms and are used in different ecosystems.

06   Inside SHA-256 - Conceptual Understanding

How Does SHA-256 Actually Compute Its Output?

You don't need to implement SHA-256 - but understanding the conceptual steps builds deep intuition for why it has the properties it does.

FIGURE 3.3-B · SHA-256 Internal Process (Conceptual)
  INPUT: "Ravi sends 1 BTC to Priya"  (any length)

  STEP 1: PADDING
  ─────────────────────────────────────────────────────────────────────
  Append a '1' bit, then '0' bits, then a 64-bit length field
  until the total message length is a multiple of 512 bits.
  Reason: SHA-256 processes data in fixed 512-bit chunks.

  STEP 2: INITIALISE 8 HASH VALUES (h0 through h7)
  ─────────────────────────────────────────────────────────────────────
  These are the first 32 bits of the fractional parts of the
  square roots of the first 8 prime numbers:
  h0 = 0x6a09e667  (from √2)
  h1 = 0xbb67ae85  (from √3)
  h2 = 0x3c6ef372  (from √5)
  ... (why primes? They have no mathematical "back door" — public verifiability)

  STEP 3: 64 ROUNDS OF COMPRESSION PER 512-BIT CHUNK
  ─────────────────────────────────────────────────────────────────────
  For each 512-bit chunk:
    Expand chunk into 64 × 32-bit "words" using bit rotations and XOR
    Run 64 rounds of: Σ (sigma), Ch (choose), Maj (majority) operations
    These are simple bitwise operations — AND, OR, XOR, bit-rotate
    Each round mixes the current state using a "round constant"
    (derived from cube roots of the first 64 primes — again, no back door)

  STEP 4: ADD COMPRESSED CHUNK TO CURRENT HASH VALUE
  ─────────────────────────────────────────────────────────────────────
  After 64 rounds, add the compressed chunk to the running hash values.
  Process the next 512-bit chunk. Repeat.

  STEP 5: OUTPUT THE FINAL HASH
  ─────────────────────────────────────────────────────────────────────
  Concatenate h0 through h7 → 8 × 32-bit = 256-bit output.
  Convert to 64 hexadecimal characters.

  OUTPUT: 3c9683017f9e4bf33d0a90e87f17f4a2625cdfede6a7572e22834f0e61fc7bf3

  ─────────────────────────────────────────────────────────────────────
  KEY INSIGHT: Each step is designed to ensure that ANY change in the input
  cascades through ALL subsequent rounds, changing the output completely.
  This is the avalanche effect — by design, not by accident.
  ─────────────────────────────────────────────────────────────────────
SHA-256 runs approximately 64 rounds of bitwise mixing operations. A modern CPU can compute ~500 million SHA-256 hashes per second. Bitcoin's network collectively computes ~500 exahashes (5×10²⁰) per second.
07   Hashing in Solidity - keccak256() in Practice

Writing Hash Functions in Smart Contracts

In Solidity (the smart contract language for Ethereum and MST), the keccak256() function is a built-in - a first-class opcode. It takes bytes as input and returns a bytes32 (256-bit) hash. Understanding how to use it correctly is essential for building secure contracts.

Solidity · keccak256 - Complete Usage Guide
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract HashingExamples {

    // ─────────────────────────────────────────────────────────────────
    // EXAMPLE 1: Hash a string
    // Use case: Commit-reveal schemes, storing verifiable secrets
    // ─────────────────────────────────────────────────────────────────
    function hashString(string memory input) public pure returns (bytes32) {
        return keccak256(abi.encodePacked(input));
    }
    // abi.encodePacked(input) converts the string to tightly packed bytes.
    // keccak256() hashes those bytes → returns bytes32.
    // "pure" means it doesn't read or modify blockchain state.

    // ─────────────────────────────────────────────────────────────────
    // EXAMPLE 2: Hash multiple values together
    // Use case: Creating unique identifiers from multiple inputs
    // ─────────────────────────────────────────────────────────────────
    function generateId(address user, uint256 nonce) public pure returns (bytes32) {
        return keccak256(abi.encodePacked(user, nonce));
    }
    // Different users + different nonces → completely different IDs.
    // Collision-resistant: virtually impossible for two inputs to produce same ID.

    // ─────────────────────────────────────────────────────────────────
    // EXAMPLE 3: Password storage WITHOUT storing the password
    // Use case: On-chain authentication — store hash, verify with hash
    // ─────────────────────────────────────────────────────────────────
    bytes32 public storedPasswordHash;

    function setPassword(string memory password) public {
        storedPasswordHash = keccak256(abi.encodePacked(password));
        // Never store raw password — only its hash. Even blockchain explorers
        // cannot reverse-engineer the original password from this hash.
    }

    function verifyPassword(string memory attempt) public view returns (bool) {
        return keccak256(abi.encodePacked(attempt)) == storedPasswordHash;
        // Compute hash of attempt, compare to stored hash. True if they match.
    }

    // ─────────────────────────────────────────────────────────────────
    // EXAMPLE 4: Commit-Reveal Scheme
    // Use case: MST FortunaX random number generation, sealed auctions
    // Phase 1 (COMMIT): Player hashes their secret choice + salt
    // Phase 2 (REVEAL): Player reveals choice + salt → contract verifies
    // ─────────────────────────────────────────────────────────────────
    mapping(address => bytes32) public commitments;

    function commit(bytes32 commitment) public {
        commitments[msg.sender] = commitment;
        // Player submits: keccak256(abi.encodePacked(choice, salt))
    }

    function reveal(uint256 choice, uint256 salt) public view returns (bool) {
        bytes32 computed = keccak256(abi.encodePacked(choice, salt));
        return computed == commitments[msg.sender];
        // True = player's revealed choice matches their original commitment.
        // They cannot change their choice after committing — the hash proves it.
    }
}
🗝️ Why abi.encodePacked() - Not Just keccak256(data)

keccak256() only accepts bytes as input. abi.encodePacked() converts any Solidity type (string, uint, address, bool, bytes) into tightly packed bytes that keccak256 can process. This is the standard pattern. Important warning: abi.encodePacked(A, B) can sometimes produce the same bytes for different A,B pairs (e.g., encodePacked("ab","c") vs encodePacked("a","bc")). For sensitive applications, use abi.encode() instead - it adds type information that prevents this collision.

08   Hashing Throughout the Blockchain Stack

Every Place Hashing Appears in a Blockchain System

1
Block Hashing - Creating the Chain
Each block's header fields are concatenated and hashed to produce the block's ID (its hash). This hash becomes the PrevHash of the next block - creating the immutable chain. In Bitcoin, miners must find a nonce such that SHA-256(SHA-256(block_header)) begins with a certain number of zero bits. Finding this nonce is "mining."
2
Transaction IDs (TXID)
Every transaction is hashed to produce its Transaction ID - the unique identifier you see on a block explorer. In Bitcoin: TXID = SHA-256(SHA-256(raw_transaction_bytes)). In Ethereum/MST: TXID = Keccak-256(RLP_encoded_transaction). This TXID is what you paste into MSTScan to look up a specific transaction.
3
Wallet Address Derivation
Your Ethereum/MST wallet address is derived from your public key using hashing. Process: Generate private key (random 256-bit number) → Compute public key (elliptic curve multiplication) → Keccak-256(public_key) → Take last 20 bytes → That's your address (0x...). The hash is one-way: nobody can compute your private key from your address.
4
Merkle Trees - Transaction Summaries
All transactions in a block are hashed, then pairs of hashes are hashed together, and so on - until a single "Merkle Root" hash summarises every transaction. This enables fraud detection (any tampered transaction changes the root) and efficient verification by light clients. Covered in depth in Sub-Module 3.8.
5
Smart Contract Storage Keys
In Ethereum/MST, the storage location of mapping values is computed as keccak256(key, slot). When you access mapping[address], the EVM computes the storage slot by hashing the key and the mapping's base slot number. This is why even complex nested mappings can be stored efficiently in the 256-bit address space.
6
Commit-Reveal in Gaming (MST FortunaX)
In MST's FortunaX random number protocol, the game operator commits a hash of a secret seed before the game begins. After the game, they reveal the seed. Players can verify: keccak256(revealed_seed) == committed_hash. If it matches, the seed was fixed before the game - the operator could not have chosen it based on player actions. This is provably fair randomness without a trusted random oracle.
09   Real-World Case Studies

Hashing in the Wild - India and Beyond

🇮🇳
Aadhaar - 1.3 Billion Biometric Hashes
UIDAI stores a biometric hash (not the fingerprint image) for all 1.3 billion enrolled Indians. The 2018 Aadhaar data breach exposed demographic data (names, addresses) but NOT biometric data - because only hashes were stored. Even if the hash database is stolen, no fingerprint can be reconstructed. This is hash security working exactly as designed. However, the breach did expose that the database was accessible - a governance failure, not a cryptographic one.
💬
WhatsApp Encryption - Hash Chains in Messaging
WhatsApp uses the Signal protocol, which uses HKDF (Hash-based Key Derivation Function - built on SHA-256) to derive new encryption keys for every single message. This is called "forward secrecy" - even if an attacker steals your current key, they cannot decrypt past messages because each message used a different derived key, and key derivation is one-way. The same hashing principle that secures blockchain transactions secures your WhatsApp messages.
📱
APK Integrity - Android App Downloads
When you download an APK (Android app package) - whether from the Play Store or directly - Google publishes its SHA-256 hash. Your device computes SHA-256 of the downloaded file and compares to the published hash. If they match, the file is authentic and unmodified. If they don't, the device warns you or refuses to install. This prevents "man-in-the-middle" attacks where a malicious party modifies the APK between Google's server and your phone.
10   Assessment Preparation

Top Facts - Know These Cold

01
SHA-256 = Bitcoin. Keccak-256 = Ethereum/MST.
Both produce 256-bit (32-byte = 64 hex character) outputs. SHA-256 was designed by the NSA. Keccak-256 won the NIST SHA-3 competition. Ethereum uses the Keccak variant - NOT the final NIST SHA-3 standard (they differ in padding).
02
The Five Properties
Deterministic, Pre-image Resistant (one-way), Avalanche Effect (sensitive), Collision Resistant, Fixed Output Size. If an exam asks "why can't you reverse a hash?" - the answer is Pre-image Resistance.
03
Solidity Syntax
keccak256(abi.encodePacked(data)) - returns bytes32. The function is pure (no state read/write). Use abi.encode() instead of encodePacked when hashing multiple variable-length arguments to avoid collisions.
04
Output Size - Always 256 bits
SHA-256 and Keccak-256 always produce exactly 256 bits regardless of input size. Hash "Hi" (2 bytes) or "The complete works of Shakespeare" (5 MB) - both produce 64 hex characters. This is why the PrevHash field in a block header is always exactly 32 bytes.
05
Commit-Reveal Pattern
Phase 1 (Commit): Publish hash(secret). Phase 2 (Reveal): Publish secret. Verifier checks: hash(secret) == commitment. Used by MST FortunaX for provably fair random numbers - the secret is fixed before the game; hashing proves you can't change it after.
11   Glossary

Key Terms - Sub-Module 3.3

Cryptographic Hash Function
A mathematical function that converts any input to a fixed-size output with five properties: deterministic, pre-image resistant, avalanche effect, collision resistant, fixed output size.
SHA-256
Secure Hash Algorithm 256-bit. Bitcoin's primary hash function. Produces 64 hex characters. Designed by NSA, standardised by NIST in 2001.
Keccak-256
Ethereum and MST's hash function. Used in Solidity as keccak256(). Won the SHA-3 competition in 2012. Note: slightly different from NIST SHA-3 due to padding.
Pre-image Resistance
The one-way property - given a hash output, it is computationally infeasible to find any input that produces it. Makes hash storage secure against database breaches.
Avalanche Effect
A tiny change in input (even one bit) causes a completely different hash output - roughly half the output bits flip. Makes subtle undetectable tampering impossible.
Collision Resistance
Computationally infeasible to find two different inputs that produce the same hash output. SHA-256 requires ~2¹²⁸ operations to find a collision - far beyond any known attack.
abi.encodePacked()
Solidity function that tightly packs multiple values into bytes for hashing. Used as: keccak256(abi.encodePacked(a, b)). Warning: can cause ambiguity with variable-length types - use abi.encode() for critical security applications.
Commit-Reveal Scheme
A two-phase protocol using hashing: Phase 1 - commit by publishing hash(secret). Phase 2 - reveal the secret. Used in MST FortunaX for provably fair random number generation.
