03   Background & Design Rationale

Why “Just Add a New Signature” Doesn’t Work - and How MST Solved It

Most blockchain quantum-resistance proposals fall into one of two traps: either they require a disruptive hard fork that splits the community, or they ask every user to manually migrate funds to a new address type - a UX catastrophe that would lock millions of non-technical users out of their assets. MST’s engineering team studied these failures and designed a backward-compatible, soft-fork-based migration that works with the existing consensus, wallet, and smart-contract infrastructure.

1
The Dual-Stack Transaction Format
MST’s transaction structure was extended with an optional pq_signature field. Before the transition height, this field is ignored. After the transition height, validators require it and verify it against a registered post-quantum public key. This means old wallets continue to work until the transition, and new wallets can add the extra signature immediately.
2
No New Address Format - Just New Keys
Instead of creating a separate “pq1…” address type, MST binds a user’s post-quantum public key to their existing ECDSA address through an on-chain key registry contract. When you deploy a SARAL wallet, a small identity contract is created that maps your address to both keys. Validators look up the pq key during signature verification - no address change, no UX disruption.
3
Governance-Scheduled Transition
The transition block height is set by MST’s on-chain governance. This allows the community to react to real-world quantum developments - if a breakthrough is announced, the height can be accelerated. The default schedule assumes a conservative 18-month lead time from governance proposal to activation, with a 6-month hybrid window after activation.
🗝️ The Soft-Fork Principle

A soft fork means that the new rules are a subset of the old rules. Before the transition, hybrid transactions are valid under both old and new rules. After the transition, only hybrid (or fully pq) transactions are accepted by upgraded nodes. Non-upgraded nodes see blocks as valid because they still observe the old consensus rules, but they can’t produce blocks without the pq signature. This prevents a chain split - a crucial requirement for financial networks.

04   Technical Deep Dive

Hybrid Signatures: From Wallet to Block Finality

Let’s walk through the life of a hybrid transaction, from the moment a user taps “Send” to the moment it’s sealed in a block.

FIGURE 17.2-A · Hybrid Transaction Lifecycle
  USER WALLET                          VALIDATOR NODE                       BLOCK
  ────────────                         ────────────────                     ─────
  1. User taps Send
  2. SARAL retrieves:
       - ecdsa_private_key (from enclave)
       - dilithium_private_key (from enclave)
  3. Wallet constructs tx:
       tx.body = {to, amount, nonce, ...}
       tx.sig_ecdsa = sign_ecdsa(tx.body, ecdsa_key)
       tx.sig_dilithium = sign_dilithium(tx.body, dilithium_key)
  4. Tx broadcast to mempool
                                          5. Node receives tx.
                                             If block.height >= TRANSITION_HEIGHT:
                                                 - Fetch user's pq_public_key from registry contract
                                                 - Verify ecdsa_sig against user's ecdsa_address
                                                 - Verify dilithium_sig against user's pq_public_key
                                                 - Both must be valid → tx enters candidate block
                                             Else:
                                                 - Only ecdsa_sig required.
                                          6. Validator proposes block containing tx.
                                          7. Block is signed by validator with
                                             its own hybrid signature (ECDSA + Falcon).
                                          8. Block finalized and added to chain.
All legacy transactions (no pq_sig) are rejected after the transition height. This enforces quantum-safe transaction submission for every user who has registered a pq key.
💻 Key Registry Contract Interface (Solidity)
// Minimal identity contract that binds a user's post‑quantum key
interface IPQKeyRegistry {
    function register(bytes calldata pqPublicKey) external;
    function getPQKey(address user) external view returns (bytes memory);
    event PQKeyRegistered(address indexed user, bytes pqPublicKey);
}

// Validator logic (simplified):
function verifyHybridSignature(address user, bytes32 txHash, bytes memory ecdsaSig, bytes memory pqSig) internal view {
    require(verifyECDSA(user, txHash, ecdsaSig), "ECDSA invalid");
    bytes memory pqKey = pqRegistry.getPQKey(user);
    require(pqKey.length > 0, "No PQ key registered");
    require(verifyDilithium(pqKey, txHash, pqSig), "Dilithium invalid");
}

What if a user hasn’t registered a PQ key? After the transition height, unregistered addresses cannot initiate transactions. However, a “force-migrate” relay contract allows anyone to call a function that uses the user’s pre-authorised migration allowance (set during SARAL wallet creation) to move assets to a fresh post-quantum address. This ensures no funds are permanently locked, even if the original owner lost access.

05   SARAL Wallet Quantum Migration

How Your Wallet Silently Becomes Quantum-Safe

From the user’s perspective, nothing changes. But under the hood, the wallet performs a choreographed key rotation that would take a manual user hours of terrifying copy-pasting.

1
Onboarding (already quantum-ready)
When a new user creates a SARAL wallet via Google/Facebook login, the secure enclave generates two keypairs: an ECDSA pair for immediate use, and a CRYSTALS‑Dilithium pair for future use. The Dilithium public key is immediately registered on-chain via a meta-transaction (gas sponsored by MST’s onboarding fund). The user never sees either private key.
2
Receiving the “Transition Notice”
Six months before the transition height, SARAL displays a non-intrusive banner: “MST is upgrading to quantum-safe signatures. Your wallet is already ready.” No action required. In the background, the wallet begins signing transactions with both keys immediately, even before the transition height, to test hybrid propagation on mainnet.
3
At Transition Height
The network begins enforcing hybrid signatures. SARAL wallets automatically append the Dilithium signature to every transaction. Legacy wallets that haven’t been upgraded stop working - but users can recover their funds through the force-migrate relay using their original SARAL biometric authentication. The migration is complete without a single seed phrase written down.

  DAY 0 (User creates wallet)
  ───────────────────────────
    Enclave generates:
      ecdsa_priv, ecdsa_pub
      dilithium_priv, dilithium_pub

The entire process is invisible to the end user, yet mathematically guarantees that a quantum adversary cannot forge a transaction from this wallet after the transition.
06   Validator Consensus Upgrade

Trika POSA Validators: From ECDSA to Falcon

Securing user transactions is only half the battle. The consensus layer - where validators propose and finalise blocks - must also become quantum-resistant. If a quantum adversary can forge a validator’s block signature, they can finalise malicious blocks, double-spend, or halt the chain.

1
Validator Key Registration (Pre-Transition)
Six months before the transition height, validators must submit a Falcon public key (or Dilithium, depending on governance) to a dedicated validator registry contract. They continue signing blocks with ECDSA. The network enters a “shadow-signing” phase: validators optionally append a Falcon signature to block proposals, and other validators verify it in addition to ECDSA. This tests the new signature scheme under live load without enforcement.
2
Dual-Signature Consensus (Transition Height)
At the transition height, the consensus rules change: every block proposal must carry a valid Falcon signature from the proposer, in addition to the ECDSA signature. Validators that have not registered a pq key are excluded from the active set. The fork-choice rule remains unchanged - the chain with the most stake-weighted votes wins - but votes themselves are now hybrid-signed.
3
Post-Transition (ECDSA Dropped)
After a second governance-defined height (typically 6 months after the first), validators stop requiring ECDSA signatures on blocks. The block size shrinks, throughput increases slightly, and the consensus layer is fully post-quantum. Any validator still relying on ECDSA is slashed for failing to produce valid blocks.
💡 Why Falcon for Validators?

Falcon signatures are ~700 bytes vs Dilithium’s ~2.5 KB. Validators sign hundreds of blocks per epoch, so bandwidth and verification speed matter. Falcon’s smaller signatures reduce block propagation time, keeping MST’s 3-second block time intact. Dilithium is used for user transactions because its simpler signing algorithm is more suitable for mobile wallets and secure enclaves with limited computational power.

09   Real-World Connection

What This Means for Developers Like You

As a developer building on MST, the quantum-ready pillar gives you a superpower: you can deploy a smart contract today, add a simple postQuantumSigner modifier, and know that your contract will still be secure decades from now. Let’s make it concrete.

🛠️ Smart Contract with Quantum-Enforced Access
contract TimeLockedVault {
    IPQKeyRegistry public pqRegistry;
    address public owner;
    uint256 public unlockTime;

    modifier onlyOwnerQuantumSafe() {
        require(msg.sender == owner, "Not owner");
        require(pqRegistry.getPQKey(msg.sender).length > 0, "Owner must register PQ key");
        _;
    }

    function withdraw() external onlyOwnerQuantumSafe {
        require(block.timestamp >= unlockTime, "Too early");
        payable(owner).transfer(address(this).balance);
    }
}

The onlyOwnerQuantumSafe modifier ensures that even if the owner’s ECDSA key is compromised (quantum attack), an attacker cannot call withdraw because they would also need to forge a valid Dilithium signature - which remains quantum-hard. By deploying with this pattern, BharatFi’s vaults are immune to the Y2Q break.

Action item: In your capstone project, add a quantum-safe modifier to any contract that holds long-term assets. Use MST’s IPQKeyRegistry interface to check that the caller has registered a post-quantum key.

11   Glossary

Key Terms - Sub-Module 17.2

Hybrid Transaction
A transaction containing both an ECDSA signature and a post-quantum signature, enforced during MST’s quantum migration window.
Key Registry Contract
An on-chain contract that stores the mapping from an MST address to its registered post-quantum public key.
Transition Height (Hₜ)
The block number at which validators begin enforcing hybrid signatures. Set by on-chain governance.
Shadow-Signing
A pre-transition phase where validators optionally append post-quantum signatures to test infrastructure without enforcement.
Force-Migrate Relay
A permissionless contract that moves assets from a non-upgraded user’s address to a quantum-safe address, using pre-authorised allowances.
Dilithium
A NIST-standardised lattice-based digital signature scheme used for user transactions in MST’s post-quantum system.
Falcon
A lattice-based signature scheme with compact signatures, used by MST validators for block signing due to its efficiency.
Soft Fork
A protocol upgrade where new rules are a subset of old rules; non-upgraded nodes still accept the new blocks, avoiding a chain split.
