03   Why Auditing Matters

The Economic Reality of Smart Contract Bugs

Smart contracts are immutable and often handle millions of dollars. Unlike Web2 applications, a single bug cannot be patched with a server update - it requires a complex governance proposal, a migration of funds, or, in the worst case, a hard fork. The cost of a single missed vulnerability can be existential. Automated analysis tools don’t replace human reasoning, but they dramatically increase the surface area of bugs you can find before an attacker does.

Real-world audits have revealed critical flaws in multi-billion-dollar protocols: the Poly Network hack ($610M) stemmed from a privilege escalation bug; the Wormhole bridge exploit ($326M) was a signature verification bypass; the Euler Finance attack ($197M) exploited a missing health-factor check. In every case, a thorough audit - especially one that combined static analysis with manual reasoning - would likely have prevented the loss. As a developer graduating from this program, you are not just writing code; you are responsible for assets that may one day exceed your entire net worth. Internalizing audit discipline now protects both your users and your career.

Moreover, investors and grant committees increasingly require audit reports before releasing funds. An MST scaleGrant review explicitly checks whether the applicant has performed a self-audit and engaged a third party. Your capstone audit report is therefore not just an assignment - it is a launchpad for real funding.

⚠️ The Uncomfortable Truth

No tool catches every bug. Slither and Mythril can detect many common vulnerability patterns, but they cannot replace expert human review. They will flag syntactic issues and known patterns, yet subtle logic bugs - like an incorrect fee calculation or a misaligned economic incentive - require a developer’s deep understanding of the protocol’s intended behavior. This sub-module teaches you to combine automated tools with manual reasoning, the same methodology used by firms like Trail of Bits and OpenZeppelin.

04   Setting Up Your Audit Toolkit

Slither, Mythril & Echidna: Your Automated Security Squad

1
Slither - Static Analysis for Solidity

Developed by Trail of Bits, Slither is the industry-standard static analyzer for Solidity. It parses the abstract syntax tree (AST) and builds a control-flow graph, enabling it to detect reentrancy, uninitialized storage variables, shadowed variables, unsafe low-level calls, and many other bug classes. Because it works directly on Solidity code without executing transactions, Slither returns results almost instantly - perfect for a CI/CD pipeline.

Slither’s detector library is extensive. Each detector targets a specific vulnerability: reentrancy-eth, unchecked-lowlevel, missing-zero-check, suicidal, arbitrary-send, etc. You can run all detectors with a single command or select a subset. The output includes a description, the exact code lines, and a reference link to a detailed documentation page. It also supports custom plugins, so you can write your own analysis rules for project-specific invariants.

2
Mythril - Symbolic Execution & Taint Analysis

Mythril, developed by ConsenSys, takes a different approach: symbolic execution. It translates smart contract bytecode into logical formulas and explores every possible execution path, treating inputs as symbolic variables. When it finds a path that violates a security property (e.g., an assertion failure, a selfdestruct call by an unauthorized user), it generates the exact transaction sequence needed to trigger it.

This makes Mythril particularly effective at finding multi-transaction exploits, complex reentrancy across multiple contracts, and integer overflow paths that static analysis might miss. However, symbolic execution is computationally expensive; Mythril may need several minutes per contract, and you sometimes have to tune the timeout and transaction count to avoid endless loops. Still, it is an invaluable complement to Slither.

3
Echidna - Property-Based Fuzzing

Echidna, also from Trail of Bits, flips the script: instead of looking for specific known bugs, it lets you define invariants (e.g., “totalSupply never exceeds initialSupply,” “only the owner can withdraw,” “the sum of all user balances equals the contract’s balance”). Then it generates random sequences of valid transactions - deposits, withdrawals, transfers, liquidations - and checks whether any sequence breaks your invariants.

Fuzzing is brutally effective at finding edge cases that neither static analysis nor human review would ever consider. Echidna supports assertion-checking, custom property functions, and even symbolic execution integration. While it is optional for this sub-module, I strongly encourage you to try it on your capstone’s core financial logic; many production protocols now run Echidna campaigns daily as part of their CI.

💻 Installation Commands
# Install Slither (requires Python 3.8+)
pip3 install slither-analyzer

# Install Mythril
pip3 install mythril

# Install Echidna (via Docker)
docker pull ghcr.io/crytic/echidna/echidna:latest

# To run Echidna after pulling:
docker run -it ghcr.io/crytic/echidna/echidna:latest

Ensure your system has solc (the Solidity compiler) available in the PATH. Slither and Mythril will invoke the compiler automatically based on the pragma in your contracts. If your project uses multiple compiler versions, ensure you have them installed or use a version manager like solc-select.

05   Running Slither on Your Capstone

From Source Code to Actionable Bug List in 5 Seconds

Navigate to your project’s root directory and run:

💻 Slither Command
slither . --print human-summary

Slither will compile your contracts (using Hardhat or solc) and output a list of detected issues, each with a severity (High/Medium/Low/Informational) and a reference to the exact line of code.

Let’s walk through a typical output. Suppose Slither finds a reentrancy in your withdraw function. The report will look like this:

FIGURE 17.3-A · Sample Slither Output - Reentrancy Detected
  INFO:Detectors:
  Reentrancy in MyLending.withdraw(address,uint256) (contracts/Lending.sol#42‑58):
    External calls:
    - (success) = msg.sender.call{value: amount}() (contracts/Lending.sol#53)
    State variables written after the call(s):
    - balances[msg.sender] = 0 (contracts/Lending.sol#58)
  Reference: https://github.com/crytic/slither/wiki/Detector-Documentation#reentrancy-vulnerabilities
Slither pinpointed a classic reentrancy bug where state is updated after an external call - the exact pattern that led to the $60M DAO hack.

For each finding, ask yourself: Can this actually be exploited in my deployment context? A reentrancy in a function that only the owner can call might be low risk. But a public withdrawal function? Critical. Slither will also flag “informational” items like missing docstrings or unused variables - fix them for completeness, but they don’t indicate security flaws.

To get a deeper view, run slither . --print vars-and-auth to see which functions can be called by which roles. This can reveal missing access controls that the detectors missed. You can also output all findings as a JSON file for programmatic processing: slither . --json output.json.

06   Running Mythril for Deeper Symbolic Analysis

Going Beyond Static: Finding Multi-Transaction Exploits

Mythril works best on individual contracts. Run it on your most critical contract (e.g., the lending core):

💻 Mythril Command
myth analyze contracts/Lending.sol --execution-timeout 60

Mythril symbolically executes the contract, considering all possible inputs. If it finds a path that leads to, say, the contract’s balance being drained, it will output the exact transaction sequence to trigger it.

FIGURE 17.3-B · Mythril Exploit Trace - Unprotected Selfdestruct

  SWC ID: 106
  Severity: High
  Contract: Lending
  Function name: emergencyStop()
  PC address: 451
  The contract can be killed by anyone.
  Anyone can call emergencyStop() and selfdestruct the contract, sending all ether to an arbitrary address.

  Initial State:
    Contract balance: 100 ETH
    Owner: 0x5B38Da6a701c568545dCfcB03FcB875f56beddC4

  Transaction Sequence:

A single forgotten access control modifier could let anyone destroy your contract and steal all assets. Mythril catches exactly that.

When reading Mythril output, note the “Transaction Sequence” carefully. It often involves multiple calls across different functions. Try to simulate the sequence mentally or in a local Hardhat node to confirm the exploit. Sometimes Mythril’s generated input may be impractical (e.g., requiring impossibly precise block timestamps), but it’s still a strong indicator that access controls are insufficient.

07   Manual Audit Checklist

The 10-Point Human Review Every Developer Must Do

Automated tools are fast, but they can’t reason about business logic. Use this checklist after your tools finish. For each point, you should physically walk through the relevant functions in your code and ask the questions below.

1
Access Control
Every state-changing function must have an appropriate modifier (onlyOwner, onlyRole). Verify that admin functions cannot be called by regular users. Check that the constructor sets the owner correctly and that there are no missing initializers in upgradeable contracts. Also verify that role-based access (e.g., OpenZeppelin’s AccessControl) is correctly configured with unique role identifiers.
2
Reentrancy
Apply the Checks-Effects-Interactions pattern: perform all validations, update internal state (e.g., deduct balance), and only then make external calls. If your logic requires interaction before state updates, use OpenZeppelin’s ReentrancyGuard. Verify that reentrancy guards are on every function that transfers value or tokens, including those that call untrusted contracts (e.g., hooks). Also check for cross-function reentrancy where two functions share state and can be called in sequence.
3
Integer Overflow / Underflow
Solidity ≥0.8 has built-in overflow checks, but beware of assembly blocks (Yul) or explicit unchecked blocks. If you use older compiler versions, verify SafeMath is applied consistently. Also check for unsafe type conversions (e.g., uint256 to uint128) - always use OpenZeppelin’s SafeCast. Examine loops that accumulate values: they could overflow the gas limit or the numeric type.
4
Front-Running & MEV
Can an attacker observe a pending transaction and profit by submitting their own first? Analyze the mempool visibility: any transaction with a predictable outcome can be front-run. Mitigations include commit-reveal schemes, time-weighted average price (TWAP) oracles, and accepting slippage parameters. Also consider sandwich attacks on AMM trades: does your DEX integration use a sensible minAmountOut?
5
Denial of Service (Gas Griefing, Block Stuffing)
Loops over unbounded arrays can exceed block gas limits, permanently locking functions. Use pull-over-push for payments: let users withdraw funds themselves instead of the contract looping over recipients. Avoid dynamic arrays that grow indefinitely; consider using mappings with iteration patterns (like OpenZeppelin’s EnumerableSet). Also, check that external calls cannot be forced to revert, blocking progress (e.g., a malicious token with a failing transfer).
6
Unchecked External Calls
Always check the return value of low-level calls (call, delegatecall). Avoid relying on transfer and send due to gas stipend limitations; prefer checked low-level call patterns (require(success, "call failed")). When interacting with unknown ERC-20 tokens, use OpenZeppelin’s SafeERC20 to handle non-standard implementations (USDT, for example, does not return a boolean).
7
Flash Loan Attacks
Can a malicious actor use a flash loan to manipulate prices in an AMM and then exploit your contract within a single block? Always fetch prices from reliable oracles (Chainlink, TWAP) - not from a spot DEX reserve. If your contract relies on a liquidity pool’s price, ensure it uses a time-weighted average. Check that governance actions (e.g., changing collateral factors) have timelocks to prevent instantaneous attack execution.
8
Upgradeability Safety
If using proxy patterns (UUPS, Transparent, Beacon), ensure storage layout compatibility between implementation versions. No constructors in implementation contracts - use initializers instead. The proxy admin must be a secure multisig; avoid centralized upgrade keys. Test upgrades in a production-like environment, including state migration scripts. Verify that the implementation contract cannot be selfdestructed (use OpenZeppelin’s UUPSUpgradeable correctly).
9
Event Emission
Every state-changing function should emit an event. This is critical for off-chain monitoring, subgraph indexing, and user transparency. Index important parameters (addresses, token IDs) for efficient filtering. Events should be emitted before external calls to follow the Checks-Effects-Interactions pattern.
10
Business Logic Consistency
Does the implementation match the specification? Walk through the intended user flow and compare it to the code. Often, logic bugs are invisible to tools. For example: does the fee calculation use basis points correctly? Are withdrawal limits applied before or after the withdrawal? Is the auction end time exclusive or inclusive? These details matter hugely in production. Write a short script (in Hardhat or Foundry) that simulates the complete user lifecycle and verifies that all balances and states end up as expected.

Spend at least 30 minutes on this checklist. Keep a text file open and note every deviation you find, even if it seems minor. Many of the worst hacks started with a “low-severity” bug that was chained with another seemingly harmless issue.

08   Writing the Audit Report

From Findings to a Document That Investors Trust

A professional audit report is not a dump of tool output. It is a structured document that communicates risk to both technical and non-technical stakeholders. Follow this gold-standard template - the same structure used by Trail of Bits, ConsenSys Diligence, and OpenZeppelin.

A
Executive Summary
One paragraph. How many contracts were audited? How many lines of code? How many findings (by severity: Critical, High, Medium, Low, Informational)? Is the code ready for deployment, or are there critical blockers? State the final recommendation clearly. Example: “We audited 4 contracts totaling 820 lines of Solidity. We discovered 2 high-severity, 3 medium, and 5 low issues. After fixes, all high and medium issues were resolved. The protocol is ready for mainnet deployment with the recommended mitigations in place.”
B
Audit Scope & Methodology
List the exact contracts and file paths. State which tools were used (Slither, Mythril, Echidna if used), the manual review approach (checklist, code walkthrough, peer review), and any limitations (e.g., “We did not perform formal verification”). Also mention the commit hash of the audited codebase for reproducibility.
C
Findings (Ordered by Severity)
Each finding includes: Severity (Critical/High/Medium/Low/Informational), Description (plain English, what is wrong), Affected Code (snippet with file name and line numbers), Exploit Scenario (step-by-step how an attacker would exploit it), Recommendation (specific code fix), and Status (Fixed, Acknowledged, Disputed). Use a consistent severity classification (see glossary). Critical findings must be fixed before launch.
D
Tool Output Appendix
Include the raw Slither and Mythril logs (and Echidna if used) as an appendix. This shows due diligence and allows others to replicate your analysis. Also include the “after-fix” re-run outputs to prove issues were resolved.
E
Conclusion
A brief paragraph summarizing the overall security posture, any remaining risks, and final deployment recommendation. If there are acknowledged issues, explain why they are acceptable (e.g., low probability, requires admin compromise).
💡 Pro Tip

A good audit report demonstrates not just that you found bugs, but that you understood their impact and fixed them correctly. Include the commit hash of the fix for each issue. This turns your report into a verifiable artifact of your engineering rigor.

09   Practical Walkthrough

Auditing Your Capstone: Step-by-Step

Let’s assume your capstone is a simple NFT marketplace with minting, listing, and buying. Here’s how you’d audit it end-to-end.

FIGURE 17.3-C · Audit Workflow for Your Capstone
  1. Install tools (Slither, Mythril) as described above.
  2. Run Slither across the whole project:
         slither . --print human-summary
  3. For each high‑severity finding, open the file and understand the bug.
  4. Run Mythril on the core contract (Marketplace.sol):
         myth analyze contracts/Marketplace.sol --execution-timeout 60
  5. For each Mythril finding, simulate the transaction sequence mentally.
  6. Go through the manual checklist (access control, reentrancy, etc.).
  7. Document all findings in a preliminary table.
  8. Fix each real vulnerability in your smart contract code.
  9. Re‑run Slither and Mythril to verify fixes; ensure no high/medium findings remain.
  10. (Bonus) Write an Echidna invariant test for a critical invariant (e.g., "Marketplace contract balance never negative").
  11. Compile the final audit report in PDF format.
  12. Commit all changes and note fix commit hashes in the report.

During manual review, pay special attention to functions that move funds: buyItem(), withdrawFunds(), royaltyPayments. Verify that the seller receives the correct amount after deducting marketplace fees, that royalties are sent to the correct address, and that re-entrancy cannot occur when transferring ETH to multiple parties in sequence.

If your capstone uses SARAL for identity, also verify that the quantum-safe modifier from Sub-Module 17.2 is correctly applied. If you integrated WASMify oracles, confirm that the data validation logic rejects impossible values (e.g., a temperature of −1000°C). Every external data source is a potential attack vector.

10   Assessment Preparation

Key Reminders Before You Submit

01
Tool output must be genuine
Run the tools on your actual capstone code. Fabricated output is easily detected and will result in a zero.
02
Severity classification is your responsibility
Tools may label everything “High.” You must apply the classification based on real exploitability in your context. A missing docstring is not High severity.
03
Fix then verify
After applying fixes, re-run the tools and include the new output. The report must show that critical/high findings were resolved or severity reduced.
04
Business logic counts
A human-identified business logic bug (e.g., incorrect fee calculation) is just as valid as a tool-found reentrancy. Document it with the same rigor.
11   Glossary

Key Terms - Sub-Module 17.3

Static Analysis
Analysis of code without executing it, by examining structure, data flow, and patterns. Slither is a static analyzer.
Symbolic Execution
A technique that treats inputs as symbolic variables and explores all possible execution paths. Mythril uses this to find multi-step exploits.
Fuzzing (Property-Based)
A testing method that feeds random inputs to a program and checks whether user-defined invariants hold. Echidna is a fuzzer for EVM.
Reentrancy
A vulnerability where an external call allows an attacker to re-enter the calling contract before state updates are complete, potentially draining funds.
Checks-Effects-Interactions
A secure coding pattern: perform all checks, update contract state, then make external calls. Prevents reentrancy.
Access Control
Restricting who can call a function. Often implemented with modifiers like onlyOwner or role-based systems (OpenZeppelin AccessControl).
Severity Classification
Categorizing a vulnerability’s impact: Critical (direct fund loss, no constraints), High (fund loss with constraints), Medium (indirect loss, DoS), Low (best practice violation), Informational (code style).
Audit Report
A structured document summarizing all vulnerabilities found, their exploitability, remediation, and final status, along with methodology and tool output.
