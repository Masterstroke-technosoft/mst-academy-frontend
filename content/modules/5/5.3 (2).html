03   Background & History

From Brain Wallets to BIP39 - The Quest for Recoverable Keys

In Bitcoin's earliest days, wallets were simply files containing randomly generated private keys. There was no standard way to back them up, no human-readable recovery mechanism, and no way to derive multiple keys from a single secret. If you lost your `wallet.dat` file, you lost your coins. If you wanted multiple addresses for privacy, you had to generate and store each private key separately. This was a usability disaster waiting to happen.

The first major improvement came with Brain Wallets - the idea that a human-memorable passphrase could be hashed to produce a private key. The passphrase "correct horse battery staple" could be turned into a deterministic key. However, humans are terrible at generating entropy. Passphrases like "ilovebitcoin" or "password123" were cracked in seconds by attackers running dictionary attacks, resulting in millions of dollars in stolen Bitcoin.

In 2012, Bitcoin Improvement Proposal 32 (BIP32) introduced Hierarchical Deterministic (HD) Wallets. Instead of random independent keys, an HD wallet uses a single master seed to derive a tree of key pairs. This means one backup (the seed) can recover every address the wallet will ever generate. BIP32 defined the mathematical structure: a root master private key and a chain code, from which child keys are derived using HMAC-SHA512. Each child can derive grandchildren, forming an infinite tree. A parent key can generate child public keys without exposing the parent private key, enabling powerful audit and accounting features.

In 2013, BIP39 (Mnemonic Code for Generating Deterministic Keys) added the crucial usability layer. Instead of backing up a long hex string like `0x1a2b3c...`, users could write down 12 or 24 common English words - like "abandon ability able about above absent absorb abstract absurd abuse access accident". These words encode a specific entropy, which is then combined with an optional passphrase to produce the BIP32 master seed. BIP39 also introduced a wordlist of 2048 words, each representing 11 bits of entropy. A 12-word mnemonic encodes 128 bits of entropy (plus 4 bits of checksum), providing 2¹²⁸ possible combinations - astronomically secure against brute force.

Finally, BIP44 (Multi-Account Hierarchy for Deterministic Wallets) defined a standard derivation path structure: `m / purpose' / coin_type' / account' / change / address_index`. This allowed a single seed to manage multiple cryptocurrencies (BTC, ETH, MSTC) across multiple accounts, with clear separation between external (receiving) and internal (change) addresses. The apostrophe indicates hardened derivation, which prevents child keys from being derived from parent public keys - a critical security feature.

Today, nearly every wallet - MetaMask, Trust Wallet, Ledger, Trezor, and MST's own BridgeKey wallet - uses the BIP39 → BIP32 → BIP44 pipeline. A 12-word phrase written on a piece of paper is the root of a cryptographic tree that can generate billions of addresses across dozens of blockchains, all recoverable from those twelve words alone.

04   Beginner-Friendly Explanation

Twelve Words = Infinite Money - How a Seed Phrase Works

🌱 The Analogy

Think of your seed phrase as a DNA sample. From a single strand of DNA, an entire organism can develop - every cell, every organ, every fingerprint. Similarly, from your 12-word seed phrase, your wallet can mathematically generate every private key, public key, and address you will ever need - across Bitcoin, Ethereum, MST, and hundreds of other chains. The seed phrase is the DNA; the wallet software is the developmental process; your addresses are the cells. If you have the DNA, you can rebuild the entire organism. If you lose the DNA, the organism can never be reconstructed.

Let's walk through the transformation chain step by step, starting from randomness and ending at a usable address:

①
ENTROPY
What is entropy and why does it matter?
Entropy is randomness - the unpredictability that makes your keys secure. A wallet starts by generating 128 bits (for 12 words) or 256 bits (for 24 words) of cryptographically random data. This randomness must come from a secure source - the operating system's random number generator (`/dev/urandom` on Linux, `crypto.randomBytes` in Node.js). If the entropy is predictable (e.g., based on the current time), an attacker can guess it and steal everything. The security of your entire wallet rests on the quality of these initial random bits.
②
MNEMONIC
How is entropy turned into words?
The entropy is divided into 11-bit chunks. Each 11-bit number (0 - 2047) maps to a word from the BIP39 English wordlist - a standardized list of 2048 words. For 128 bits, that's 11 chunks (121 bits) plus a 4-bit checksum (the first 4 bits of SHA256(entropy)) to make 132 bits = 12 words. The wordlist is carefully chosen: no two words have the same first four letters, making it easy to auto-complete. Words like "abandon" and "zoo" are deliberately distinct. A 12-word mnemonic has 2¹²⁸ possible combinations - more than the number of atoms in the observable universe. Brute-forcing it is physically impossible.
③
SEED
How does the mnemonic become a seed?
The mnemonic is fed into PBKDF2 (Password-Based Key Derivation Function 2) with HMAC-SHA512, using the string "mnemonic" + an optional passphrase as the salt, and iterating 2048 times. This produces a 512-bit seed. The passphrase is crucial: it acts as a "13th word" (or "25th word" for 24-word mnemonics). An empty passphrase yields the default wallet; a different passphrase yields a completely different wallet from the same mnemonic. This allows for plausible deniability: you can store a small amount in the default wallet and the real funds in a passphrase-protected wallet.
④
MASTER KEY
How does the seed become a master key?
The 512-bit seed is split in half: the left 256 bits become the master private key, and the right 256 bits become the chain code. The master private key can be used to derive child keys; the chain code ensures that each child's derivation is unique and that children cannot be derived from public keys alone (when hardened). This master key pair is the root of the HD tree.
⑤
CHILD KEYS
How are individual addresses created?
Using the BIP44 path `m/44'/60'/0'/0/0` (for the first Ethereum-compatible account on the first external address), the wallet iteratively applies HMAC-SHA512 to derive child private keys. The apostrophe (`'`) indicates hardened derivation, which means the parent private key is used - this prevents an attacker who knows a child private key and the parent chain code from deriving sibling keys. Each derived private key can then be converted to a public key using elliptic curve multiplication (secp256k1), and then hashed with Keccak-256 to produce the 20-byte address. The final address is the last 20 bytes of the hash, prefixed with `0x`. This whole process is deterministic: given the same mnemonic and path, you'll always get the same address.
05   Technical Deep Dive

The Mathematics of HD Wallets - BIP32, BIP39, BIP44 in Code

Let's now examine the precise algorithms that turn twelve words into a secure, recoverable wallet. We'll focus on the most common implementation: Ethereum-compatible wallets (like MetaMask and MST's BridgeKey) that use the derivation path `m/44'/60'/0'/0/0`.

BIP39: Mnemonic Generation Step by Step

  1. Generate entropy. Produce 128 bits (16 bytes) of cryptographically secure random data. In Node.js: `const entropy = crypto.randomBytes(16);`
  2. Compute checksum. Take the SHA-256 hash of the entropy. The first `entropy_length / 32` bits are the checksum. For 128 bits, that's 4 bits. So checksum = `sha256(entropy).slice(0, 1) >> 4` (the first 4 bits of the first byte).
  3. Concatenate. Append the checksum bits to the entropy, resulting in 132 bits (128 + 4).
  4. Split into 11-bit chunks. 132 bits ÷ 11 bits = 12 chunks. Each chunk is an integer from 0 to 2047.
  5. Map to words. Each integer is used as an index into the BIP39 wordlist (a sorted array of 2048 English words). For example, index 0 → "abandon", index 1 → "ability", index 2047 → "zoo".
  6. Join with spaces. The 12 words are concatenated with a space separator, forming the mnemonic sentence.

BIP39: Mnemonic to Seed (PBKDF2)

The mnemonic sentence is used to derive a 512-bit seed via PBKDF2:

seed = PBKDF2(password: mnemonic, salt: "mnemonic" + passphrase, iterations: 2048, keylen: 512 bits, hash: HMAC-SHA512)

The salt is the string `"mnemonic"` concatenated with the passphrase (which defaults to empty string if not provided). The 2048 iterations make brute-forcing the passphrase computationally expensive, adding a layer of security even if the mnemonic is exposed. The result is 64 bytes (512 bits) of pseudo-random data.

BIP32: Master Key Derivation from Seed

The 64-byte seed is used to generate the master private key and master chain code using HMAC-SHA512 with the key `"Bitcoin seed"` (yes, even for Ethereum, the string is literally `Bitcoin seed` - a historical artifact):

I = HMAC-SHA512(key: "Bitcoin seed", data: seed)
master private key = I[0:32]
master chain code = I[32:64]

The master private key is a 256-bit integer, which is then used as the starting point for the elliptic curve secp256k1. The master public key is derived by multiplying the generator point G by this integer: `master_public_key = master_private_key * G`.

BIP32: Child Key Derivation (CKD)

To derive a child key at index `i`, the parent's private key, chain code, and the index are used:

  • If hardened (i >= 2³¹): data = 0x00 || parent_private_key || i (big-endian 4 bytes). This uses the parent private key, so a child key cannot be derived from a parent public key - it's hardened.
  • If non-hardened (i < 2³¹): data = parent_public_key (compressed) || i. This allows public derivation (useful for watch-only wallets).

The HMAC-SHA512 is computed with the chain code as the key: `I = HMAC-SHA512(key: chain_code, data: data)`. The left 32 bytes of I are the child private key (added to parent private key modulo the curve order), and the right 32 bytes become the child chain code. The process can be repeated to any depth.

BIP44: The Derivation Path

BIP44 defines a standardized path structure so that wallets from different vendors can generate the same addresses from the same mnemonic. The path for Ethereum (and all EVM chains like MST) is:

m / 44' / 60' / 0' / 0 / 0

  • `m` - master key
  • `44'` - purpose (BIP44)
  • `60'` - coin type (60 = Ethereum, also used by all EVM-compatible chains like MST, Polygon, BSC)
  • `0'` - account (first account; you can have multiple accounts: 0', 1', 2'...)
  • `0` - change (0 = external/receiving addresses, 1 = internal/change addresses)
  • `0` - address index (first address; increment for more addresses)

The apostrophe means hardened derivation. The account level is hardened, so a compromised child private key at the address level cannot expose sibling addresses in the same account. This path is used by MetaMask, BridgeKey, and nearly every EVM wallet. A wallet that supports multiple blockchains simply uses different coin types (e.g., 0' for Bitcoin, 60' for Ethereum, and a custom coin type for MST if needed, though MST uses the same 60' path for EVM compatibility).

This is why backing up the mnemonic is equivalent to backing up every wallet you will ever own.
🗝️ The Immutable Truth

Not your keys, not your coins. A wallet does not "hold" your cryptocurrency. Your assets exist as entries on the blockchain. The wallet holds the private keys that allow you to sign transactions proving you own those assets. The mnemonic is the master key to all your private keys. Lose the mnemonic, and you lose the ability to prove ownership - forever. Share the mnemonic, and anyone who has it can take everything. There is no customer support, no recovery email, no KYC reset. The keys are the kingdom.

06   Wallet Types & Security

Hot, Cold, Hardware, and Paper - The Spectrum of Key Custody

Wallets are categorized by how they store and use private keys. Understanding the trade-offs is essential for protecting funds of any significant value.

Software (Hot) Wallets

MetaMask, Trust Wallet, and MST's BridgeKey browser extension are hot wallets - they store keys encrypted on a device connected to the internet. They are convenient for daily use and dApp interaction, but they are vulnerable to malware, keyloggers, and phishing. A malicious browser extension can read your keystrokes or intercept transactions. Hot wallets are best for small, transactional amounts - what you'd carry in a physical wallet, not your life savings.

Hardware (Cold) Wallets

Ledger, Trezor, and similar devices are cold wallets - they store private keys on a dedicated hardware device that never exposes them to the internet. When you sign a transaction, the transaction data is sent to the device, which signs it internally and returns only the signature. The private key never leaves the device. This protects against malware and phishing, even if the computer is compromised. Hardware wallets are the gold standard for storing large amounts. They still use BIP39 mnemonics, so recovery is identical to hot wallets.

Paper Wallets & Metal Backups

A paper wallet is simply a physical copy of a mnemonic or private key. Paper burns, gets wet, and fades. For long-term storage, metal backups (like Cryptosteel or stamped steel plates) are recommended - they survive fires, floods, and decades of neglect. Regardless of medium, the mnemonic must be stored securely, ideally in multiple geographically separated locations, and never in a digital photograph or cloud storage.

MST BridgeKey: MST's native wallet follows the BIP39/44 standards and integrates directly with the MST ecosystem. A mnemonic created in BridgeKey can be imported into MetaMask or any compliant wallet, and vice versa. Because MST is EVM-compatible, the same 0x address that receives MSTC can also receive any ERC-20 token deployed on MST. One mnemonic controls all assets on MST.
07   Real-World Case Studies

When Key Management Goes Right - and Horribly Wrong

🔥
The Lost Hard Drive - 7,500 BTC in a Landfill
In 2013, James Howells accidentally threw away a hard drive containing the private keys to 7,500 BTC (worth over ₹2,000 crore today). The hard drive is buried in a Newport landfill. Despite years of legal battles and offers to share the recovery with the city, the drive has never been found. The coins remain unspent on the blockchain, visible to everyone, but permanently inaccessible. This is the ultimate cautionary tale: a private key stored in only one place is a disaster waiting to happen. A simple mnemonic backup in a safety deposit box would have prevented this entirely.
🎣
The MetaMask Phishing Epidemic - "Enter Your Seed Phrase to Verify"
Since 2020, phishing scams targeting MetaMask users have exploded. Victims receive a fake email or website that looks exactly like MetaMask, asking them to "verify their wallet" by entering their 12-word seed phrase. Once entered, the attacker immediately imports the seed into their own wallet and drains all assets. This is successful because many users do not understand that no legitimate service will ever ask for your seed phrase. The seed phrase is for you and you alone. It should never be typed into a website, shared on Telegram, or stored in a password manager. This attack vector has stolen hundreds of millions of dollars. Education - like this sub-module - is the only defense.
💼
The Institutional Custody Model - Multi-Sig and MPC
For exchanges and institutions holding billions in customer funds, a single mnemonic is unacceptable. Instead, they use multi-signature (multi-sig) wallets, where multiple private keys are required to authorize a transaction (e.g., 3 of 5 signatures). More recently, Multi-Party Computation (MPC) has gained traction, where the private key is split into shares held by different parties, and no single party ever has the full key. Both approaches prevent a single point of failure. MST's smart contract platform supports multi-sig wallets natively - a simple Solidity multi-sig contract can be deployed with low gas fees, making it practical for team treasury management.
08   Assessment Preparation

Top Facts - Know These Cold for the Exam

01
BIP39 Mnemonic = Entropy + Checksum → 12/24 Words
A 12-word mnemonic encodes 128 bits of entropy + 4 bits checksum. Each word is an index (0 - 2047) into a standardized 2048-word list. The words can be converted back to the seed using PBKDF2 with 2048 iterations.
02
BIP32 HD Wallets: Master Seed → Tree of Keys
A single 512-bit seed generates a master private key and chain code. Child keys are derived using HMAC-SHA512, either hardened (uses private key) or non-hardened (uses public key). Hardened derivation is marked with an apostrophe (e.g., 44').
03
Standard EVM Derivation Path: m/44'/60'/0'/0/0
This BIP44 path is used by MetaMask, BridgeKey, and all EVM-compatible wallets for the first external address. Coin type 60' = Ethereum and all EVM chains (MST, Polygon, BSC).
04
Seed Phrase = Unlimited Recovery. Lose It = Lose Everything.
Anyone with the mnemonic can recreate every private key and address. There is no "forgot password" option. Store it offline, never digitally, and never share it with anyone - including support teams.
05
Hardware Wallets Sign Offline; Hot Wallets Are for Daily Use
Use hardware wallets (Ledger, Trezor) for large holdings - they keep keys offline. Use hot wallets (MetaMask, BridgeKey) with small amounts for dApp interaction. Never enter your hardware wallet's seed phrase into a hot wallet.

Key Terms - Sub-Module 5.3

Private Key
A 256-bit secret number that allows signing of transactions. Whoever knows the private key controls the associated assets. Never share it.
Public Key
Derived from the private key using elliptic curve multiplication. Can be shared freely. Used to verify signatures and generate addresses.
Address
A 20-byte hex string (0x...) derived from the public key via Keccak-256 hashing. Used to receive funds on EVM chains like MST.
Mnemonic (Seed Phrase)
A set of 12 or 24 English words that encode entropy. Used to generate the master seed for an HD wallet. The ultimate backup.
BIP39
Standard for mnemonic generation. Defines the wordlist, entropy encoding, and PBKDF2 seed derivation.
BIP32
Standard for Hierarchical Deterministic (HD) wallets. Defines how to derive a tree of child keys from a master seed.
BIP44
Defines the multi-asset derivation path structure: m / purpose' / coin_type' / account' / change / address_index.
PBKDF2
Password-Based Key Derivation Function 2. Used in BIP39 to convert a mnemonic + passphrase into a 512-bit seed, with 2048 iterations for brute-force resistance.
Hardened Derivation
Child key derivation that uses the parent private key, preventing a compromised child key from exposing parent or sibling keys. Marked with `'` in paths.
Hot Wallet
A wallet that stores keys on an internet-connected device. Convenient but vulnerable to malware. Use for small, daily amounts.
Cold Wallet
A wallet that keeps keys offline (hardware device, paper, metal). Much more secure; used for long-term storage of significant funds.
BridgeKey
MST's native wallet. Fully EVM-compatible, BIP39/44 compliant. Can import mnemonics from MetaMask and vice versa. Integrated with the MST ecosystem.
