02 - Real World Introduction Story

The Sealed Envelope That Knows If It Was Opened

📖 Story Time

Imagine you are the Chief Justice of India. Every day, thousands of court documents arrive at your desk - land deeds, affidavits, contracts. Each one claims to be original and untampered.

Now imagine a magic machine: you feed any document into it - one page or one thousand pages - and it spits out a unique 64-character code in under a millisecond. Feed the same document again - identical code. Change even a single full stop anywhere in the document - completely different code. And most importantly: given only the 64-character code, it is impossible to reconstruct the original document.

This machine is a cryptographic hash function. And it already runs on every document you've ever stored in DigiLocker, every message you've ever sent on WhatsApp, and every transaction on the MST blockchain.

The 64-character code is called a hash - or a digest. It is the document's unforgeable fingerprint. Not the document itself. Just its identity proof.

💡
Hash Function ≠ Encryption

This is the most important distinction in this sub-module. Encryption is reversible - given the key, you get the original back. A hash function is one-way - given the hash, you cannot get the original back. There is no key. There is no decrypt. A hash is a permanent, irreversible fingerprint.

03 - Background & History

From Checksums to Cryptographic Hashes

The Old Way: Checksums (Not Secure)

Before cryptographic hashes, computers used simple checksums - adding up all the bytes in a file to produce a number. If the file changed, the checksum changed. Simple, fast - but completely insecure. A malicious actor could modify a file and adjust a few bytes to produce the same checksum. Checksums catch accidents, not attacks.

The Evolution of Cryptographic Hash Functions

Hash functions were designed to resist deliberate tampering by intelligent adversaries. The key milestones:

  • MD5 (1991): Produced a 128-bit hash. Widely used - and broken in 2004 when researchers found practical collision attacks. Never use MD5 for security.
  • SHA-1 (1995): 160-bit hash from NIST. Deprecated in 2017 when Google's CWI team found the first practical SHA-1 collision (the SHAttered attack). Never use SHA-1 for security.
  • SHA-256 (2001): Part of the SHA-2 family. 256-bit output. Used by Bitcoin for proof-of-work and block headers. Still considered secure.
  • Keccak-256 / SHA-3 (2012): Keccak won NIST's SHA-3 competition. Ethereum (and MST Blockchain) use a specific variant called Keccak-256 for transaction hashing, address derivation, and smart contract storage.
  • BLAKE3 (2020): The fastest secure hash function available today - 10× faster than SHA-256 on modern CPUs. Used in ZFS filesystems, Rust toolchains, and increasingly in newer blockchain protocols.

04 - Beginner-Friendly Deep Explanation

The Four Properties Every Secure Hash Must Have

Not every "hash" is cryptographically secure. A function earns the title of cryptographic hash function only if it satisfies all four of these properties - without exception:

🔄

Deterministic

Same input always produces exactly the same output. Always. No randomness.

🌊

Avalanche Effect

Changing even 1 bit of input changes roughly 50% of the output bits. Wildly unpredictable.

🚪

Pre-image Resistant

Given only the hash output, it is computationally impossible to find any input that produces it.

💥

Collision Resistant

Computationally impossible to find two different inputs that produce the same output hash.

Understanding Each Property With Indian Analogies

1. Deterministic - The Biometric Scanner

When you scan your fingerprint at an Aadhaar e-KYC kiosk, the same finger always produces the same biometric template. If the machine gave different results every time, you could never authenticate. Hash functions work exactly like this - the same document, run through SHA-256 a million times, produces the same 64-character output every single time.

2. Avalanche Effect - The Butterfly in the Storm

Consider these two inputs to SHA-256:

3. Pre-image Resistance - The One-Way Grinder

Imagine a powerful kitchen grinder. You put a whole mango in - you get mango juice out. Given only the mango juice, you cannot reconstruct the original mango. A hash function is mathematically identical - it grinds the input into a fixed-size output from which the input cannot be reconstructed. This is why storing password hashes in databases is safe - even if the database is stolen, the attacker cannot reverse the hashes to find the original passwords.

🗝️
How WhatsApp Protects Your Messages

WhatsApp's end-to-end encryption uses the Signal Protocol, which relies heavily on hash functions for key derivation. When you and your contact establish an encrypted session, both sides compute the same hash of shared key material - this hash becomes the encryption key. An attacker who intercepts the key material still cannot compute the key without knowing a secret input - because hash functions are one-way.

4. Collision Resistance - Why Two Documents Can Never Match

SHA-256 produces a 256-bit output - that means there are 2²⁵⁶ possible different hash values. The number 2²⁵⁶ is approximately 10⁷⁷ - more than the number of atoms in the observable universe. The probability of two different documents producing the same SHA-256 hash by accident is so small it is effectively zero. And finding a deliberate collision requires computation that would take longer than the age of the universe on today's fastest computers.

⚠️
MD5 and SHA-1 Are Broken - Never Use Them for Security

MD5 collisions can be found in seconds on a laptop. SHA-1 collisions have been demonstrated practically (the SHAttered attack, 2017). If you ever see MD5 or SHA-1 used for anything security-critical in a codebase - that is a critical vulnerability. Only SHA-256, SHA-3 / Keccak-256, and BLAKE2/BLAKE3 are acceptable for modern security applications.

Hash Functions in Blockchain - Four Specific Uses

1

Transaction Hashing

Every transaction is hashed to produce a unique Transaction ID (TxID). This is what you see on MSTScan.com - a 64-character hex string that uniquely identifies your transfer.

2

Block Header Hashing

Each block's header is hashed to produce the block hash. This hash is then included in the next block as its "previous block hash" - creating the unbreakable chain. Change one block → its hash changes → every subsequent block's "previous hash" pointer breaks → the forgery is instantly detected.

3

Wallet Address Derivation

As studied in Sub-Module 4.1: Keccak-256 of the public key → take last 20 bytes → your wallet address. The hash makes address derivation one-way - you cannot reverse an address to find the public key.

4

Merkle Trees - Efficient Transaction Verification

All transactions in a block are combined into a Merkle Tree - a structure that lets you prove any single transaction is included without downloading every transaction. This is the main focus of the second half of this sub-module.

05 - Technical Deep Dive

SHA-256, Keccak-256, and BLAKE3 - Under the Hood

SHA-256 - Bitcoin's Engine

SHA-256 (Secure Hash Algorithm 256-bit) belongs to the SHA-2 family, designed by the NSA and published by NIST in 2001. It processes input in 512-bit blocks using a series of 64 rounds of bitwise operations - AND, OR, XOR, rotations, and additions modulo 2³². The internal state is 256 bits (eight 32-bit words). The design ensures extreme sensitivity to input changes (avalanche effect) and one-wayness through irreversible mixing operations.

Bitcoin uses SHA-256 in two specific ways:

  • Double SHA-256 for transaction IDs and block headers: Hash = SHA-256(SHA-256(data)). The double application eliminates a theoretical length-extension attack vulnerability in SHA-2.
  • Proof-of-Work: Miners must find an input (nonce) such that SHA-256(SHA-256(block_header)) starts with a required number of zero bits. This is computationally hard to find but trivially easy to verify.

Keccak-256 - The Ethereum and MST Standard

Keccak-256 is the specific variant of Keccak that Ethereum (and MST) chose before the NIST SHA-3 standardisation process slightly modified Keccak's padding. Important: Keccak-256 ≠ SHA3-256. They use the same algorithm but different padding - producing different outputs for the same input. Ethereum/MST use Keccak-256 everywhere: address derivation, transaction hashing, event topics in smart contracts, and keccak256() in Solidity.

Keccak uses a "sponge construction" - absorbing input bits into a 1600-bit state and squeezing out the hash. This is architecturally different from SHA-2 and offers elegant security proofs.

BLAKE3 - The Modern Speed Champion

BLAKE3 (2020) is the fastest cryptographically secure hash function on modern hardware - achieving speeds above 1 GB/s on a single CPU core without hardware acceleration. It is highly parallelisable (unlike SHA-256) and is used in Rust, ZFS, and newer blockchain protocols that need high throughput. MST's future ZK-proof systems may use BLAKE3 for efficiency.

Merkle Trees - The Data Structure That Powers Light Clients

A Merkle Tree (named after computer scientist Ralph Merkle, 1979) is a binary tree where every leaf node contains the hash of a data block, and every non-leaf node contains the hash of its two children. The single hash at the very top is called the Merkle Root.

"The Merkle Root is a single 32-byte value that mathematically represents every transaction in the block. If any transaction changes, the Merkle Root changes."

The Merkle Root is stored in the block header - so every block header commits to every transaction it contains, in an extremely compact way.

Why Do We Need Merkle Trees? The Light Client Problem.

A full MST blockchain node stores every block and every transaction - tens of gigabytes. But a mobile wallet on your phone cannot store all of that. A light client (like BridgeKey Wallet) only downloads block headers - a few kilobytes per block. But then how does it verify that a specific transaction is really inside a block?

With a Merkle Proof - a small set of sibling hashes that lets you reconstruct the path from your transaction up to the Merkle Root. If your computed root matches the root in the block header - your transaction is in the block. Verified. Without downloading a single other transaction.

07 - Step-by-Step Workflow

The Five "W" Questions Answered

❓ WHY Do We Hash Transactions Into a Merkle Tree - Not Just Hash All of Them Together?

If you simply concatenated all 10,000 transactions and hashed them, you would get one root hash - but verifying a single transaction would require you to have all 10,000 transactions available. A Merkle Tree structure allows logarithmic proof size: with 10,000 transactions, you only need log₂(10,000) ≈ 14 hashes to prove inclusion. The tree structure is the key - it lets you audit a path without the full dataset.

❓ WHAT Is the Difference Between a Transaction Hash and a Block Hash?

Transaction Hash (TxID)

Fingerprint of One Transfer

keccak256(signed_transaction_data). Uniquely identifies one specific transaction. What you share on a blockchain explorer to prove a payment was made.

Block Hash

Fingerprint of the Entire Block

keccak256(block_header) where the header contains the Merkle Root of all transactions. One hash represents everything in the block - and links to the previous block.

❓ HOW Are Hash Functions Used in DigiLocker?

When CBSE issues your Class 12 certificate to DigiLocker, it doesn't upload the PDF. It uploads:

  1. The PDF itself
  2. A SHA-256 hash of the PDF
  3. A digital signature of that hash (using CBSE's private key)

When you share the document with a university, the university runs SHA-256 on the PDF you provide, verifies the hash matches the published hash, and verifies CBSE's digital signature. They have mathematically proven the document is genuine - without calling CBSE. The hash is the glue between the document and the signature.

❓ WHERE Else Are Merkle Trees Used Besides Blockchain?

  • Git version control: Every Git commit is a Merkle Tree of file hashes. git clone uses Merkle proofs to download only changed files, not the entire repository history.
  • BitTorrent: Large files are split into pieces. Each piece's hash is included in a Merkle Tree in the torrent file. Your client verifies each downloaded piece matches its hash before accepting it - preventing corrupted or tampered downloads.
  • Certificate Transparency (Google CT): All SSL certificates are logged in a Merkle Tree. Anyone can verify their certificate is logged without downloading the full log.
  • ZK-SNARKs / ZK-STARKs: Zero-knowledge proof systems use Merkle Trees to efficiently commit to large datasets (covered in Sub-Module 4.4).

❓ WHAT Are the Limitations of Hash Functions?

  • ⚠️ Quantum threat: Grover's Algorithm on a quantum computer reduces the effective security of a 256-bit hash to 128 bits (searching 2²⁵⁶ possibilities in 2¹²⁸ operations). SHA-256 and Keccak-256 still survive this - 128-bit security is considered adequate. But MD5 and SHA-1 would be catastrophically weakened.
  • ⚠️ Rainbow table attacks on passwords: If you hash passwords without a salt (a random per-user value added before hashing), attackers can precompute hashes for common passwords. Always use bcrypt, scrypt, or Argon2 for password storage - these are memory-hard functions specifically designed to slow down brute-force attacks.
  • ⚠️ Length extension attacks on SHA-2: SHA-256 is vulnerable to length extension attacks - an attacker who knows Hash(secret || message) can compute Hash(secret || message || extra) without knowing the secret. Bitcoin's double-SHA-256 mitigates this. Keccak-256 is immune by design.
🔮
MST Blockchain Context

MST uses Keccak-256 for transaction hashing (EVM-compatible), and the Merkle Root in each block header commits to every transaction in that block. BridgeKey Wallet, as a light client, uses Merkle Proofs to verify transaction inclusion without running a full node - making it practical for mobile devices with limited storage and bandwidth.

10 - Real World Case Studies

Hash Functions in the Wild - Three Stories

Case Study 1: How Aadhaar Protects Your Biometric Data

UIDAI does not store your actual fingerprint image. It stores the hash of your biometric template - a mathematical fingerprint of your fingerprint. When you authenticate at a bank KYC kiosk, your live fingerprint is scanned, hashed, and the hash is compared to the stored hash. Your actual biometric data never leaves UIDAI's secure servers. Even if UIDAI's hash database were stolen, the attacker cannot reconstruct your fingerprints from the hashes.

Case Study 2: Git - The Blockchain Before Blockchain

Linus Torvalds created Git in 2005 - years before Bitcoin - and it uses a Merkle-like tree of SHA-1 hashes (now transitioning to SHA-256). Every file in a Git repository has a hash. Every directory (tree) has a hash of its contents' hashes. Every commit has a hash of the tree hash plus the parent commit hash. This is structurally identical to a blockchain - an append-only, hash-linked chain of states. Satoshi Nakamoto's insight was to add a distributed consensus mechanism on top of this data structure.

Case Study 3: Verifying a Payment on MSTScan.com

When Priya pays for a product using MST tokens, the merchant receives a Transaction ID (TxID): 0x3f2a...c291. The merchant visits MSTScan.com, enters the TxID, and instantly sees: Block number, timestamp, sender, receiver, amount, and confirmation count. They trust this because:

  1. The TxID is the Keccak-256 hash of the signed transaction - it cannot be fabricated
  2. The transaction's hash is included in the block's Merkle Root - any node can verify this
  3. The block hash is included in subsequent blocks - making reversal exponentially harder with each new block

After 6 block confirmations on MST (approximately 18 seconds at 3-second block times), the payment is considered final. The merchant ships the product. No bank, no payment processor, no fraud department - just mathematics.

11 - Assessment Preparation

Top Must-Know Facts Before the Test

  • A cryptographic hash function is one-way (not reversible) and deterministic (same input → same output always).
  • The four properties: Determinism, Avalanche Effect, Pre-image Resistance, Collision Resistance. Know all four names and what each means.
  • MD5 and SHA-1 are broken - practical collisions have been demonstrated. Never use for security.
  • Bitcoin uses SHA-256 (double SHA-256 for block headers and TxIDs). Ethereum and MST use Keccak-256.
  • Keccak-256 ≠ SHA3-256 - they use the same algorithm but different padding, producing different outputs.
  • A Merkle Tree is a binary tree of hashes: leaf nodes = hash of each transaction; parent nodes = hash of two children; root = Merkle Root.
  • The Merkle Root is stored in the block header and represents every transaction in the block in 32 bytes.
  • A Merkle Proof requires only log₂(N) hashes to prove any transaction is in a block of N transactions.
  • If N = 1,048,576 transactions (2²⁰), a Merkle Proof requires only 20 hashes (640 bytes) instead of 33 MB.
  • The hash chain between blocks (each block contains the previous block's hash) is what makes blockchain tamper-evident.
  • Hash functions in blockchain serve four roles: transaction IDs, block headers, wallet address derivation, and Merkle Tree construction.
  • Git, BitTorrent, and Certificate Transparency all use Merkle Trees for efficient data integrity verification.
⚠️
Common Exam Trap - Keccak-256 vs SHA3-256

Many students write "Ethereum uses SHA-3" - this is technically wrong. Ethereum uses Keccak-256, which was submitted to the SHA-3 competition. When NIST standardised SHA-3, they slightly modified the padding. Ethereum was built with the original Keccak padding and did not update. So keccak256("abc") ≠ sha3_256("abc"). In exam answers, always write "Keccak-256" for Ethereum and MST, not "SHA-3."

12 - Security Notes

Critical Security Principles

🔴
Never Use Hash Functions Alone to Store Passwords

A plain hash of a common password like keccak256("password123") can be cracked instantly using rainbow tables - precomputed databases of hashes for common passwords. Always use a password hashing function like bcrypt, scrypt, or Argon2, which adds a salt and is intentionally slow. Regular hash functions (SHA-256, Keccak-256) are designed to be fast - which is a security weakness for password storage.

🔴
Hash-Based Commit-Reveal Schemes Need a Secret Salt

A common pattern in smart contracts: commit keccak256(secret_value) on-chain, then reveal later. If the set of possible values is small (e.g., a vote between 3 options), an attacker can precompute all three hashes and compare to your commitment - breaking the secrecy. Always include a random salt: keccak256(abi.encodePacked(secret_value, random_salt)). This is used in MST's FortunaX randomness protocol.

🛡️
Second Pre-image Attack vs Collision Attack

There are two types of attack harder than finding a collision: a pre-image attack (find any input with a given hash) and a second pre-image attack (given input A, find a different input B with the same hash as A). Both are harder than finding a collision. Collision resistance (the weakest) implies second pre-image resistance, which implies pre-image resistance. In exam scenarios, always distinguish which type of attack is being described.

13 - Industry Insights

How the Industry Uses This Knowledge

Where Hash Functions Appear in Smart Contract Development

  • keccak256() in Solidity: Used to compute function selectors (first 4 bytes of keccak256("functionName(types)")), event topic IDs, mapping keys for EnumerableMap, and commit-reveal schemes.
  • EIP-712 typed data hashing: Structured data is hashed using a domain separator + type hash + field hashes - all using Keccak-256 - before signing. Prevents cross-domain replay attacks.
  • Merkle Airdrops: NFT projects and token distributions use a Merkle Tree of eligible addresses. Only the Merkle Root is stored on-chain (32 bytes, negligible gas). Each claimant provides a Merkle Proof to verify they're on the list. Used by Uniswap's UNI airdrop and many MST ecosystem projects.
  • Storage slot computation: In Solidity, the storage slot of a mapping value is keccak256(key ++ slot_number). Understanding this is essential for writing storage-efficient contracts and understanding proxy upgrade patterns.

Merkle Airdrops - A Practical Example

Imagine MST Academy wants to airdrop 100 MST tokens to 50,000 graduated students. Storing 50,000 addresses on-chain would cost enormous gas. Instead:

  1. Build a Merkle Tree of all 50,000 student addresses off-chain
  2. Store only the Merkle Root (32 bytes) in the smart contract - one transaction, minimal gas
  3. Each student calls claim(proof[]) with their Merkle Proof (14 hashes for 50,000 addresses)
  4. The contract verifies the proof against the root - if valid, transfers 100 MST tokens

This pattern saves 99.97% of the gas cost compared to storing all addresses on-chain.

14 - Glossary

Key Terms You Must Know

Hash Function
A mathematical function that takes any input and produces a fixed-length output (digest). Cryptographic hash functions are one-way, deterministic, and collision-resistant.
Hash / Digest
The fixed-length output of a hash function. For SHA-256 and Keccak-256: 32 bytes = 64 hexadecimal characters. The unique "fingerprint" of an input.
Avalanche Effect
The property that a tiny change in input (even 1 bit) causes approximately 50% of the output bits to change - making the output appear completely different and unrelated.
Pre-image Resistance
Given a hash output H, it is computationally infeasible to find any input m such that Hash(m) = H. This is the "one-way" property.
Collision Resistance
It is computationally infeasible to find two different inputs m₁ and m₂ such that Hash(m₁) = Hash(m₂).
SHA-256
Secure Hash Algorithm 256-bit. Part of NIST's SHA-2 family (2001). Produces a 256-bit hash. Used in Bitcoin for transaction IDs, block headers, and proof-of-work. Still considered secure.
Keccak-256
The specific hash function used by Ethereum and MST Blockchain. Based on the Keccak sponge construction. Distinct from the NIST SHA-3 standard due to different padding. Used in Solidity's keccak256() builtin.
Merkle Tree
A binary tree of hashes where each leaf is a transaction hash, each parent is the hash of its two children, and the root (Merkle Root) represents the entire dataset in 32 bytes.
Merkle Root
The single root hash of a Merkle Tree. Stored in a block header - it commits to every transaction in the block. Any change to any transaction changes the Merkle Root.
Merkle Proof
A set of log₂(N) sibling hashes that allows verification that a specific transaction is included in a block of N transactions, without needing all N transactions.
Light Client
A blockchain node that only downloads block headers (not full transaction data), using Merkle Proofs to verify specific transactions. BridgeKey Wallet operates as a light client.
BLAKE3
A modern cryptographic hash function (2020) that is faster than SHA-256 and SHA-3 while maintaining full security. Highly parallelisable. Used in Rust and ZFS.
Merkle Airdrop
A gas-efficient token distribution pattern where only the Merkle Root of eligible addresses is stored on-chain. Each recipient proves eligibility by submitting a Merkle Proof.

MCQ
● Easy
1 mark
Which of the following correctly describes the "Avalanche Effect" in a cryptographic hash function?
  • A) A large input always produces a larger hash output than a small input
  • B) The hash function slows down significantly when processing very large files
  • C) Changing even a single bit of the input causes approximately 50% of the output bits to change, making the output appear completely unrelated
  • D) The same input can occasionally produce two different outputs depending on system load
✅ Correct Answer: C

The Avalanche Effect is the property that even the smallest possible input change - flipping a single bit - causes roughly half of all output bits to flip. This means there is no pattern or gradual relationship between similar inputs and their hashes. For example, hashing "Send ₹500 to Rahul" and "Send ₹501 to Rahul" produces two completely different, unpredictable 256-bit outputs. This is critical for blockchain security - any tampering with a transaction makes its hash completely different, instantly exposing the forgery.

● Easy

Ethereum and MST Blockchain use which hash function for transaction hashing and within the Solidity keccak256() function?
  • A) SHA-256
  • B) MD5
  • C) Keccak-256
  • D) SHA3-256 (NIST standardised)
✅ Correct Answer: C

Ethereum and MST use Keccak-256 - not SHA3-256. Although both are based on the Keccak algorithm, NIST modified the padding scheme when standardising SHA-3. Ethereum was built with the original Keccak padding (before the standard was finalised) and did not change. This means keccak256("abc") and sha3_256("abc") produce different outputs. In exams and professional settings, always specify "Keccak-256" when referring to Ethereum/MST - writing "SHA-3" is technically incorrect and will be penalised.

● Medium

A blockchain has 1,048,576 (2²⁰) transactions in a block. A light client wants to verify that one specific transaction is in the block using a Merkle Proof. How many sibling hashes does the proof need to provide?
  • A) 1,048,575 (all other transaction hashes)
  • B) 524,288 (half of all transaction hashes)
  • C) 20 (log₂ of 1,048,576)
  • D) 256 (fixed regardless of tree size)
✅ Correct Answer: C

A Merkle Proof requires exactly log₂(N) sibling hashes, where N is the total number of transactions. For N = 2²⁰ = 1,048,576 transactions, the proof needs log₂(2²⁰) = 20 hashes. Each hash is 32 bytes, so the entire proof is 20 × 32 = 640 bytes. Contrast this with downloading all 1,048,576 transaction hashes: 1,048,576 × 32 = 33.5 MB. The Merkle Proof is 52,000× smaller. This logarithmic efficiency is what makes light clients practical on mobile devices.

MCQ
● Medium
1 mark
Which of the following correctly describes the Merkle Root stored in a blockchain block header?
  • A) The concatenation of all transaction IDs in the block
  • B) The hash of the first transaction in the block only
  • C) The root hash of a binary hash tree built from all transaction hashes - representing all transactions in 32 bytes
  • D) The SHA-256 hash of the block's body (all transactions concatenated)
✅ Correct Answer: C

The Merkle Root is the top-level hash of a Merkle Tree constructed by repeatedly hashing pairs of transaction hashes up from the leaf level. The final single 32-byte value (the root) mathematically represents every transaction in the block - if any transaction changes, the Merkle Root changes. It is this root - not all individual transaction hashes - that is stored in the compact block header. This allows full blocks to be audited by verifying the header alone. Option D is wrong because simply concatenating all transactions and hashing them once would not allow logarithmic-size Merkle Proofs.

● Medium
1 mark
Why is SHA-256(password) NOT a safe way to store user passwords in a database, even though SHA-256 is a secure hash function?
  • A) SHA-256 outputs are too short to be unique for passwords
  • B) SHA-256 is a reversible function - attackers can decrypt the hash
  • C) SHA-256 is too fast - attackers can compute billions of hashes per second and use precomputed rainbow tables for common passwords
  • D) SHA-256 produces the same hash for all passwords shorter than 8 characters
✅ Correct Answer: C

SHA-256's speed is a strength for most applications but a weakness for password hashing. A modern GPU can compute billions of SHA-256 hashes per second - allowing an attacker with a stolen password database to try every common password, dictionary word, and variation in seconds. Additionally, precomputed "rainbow tables" map common passwords to their SHA-256 hashes. For passwords, use bcrypt, scrypt, or Argon2 - these are intentionally slow (100ms per hash) and memory-intensive, making brute-force and rainbow table attacks computationally impractical. Option B is wrong - SHA-256 is definitively not reversible.

MCQ
● Hard
1 mark
An MST smart contract stores keccak256(abi.encodePacked(vote)) as a commitment, where vote can only be one of three values: "YES", "NO", or "ABSTAIN". A malicious observer can determine which option was committed. Which property of cryptographic hash functions does this scenario violate - and what is the fix?
  • A) Avalanche Effect is violated; fix by using SHA-256 instead of Keccak-256
  • B) Collision Resistance is violated; fix by using a longer hash output
  • C) Pre-image Resistance is practically broken due to the tiny input space; fix by including a large random salt: keccak256(abi.encodePacked(vote, randomSalt))
  • D) Determinism is violated; fix by adding a timestamp to the input
✅ Correct Answer: C

Pre-image resistance guarantees that given a hash H, you cannot find any input m. But it assumes the input space is enormous. When the input space has only 3 possibilities ("YES", "NO", "ABSTAIN"), an attacker can simply compute all three hashes in microseconds and compare to the committed hash - trivially revealing the vote. The hash function itself is not broken - its security properties hold - but they are useless when the input can be enumerated. The fix is adding a large random salt (at least 32 bytes of secure randomness) to the commitment: keccak256(abi.encodePacked(vote, random_256_bit_salt)). The salt makes the actual pre-image space 2²⁵⁶ × 3 - practically infinite.

● Hard
3 marks
Study the following Merkle Tree and answer all three sub-questions (1 mark each).
A block contains exactly 4 transactions: Tx1, Tx2, Tx3, Tx4. The Merkle Tree is built as follows: H1 = keccak256(Tx1) = 0xAA... H2 = keccak256(Tx2) = 0xBB... H3 = keccak256(Tx3) = 0xCC... H4 = keccak256(Tx4) = 0xDD... H12 = keccak256(H1 ++ H2) = 0x11... H34 = keccak256(H3 ++ H4) = 0x22... MerkleRoot = keccak256(H12 ++ H34) = 0xFF... Visual Tree: [ Root: 0xFF... ] / \ [H12: 0x11...] [H34: 0x22...] / \ / \ [H1:0xAA] [H2:0xBB] [H3:0xCC] [H4:0xDD] Tx1 Tx2 Tx3 Tx4 A light client wants to verify that Tx3 is in this block. It already has: Tx3 itself, and the MerkleRoot (0xFF...) from the block header. Sub-questions: (a) List the exact sibling hashes the full node must provide as the Merkle Proof for Tx3. (b) Write out the three computation steps the light client performs to verify the proof, in order. (c) An attacker provides a fake proof claiming Tx3' (a modified transaction) is in the block. The attacker provides the same proof hashes (H4 and H12). Explain precisely why this proof will fail verification.
✅ Model Answers - 1 mark each

(a) Sibling hashes for Tx3 Merkle Proof: The full node must provide exactly 2 hashes:

1. H4 = 0xDD... - the sibling of H3 at the leaf level (needed to compute H34)

2. H12 = 0x11... - the sibling of H34 at the parent level (needed to compute the Root)

Only 2 hashes (64 bytes total) are needed - not all 4 transaction hashes.

(b) Verification steps the light client performs:

Step 1: Compute leaf hash from owned transaction: H3' = keccak256(Tx3). This should equal 0xCC....

Step 2: Combine with sibling H4 (from proof, position: right sibling): H34' = keccak256(H3' ++ H4) = keccak256(0xCC... ++ 0xDD...). This should equal 0x22....

Step 3: Combine with sibling H12 (from proof, position: left sibling): Root' = keccak256(H12 ++ H34') = keccak256(0x11... ++ 0x22...). Compare Root' with 0xFF... from the block header. If equal → Tx3 is genuinely in the block. ✅

(c) Why the fake proof fails: The attacker has a modified transaction Tx3' (different bytes). When the light client computes Step 1: H3_fake = keccak256(Tx3'). Due to the Avalanche Effect of Keccak-256, Tx3' ≠ Tx3 means H3_fake ≠ H3 = 0xCC.... In Step 2, the client computes H34_fake = keccak256(H3_fake ++ H4) - this is a completely different value from H34 = 0x22.... In Step 3, Root_fake = keccak256(H12 ++ H34_fake) - completely different from 0xFF.... The comparison fails: Root_fake ≠ 0xFF.... The proof is rejected. The attacker cannot forge a valid Merkle Proof without finding a collision in Keccak-256 - computationally infeasible.

True / False + Justification
● Medium
2 marks
Statement: "Given the SHA-256 hash of a document - 0x3f2a...c291 - it is theoretically possible to reconstruct the original document if you have a powerful enough computer, because SHA-256 uses only mathematical operations."
TRUE ✗
FALSE ✅
✅ Correct Answer: FALSE - Full Justification Required (2 marks)

This statement is FALSE.

This misconception arises because students assume that if something is computed mathematically, it can be reversed mathematically. SHA-256 is specifically designed so that reversal is computationally infeasible - and this is not merely a practical limitation of today's computers; it is a fundamental security property.

Why reversal is infeasible: SHA-256 maps any arbitrary-length input to a fixed 256-bit output. This means infinitely many different inputs produce the same 256-bit hash (pigeonhole principle) - the function is not injective and mathematically cannot be inverted to a unique answer. The best possible attack (brute-force pre-image search) requires trying 2²⁵⁶ possible inputs on average. Even with every atom in the observable universe converted into a computing unit, and running since the Big Bang, this computation could not be completed.

The "powerful enough computer" framing is the trap - no classical computer, however powerful, can overcome a 256-bit security parameter through brute force. Quantum computers (Grover's Algorithm) reduce this to 2¹²⁸ operations - still utterly infeasible. SHA-256 with 256-bit output is considered quantum-resistant for pre-image attacks. (1 mark for correct False; 1 mark for justification mentioning pre-image resistance / infeasibility / 2²⁵⁶.)

True / False + Justification
● Easy
1 mark
Statement: "The Merkle Root stored in a block header is sufficient to verify that any specific transaction is included in that block - without downloading any other transaction data - using only a Merkle Proof and the transaction itself."
TRUE ✅
FALSE ✗
✅ Correct Answer: TRUE - Justification Required (1 mark)

This statement is TRUE. A Merkle Proof consists of log₂(N) sibling hashes that define the path from a specific transaction up to the Merkle Root. Given: (1) the transaction itself, (2) the Merkle Proof hashes (the siblings at each level of the tree), and (3) the Merkle Root from the block header - a light client can compute a chain of hashes from the transaction up through the tree and arrive at a root hash. If this computed root matches the block header's Merkle Root, the transaction's inclusion is cryptographically verified. No other transaction needs to be downloaded. This is the entire basis of SPV (Simplified Payment Verification) and light client operation in all major blockchains including MST.
