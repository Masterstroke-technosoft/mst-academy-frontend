08 - Hands-On Lab Code

Building the Wallet CLI - Step by Step

We will build wallet.js in six progressive steps. Each step is self-contained - run it, observe the output, then add the next step. The final file will be your submission.

STEP 1 - Generate a Private Key

Cryptographically Secure Random Number Generation

A private key is a 256-bit (32-byte) random number. It must come from a cryptographically secure pseudorandom number generator (CSPRNG) - not Math.random(). Node.js provides crypto.randomBytes() for this purpose.

wallet.js - Step 1 JavaScript
import { ethers } from 'ethers'; import { randomBytes } from 'crypto'; // ── STEP 1: Generate a private key ───────────────────────────────── // A private key = 32 cryptographically random bytes = 64 hex chars // NEVER use Math.random() for cryptographic keys - it is predictable! const privateKeyBytes = randomBytes(32); // 32 bytes from OS CSPRNG const privateKey = '0x' + privateKeyBytes.toString('hex'); // prefix with 0x console.log('\n════════════════════════════════════════════'); console.log(' MST WALLET CLI - STEP 1: Private Key'); console.log('════════════════════════════════════════════'); console.log('Private Key (32 bytes / 64 hex chars):'); console.log(' ', privateKey); console.log('⚠️ NEVER share this with anyone!');
Terminal - node wallet.js
$ node wallet.js ════════════════════════════════════════════ MST WALLET CLI - STEP 1: Private Key ════════════════════════════════════════════ Private Key (32 bytes / 64 hex chars): 0xe9873d79c6d87dc0fb6a5778633389f4453213303da61f20bd0ea2eefec5b2fa ⚠️ NEVER share this with anyone!

Your output will differ - every run generates a fresh random key. That is correct.

STEP 2 - Derive the Public Key

secp256k1 Elliptic Curve Multiplication

The public key is derived by multiplying the private key scalar by the generator point G on the secp256k1 curve: pubKey = privKey × G. This is a one-way operation - you cannot reverse it. Ethers.js handles the curve math internally.

wallet.js - Step 2 (add after Step 1) JavaScript
// ── STEP 2: Derive the Public Key ────────────────────────────────── // ethers.SigningKey wraps secp256k1 operations. // It computes pubKey = privKey × G (generator point of secp256k1). const signingKey = new ethers.SigningKey(privateKey); const publicKey = signingKey.publicKey; // uncompressed: 65 bytes, "04" prefix const compressedPubKey = signingKey.compressedPublicKey; // compressed: 33 bytes console.log('\n─── Public Key ───────────────────────────────'); console.log('Uncompressed (04 + x + y, 65 bytes = 130 hex):'); console.log(' ', publicKey); console.log('Compressed (02/03 + x, 33 bytes = 66 hex):'); console.log(' ', compressedPubKey); console.log('✅ Public key is safe to share publicly.');
Expected output
─── Public Key ─────────────────────────────── Uncompressed (04 + x + y, 65 bytes = 130 hex): 0x04a478...8f3c2b (130 hex chars - 04 prefix + 64-byte X + 64-byte Y coordinate) Compressed (02/03 + x, 33 bytes = 66 hex): 0x02a478...8f3c2b (66 hex chars - parity prefix + 64-byte X coordinate) ✅ Public key is safe to share publicly.
STEP 3 - Derive the Wallet Address

Keccak-256 Hash → Last 20 Bytes

The wallet address is derived by: (1) taking the uncompressed public key bytes (without the 04 prefix), (2) hashing them with Keccak-256, (3) taking the last 20 bytes (40 hex chars) of the hash, (4) applying EIP-55 checksum capitalisation.

wallet.js - Step 3 (add after Step 2) JavaScript
// ── STEP 3: Derive the Wallet Address ───────────────────────────── // Method 1 - Use ethers.js built-in (recommended for production): const wallet = new ethers.Wallet(privateKey); const address = wallet.address; // EIP-55 checksummed address // Method 2 - Manual derivation (shows every step explicitly): const pubKeyBytes = ethers.getBytes(publicKey); // 65-byte array const pubKeyNoPrefix = pubKeyBytes.slice(1); // remove the 04 prefix byte const hash = ethers.keccak256(pubKeyNoPrefix); // 32-byte Keccak-256 hash const rawAddress = '0x' + hash.slice(-40); // last 20 bytes = 40 hex chars const checksumAddress = ethers.getAddress(rawAddress); // apply EIP-55 checksum console.log('\n─── Wallet Address ───────────────────────────'); console.log('Keccak-256(pubKey) full hash:', hash); console.log('Last 20 bytes (raw) :', rawAddress); console.log('EIP-55 Checksummed Address :', checksumAddress); console.log('Matches ethers.Wallet.address?', checksumAddress === address);
Expected output
─── Wallet Address ─────────────────────────── Keccak-256(pubKey) full hash: 0x2a9b5e...f3c1d8a2 (64 hex = 32 bytes) Last 20 bytes (raw) : 0x...f3c1d8a2 (40 hex = 20 bytes) EIP-55 Checksummed Address : 0x71C7656EC7ab88b098defB751B7401B5f6d8976F Matches ethers.Wallet.address? true
💡
What Is EIP-55 Checksum?

A raw Ethereum address is all lowercase hex. EIP-55 applies mixed capitalisation based on the Keccak hash of the lowercase address. This creates a built-in checksum - a typo in the address usually produces invalid capitalisation, preventing accidental token sends to wrong addresses. Ethers.js uses getAddress() to apply and validate EIP-55 checksum.

STEP 4 - Sign a Message

ECDSA Signing with Your Private Key

Now for the core action: sign a message. Ethers.js automatically prepends the Ethereum personal sign prefix (\x19Ethereum Signed Message:\n{length}) before hashing, which prevents signed messages from being mistaken for valid transaction data.

wallet.js - Step 4 (add after Step 3) JavaScript
// ── STEP 4: Sign a Message ───────────────────────────────────────── // signMessage() uses EIP-191 personal sign: // hash = keccak256("\x19Ethereum Signed Message:\n" + message.length + message) // Then signs the hash using ECDSA with the private key. const message = 'Hello MST! I am signing this message to prove I own this wallet.'; // signMessage is async - it returns a Promise const signature = await wallet.signMessage(message); // The signature is 65 bytes = 130 hex chars // Structure: r (32 bytes) + s (32 bytes) + v (1 byte) const sigParts = ethers.Signature.from(signature); console.log('\n─── Signing ──────────────────────────────────'); console.log('Message :', message); console.log('Signature :', signature); console.log(' r (32 bytes):', sigParts.r); console.log(' s (32 bytes):', sigParts.s); console.log(' v (recovery):', sigParts.v);
Expected output
─── Signing ────────────────────────────────── Message : Hello MST! I am signing this message to prove I own this wallet. Signature : 0x3d5a...c291b3f1...44ae1b (130 hex chars = 65 bytes) r (32 bytes): 0x3d5a...c291 (the x-coordinate of R = k×G) s (32 bytes): 0xb3f1...44ae (the proof value) v (recovery): 27 or 28 (recovery ID for ecrecover)
STEP 5 - Verify the Signature

Recover the Signer's Address From the Signature

Verification uses Ethereum's ecrecover: given the original message and the signature, recover the signer's address. If the recovered address matches the expected address - the signature is valid. The private key is never needed for verification.

wallet.js - Step 5 (add after Step 4) JavaScript
// ── STEP 5: Verify the Signature ────────────────────────────────── // verifyMessage() recovers the signer's address from (message, signature). // It does NOT need the private key or even the public key. // This is the same ecrecover used by smart contracts on MST/Ethereum. const recoveredAddress = ethers.verifyMessage(message, signature); const isValid = (recoveredAddress.toLowerCase() === address.toLowerCase()); console.log('\n─── Verification ─────────────────────────────'); console.log('Recovered address :', recoveredAddress); console.log('Our wallet address:', address); console.log('Signature valid? :', isValid ? '✅ YES - addresses match!' : '❌ NO'); // ── Tamper Test: verify a MODIFIED message ────────────────────── // What happens if someone alters the message after signing? const tamperedMessage = 'Hello MST! I am signing this message to prove I own this wallet!!!'; const tamperedRecovered = ethers.verifyMessage(tamperedMessage, signature); console.log('\n─── Tamper Test ───────────────────────────────'); console.log('Tampered message signed with same sig:'); console.log('Recovered address :', tamperedRecovered); console.log('Is our address? :', tamperedRecovered.toLowerCase() === address.toLowerCase() ? '⚠️ MATCHES (impossible in practice)' : '✅ DIFFERENT - tampering detected! Signature is invalid for altered message.' );

Troubleshooting Tips

  • If you get Cannot use import statement - add "type": "module" to your package.json.
  • If you get Cannot find package 'ethers' - run npm install ethers in the project directory.
  • If your addresses don't match in Step 3 - verify you stripped the 04 prefix byte (index 0) from the public key before hashing. pubKeyBytes.slice(1) removes it.
  • If verification returns false - ensure you are verifying the exact same message string, with no trailing spaces or line breaks added.

  • 🔵 Sign multiple messages: Sign 3 different messages with the same wallet. Observe that each signature is completely different (different r, s) even though the same private key is used - this is ECDSA's random nonce k at work.
11 - Assessment Preparation

Top Must-Know Facts Before the Submission

  • A private key is 32 bytes (256 bits) of cryptographically secure random data. Always use a CSPRNG - never Math.random().
  • The public key is derived using secp256k1 elliptic curve multiplication: pubKey = privKey × G. This is one-way - cannot be reversed.
  • The uncompressed public key is 65 bytes: 04 prefix + 32-byte X + 32-byte Y coordinate.
  • The compressed public key is 33 bytes: 02 or 03 prefix (parity of Y) + 32-byte X coordinate.
  • The wallet address is: keccak256(uncompressed_pubKey_without_04_prefix) → last 20 bytes.
  • EIP-55 checksum applies mixed-case capitalisation to the hex address, providing a built-in typo-detection mechanism.
  • EIP-191 personal sign prefixes the message with "\x19Ethereum Signed Message:\n{length}" before hashing - preventing signed messages from being mistakenly submitted as transactions.
  • An ECDSA signature is 65 bytes: 32-byte r + 32-byte s + 1-byte v (recovery ID, typically 27 or 28).
  • ethers.verifyMessage(message, signature) recovers the signer's address without needing the private key - this is the JavaScript equivalent of Solidity's ecrecover.
  • A tampered message produces a completely different recovered address - demonstrating the Integrity property of digital signatures (Sub-Module 4.2).
13 - Industry Insights

What You Just Built - In Context

The pipeline you built today - generate → derive → sign → verify - is the backbone of every Web3 application ever built. MetaMask, BridgeKey Wallet, Coinbase Wallet - they all run this exact pipeline. The only difference is they add a Secure Enclave to protect the private key from the device memory, a seed phrase (BIP39 mnemonic) to recover the key, and a UI to make it human-friendly. The cryptographic core is identical to what you just wrote.

Understanding this pipeline at the code level makes you a dramatically better smart contract developer, security auditor, and Web3 architect. When you write require(msg.sender == owner) in Solidity, you now understand exactly how the EVM knows who msg.sender is - it ran the same ecrecover on your signed transaction.

14 - Glossary

Key Terms You Must Know

ethers.SigningKey
Ethers.js class that wraps secp256k1 operations. Accepts a private key and exposes publicKey, compressedPublicKey, and sign() methods.
ethers.Wallet
Ethers.js class for a full wallet. Accepts a private key; exposes address, signMessage(), signTransaction(), and signTypedData().
EIP-191
Ethereum Improvement Proposal 191 - defines the personal sign message format. Prefix: "\x19Ethereum Signed Message:\n{length}". Prevents message signatures being replayed as transactions.
EIP-55
Ethereum checksum address encoding. Applies mixed-case capitalisation to hex address based on its Keccak-256 hash. Invalid capitalisation indicates a transcription error.
verifyMessage
ethers.js function that recovers the Ethereum address of the signer from a message and its signature. JavaScript equivalent of Solidity's ecrecover precompile.
CSPRNG
Cryptographically Secure Pseudorandom Number Generator. Produces random numbers that are computationally indistinguishable from true randomness. Node.js: crypto.randomBytes(). Never use Math.random() for key generation.
04 Prefix
The leading byte (0x04) of an uncompressed secp256k1 public key, indicating it contains both X and Y coordinates (65 bytes total). Must be stripped before Keccak-256 hashing for address derivation.

2
MCQ
● Medium
1 mark
In the wallet derivation pipeline, why must the 0x04 prefix byte be stripped from the public key before computing the Keccak-256 hash for address derivation?
  • A) The 0x04 prefix causes Keccak-256 to produce a different number of output bytes
  • B) Stripping the prefix is optional - it is done only for gas optimisation on-chain
  • C) By convention, Ethereum's address derivation hashes only the 64-byte (X, Y) coordinate data - not the prefix byte. Including it would produce a different hash and incorrect address
  • D) The compressed public key (without 04) must be used instead of the uncompressed key for address derivation
✅ Correct Answer: C

Ethereum's address derivation specification (Ethereum Yellow Paper) defines the address as the last 20 bytes of keccak256(pubKey[1:]) - explicitly slicing off the first byte (0x04 prefix). This prefix is a serialisation marker from the SEC (Standards for Efficient Cryptography) specification indicating "uncompressed point follows." It is not part of the actual curve coordinates. Hashing the raw 64-byte (X, Y) coordinates (without the prefix) produces the canonical Ethereum address. Including the 0x04 byte would hash 65 bytes instead of 64, producing a completely different 32-byte Keccak output and therefore a completely different address. Option D is wrong - the uncompressed public key is used (both X and Y), not the compressed form (X only).

3
MCQ
● Hard
1 mark
In the lab, signing the same message twice with wallet.signMessage(message) would produce two different signatures each time. But ethers.verifyMessage(message, sig1) and ethers.verifyMessage(message, sig2) both return the same address. Why do the signatures differ even though the same key signs the same message?
  • A) ethers.js adds a random salt to the message before signing - making each hash different
  • B) ECDSA uses a random nonce k per signing operation. Different k values produce different r and s values, even for the same message and private key - yet all verify correctly
  • C) The Ethereum network assigns a unique transaction ID to each signing operation, which is embedded in the signature
  • D) The block timestamp is included in the message hash before signing, making each signature unique
✅ Correct Answer: B

This directly applies the knowledge from Sub-Module 4.2. ECDSA's signing formula is r = (k × G).x mod n and s = k⁻¹(hash + r × privKey) mod n - where k is a fresh cryptographically random nonce per signature. Different k values produce different r values (the x-coordinate of the random curve point k × G), which in turn produce different s values. However, the ECDSA verification algorithm uses the public key and signature (r, s) to recover the same address regardless of which valid k was used - because the relationship between r, s, the message hash, and the private key is consistent. This is also why reusing k for two different messages is catastrophic (Sony PS3 attack) - it is the unique k per message that provides non-determinism. Ethers.js uses Node.js's CSPRNG for each k generation.
