02   Real-World Introduction Story

The Two Betting Platforms - One Survived, One Didn't

📖 Story

In 2023, two sports betting dApps launched within a month of each other. Both allowed users to bet on IPL cricket match outcomes, and both used smart contracts to hold funds and pay out winners automatically.


BetFast used a centralised oracle: a single server run by the founding team that scraped ESPNcricinfo for match results and pushed them on-chain. It was lightning-fast - results appeared within 30 seconds of a match ending. Gas costs were minimal because only one transaction was needed per match. For six months, everything ran perfectly. Then, during the IPL final, the server crashed under heavy load. The match result never reached the smart contract. ₹2 crore in user funds sat locked in the contract for three days while the founders scrambled to reboot the server. Users rioted on Twitter. The platform never recovered.


BetDecentral used a decentralised oracle network. Five independent node operators, each pulling from different sports data APIs, submitted the match result to an on-chain aggregation contract. The median result was accepted. When one node submitted a wrong score (its API had a bug), the median filter caught it. When another node went offline, the remaining four still delivered the result. The platform processed the IPL final without a hitch. The cost? Five on-chain transactions instead of one, totalling about ₹0.03 on MST. The lesson: redundancy is expensive only in gas; trust is expensive in reputation.

03   Background & History

The Evolution of Oracle Trust Models - From Single Servers to Zero-Knowledge Proofs

The history of oracles is a history of trust reduction. Every generation has attempted to push the trust boundary further out - from trusting a person, to trusting a committee, to trusting mathematics itself.

Phase 1: The Centralised Era (2015 - 2018)

In the earliest dApps, the developer was the oracle. They ran a Node.js server with a simple script: watch the blockchain for requests, fetch data from an API, call a function on the contract with the result. This was the era of Oraclize (later Provable), which provided "authenticity proofs" - cryptographic evidence that the data from a specific API had not been tampered with. However, Oraclize was still a single company; users had to trust that the company's server was honest and the TLS proofs were genuine. The single point of failure was glaring.

Phase 2: The Decentralised Era (2019 - Present)

Chainlink launched in 2017 and matured into the dominant decentralised oracle network by 2019. The key innovation: don't trust one oracle; trust a committee of independent oracles, economically incentivised to be honest. This model proved itself during the DeFi summer of 2020, securing billions in lending and trading protocols. Today, Chainlink's price feeds are the industry standard, and other decentralised networks like Tellor, Band Protocol, and API3 compete with variations on the committee model.

However, decentralised oracles are not perfect. They are slow (multiple nodes must submit), expensive (multiple transactions), and still carry residual trust in the committee majority. A 51% attack on the oracle nodes, or collusion, is theoretically possible. Also, they are ill-suited for large or private data, where submitting the full dataset on-chain is impractical or undesirable.

Phase 3: The ZK Era (Emerging)

The next frontier, which MST's WASMify spearheads, is the zero-knowledge oracle. Instead of trusting a committee, a single entity can perform off-chain computation and produce a ZK proof that the computation was correct. The smart contract verifies the proof. Trust shifts from people to mathematics. This model is trust-minimised, fast (single transaction), and can handle complex, private data. It's the holy grail of oracles - and it's becoming practical today.

Key insight: Every oracle model makes a trade-off between trust, cost, speed, and data complexity. There is no one-size-fits-all. The skill is knowing which model fits which use case.
04   Beginner-Friendly Explanation

One Referee vs a Panel of Judges - Which Do You Trust More?

⚽ The Analogy

A centralised oracle is like a single referee in a football match. He's fast, decisive, and costs less than a panel. But if he's bribed, or if he simply makes a mistake, the entire game is affected with no recourse. A decentralised oracle is like a panel of five judges in an Olympic gymnastics event. Each judge submits a score independently. The highest and lowest scores are discarded, and the average of the remaining three is taken. Bribing one judge is useless because their score is thrown out. Bribing three judges is expensive and risky. The panel is slower and more expensive, but far more trustworthy. A ZK oracle is like having a robot referee that records the entire game on video, runs a computer vision algorithm to detect fouls, and produces a cryptographic certificate that the algorithm was run correctly. You don't need to trust the robot - you can verify the certificate yourself.

Let's examine each model's strengths and weaknesses in a structured way:

C
CENTRAL
Centralised Oracle: The Single Source
A single server or service that fetches data and submits it on-chain. Strengths: Fast (one transaction), cheap (one gas fee), simple to implement. Weaknesses: Single point of failure - server downtime, hacking, or malicious operator can compromise the entire dApp. Also a single point of censorship - a government could pressure the operator to manipulate data. Best for: low-value, non-financial applications where speed matters more than absolute trust, or as a temporary solution during prototyping.
D
DECENTRAL
Decentralised Oracle Network: The Committee
Multiple independent node operators fetch data from multiple sources, aggregate it, and submit it on-chain via a coordinating contract. Strengths: No single point of failure, economic incentives for honesty (staking/slashing), battle-tested in DeFi securing billions. Weaknesses: Slower (multiple transactions must be submitted and aggregated), more expensive (multiple gas fees), still has residual trust in the committee majority, and complex to set up. Best for: high-value DeFi protocols, insurance, prediction markets - anywhere financial loss from bad data would be catastrophic.
Z
ZK-PROOF
ZK Oracle (WASMify): The Mathematical Certificate
A single entity executes off-chain computation and produces a zero-knowledge proof of correct execution. The smart contract verifies the proof. Strengths: Trustless (no need to trust any operator), fast (single transaction), can handle complex/private data, no need for a network of staked nodes. Weaknesses: ZK proof generation is computationally intensive (though improving rapidly), requires specialised tooling, and is still an emerging technology with a smaller developer ecosystem. Best for: supply chain (complex IoT data), identity verification (privacy-critical), AI oracles, and any use case where the data is too large, complex, or sensitive for traditional oracles.
05   Technical Deep Dive - Centralised Oracles

The Single-Provider Model - Simple, Fast, and Fragile

A centralised oracle is the simplest possible data bridge. The architecture is straightforward:

One server, one data source, one transaction. The simplicity is seductive, but the trust assumption is absolute: the oracle operator must be honest and always available.

Implementation Example

Here's a minimal centralised oracle in JavaScript using ethers.js, fetching the ETH/USD price from CoinGecko and submitting it to a smart contract on MST every 10 minutes:

const { ethers } = require("ethers");
const axios = require("axios");

// Connect to MST Testnet
const provider = new ethers.JsonRpcProvider("https://testnet-rpc.mstblockchain.com");
const wallet = new ethers.Wallet(process.env.PRIVATE_KEY, provider);

// Consumer contract ABI and address
const oracleConsumer = new ethers.Contract(
  "0xYourConsumerContract",
  ["function updatePrice(uint256 price) external"],
  wallet
);

async function fetchAndSubmit() {
  try {
    // Fetch price from CoinGecko API
    const response = await axios.get(
      "https://api.coingecko.com/api/v3/simple/price?ids=ethereum&vs_currencies=usd"
    );
    const price = Math.round(response.data.ethereum.usd); // e.g., 260000 (in cents)

    // Submit to smart contract
    const tx = await oracleConsumer.updatePrice(price);
    await tx.wait();
    console.log(`Price updated: ${price} at block ${tx.blockNumber}`);
  } catch (error) {
    console.error("Oracle update failed:", error.message);
  }
}

// Run every 10 minutes
setInterval(fetchAndSubmit, 10 * 60 * 1000);
fetchAndSubmit();

Failure Modes of Centralised Oracles

⚠️
Server Downtime
If the oracle server crashes, the data stops flowing. The smart contract operates on stale data or freezes entirely. In DeFi, stale prices can lead to incorrect liquidations or prevent them entirely, causing bad debt.
⚠️
Data Source Manipulation
Even if the oracle server is honest, the API it fetches from might be compromised. If a single exchange's price feed is hacked to show ETH = $1, the oracle will faithfully report that price, and the smart contract will act on it - potentially liquidating all positions.
⚠️
Malicious Operator
The oracle operator could intentionally submit false data to profit - for example, manipulating a price feed to trigger liquidations and then buying the liquidated collateral at a discount. This is an "oracle attack" and has happened multiple times in DeFi history.
⚠️ When is a centralised oracle acceptable?

Centralised oracles are acceptable for prototyping, testnets, low-value applications (where the cost of an attack exceeds the potential gain), and non-financial data (e.g., displaying a blog feed on a dApp). They are not acceptable for any application handling significant value, where a single malicious or failed data point could cause real financial damage.

06   Technical Deep Dive - Decentralised Oracles

Chainlink's Committee Model - Strength in Numbers

Chainlink's architecture solves the centralised oracle's single point of failure by introducing redundancy at every layer: multiple nodes, multiple data sources, and on-chain aggregation. Let's examine how each component contributes to trust minimisation.

The Five Layers of Chainlink Security

  1. Multiple Independent Node Operators: Instead of one server, a feed might use 7 - 21 independent nodes, each run by different entities with different infrastructure, located in different jurisdictions. No single operator controls the feed.
  2. Multiple Data Sources: Each node fetches from multiple APIs (e.g., Binance, Coinbase, Kraken, Gemini) and aggregates locally before submitting. This prevents a single exchange's API bug or manipulation from affecting the result.
  3. On-Chain Aggregation: The Aggregator contract receives all submissions and computes a weighted median. The median is resistant to outliers - if one node submits a wildly wrong price, it has zero effect on the final answer.
  4. Economic Incentives (Staking): Nodes stake LINK tokens. If they submit data that deviates from the consensus, their stake can be slashed. This creates a financial disincentive for dishonesty.
  5. Reputation System: Nodes build on-chain reputation over time. Consumers can choose feeds that use high-reputation nodes, creating a market for reliable data.

Node 5 submitted a bug price of $99, but the median ($2,000) was unaffected. Even if Nodes 4 and 5 colluded, they'd need to control a majority of nodes to influence the median.

Remaining Trust Assumptions in Chainlink

Despite its robustness, Chainlink is not fully trustless:

  • Committee majority collusion: If a majority of nodes collude, they can control the median. The cost of this attack is high (acquiring 51% of the staked LINK), but it's theoretically possible.
  • Data source manipulation at scale: If a major exchange's API is hacked across all nodes that use it, the median could be skewed if multiple nodes rely on the same compromised source.
  • Smart contract risk: The Aggregator contract itself could have a bug. Chainlink's contracts are audited, but no code is bug-free.
  • Liveness: If too many nodes go offline simultaneously, the feed may stop updating. Chainlink requires a minimum number of submissions before it considers an answer valid.

08   The Third Way - WASMify ZK Oracles

Verifiable Computation: Neither Centralised Nor Decentralised - Trustless

WASMify sidesteps the entire trust debate. Instead of asking "who do I trust to give me data?", it asks: "how can I verify the data is correct, regardless of who submitted it?"

How It Works - Step by Step

  1. Write a WASM module: Compose a program (in Rust, C, Python compiled to WASM) that defines the computation - e.g., "fetch temperature data from IoT sensor X, verify it stayed below -20°C for the entire journey, return TRUE/FALSE".
  2. Execute off-chain: Any party - the dApp operator, a third-party service, even a user - runs the WASM module on their hardware. The execution produces a result and a ZK proof that the computation was performed exactly as specified, with the correct inputs.
  3. Submit proof on-chain: The proof (a few hundred bytes) is submitted to an MST smart contract.
  4. Verify on-chain: The contract runs a verification algorithm. If the proof is valid, it accepts the result as truth and proceeds with its logic. If the proof is invalid, it rejects the submission.

🗝️ When to Use Which

Chainlink (Decentralised): Best for simple, high-frequency, public data where a well-established network already exists (e.g., ETH/USD price feeds). The infrastructure is mature and battle-tested.

WASMify (ZK): Best for complex, private, or novel data where no oracle network exists (e.g., supply chain IoT data, identity verification, AI inference). Also ideal when you want minimal trust assumptions and can tolerate the proof generation overhead.

Centralised: Best for prototypes, testnets, and non-critical applications. Never for production financial applications.

10   Common Pitfalls

Oracle Architecture Mistakes That Cost Real Money

⚠️
Using a centralised oracle for a production DeFi protocol
This is the #1 mistake. A centralised oracle may work for months, but it will fail at the worst possible moment - during a market crash, when liquidations are critical and the server is overloaded. The cost of a decentralised oracle (₹0.04/update on MST) is negligible compared to the reputational and financial damage of a failure.
⚠️
All nodes using the same data source
A decentralised oracle with 21 nodes, all fetching from the same API, is effectively a centralised oracle with extra steps. True decentralisation requires source-level diversity. Each node should fetch from different, independent data providers.
⚠️
Ignoring the freshness of oracle data
An oracle that updates every hour is useless for a lending protocol where prices can move 10% in minutes. Always define acceptable staleness thresholds in your smart contract (e.g., require(block.timestamp - updatedAt < 300, "Stale price")).

12   Glossary

Key Terms - Sub-Module 13.2

Centralised Oracle
A single data provider that fetches external data and submits it on-chain. Fast and simple but creates a single point of failure.
Decentralised Oracle
A network of independent node operators that fetch data from multiple sources, aggregate it, and submit on-chain. Trust-minimised through redundancy and economic incentives.
Chainlink
The leading decentralised oracle network. Uses staking, reputation, and median aggregation to provide reliable data feeds for DeFi.
ZK Oracle
An oracle that uses zero-knowledge proofs to verify off-chain computation, eliminating the need to trust the data provider. MST's WASMify implements this model.
Median Aggregation
A statistical method where the middle value of sorted submissions is selected, making the result resistant to outliers and manipulation.
Source-Level Diversity
The practice of having oracle nodes fetch data from different, independent data providers to avoid a shared single point of failure at the data source.
WASMify
MST's framework for verifiable off-chain computation. Executes WASM modules off-chain, generates ZK proofs, and verifies them on-chain.
Staking (in oracles)
Node operators locking tokens as collateral to incentivise honest behaviour. Bad data results in slashing (loss of stake).
