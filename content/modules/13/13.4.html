02   Real-World Introduction Story

The Logistics Company That Replaced Auditors with Code

📖 Story

For years, a global logistics company had relied on third-party auditors to certify that temperature-sensitive pharmaceutical shipments had remained within safe ranges during transit. The auditors charged ₹20,000 per shipment, took three days to produce a report, and required access to the company's sensitive data. The company dreamed of a system where compliance was proven automatically, instantly, and without revealing the full temperature log to anyone.


An engineer on the team learned about MST's WASMify. She designed a feed: a small Rust program that would read the temperature log, check every data point against the 8°C threshold, and output a simple true or false. This program was compiled to WebAssembly. An off-chain server executed it against the real shipment data, and a zero-knowledge proof of correct execution was generated. The proof - just 200 bytes - was submitted to an MST smart contract. The contract verified the proof in milliseconds and released payment. The entire pipeline cost ₹0.024 in gas and ran in under a minute (excluding proof generation). No auditor, no data leak, no trust required. The company saved ₹2 crore annually in audit costs.

03   Background & History

The Evolution of Oracle Feed Design - From Push to Prove

Traditional oracle feeds follow a "push" model: an operator fetches data and pushes it on-chain via a transaction. The design is simple: a cron job calls an API, signs a transaction, and updates a storage variable. But this model bakes in trust assumptions. What if the API lies? What if the server is compromised? What if the operator's private key is stolen?

The Chainlink model improved on this by distributing trust across a committee of nodes, using economic incentives to discourage dishonesty. But it still involves multiple parties submitting data on-chain - which is expensive for large datasets and exposes data to multiple operators.

WASMify introduces a fundamentally different design paradigm: prove, don't trust. Instead of a trusted operator or a committee, a WASMify feed has one executor that runs a computation and produces a ZK proof. The smart contract verifies the proof. This shifts the trust from people to mathematics. But it also shifts the design: you no longer design a "data feed", you design a computation feed. The output is not the data itself, but a truth claim about the data - certified by a proof.

This sub-module teaches you how to think like a WASMify architect. You'll learn to decompose real-world problems into verifiable computations, design the off-chain module, and wire it up to MST contracts.

Key insight: In a WASMify feed, you're not asking "What data do I need on-chain?" You're asking "What computation on external data can I prove, such that the result is sufficient for my smart contract?"
04   Beginner-Friendly Explanation

The Recipe for a Trustless Data Meal

👨‍🍳 The Analogy

Designing a WASMify feed is like writing a recipe for a dish that must be certified organic. You don't ship the raw vegetables to the inspector. You write a recipe that takes the vegetables as input, cooks the dish, and produces a certificate that says "This dish was made from organic ingredients following the exact recipe, and the chef never saw the inspector." The inspector just verifies the certificate. The recipe is your WASM module, the certificate is the ZK proof, and the inspector is the MST smart contract.

Every WASMify feed design answers five questions:

①
INPUT
What external data does the feed need, and where does it come from?
Identify the data sources (IoT sensors, APIs, databases) and the format. The data may be private (only the prover sees it) or public (submitted alongside the proof). For privacy, the data stays off-chain and the ZK proof hides it. For public verifiability, the data or a hash of it can be included in the proof's public inputs.
②
COMPUTE
What computation does the module perform, and what is the output?
Define the WASM module's logic. It should be deterministic and produce a simple output (boolean, integer, hash). The computation can be as complex as needed (machine learning inference, data stream processing) because it runs off-chain. The output is what the smart contract will consume.
③
PROOF
How is the ZK proof generated, and what does it certify?
Choose a ZK proving backend (RISC Zero, zkWASM). The prover takes the WASM binary, the input data, and produces a proof that the WASM module executed correctly, producing the claimed output. The proof is succinct and can optionally conceal the input.
④
VERIFY
How does the MST smart contract verify the proof and use the result?
Deploy a consumer contract that calls the WASMify Verifier contract with the proof and the expected output. If the proof is valid, the contract accepts the output and proceeds with its logic. Design the contract's state machine to handle the result (e.g., release escrow, update a status).
⑤
TRIGGER
What triggers the off-chain execution and proof submission?
Define the automation: a cron job, an event listener on the contract, or a manual user action. The off-chain executor watches for requests, runs the module, generates the proof, and submits it. Ensure the system can handle failures gracefully (e.g., retry on proof generation failure).
05   Technical Deep Dive - Designing a WASMify Feed

A Complete Walkthrough: Age Verification for Web3 Delivery

Let's design a feed that proves a user is over 18 without revealing their birthdate. This is a classic privacy-preserving oracle use case.

Step 1: Define the Computation and Interfaces

The WASM module will accept a government-issued digital ID (e.g., a signed XML document) as input. It will parse the birthdate, compute the age, and output a boolean: 1 if age ≥ 18, 0 otherwise. The input must be kept private - only the prover sees it. The output and the proof are public.

WASM module interface (pseudocode in Rust):

// age_verify.rs
#[no_mangle]
pub extern "C" fn verify_age(id_data_ptr: *const u8, id_len: usize) -> u32 {
    // id_data: raw bytes of the digital ID
    let id_bytes = unsafe { std::slice::from_raw_parts(id_data_ptr, id_len) };
    // Parse the ID (simplified: assume it's a JSON with a "birthdate" field)
    let id: Identity = serde_json::from_slice(id_bytes).unwrap_or_default();
    let birthdate = id.birthdate; // format: "YYYY-MM-DD"
    let age = calculate_age(birthdate);
    if age >= 18 { 1 } else { 0 }
}

Note: In a real module, you'd verify the ID's digital signature to ensure it's a valid government ID, but that's beyond the scope of this example.

Step 2: Generate the ZK Proof

Compile the Rust code to WASM. Use RISC Zero's cargo risczero tooling to create a prover program that executes the WASM binary and generates a proof. The prover outputs a proof file and the public output (1 or 0).

Pseudocode for the off-chain executor:

// offchain_executor.js (Node.js pseudocode)
const { exec } = require('child_process');
const fs = require('fs');

async function processRequest(userIdDataBase64) {
  // 1. Save the ID data to a file (private)
  fs.writeFileSync('/tmp/input.dat', Buffer.from(userIdDataBase64, 'base64'));

  // 2. Run the RISC Zero prover
  await execPromise(`./prover --input /tmp/input.dat --output /tmp/proof.json`);

  // 3. Read the proof and public output
  const proof = JSON.parse(fs.readFileSync('/tmp/proof.json'));
  const output = proof.output; // 1 or 0
  const zkProof = proof.proof; // hex‑encoded proof

  // 4. Submit to MST contract
  const tx = await contract.verifyAge(zkProof, output);
  await tx.wait();
}

Step 3: On-Chain Consumer Contract

// AgeVerificationConsumer.sol
interface IWASMifyVerifier {
    function verify(bytes calldata proof, uint256 output) external view returns (bool);
}

contract AgeVerification {
    IWASMifyVerifier public verifier;
    mapping(address => bool) public isVerifiedAdult;

    event AgeVerified(address indexed user, bool isAdult);

    constructor(address _verifier) {
        verifier = IWASMifyVerifier(_verifier);
    }

    function verifyAge(bytes calldata proof, uint256 result) external {
        require(verifier.verify(proof, result), "Invalid proof");
        bool adult = (result == 1);
        isVerifiedAdult[msg.sender] = adult;
        emit AgeVerified(msg.sender, adult);
    }
}

The contract is simple: it verifies the proof and stores the result. A delivery dApp can then read isVerifiedAdult to gate access.

Step 4: Automation and Scalability

For a high-volume dApp, you'd run a lightweight service that listens for on-chain events (e.g., VerificationRequested) and automatically triggers the prover. The service can be hosted on a small cloud VM. Because MST gas is cheap, each verification costs ₹0.024 - easily covered by a small dApp fee.

06   Design Patterns

Feed Structures for Common Scenarios

While the core flow is similar, different applications require tweaks. Let's explore patterns with pseudocode.

Pattern 1: Supply Chain Batch Verification

Problem: Prove an entire shipment's temperature log never exceeded 8°C. The log has 50,000 data points.

WASM module (pseudocode):

pub fn check_shipment(temps: &[f32]) -> bool {
    temps.iter().all(|&t| t <= 8.0)
}

The prover runs this on the entire log. The proof certifies that all 50,000 values were checked. The contract only sees the boolean output. Gas: ~₹0.024 per shipment.

Pattern 2: AI Model Inference as Oracle

Problem: A DeFi protocol wants to use a credit risk model to approve loans. The model is proprietary, and the applicant's data is sensitive.

WASM module (pseudocode):

pub fn predict_risk(applicant_data: &[u8]) -> u8 {
    let model = load_model(MODEL_WEIGHTS); // embedded in WASM binary
    let risk_score = model.predict(applicant_data);
    risk_score // 0‑100
}

The model weights are compiled into the WASM binary (or loaded from a private source). The proof guarantees the model ran correctly on the given input. The contract receives the risk score and decides loan terms. The model's intellectual property is protected because the weights are never revealed on-chain.

Pattern 3: Periodic Price Feed with Historical Average

Problem: Provide a trustless 24-hour TWAP (time-weighted average price) of MSTC/USD, updating every hour.

WASM module: Fetches price data from multiple exchange APIs over the last 24 hours, computes the TWAP, and outputs the result. The input data (the price history) can be public, included alongside the proof for transparency.

pub fn compute_twap(prices: &[(u64, f64)]) -> f64 {
    // prices: array of (timestamp, price)
    let sum: f64 = prices.iter().map(|&(_, p)| p).sum();
    sum / prices.len() as f64
}

Here the prover can submit the price array alongside the proof, making it publicly verifiable. The contract can log the data for historical reference.

09   Common Pitfalls in WASMify Feed Design

Avoid These Mistakes

⚠️
Forgetting input authenticity
A ZK proof proves correct computation, but if the input data is fake, the output is meaningless. Always sign inputs at the source (e.g., a hardware root of trust for IoT, a government-signed ID). The proof should include verification of these signatures.
⚠️
Neglecting to pin the WASM binary's hash
The verifier contract must know exactly which WASM module was executed. If the module can be swapped out, the proof is meaningless. The contract should either contain the hash of the approved WASM binary or verify the proof against a known commitment. In practice, the ZK proving key itself encodes the binary; publishing the verification key on-chain solves this.
⚠️
Ignoring proof generation time
Complex WASM modules can take minutes to prove. For time-sensitive applications (e.g., real-time trading), you may need to batch proofs or use a faster proving backend. Design your feed with the expected latency in mind.

11   Glossary

Key Terms - Sub-Module 13.4

WASMify Feed
A trustless oracle feed using off-chain WebAssembly computation and on-chain ZK proof verification to provide smart contracts with real-world data.
Off-Chain Executor
The server or service that runs the WASM module, generates the ZK proof, and submits it to the blockchain.
ZK Prover
Software that takes a computation trace and produces a zero-knowledge proof. Runs off-chain and is computationally heavy.
WASMify Verifier
A pre-deployed MST smart contract that cryptographically verifies ZK proofs generated by WASMify feeds.
Deterministic Module
A WASM module that always produces the same output for the same input, a requirement for ZK proving.
Consumer Contract
The smart contract that uses the verified data from a WASMify feed to execute business logic (e.g., release payment).
