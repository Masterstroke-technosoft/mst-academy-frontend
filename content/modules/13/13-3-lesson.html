02   Real-World Introduction Story

The Cold-Chain Miracle - How a Mumbai Pharma Company Saved ₹50 Lakh in Lost Vaccines

📖 Story

In 2024, a pharmaceutical logistics company in Mumbai was shipping a ₹2 crore batch of temperature-sensitive vaccines from Pune to Dubai. The contract with the buyer specified that the vaccines must never exceed 8°C during transit. If they did, the entire shipment would be rejected and destroyed - a ₹2 crore loss.


The shipping container was fitted with IoT sensors that recorded temperature every 30 seconds. That's over 57,000 data points for a 48-hour journey. The buyer didn't trust the seller's word; the seller needed a trustless way to prove compliance.


Enter WASMify on MST. The logistics company wrote a simple program in Rust that read the entire temperature log, checked every data point against the 8°C threshold, and output a single boolean: true if the shipment was compliant, false otherwise. This program was compiled to WebAssembly and run on an off-chain server. The server produced a zero-knowledge proof - a cryptographic certificate that the program was executed correctly with the actual temperature data - and submitted only the proof and the boolean result to an MST smart contract. The contract verified the proof in milliseconds, and the payment was released automatically. The buyer received cryptographic certainty; the seller received instant payment. Neither party had to trust the other. The total gas cost on MST? ₹0.008. The equivalent using a traditional oracle that submitted all 57,000 data points on-chain would have cost over ₹3,00,000 in gas on Ethereum and exposed sensitive logistics data to the world. WASMify made the impossible practical.

03   Background & History

From WebAssembly to Zero-Knowledge - The Twin Revolutions

WASMify sits at the intersection of two transformative technologies: WebAssembly (WASM) and zero-knowledge proofs (ZKPs). Understanding each is key to grasping why WASMify is a paradigm shift.

WebAssembly: The Universal Portable Binary

WebAssembly was announced in 2015 as a joint effort by Google, Mozilla, Microsoft, and Apple. The goal was to create a fast, safe, portable binary format that could run anywhere - browsers, servers, embedded devices. Unlike JavaScript, WASM is a compilation target: you write code in Rust, C, C++, Go, or Python (via toolchains), and compile it to .wasm. The WASM runtime executes this binary in a sandboxed environment with near-native performance. In the blockchain world, WASM is revolutionary because it allows complex, non-deterministic logic to be run off-chain, while still being verifiable. Instead of trying to cram everything into the EVM, developers can write in familiar languages, run the code wherever they want, and only submit a proof of correct execution to the chain.

Zero-Knowledge Proofs: Verifying Without Revealing

Zero-knowledge proofs were first conceptualized in the 1980s by Shafi Goldwasser, Silvio Micali, and Charles Rackoff. A ZKP allows a prover to convince a verifier that a statement is true without revealing any additional information. In the context of computation, a ZK-SNARK (Succinct Non-Interactive Argument of Knowledge) or ZK-STARK (Scalable Transparent Argument of Knowledge) can prove that a specific program was executed correctly on specific inputs, producing a specific output - all without revealing the inputs or requiring the verifier to re-run the program. The proof is tiny (a few hundred bytes) and verification is fast (a few milliseconds). This is exactly what blockchains need: a way to verify off-chain computation without trusting the executor.

MST's Synthesis: WASMify

MST's WASMify bridges these two worlds. It provides a framework where developers can write a WASM module, execute it off-chain, and generate a ZK proof of execution using a proving backend (like RISC Zero, SP1, or zkWASM). The proof and the output are submitted to an MST smart contract, which verifies the proof using a pre-deployed verifier contract. If the proof is valid, the contract accepts the output as truth. Because the verification is purely mathematical, there is no trust assumption - not in the executor, not in the data provider, not in any committee. Trust the math, not the messenger.

The WASMify Advantage: Write complex logic in Rust, Python, or C → compile to WASM → execute anywhere → generate ZK proof → submit proof to MST → verify instantly → trustless result. All with fees measured in paise.
04   Beginner-Friendly Explanation

The Magic Box That Proves It Did Its Homework

🎁 The Analogy

Imagine you're a teacher, and a student claims they solved a very complex math problem. You don't have time to check every step of their work - that would take hours. Instead, the student hands you a sealed magic box. You press a button, and a green light flashes along with a piece of paper that says "Correct." You trust the green light because the box is mathematically guaranteed to only flash green if the student did the work correctly. You don't need to see their notebook, their scratch paper, or even know what numbers they used. The box proves it. WASMify is that magic box for blockchain: the student is the off-chain server, the math problem is the real-world data processing, and the green light is the ZK proof verified by the smart contract.

Let's explore the architecture of WASMify step by step:

①
WASM
What is a WASM module in WASMify?
A WASM module is a compiled program that performs a specific off-chain computation. It can be written in Rust, C, C++, Python (via Pyodide or similar), or any language that targets WebAssembly. The module defines: (1) the inputs it expects, (2) the computation logic, and (3) the output it produces. For example, a module might take a temperature log file, check every data point against a threshold, and return a boolean. The module is deterministic: given the same inputs, it always produces the same output. This determinism is essential for generating a ZK proof.
②
EXEC
How is the module executed off-chain?
Any party - the dApp operator, a third-party service, or even the user - can run the WASM module on their own hardware. They provide the required inputs (the temperature log, an identity document, a GPS trace) and execute the module. The execution produces an output (the result) and an execution trace (a record of every computational step). The execution trace is the raw material for generating the ZK proof. The executor never needs to be trusted - if they lie about the output, the proof will be invalid. If they use incorrect inputs, the proof will be invalid (unless the inputs themselves are also proven, which is possible with recursive proofs).
③
PROOF
How is the ZK proof generated?
The execution trace is fed into a ZK prover - a specialized piece of software that mathematically encodes the entire computation into a cryptographic proof. This proof is succinct (small, typically a few hundred bytes) and non-interactive (the verifier doesn't need to communicate with the prover). The proof certifies: "There exists some input for which this WASM module, when executed, produced this output, and the execution was valid." The proof does not reveal the input, preserving privacy. The prover is computationally intensive - generating a proof for a complex program can take seconds to minutes - but it only needs to be done once per submission. The result: a tiny, trustless certificate of computation.
④
VERIFY
How does the MST smart contract verify the proof?
The proof and the output are submitted to an MST smart contract. The contract calls a verifier function - a piece of Solidity code that implements the ZK verification algorithm. This verification is fast and cheap: it takes a few milliseconds of EVM time and costs about 200,000 - 300,000 gas (approximately ₹0.02-₹0.03 on MST). If the proof is valid, the contract accepts the output as truth and proceeds with its logic - releasing a payment, minting an NFT, or updating a state variable. If the proof is invalid, the transaction reverts. The verifier does not need to know the original inputs; it only checks the mathematical validity of the proof.
05   Technical Deep Dive - The WASMify Pipeline

From Rust Code to Verifiable On-Chain Truth - A Complete Walkthrough

Let's follow a concrete WASMify feed from start to finish. We'll use the cold-chain vaccine example.

Step 1: Write the WASM Module in Rust

// cold_chain_check.rs
#[no_mangle]
pub extern "C" fn check_temperature(temps_ptr: *const u8, len: usize) -> u32 {
    let temps = unsafe { std::slice::from_raw_parts(temps_ptr, len) };
    for &temp in temps {
        if temp > 8 { // Temperature exceeded 8°C
            return 0; // false — shipment failed
        }
    }
    1 // true — shipment compliant
}

This simple Rust function takes a pointer to a byte array of temperatures and its length, checks each byte, and returns 1 (pass) or 0 (fail). It is compiled to WASM using wasm-pack or rustc --target wasm32-unknown-unknown.

Step 2: Generate the ZK Proof

The WASM binary is loaded into a ZK proving framework that supports WASM execution traces, such as RISC Zero (which uses the RISC-V ISA as an intermediate layer) or zkWASM (a specialized ZK prover for WASM). The framework executes the WASM binary with the actual temperature data as input, producing:

  • The output (e.g., 1)
  • A ZK proof (a few hundred bytes, often serialized as a hex string)
  • Optionally, a public input commitment - but in this privacy-preserving example, the temperature data is kept private

Step 3: Submit to the MST Smart Contract

A transaction is sent to the WASMify consumer contract on MST, calling a function like verifyAndSettle(bytes calldata proof, uint256 output). The proof is passed as calldata (cheap on MST). The gas cost for the transaction is the base 21,000 gas plus the cost of the verification algorithm (e.g., 200,000 gas). Total: ~221,000 gas, or ~₹0.021 on MST.

Step 4: On-Chain Verification and Action

The consumer contract calls the WASMify Verifier Contract, a pre-deployed contract on MST that contains the ZK verification logic. The verifier checks the proof against the claimed output. If valid, the consumer contract executes its business logic:

function verifyAndSettle(bytes calldata proof, uint256 result) external {
    require(wasmifyVerifier.verify(proof, result), "Invalid proof");
    if (result == 1) {
        // Shipment compliant — release payment
        payable(seller).transfer(amount);
        emit ShipmentCertified(shipmentId);
    } else {
        // Shipment failed — refund buyer
        payable(buyer).transfer(amount);
        emit ShipmentRejected(shipmentId);
    }
}

🗝️ The WASMify Paradigm

Traditional oracles ask: "Who can I trust to bring data on-chain?" WASMify asks: "No one. I'll verify the computation mathematically." This shift from economic trust (staking, reputation, committees) to cryptographic trust (ZK proofs) is the defining feature of WASMify.

🚚 Supply Chain: Verifying Cold-Chain Compliance

Problem: Pharmaceutical shipments must be proven to stay within temperature limits without exposing the full sensor log to competitors or regulators.

Problem: A Web3 delivery app needs to verify users are over 18, but storing or transmitting government IDs is a privacy and legal liability.

WASMify Solution: A WASM module takes the user's digital ID (e.g., Aadhaar XML) as a private input, extracts the birthdate, computes age, and outputs a boolean (over18 = true/false). The ZK proof guarantees the computation was correct without revealing the birthdate or ID number. The smart contract verifies the proof and grants access if over18 is true.

Why not Chainlink? Chainlink nodes would need to see the ID to verify it, which is a non-starter for privacy regulations. WASMify keeps the ID entirely off-chain.

🤖 AI Inference: Verifying a Machine Learning Model's Prediction

Problem: A DeFi protocol wants to use an AI model to predict loan default risk, but users need assurance that the model was actually run - not a fake result.

WASMify Solution: The AI model is compiled to a WASM module (using frameworks like ONNX-WASM or tract). The loan application data is passed as input. The module runs inference and outputs a risk score. A ZK proof is generated showing the model was executed correctly. The smart contract verifies the proof and uses the risk score to approve or deny the loan. The model itself remains proprietary, and the user's data remains private.

Why not Chainlink? AI inference is computationally heavy and would be prohibitively expensive to run on-chain or to submit via multiple oracle nodes. WASMify off-loads the heavy lifting off-chain while preserving trust.

🎲 Gaming: Provably Fair Randomness with FortunaX + WASMify

It produces random numbers and a ZK proof that the numbers were generated according to the specified algorithm. The smart contract verifies the proof and uses the random numbers for the game. Players can verify the proof themselves to ensure fairness, but the seed remains secret to prevent prediction.

09   Developer Workflow

Your First WASMify Feed - A Hands-On Guide

Ready to build? Here's the step-by-step developer workflow for creating a WASMify oracle on MST.

🛠️ Step 1: Write and compile your WASM module

Write your computation logic in Rust (recommended for ZK ecosystem support). Use the wasm32-unknown-unknown target. Keep the interface simple: a single function that takes a pointer and length, and returns an integer or a boolean. Test it locally with sample data.

🔐 Step 2: Generate a ZK proof for your first execution

Use a ZK proving backend compatible with WASM. MST's developer toolkit provides a CLI that integrates with RISC Zero or zkWASM. Run your module with real data and generate a proof file (JSON or hex). Store the proof - you'll need it for the on-chain submission.

📜 Step 3: Deploy your consumer contract on MST Testnet

Write a Solidity contract that imports the WASMify Verifier interface and calls verifier.verify(proof, output). Deploy it using Hardhat with MST Testnet configuration. Cost: ~₹0.02.

🚀 Step 4: Submit the proof and watch the magic

From a script or your frontend, call verifyAndSettle(proof, output) on your consumer contract. The transaction will verify the proof on-chain. If it succeeds, your business logic executes. If the proof is bad, it reverts.

🔄 Step 5: Automate the pipeline for production

Set up a lightweight server (could be a serverless function) that listens for events on your consumer contract, runs the WASM module with fresh data, generates a proof, and submits it. This closes the loop, creating a fully automated, trustless oracle feed.

11   Common Pitfalls

Avoid These Mistakes When Building with WASMify

⚠️
Assuming the proof guarantees input correctness
A ZK proof certifies that if the prover used the claimed inputs, the output is correct. It does not certify that the inputs themselves are true. For example, a WASM proof can show that a temperature log was checked correctly, but it cannot guarantee that the temperature log accurately reflects the actual temperatures. That requires hardware attestation or trusted sensors - a separate problem.
⚠️
Underestimating proof generation time
ZK proof generation is computationally intensive. For complex WASM modules, generating a proof can take several minutes on consumer hardware. This is fine for hourly updates but may not suit sub-second applications. Use it for periodic certifications, not real-time trading.
⚠️
Hardcoding the verifier contract address
The WASMify Verifier contract may be upgraded or moved. Always use a registry or a proxy to look up the latest verifier address, rather than hardcoding it, to ensure your consumer contract remains functional after updates.

13   Glossary

Key Terms - Sub-Module 13.3

WebAssembly (WASM)
A portable binary instruction format enabling high-performance execution of code written in multiple languages in a sandboxed environment.
Zero-Knowledge Proof (ZKP)
A cryptographic method that proves a statement is true without revealing the underlying data or requiring the verifier to re-compute.
WASMify
MST's framework combining WASM off-chain execution with ZK proof generation and on-chain verification for trustless oracles.
ZK Prover
Software that takes an execution trace and produces a ZK proof. Runs off-chain and is computationally heavy.
Verifier Contract
An MST smart contract that cryptographically verifies a ZK proof. Fast and cheap on-chain.
Execution Trace
A record of every step taken during a computation, used by the ZK prover to construct a proof of correct execution.
RISC Zero
A popular ZK proving framework that uses the RISC-V instruction set as an intermediate representation for proving WASM execution.
zkWASM
A specialized ZK prover designed to work directly with WebAssembly bytecode, generating proofs of WASM execution.
