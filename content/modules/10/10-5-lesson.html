02   Real-World Story

The Bug That Testing Missed - But Mathematics Caught

📖 Story

In 2023, a leading DeFi protocol was preparing to launch a new version of their staking contract. The contract had been through three rounds of professional auditing. It had over 200 unit tests, all passing. It had been fuzzed with Foundry for a million runs, and no invariant violations were found. The developers were confident.


As a final check, they wrote a formal specification using Certora and ran the prover. Within minutes, Certora returned a counterexample: a specific sequence of operations where a user could claim more rewards than they had earned. The bug was in a subtle interaction between the stake and compound functions: if a user called compound at precisely the right moment after a large deposit, the reward accumulator was updated incorrectly due to a rounding edge case.


This bug had survived three audits. It had survived a million fuzz runs. But formal verification found it, because formal verification doesn't sample - it explores the entire mathematical space of possible inputs and states. The protocol delayed their launch, fixed the bug, and ran the prover again. This time, the rule was verified: the prover mathematically proved that no sequence of operations could cause the reward accounting to break.


This story is not an exception. It's the new normal. Formal verification is becoming standard practice for high-value protocols precisely because testing - no matter how thorough - can only ever check a tiny fraction of the infinite possibilities. Formal verification checks them all.

03   What Is Formal Verification?

Proving Correctness Instead of Testing for Bugs

Formal verification is a mathematical technique that proves a program behaves correctly for all possible inputs and all possible states. Instead of running the program with specific values and checking the output (testing), formal verification analyses the program's logic and proves that certain properties always hold, no matter what.

The fundamental difference:

Testing asks: "For this specific input, does the contract behave correctly?" If the test passes, you know the contract works for that one case. You don't know anything about any other case.

Fuzzing asks: "For these thousands of random inputs, does the contract behave correctly?" If the fuzz tests pass, you have statistical confidence. But there's always a chance that the fuzzer missed the one input that breaks the contract.

Formal verification asks: "Can there exist ANY input or state that breaks this property?" If the prover says "no," you have mathematical certainty that the property holds for every possible scenario. This is a fundamentally different level of assurance.

Formal verification provides the highest level of assurance, but it requires more effort to write specifications.

How formal verification works: The core technology behind modern formal verification tools is SMT solvers (Satisfiability Modulo Theories). An SMT solver is a program that can determine whether a set of mathematical equations and logical constraints is satisfiable - that is, whether there exists any assignment of values that makes all the equations true simultaneously.

When you write a formal specification, you're describing the expected behavior of your contract as mathematical equations. The prover then combines your specification with the actual Solidity code (also translated into mathematical equations) and asks the SMT solver: "Is there any way the code can violate the specification?" If the solver finds a way, it produces a counterexample - the exact input values and function call sequence that triggers the violation. If the solver cannot find any violation, the specification is proven.

04   Certora - The Formal Verification Tool for EVM Contracts

Writing Specifications in Certora Verification Language (CVL)

Certora is the leading formal verification platform for Ethereum-compatible smart contracts. It works by having you write specifications in a language called CVL (Certora Verification Language), which are then checked against your contract's bytecode (or Solidity source) by the Certora Prover. Certora is used by Aave, Compound, MakerDAO, and many other top protocols.

How Certora works, step by step:

1. You write your contract in Solidity, exactly as you normally would. No changes are needed to the contract itself.

2. You write a specification file (a .spec file) that contains rules describing the expected behavior of your contract.

3. You run the Certora Prover on your contract and specification. The prover translates both into mathematical formulas and uses SMT solvers to check whether any violation exists.

4. The result: Either the rule is verified (the property is proven to hold for all possible executions), or a counterexample is found (a specific input and call sequence that breaks the property).

Types of specifications in Certora:

Rules are checked against specific function calls. They define what should happen when a particular function is called. For example: "After calling transfer, the sender's balance must decrease by exactly the transferred amount."

Invariants are checked after every possible function call (or after every call to a specified set of functions). They define properties that must always hold, no matter what sequence of operations has been performed. For example: "The total supply must always equal the sum of all balances."

// Example Certora specification for an ERC‑20 token

// Import the contract interface
import "./Token.sol";

// Define methods that the prover should use
methods {
    function balanceOf(address) external returns (uint256) envfree;
    function totalSupply() external returns (uint256) envfree;
}

// Rule: transfer must correctly update balances
rule transferCorrectness(address sender, address recipient, uint256 amount) {
    // Store the balances before the transfer
    uint256 senderBalanceBefore = balanceOf(sender);
    uint256 recipientBalanceBefore = balanceOf(recipient);

    // Perform the transfer (from sender to recipient)
    env e; // create an environment (msg.sender, block.timestamp, etc.)
    transfer(e, recipient, amount);

    // Check the balances after the transfer
    assert balanceOf(sender) == senderBalanceBefore - amount,
        "Sender balance not decreased correctly";

    assert balanceOf(recipient) == recipientBalanceBefore + amount,
        "Recipient balance not increased correctly";
}

// Invariant: total supply must always equal sum of all balances
// This is checked after every possible function call
invariant totalSupplyEqualsSumOfBalances()
    totalSupply() == balanceOf(address(0x1)) + balanceOf(address(0x2)) + balanceOf(address(0x3));

This specification defines two properties. The transferCorrectness rule checks that a transfer correctly updates both the sender's and recipient's balances. The totalSupplyEqualsSumOfBalances invariant checks that the total supply remains consistent with the sum of all individual balances after any operation.

05   Writing Your First Formal Rule - A Step-by-Step Guide

Turning English Requirements into Mathematical Guarantees

Let's walk through the complete process of formally verifying a simple contract. We'll use a basic savings contract where users can deposit and withdraw ETH, and the contract tracks each user's balance.

Step 1: Write the contract.

// Savings.sol – A simple savings contract
contract Savings {
    mapping(address => uint256) public balances;
    uint256 public totalDeposits;

    function deposit() public payable {
        balances[msg.sender] += msg.value;
        totalDeposits += msg.value;
    }

    function withdraw(uint256 amount) public {
        require(balances[msg.sender] >= amount, "Insufficient balance");
        balances[msg.sender] -= amount;
        totalDeposits -= amount;
        payable(msg.sender).transfer(amount);
    }
}

Step 2: Identify the properties you want to verify. Think about what must always be true, no matter what sequence of deposits and withdrawals happens. Here are some good properties:

P1: A user's balance can never be negative.

P2: A user cannot withdraw more than they deposited.

P3: The total deposits must equal the sum of all users' balances. (This is an invariant.)

P4: After a deposit of X, the user's balance must increase by exactly X.

Step 3: Translate these properties into Certora rules.

// Savings.spec – Certora specification
import "./Savings.sol";

methods {
    function balances(address) external returns (uint256) envfree;
    function totalDeposits() external returns (uint256) envfree;
}

// Rule: After depositing X, the user's balance increases by X
rule depositIncreasesBalance(address user, uint256 amount) {
    uint256 balanceBefore = balances(user);

    env e;
    require e.msg.value == amount;
    deposit(e, user);  // call deposit as user with `amount` ETH

    assert balances(user) == balanceBefore + amount,
        "Deposit did not increase balance correctly";
}

// Rule: After withdrawing X, the user's balance decreases by X
rule withdrawDecreasesBalance(address user, uint256 amount) {
    uint256 balanceBefore = balances(user);
    require balanceBefore >= amount;

    env e;
    withdraw(e, user, amount);

    assert balances(user) == balanceBefore - amount,
        "Withdraw did not decrease balance correctly";
}

// Invariant: Total deposits equals sum of all balances
invariant totalDepositsMatchesBalances(address[] users)
    totalDeposits() == sumBalance(users);

Step 4: Run the Certora Prover. The prover will explore every possible deposit and withdrawal sequence, checking each rule and the invariant. If any property can be violated, Certora will show the exact steps that cause it - for example, "deposit 10 ETH, withdraw 15 ETH, and the balance becomes negative."

🔑 What "Verified" Really Means

When Certora says a rule is verified, it means the SMT solver has mathematically proven that no possible execution of the contract can violate that rule. This is not a statistical likelihood. This is not "we tested it a lot." This is a mathematical proof. The property holds for every possible input value, every possible call sequence, and every possible blockchain state.

However, there is an important caveat: the verification is only as strong as the specification. If you forgot to specify a critical property, Certora won't check it. Formal verification proves that your code matches your specification - it doesn't guarantee that your specification captured every important requirement. This is why writing good specifications is a skill that requires deep understanding of the contract's intended behavior.

06   Common Patterns in Formal Specifications

Rules You Should Write for Every Contract

Over years of verifying DeFi protocols, the Certora community has identified patterns that catch the majority of bugs. Here are the specifications you should write for any new contract.

1. Balance Conservation: In any token or vault contract, the total amount of assets must be conserved. Tokens cannot be created from nothing or destroyed into nothing (unless there's an explicit mint/burn function). Write an invariant that checks: sum of all user balances = total supply (or total deposits).

2. Monotonicity of Accumulators: Variables that should only increase (like "total fees collected" or "total rewards distributed") must never decrease. Write a rule that checks: after any operation, the accumulator is ≥ its previous value.

3. No Underflow/Overflow: While Solidity 0.8+ catches arithmetic overflow, semantic overflow (where the logic allows a value to go below its intended minimum) is still possible. Write rules that check: after any operation, all balances are ≤ some maximum and ≥ some minimum.

4. Rounding Correctness: In financial contracts, rounding must always favor the protocol (or at least be consistent). Write a rule that checks: after a sequence of deposits, withdrawals, and compounding, no user can have received more than they mathematically earned.

5. Authorization: Critical functions must only be callable by authorized addresses. Write a rule that checks: if a non-owner calls an admin function, the transaction must revert.

⚠️ The Limits of Formal Verification

Formal verification is powerful but not magical. It cannot check properties you haven't specified. It cannot verify that your contract correctly implements the business logic - only that it matches the mathematical rules you wrote. It can be computationally expensive for very complex contracts. And it typically cannot reason about external contracts (oracles, other protocols) unless they are also formally modeled. Use formal verification as part of a comprehensive security strategy that includes manual review, testing, fuzzing, and auditing.

07   Formal Verification and MST Blockchain

Why Formal Verification Matters for MST

MST Blockchain is fully EVM-compatible, which means every formal verification tool that works for Ethereum works for MST. Certora, SMT solvers, and other verification tools can be applied to MST smart contracts without any modifications. This is a direct benefit of MST's decision to maintain EVM compatibility.

Furthermore, MST's unique features like WASMify can benefit from formal verification. WASMify allows running WebAssembly modules on-chain, and these modules can also be formally verified using the same techniques. The Certora Prover can operate on the bytecode of compiled WASM contracts, or you can write specifications for the Solidity wrappers that call WASM functions.

For high-value protocols launching on MST, formal verification should be a standard step in the development pipeline, right alongside unit testing and fuzzing. The low gas fees and high throughput of MST make it attractive for complex financial applications, which are exactly the kind of applications that benefit most from mathematical correctness guarantees.

08   Writing a Formal Specification in Plain English

The Most Important Skill: Thinking Before You Code

Before you write a single line of Certora code, you must be able to describe what your contract should do in clear, precise English. This is actually the hardest part of formal verification - and the most valuable. The process of writing a formal specification forces you to think deeply about your contract's invariants and edge cases.

Example: English specification for an ERC-20 token.

1. Balance Conservation: At all times, the sum of all token balances must equal the total supply.

2. Transfer Correctness: After a transfer of amount X from A to B (where A ≠ B and A has at least X tokens), A's balance must decrease by exactly X, and B's balance must increase by exactly X.

3. Transfer to Self: A transfer from A to A must not change A's balance (or, alternatively, must revert).

4. Transfer of Zero: A transfer of 0 tokens must not change any balances and must succeed (or, if your design requires, must revert).

5. Approve and TransferFrom: After B is approved to spend up to Y tokens from A, and B calls transferFrom(A, B, X) where X ≤ Y and X ≤ A's balance, then A's balance decreases by X, B's balance increases by X, and B's allowance from A decreases by X.

6. Mint: Only the minter address can mint tokens. After minting X tokens to address A, the total supply increases by X, and A's balance increases by X. No other balances are affected.

7. Burn: After burning X tokens from address A (where A has at least X tokens), the total supply decreases by X, and A's balance decreases by X.

This plain-English specification is the foundation. From here, you can translate each property into a Certora rule. The act of writing this specification will often reveal edge cases you hadn't considered: "What if someone transfers to the zero address?" "What if the total supply overflows?" "What if we mint to the contract itself?" Answering these questions before writing code prevents entire categories of bugs.

10   Glossary

Key Terms - Sub-Module 10.5

Formal Verification
A mathematical technique that proves a program satisfies a specification for all possible inputs and states. Uses SMT solvers and logical reasoning to guarantee correctness.
Certora
The leading formal verification platform for EVM-compatible smart contracts. Uses CVL (Certora Verification Language) to write specifications that are checked by the Certora Prover.
SMT Solver
Satisfiability Modulo Theories solver - a program that determines whether a set of mathematical equations and logical constraints can be satisfied. The core engine behind formal verification tools.
Invariant
A property that must hold at all times, regardless of what sequence of operations has been performed. Checked after every function call (or every call to a specified set).
Rule
A specification that describes the expected behavior of a specific function. Defines what should happen when that function is called, and is checked against all possible inputs.
Counterexample
A specific set of input values and function call sequences that violates a rule or invariant. Found by the prover when a property cannot be proven.
CVL
Certora Verification Language - the language used to write specifications for the Certora Prover. Combines Solidity-like syntax with mathematical operators.
Balance Conservation
An invariant stating that the total amount of assets in a contract must equal the sum of all individual balances. Prevents tokens from being created or destroyed illegally.
