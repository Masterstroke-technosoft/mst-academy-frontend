02   Real-World Story

The Missing Review That Cost ₹2 Crore

📖 Story

A Web3 gaming startup in Hyderabad had built a play-to-earn platform where players earned tokens by completing quests. The lead developer, working alone under a tight deadline, wrote a 400-line contract over two nights. The code worked. The test suite passed. The contract was deployed.


Three weeks later, a player noticed that if they called the claimReward function at exactly the right millisecond, they could claim the same reward twice. The bug was a classic reentrancy issue - the contract sent tokens before updating the reward counter. But there was a second, subtler problem: the contract used a for loop to iterate over all players to distribute daily bonuses. With 2,000 active players, the loop consumed 2.8 million gas - within the block limit. But the developer never asked: what happens at 10,000 players? At 50,000?


Five months later, the player base hit 8,000. The daily bonus function became permanently unusable - the loop consumed more gas than the block limit. The contract was frozen. ₹2 crore worth of tokens were locked forever. The post-mortem revealed that a single one-hour code review by another developer would have caught both issues immediately. The reentrancy was visible to a trained eye, and the unbounded loop was flagged by every static analysis tool.


Code review is not a formality. It is the last line of defense before immutability. In traditional software, you can fix bugs post-deployment. In blockchain, deployment is forever. Every minute spent reviewing code pays back in prevented disasters. This sub-module teaches you exactly how to perform that review - on your own code, on a teammate's code, and on the contracts you'll evaluate professionally.

03   Why Code Review Is Non-Negotiable for Smart Contracts

Immutability + Money = Every Line Must Be Justified

In traditional software development, code review is a best practice. In blockchain development, it is a survival requirement. Smart contracts are immutable, handle real money, and are publicly visible to attackers worldwide. A single un-reviewed line can be exploited within hours of deployment.

Code review catches what testing cannot:

1. Logical errors: Tests verify that code behaves as written. Reviews verify that code behaves as intended. A test might pass because the developer wrote the test to match the buggy code. A reviewer asks: "Is this actually correct?"

2. Architecture flaws: A function might work correctly in isolation but create a vulnerability when combined with other functions. Reviewers see the big picture that individual tests miss.

3. Gas inefficiencies: Tests rarely check gas consumption. A reviewer notices that a loop will become impossible at scale, or that a storage variable is read three times when once would suffice.

4. Readability: Code that is hard to read is hard to audit. A reviewer ensures that variable names are clear, comments are helpful, and the logic flows logically - so that future auditors and developers can understand the contract quickly.

5. Standards compliance: Is your ERC-20 token actually compliant with the standard? Does it emit the correct events with the correct parameters? A reviewer verifies these details that tests often overlook.

04   The Code Review Checklist - A Systematic Approach

Every Line, Every Function, Every Pattern

Professional auditors and senior developers don't review code randomly. They follow a structured checklist that ensures nothing is missed. Below is the checklist you should use for every smart contract you review - starting with your capstone project.

1
Security Vulnerabilities
Reentrancy: Any external call before state updates? Any function that sends ETH or calls an external contract? Apply Checks-Effects-Interactions or ReentrancyGuard.
Integer overflow/underflow: Any arithmetic that could overflow? Downcasting without validation?
Access control: Every state-changing function has a proper modifier? No public functions that should be restricted?
tx.origin: Search the entire codebase for tx.origin. If found, it's almost certainly a bug.
Front-running susceptibility: Can a transaction be front-run for profit? Does the contract use commit-reveal or Flashbots?
Denial of Service: Any unbounded loops? Any single-point dependency (single owner, single oracle)?
Signature verification: Are signed messages protected against replay attacks with nonce and expiry?
2
Gas Efficiency
Storage reads: Are the same storage variables read multiple times? Cache them in memory.
Storage writes: Are multiple writes happening when one would do? Batch updates.
Loops: Any loops that could grow unboundedly? Replace with mappings and counters.
Data types: Are smaller types (uint64, uint128) used appropriately for packing? Or are they used unnecessarily where uint256 would be cheaper?
String operations: Are strings stored on-chain? Could they be stored as bytes32 or moved off-chain?
Events: Are events emitted for all state changes? Are parameters indexed for efficient filtering?
Constructor vs. immutable: Variables set once at deployment should use immutable to save storage reads.
3
Readability & Maintainability
Function length: Any function over 30 lines? Extract helper functions.
Naming: Are variable and function names descriptive? calculateReward vs. calc?
Magic numbers: Any unexplained constants? Replace with named constants (e.g., uint256 public constant MAX_FEE = 1000;).
NatSpec comments: Every function has @notice and @param? Every contract has @title and @dev?
Consistent style: Indentation, brace placement, spacing - all consistent? Run a linter (Solhint) before review.
4
Standards & Interoperability
ERC compliance: Does your token correctly implement the standard? All mandatory functions present and correct? Events emitted correctly?
ABI compatibility: Does the ABI match the interface? Are function selectors correct?
Upgradeability: If using a proxy, is the storage layout preserved? Unstructured storage used for proxy variables?
External contract interactions: Are calls to other contracts checked for success? Are return values validated?
5
Testing Coverage
Unit tests: Every function tested? Happy path, edge cases, reverts?
Fuzz tests: Invariant tests written? Random inputs tested?
Coverage report: Branch coverage ≥ 90%? Uncovered lines reviewed and justified?
Formal verification: Critical invariants specified and proven? (If applicable)
05   Refactoring - Transforming Messy Code into Professional Contracts

Why Refactoring Matters and When to Do It

Refactoring is the process of restructuring existing code without changing its external behavior. The contract does the same thing - it's just cleaner, more efficient, and easier to audit. You refactor after you have a working, tested contract, but before deployment. You refactor to improve gas efficiency, readability, and security posture.

Common refactoring patterns:

1. Extract helper functions: If a function does three different things, break it into three functions. Each function should do one thing well.

2. Replace magic numbers with constants: require(msg.value >= 1000000000000000000) becomes require(msg.value >= MIN_DEPOSIT) where MIN_DEPOSIT is a named constant.

3. Cache storage in memory: If you read a state variable three times in a function, read it once into memory and use the local copy. This can save thousands of gas.

4. Use modifiers for shared conditions: If multiple functions check require(msg.sender == owner), create an onlyOwner modifier and apply it to all of them.

5. Replace arrays with mappings for large lists: If an array can grow without bound, replace it with a mapping and a counter. This prevents the contract from becoming unusable.

6. Short-circuit evaluation: Put the cheapest condition first in if and require statements. If the cheap condition fails, the expensive one is never evaluated, saving gas.

7. Remove dead code: Any variable that is written but never read? Any function that is never called? Delete it. Every line costs gas to deploy.

06   Before-And-After Refactoring Example

From a Messy First Draft to a Production-Ready Contract

Below is a contract as it might look after a first pass - functional, but with multiple issues. Then we refactor it step by step, explaining every change.

BEFORE - The messy contract:

// BEFORE: Messy, inefficient, hard to audit
contract BadVault {
    mapping(address => uint256) public b;
    address public o;
    uint256 public t;
    bool public l;

    constructor() { o = msg.sender; }

    function d() public payable { b[msg.sender] += msg.value; t += msg.value; }

    function w(uint256 a) public {
        require(a > 0);
        require(b[msg.sender] >= a);
        b[msg.sender] -= a;
        t -= a;
        (bool s, ) = msg.sender.call{value: a}("");
        require(s);
    }

    function p() public { require(msg.sender == o); l = !l; }

    function f() public {
        require(l);
        uint256 total = 0;
        for (uint256 i = 0; i < 1000; i++) {
            total += i;
        }
        payable(o).transfer(total);
    }
}

Issues found: (1) Reentrancy in w() - ETH sent before balance updated. (2) tx.origin not used but msg.sender in p() is fine. (3) Unbounded loop in f() - but bounded to 1000, so okay. However, the loop does nothing useful. (4) No events. (5) Terrible naming. (6) Magic numbers. (7) No NatSpec. (8) No access control on d() (deposit should be public anyway). (9) t (totalDeposits) and b (balances) - inconsistent naming. (10) l (locked) is a boolean with unclear purpose.

AFTER - The refactored, production-ready contract:

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

/**
 * @title SecureVault
 * @notice A simple ETH vault with deposit, withdraw, and emergency pause.
 * @dev Uses Checks‑Effects‑Interactions to prevent reentrancy.
 */
contract SecureVault {
    // ─── State Variables ───
    mapping(address => uint256) public balances;
    uint256 public totalDeposited;
    address public immutable owner;
    bool public paused;

    // ─── Constants ───
    uint256 private constant MIN_WITHDRAWAL = 0.001 ether;

    // ─── Events ───
    event Deposited(address indexed user, uint256 amount);
    event Withdrawn(address indexed user, uint256 amount);
    event Paused(address indexed owner);
    event Unpaused(address indexed owner);

    // ─── Modifiers ───
    modifier onlyOwner() {
        require(msg.sender == owner, "Not owner");
        _;
    }
    modifier whenNotPaused() {
        require(!paused, "Contract paused");
        _;
    }

    // ─── Constructor ───
    constructor() {
        owner = msg.sender;
    }

    // ─── Functions ───
    function deposit() public payable whenNotPaused {
        require(msg.value > 0, "Zero deposit");
        balances[msg.sender] += msg.value;
        totalDeposited += msg.value;
        emit Deposited(msg.sender, msg.value);
    }

    function withdraw(uint256 amount) public whenNotPaused {
        require(amount >= MIN_WITHDRAWAL, "Below minimum withdrawal");
        uint256 userBalance = balances[msg.sender];
        require(userBalance >= amount, "Insufficient balance");

        // EFFECTS: update state BEFORE external call
        balances[msg.sender] = userBalance - amount;
        totalDeposited -= amount;

        emit Withdrawn(msg.sender, amount);

        // INTERACTIONS: send ETH LAST
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success, "Transfer failed");
    }

    function togglePause() public onlyOwner {
        paused = !paused;
        if (paused) {
            emit Paused(msg.sender);
        } else {
            emit Unpaused(msg.sender);
        }
    }
}

What changed and why:

1. Naming: b → balances. o → owner. d() → deposit(). w() → withdraw(). p() → togglePause(). l → paused. Now anyone can understand what the contract does without reading the logic.

2. Reentrancy fix: Withdraw now updates balances[msg.sender] and totalDeposited before the external call. The contract uses the Checks-Effects-Interactions pattern.

3. Events: Every state change now emits an event. Frontend applications can now react to deposits, withdrawals, and pause toggles without polling.

4. NatSpec: The contract header explains what the contract does. Every function describes its purpose. Future auditors will understand the intent immediately.

5. Constants: MIN_WITHDRAWAL is a named constant instead of a magic number. If the minimum changes, it's updated in one place.

6. immutable owner: Since owner is set only in the constructor, it's marked immutable. This stores it in the contract bytecode instead of storage, saving gas on every read.

7. Modifiers: onlyOwner and whenNotPaused are defined once and reused, making the code DRY (Don't Repeat Yourself).

8. Removed dead code: The meaningless loop in f() is gone. Every line of bytecode costs gas to deploy - don't deploy code you don't need.

9. Gas optimization: In withdraw(), balances[msg.sender] is read once into userBalance (memory) and used for both the balance check and the update. This saves a storage read (~2,100 gas).

07   Peer Review vs. Self-Review vs. Professional Audit

Three Levels of Code Review - When to Use Each

Self-review: Before you ask anyone else to look at your code, review it yourself. Use the checklist. Pretend you are an auditor who has never seen the code. Question every line: "Why is this here? Could it fail? Could it be simpler?" Self-review catches 60-70% of issues and saves your reviewer's time.

Peer review: Ask a classmate or colleague to review your code. A fresh pair of eyes will see things you missed - assumptions you made, edge cases you didn't consider, naming that makes sense to you but no one else. Peer reviews are the standard in professional development teams.

Professional audit: For contracts managing significant value (>$1M), hire a professional auditing firm (Trail of Bits, OpenZeppelin, CertiK). They bring specialized tools, deep expertise, and a legal reputation that gives users confidence. Audits cost $20K-$100K+ depending on complexity. The MST grant program includes audit support for scaleGrant recipients.

08   Review Tools - Automate What You Can

Static Analysis, Linters, and Formatters

Automated tools can catch many issues before human review even begins. Run these tools on your code before submitting it for peer review.

1. Slither: A static analysis tool that detects common vulnerabilities (reentrancy, unchecked returns, shadowed variables). Install with pip install slither‑analyzer, run with slither contracts/.

2. Mythril: A symbolic execution tool that explores all possible paths through your contract. Finds more complex bugs than Slither but is slower. Run with myth analyze contracts/MyContract.sol.

3. Solhint: A Solidity linter that enforces style rules (naming, ordering, line length). Install with npm install solhint, configure with a .solhint.json file.

4. Prettier + solhint: Format your code automatically with Prettier (for JS/TS) and Solhint (for Solidity). Consistent formatting makes code reviews faster because reviewers can focus on logic, not whitespace.

10   Writing the Audit Report - Your Assessment Deliverable

How to Document Issues Like a Professional Auditor

An audit report is a structured document that describes every vulnerability, inefficiency, and improvement opportunity found during review. Each finding includes a severity rating, a description of the issue, the impact, and the recommended fix.

Severity classification:

Critical: Can result in complete loss of funds, permanent freezing of the contract, or theft by any user. Must be fixed before deployment.

High: Can result in partial loss of funds under specific conditions, or make the contract unusable over time. Fix before deployment.

Medium: Significant gas inefficiency, missing access control that could be exploited with effort, or missing event emissions that break frontend integration. Fix before deployment.

Low: Style violations, missing comments, inconsistent naming. Fix eventually - not a security risk, but reduces auditability.

Informational: Suggestions for improvement that are not bugs or vulnerabilities. Optional.

Report format:

ISSUE #1: Reentrancy vulnerability in withdraw()
Severity: CRITICAL
Location: contracts/Vault.sol, line 42
Description: The withdraw() function sends ETH to the caller before updating
the user's balance. An attacker can re‑enter the function through a fallback
and withdraw more than their balance.
Impact: An attacker can drain the entire contract's ETH balance.
Recommendation: Update the balance BEFORE the external call, following the
Checks‑Effects‑Interactions pattern.

ISSUE #2: Unbounded loop in distributeRewards()
Severity: HIGH
Location: contracts/Vault.sol, line 78
Description: The distributeRewards() function loops over all users using a
dynamic array that grows with each new user. Eventually, the gas consumed
will exceed the block gas limit.
Impact: The distributeRewards() function will become permanently unusable,
trapping funds for all users.
Recommendation: Replace the array with a mapping and a counter. Use a
withdrawal pattern where users claim their own rewards rather than the
contract pushing rewards to all users.

12   Glossary

Key Terms - Sub-Module 10.6

Code Review
A systematic examination of source code by one or more developers to identify bugs, security vulnerabilities, and style issues before the code is deployed.
Refactoring
Restructuring existing code to improve readability, efficiency, or maintainability without changing its external behavior.
NatSpec
Ethereum Natural Specification Format - a documentation standard for Solidity using tags like @notice, @dev, @param, and @return.
Static Analysis
Automated examination of source code without executing it. Tools like Slither detect patterns that indicate potential vulnerabilities.
Severity Classification
A rating system (Critical, High, Medium, Low, Informational) that communicates the potential impact of a finding in an audit report.
Checks-Effects-Interactions
A design pattern that prevents reentrancy by structuring functions in three steps: validate conditions, update state, then make external calls.
Solhint
A Solidity linter that enforces style rules and best practices automatically, similar to ESLint for JavaScript.
Audit Report
A structured document delivered by smart contract auditors listing every vulnerability, its severity, impact, and recommended fix.
