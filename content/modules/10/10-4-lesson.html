02   Real-World Story

The Overflow That Only Happened on Tuesdays

📖 Story

A DeFi protocol had a reward calculation that multiplied three numbers: user's stake, a time multiplier, and a bonus factor. The team wrote unit tests with typical values: stake=100, time=30, bonus=2. All tests passed. They deployed. Six months later, a user with a stake of 50,000 tokens, a time multiplier of 10,000 (from a two-year lock), and a bonus of 5 (from a referral program) triggered an integer overflow. The multiplication of 50,000 × 10,000 × 5 exceeded uint256 max, and the overflow caused the reward to wrap to a tiny number. The user lost their rewards, and the protocol's accounting was permanently corrupted.


Here's the painful part: this bug would have been caught by a fuzz test in seconds. A fuzzer would have randomly combined large numbers and immediately hit the overflow. But the developers only tested the small, "normal" values they could think of. They never tested the extreme combinations that real users - and attackers - would eventually produce.


This is why fuzzing matters. You cannot write a unit test for every possible input combination - there are 2²⁵⁶ possibilities for a single uint256. Fuzzing explores millions of random combinations automatically, finding the edge cases that your imagination missed. In this sub-module, you'll learn to write fuzz tests that catch these bugs before they cost real money.

03   What Is Fuzzing? The Hammer That Finds Every Crack

From Unit Tests to Property-Based Testing

A unit test checks one specific scenario with hard-coded values: "deposit 100 tokens, withdraw 50, expect balance 50." It's precise, but it only checks one path through a forest of possibilities.

A fuzz test (also called property-based testing) defines a property - a rule that should always hold - and lets the testing framework randomly generate thousands of inputs to try to break it. If the property ever fails, the fuzzer reports the exact values that caused the failure, and you have found a bug.

Fuzzing replaces hard-coded values with randomness, exploring the input space automatically.

Why fuzzing is critical for smart contracts: Smart contracts are immutable and control real money. A single unhandled edge case can drain the contract. Fuzzing systematically searches for those edge cases, giving you confidence that your contract is correct across the entire input space, not just the few examples you tested manually.

04   Foundry - The Fuzzing Powerhouse

Setting Up Foundry for Solidity Fuzzing

Foundry is a blazing-fast Ethereum development toolkit written in Rust. It includes Forge for testing, which has a built-in fuzzer. Unlike Hardhat, which runs JavaScript tests, Foundry tests are written in Solidity itself. This gives you direct access to contract internals, cheatcodes for manipulating blockchain state, and significantly faster execution.

Installation:

# Install Foundry (macOS, Linux, Windows WSL)
curl -L https://foundry.paradigm.xyz | bash
foundryup

# Create a new project
forge init my-fuzz-project
cd my-fuzz-project

Project structure:

my-fuzz-project/
├── src/          ← Your contracts
│   └── Vault.sol
├── test/         ← Your tests (written in Solidity!)
│   └── Vault.t.sol
├── script/       ← Deployment scripts
└── foundry.toml  ← Configuration

Writing your first fuzz test: Foundry automatically treats any test function parameter as a fuzzing input. If you define function testDeposit(uint256 amount), Foundry will call this function 256 times (by default) with randomly generated amount values.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "forge-std/Test.sol";
import "../src/Vault.sol";

contract VaultTest is Test {
    Vault public vault;

    function setUp() public {
        vault = new Vault();
    }

    // ✅ FUZZ TEST: Foundry randomly generates `amount` 256 times
    function testDeposit(uint256 amount) public {
        // Bound amount to avoid overflow in test setup
        amount = bound(amount, 1, address(this).balance);
        vault.deposit{value: amount}();
        assertEq(vault.balanceOf(address(this)), amount);
    }

    // ✅ FUZZ TEST with multiple random parameters
    function testWithdraw(uint256 depositAmt, uint256 withdrawAmt) public {
        depositAmt = bound(depositAmt, 1, 100 ether);
        vault.deposit{value: depositAmt}();

        withdrawAmt = bound(withdrawAmt, 1, depositAmt);
        uint256 balBefore = address(this).balance;
        vault.withdraw(withdrawAmt);
        assertEq(address(this).balance, balBefore + withdrawAmt);
    }
}

The bound cheatcode: bound(x, min, max) constrains a fuzzed value to a specific range. This prevents the fuzzer from wasting runs on values that are trivially invalid (like amount=0 or amount exceeding balance). You should still have separate tests for those boundary conditions.

05   Invariant Testing - The Ultimate Safety Net

Rules That Must Never, Ever Break

An invariant is a property of your system that must hold true at all times, regardless of what sequence of operations is performed. For example, in an ERC-20 token: "The sum of all balances must always equal the total supply." No matter who transfers to whom, this equation must hold. If it ever breaks, the contract has a critical bug.

Foundry supports invariant testing through a special kind of test contract. You define one or more invariant functions (prefixed with invariant_), and Foundry randomly calls sequences of your contract's functions, checking the invariants after each call. This is far more powerful than fuzzing individual functions because it tests compositions of operations - exactly how a real attacker would probe your system.

// Invariant test contract
import "forge-std/Test.sol";
import "../src/Token.sol";

contract TokenInvariantTest is Test {
    Token public token;
    address[] public holders;

    function setUp() public {
        token = new Token("Test", "TST");
        // Mint to several accounts
        address user1 = address(1);
        address user2 = address(2);
        token.mint(user1, 1000 ether);
        token.mint(user2, 500 ether);
        holders = [user1, user2];
    }

    // Foundry will call these functions randomly
    function transfer_random(address from, address to, uint256 amount) public {
        // Bound to realistic values
        vm.assume(from != to);
        vm.assume(amount <= token.balanceOf(from));
        vm.prank(from);
        token.transfer(to, amount);
    }

    // Invariant: total supply must equal sum of all balances
    function invariant_totalSupplyEqualsSumOfBalances() public view {
        uint256 sum;
        for (uint256 i = 0; i < holders.length; i++) {
            sum += token.balanceOf(holders[i]);
        }
        assertEq(sum, token.totalSupply());
    }
}

Run invariant tests with forge test --match-contract TokenInvariantTest. Foundry will execute thousands of random function call sequences, checking the invariant after each step. If the invariant ever fails, you get a detailed call trace showing exactly which sequence of operations broke it.

06   Differential Fuzzing - Comparing Two Implementations

Catching Logic Bugs by Comparing Against a Reference

Differential fuzzing compares your smart contract against a simpler, trusted reference implementation. Both receive the same random inputs, and their outputs are compared. If they ever differ, you've found a bug in your contract.

This is incredibly powerful for testing upgradeable contracts (compare old implementation vs. new), gas-optimized rewrites (compare the optimized version against the readable but slow version), and cross-chain compatibility (compare your MST contract against an Ethereum reference).

// Differential fuzzing example
contract DifferentialTest is Test {
    OptimizedVault public optimized;
    ReferenceVault public reference;

    function setUp() public {
        optimized = new OptimizedVault();
        reference = new ReferenceVault();
    }

    function testDepositAndWithdraw(uint256 amount) public {
        amount = bound(amount, 1, 100 ether);

        // Apply same operations to both
        optimized.deposit{value: amount}();
        reference.deposit{value: amount}();

        uint256 optimizedBal = optimized.balanceOf(address(this));
        uint256 referenceBal = reference.balanceOf(address(this));

        // They must always match!
        assertEq(optimizedBal, referenceBal, "Balance mismatch");
    }
}

Real-world use: When Uniswap V3 was developed, the team used differential fuzzing against a Python model of the math. Every tick calculation, every swap formula was fuzzed against the Python reference to ensure the Solidity implementation was mathematically identical.

07   Advanced Fuzzing Techniques

Cheatcodes, Assumptions, and Gas Tracking

1. vm.assume(condition): Skips the current fuzz run if the condition is false. Use this to reject inputs that are trivially invalid, so the fuzzer spends its energy exploring meaningful inputs.

function testTransfer(address to, uint256 amount) public {
    vm.assume(to != address(0));     // Skip zero‑address runs
    vm.assume(to != address(this));   // Skip self‑transfers
    amount = bound(amount, 1, token.balanceOf(address(this)));
    token.transfer(to, amount);
}

2. vm.prank(address): Sets msg.sender for the next call. Essential for testing access-controlled functions with random users.

3. Gas tracking: Foundry can measure gas usage of fuzzed functions, helping you identify gas-hungry edge cases.

function testGasUsage(uint256 amount) public {
    amount = bound(amount, 1, 1000 ether);
    uint256 gasBefore = gasleft();
    vault.deposit{value: amount}();
    uint256 gasUsed = gasBefore - gasleft();
    assertLt(gasUsed, 100000, "Gas too high");
}
08   Integrating Fuzzing into Your Workflow

From Local Machine to CI/CD Pipeline

1. Run fuzz tests locally during development: forge test --fuzz‑runs 1000 (increase from default 256 for more thorough exploration).

2. Add invariant tests to your continuous integration (CI) pipeline. Every pull request triggers a fuzzing run. If any invariant breaks, the PR cannot be merged.

3. Use differential fuzzing for every contract upgrade. Deploy the old and new implementations side-by-side, fuzz them with identical random inputs, and ensure they produce identical outputs (for unchanged functions).

4. Schedule deep fuzzing runs overnight with --fuzz‑runs 100000 for contracts managing significant value. The more runs, the higher the probability of finding rare edge cases.

🔑 The Fuzzing Mindset

Fuzzing doesn't prove your contract is correct - it proves your contract survives a certain number of random attempts. But every run that doesn't find a bug increases your confidence. Combine fuzzing with unit tests, formal verification, and manual code review for defense in depth.

10   Glossary

Key Terms - Sub-Module 10.4

Fuzz Testing
An automated testing technique that generates random inputs to discover edge cases and vulnerabilities. Also called property-based testing.
Foundry
A fast Ethereum development toolkit written in Rust. Includes Forge for Solidity-native testing with built-in fuzzing and cheatcodes.
Invariant Testing
Testing that a specific property (e.g., "total supply equals sum of balances") holds after any random sequence of function calls. Catches complex state corruption bugs.
Differential Fuzzing
Comparing two implementations by feeding them identical random inputs and asserting that their outputs match. Used to verify upgrades and optimizations.
Cheatcodes
Special functions provided by Foundry (like vm.prank, vm.assume, bound) that manipulate the test environment, filter inputs, and control blockchain state.
bound
A Foundry cheatcode that constrains a fuzzed value to a specific numeric range, preventing wasted runs on trivially invalid inputs.
vm.assume
A cheatcode that skips the current fuzz run if a condition is not met. Used to filter out invalid test scenarios without failing the test.
Gas Tracking
Measuring the gas consumption of fuzzed functions to identify unexpectedly expensive edge cases that could make the contract unusable.
