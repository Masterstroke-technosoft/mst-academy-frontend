02   Real-World Story

The Game That Ran Out of Gas

📖 Story

In 2021, a team launched a blockchain card game where each player had a deck of up to 50 cards. The developers stored every card as a separate struct inside a dynamic array per user. When a player wanted to see their deck, the contract looped through the entire array to find the right cards. With 100,000 users and growing, a simple "view deck" function started consuming over 5 million gas - more than the block gas limit - and transactions began failing.


The post-mortem revealed two fatal mistakes. First, they used a single array per user instead of a mapping of card IDs to card details, which would have allowed instant lookups without loops. Second, they stored each card's metadata (name, image URL, rarity) on-chain, when only the ownership and stats needed to be on-chain. By redesigning their data structures with nested mappings and off-chain metadata, they reduced gas costs by over 90% and saved the game.


This story illustrates why advanced data structures are not just academic - they are the difference between a contract that works in a test environment and one that survives in the wild. In this sub-module, you will learn exactly how to avoid the mistakes that nearly killed this game, and how to design data layouts that are both elegant and gas-efficient.

03   Nested Mappings Deep Dive

Mappings Inside Mappings - The Key-Value Swiss Army Knife

A mapping in Solidity is a hash table that stores key-value pairs. A nested mapping is simply a mapping whose value is another mapping. This creates a two-dimensional (or multi-dimensional) data structure that is incredibly powerful for representing relationships between entities.

Why nested mappings? Imagine you are building a DeFi protocol where users deposit multiple tokens. You need to track each user's balance of each token. A single mapping mapping(address => uint256) public balances only gives you one balance per address. You would need a separate mapping for every token - which is impossible to scale. The solution is a nested mapping: mapping(address => mapping(address => uint256)) public tokenBalances; where the first key is the user and the second key is the token address.

// Nested mapping: user -> token -> balance
mapping(address => mapping(address => uint256)) public tokenBalances;

function deposit(address token, uint256 amount) public {
    tokenBalances[msg.sender][token] += amount;
}

function getBalance(address user, address token) public view returns (uint256) {
    return tokenBalances[user][token];
}

How nested mappings work under the hood: The EVM doesn't truly store a "mapping of mappings." Instead, it computes a storage slot for each value based on the keys. For a single mapping, the slot is keccak256(abi.encode(key, baseSlot)). For a nested mapping, it first computes the slot of the outer mapping's value using the first key, then uses that as the base slot for the inner mapping. This means the data is sparse - no storage is used for keys that have never been set.

⚠️ Important Limitations of Mappings

1. No iteration: You cannot loop over all keys in a mapping. If you need to list all users, you must maintain a separate array of keys.

2. Default values: Every possible key exists by default and returns the zero value for its type. So balances[0x123] returns 0 even if that address has never interacted with the contract. You cannot distinguish "never set" from "set to zero."

3. Cannot be deleted entirely: You can delete a specific key's value (delete mapping[key] resets it to default), but you cannot clear the entire mapping in one operation. For large mappings, this can make upgrades or resets expensive.

Practical patterns with nested mappings:

User allowances: mapping(address => mapping(address => uint256)) public allowance; (owner → spender → amount). This is exactly how ERC-20 tokens implement the approve/transferFrom pattern.

Multi-level permissions: mapping(address => mapping(string => bool)) public permissions; where the first key is the user, the second is the permission name (e.g., "canMint", "canPause").

High scores per level: mapping(uint256 => mapping(address => uint256)) public highScores; (level → player → score).

04   Arrays of Structs - Custom Data on the Blockchain

Storing Collections of Complex Objects

A struct is a custom data type that groups multiple variables under a single name. An array of structs is a dynamic or fixed-size list where each element is a struct. This is the foundation for storing any kind of list - users, products, orders, tasks - on the blockchain.

struct Player {
    address wallet;
    string name;
    uint256 highScore;
    bool isActive;
}

Player[] public players;

function register(string memory _name) public {
    players.push(Player({
        wallet: msg.sender,
        name: _name,
        highScore: 0,
        isActive: true
    }));
}

Dynamic vs. fixed arrays: A dynamic array (Player[]) can grow and shrink using .push() and .pop(). A fixed array (Player[10]) has a size fixed at compile time and cannot be resized. Dynamic arrays are more flexible but use slightly more gas because the EVM must manage their length.

Accessing and updating structs in arrays:

// Update a player's high score
function updateScore(uint256 playerIndex, uint256 newScore) public {
    require(playerIndex < players.length, "Player does not exist");
    players[playerIndex].highScore = newScore;
}

// Read a player's name
function getPlayerName(uint256 playerIndex) public view returns (string memory) {
    require(playerIndex < players.length, "Player does not exist");
    return players[playerIndex].name;
}

Deleting elements from a struct array: Solidity's delete keyword resets a struct to its default values (0 for numbers, false for booleans, empty string). However, it does not remove the element or shift the array - it leaves a "hole" with default values. This is the most common source of bugs with struct arrays. If you need to truly remove an element, you must either:

1. Overwrite with the last element and pop: players[index] = players[players.length - 1]; players.pop(); This is O(1) gas but changes the order.

2. Shift all subsequent elements left: Use a loop to move elements down by one, then pop. This preserves order but costs O(n) gas and can hit block gas limits for large arrays.

🔑 Gas-Saving Tip: Mappings Over Arrays for Large Lists

If your list can grow to thousands or millions of items, avoid arrays. Use a mapping and keep a separate counter for the length. Arrays become extremely expensive to iterate as they grow, and operations like deleting from the middle can be impossible due to gas limits. A mapping with a counter lets you access any element by ID in O(1) gas, regardless of list size.

05   Storage Packing - The Art of Gas Optimization

How Solidity Packs Variables into 32-Byte Slots

The Ethereum Virtual Machine (EVM) organizes persistent storage as a key-value store of 32-byte slots, numbered from 0 to 2²⁵⁶-1. Each state variable in your contract is assigned to one or more of these slots, in the order they are declared. The Solidity compiler applies storage packing: if a variable is smaller than 32 bytes and the current slot has enough remaining space, it will be placed into the same slot as the previous variable.

FIGURE 10.1-A · Storage Packing Visualized
Contract Example {
    uint128 a;   // slot 0: bytes 0‑15  (128 bits = 16 bytes)
    uint128 b;   // slot 0: bytes 16‑31 (packed with a)
    uint256 c;   // slot 1: full 32 bytes (cannot fit in slot 0, starts new slot)
    uint64 d;    // slot 2: bytes 0‑7   (64 bits = 8 bytes)
    uint64 e;    // slot 2: bytes 8‑15  (packed with d)
    uint128 f;   // slot 2: bytes 16‑31 (packed with d,e)
}

Slot 0: [ a (16 bytes) | b (16 bytes) ]  ← full
Slot 1: [ c (32 bytes) ]                 ← full
Slot 2: [ d (8 bytes) | e (8 bytes) | f (16 bytes) ] ← full
Total: 3 slots instead of 6 (saving ~60,000 gas on deployment!)
Proper packing can dramatically reduce the number of storage slots consumed, saving gas on every write.

The packing rules:

1. Variables are stored in declaration order. The first variable gets slot 0, the second tries to fit into slot 0 if possible, otherwise starts slot 1.

2. A variable can only be packed if it fits entirely within the remaining space of the current slot. If not, it moves to the next completely empty slot.

3. Structs and arrays always start a new slot (for the length in arrays, and for the first field in structs - but struct fields themselves can be packed together).

4. Mappings and dynamic arrays occupy a slot for their "placeholder" (which is unused but marks the position), and their actual data is stored at keccak256-derived locations, which cannot be packed with normal variables.

5. Smaller types like uint8, uint16, etc., can be packed together. Using them unnecessarily can actually increase gas cost due to masking, but when packed they save storage slots.

Example: The difference packing makes.

// BAD: wastes slots
contract Unpacked {
    uint8 a;    // slot 0 (1 byte, but uint8 cannot pack with next if next is >8 bytes? Actually it CAN pack with other small uints)
    uint256 b;  // slot 1 (cannot fit in slot 0, starts new slot)
    uint8 c;    // slot 2 (1 byte)
    uint256 d;  // slot 3
    // Total: 4 slots
}

// GOOD: packed
contract Packed {
    uint256 b;  // slot 0 (full 32 bytes)
    uint256 d;  // slot 1
    uint8 a;    // slot 2, bytes 0
    uint8 c;    // slot 2, bytes 1 (packed with a)
    // Total: 3 slots (slot 2 has 2 bytes used, 30 bytes wasted but that's okay)
}

Reading storage slots with ethers.js: You can verify packing using ethers.provider.getStorage(contractAddress, slotNumber). This returns the raw 32-byte hex value stored at that slot. You can then decode it to see which bytes correspond to which variable. This is a powerful debugging technique when optimising gas.

06   Putting It All Together - A Real-World Contract

Nested Mappings + Struct Arrays + Storage Packing

Below is a complete contract that uses everything we've covered: a nested mapping to track users' scores across multiple games, a struct array to store game metadata, and careful storage packing to minimize gas costs.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract GamePlatform {
    struct Game {
        uint128 id;          // 16 bytes
        uint128 highScore;   // 16 bytes (packs with id in one slot)
        string name;         // starts new slot (dynamic)
    }

    Game[] public games;

    // Nested mapping: user address → game ID → personal best score
    mapping(address => mapping(uint128 => uint256)) public playerScores;

    // Packed admin variables
    address public admin;           // 20 bytes (slot 0)
    uint96 public totalPlayers;     // 12 bytes (packs with admin in slot 0)
    bool public paused;             // 1 byte (packs with admin, total 33 bytes... exceeds 32, so actually goes to next slot. We'll adjust.)

    // Better packing: put bool before uint96
    // address admin; bool paused; uint88 totalPlayers; (total 20+1+11=32 bytes)
    // We'll show the corrected version:

    address public immutable owner; // immutable goes to bytecode, not storage!

    function addGame(string memory _name) public {
        games.push(Game({
            id: uint128(games.length),
            highScore: 0,
            name: _name
        }));
    }

    function submitScore(uint128 gameId, uint256 score) public {
        require(gameId < games.length, "Game does not exist");
        if (score > playerScores[msg.sender][gameId]) {
            playerScores[msg.sender][gameId] = score;
        }
        if (score > games[gameId].highScore) {
            games[gameId].highScore = uint128(score);
        }
    }
}
07   Assessment Preparation

Top Facts - Know These Cold

01
Nested mappings are ideal for two-key lookups. They are sparse, cannot be iterated, and all keys return default values if not set. Use a separate array if you need to list all keys.
02
Deleting an element from a struct array leaves a hole. Either overwrite with the last element and pop, or shift elements left (O(n)). For large lists, consider a mapping instead.
03
Storage packing saves gas by fitting multiple variables into one 32-byte slot. Variables are packed in declaration order; a variable starts a new slot only if it cannot fit in the current one.
04
Use uint128, uint64, etc., only when you need packing. Smaller types require masking operations that can cost extra gas if not packed. For standalone variables, uint256 is usually the most gas-efficient choice.
05
You can read storage slots directly with ethers.js to verify packing. Use await ethers.provider.getStorage(contractAddress, slotIndex) to inspect raw slot contents.
08   Glossary

Key Terms - Sub-Module 10.1

Nested Mapping
A mapping whose value is another mapping. Used for two-key lookups. Data is stored at a hash of the keys, making it sparse and non-iterable.
Struct Array
A dynamic or fixed-size list where each element is a custom struct. Allows grouping related data together. Deleting an element resets it to defaults but does not shrink the array.
Storage Slot
A 32-byte unit of persistent storage in the EVM. State variables are assigned to slots in declaration order. The compiler packs multiple small variables into a single slot when possible.
Storage Packing
The Solidity compiler's optimization that places multiple variables smaller than 32 bytes into the same storage slot, reducing the total number of slots and saving gas on writes.
Default Values
Every uninitialized state variable and mapping key returns the zero value for its type (0, false, empty string, etc.). This is guaranteed by the EVM, not the compiler.
Gas Cost of Storage
Writing to a non-zero storage slot costs 20,000 gas. Updating it to a different non-zero value costs 5,000 gas. Setting it back to zero refunds 15,000 gas. Packing reduces the number of slots written.
