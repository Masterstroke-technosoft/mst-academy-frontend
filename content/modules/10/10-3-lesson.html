02   Real-World Story

The DAO Hack - How a Single Missing Line Cost $60 Million

📖 Story

June 17, 2016. The DAO (Decentralized Autonomous Organization) had raised $150 million in ETH - the largest crowdfunding in history. It was meant to be a revolutionary venture fund where token holders voted on investments. Then, around 3:30 AM UTC, an attacker began siphoning ETH from the DAO's smart contract. By the time the community could react, over 3.6 million ETH (worth approximately $60 million at the time) had been drained into a child DAO controlled by the attacker.


The vulnerability was a reentrancy attack. The DAO's withdraw function sent ETH to the user before updating the user's balance. The attacker deployed a malicious contract with a fallback function that, upon receiving ETH, called the withdraw function again - before the first withdrawal had updated the balance. This recursive loop repeated until the DAO was drained. A single missing line of code - updating the balance before sending - had allowed the largest hack in blockchain history at that time.


The fallout was seismic. The Ethereum community was so divided on how to respond that it led to a hard fork, splitting Ethereum into two chains: Ethereum (ETH) and Ethereum Classic (ETC). The incident changed how smart contracts were written forever. It gave birth to the Checks-Effects-Interactions pattern and the ReentrancyGuard modifier that are now standard in every project. Every line of security advice in this sub-module was written in blood - or more precisely, in stolen ether.

03   Reentrancy Attacks - The Most Famous Vulnerability in Blockchain

How They Work, How to Prevent Them, and Why They Still Happen

A reentrancy attack occurs when a contract calls an external contract, and that external contract calls back into the original contract before the original contract has finished updating its state. It's like someone withdrawing money from a bank, but before the teller updates your balance, you walk back to the counter and withdraw again - and again, and again - because the system still thinks you have the full amount.

The vulnerable pattern:

// VULNERABLE CONTRACT – DO NOT USE
contract VulnerableVault {
    mapping(address => uint256) public balances;

    function deposit() public payable {
        balances[msg.sender] += msg.value;
    }

    function withdraw() public {
        uint256 amount = balances[msg.sender];
        require(amount > 0, "No balance");

        // ❌ BUG: ETH sent BEFORE state update
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success, "Transfer failed");

        // ❌ Balance updated AFTER external call – too late!
        balances[msg.sender] = 0;
    }
}

How the attacker exploits it: The attacker deploys a malicious contract with a fallback function. When the Vault sends ETH to the attacker's contract, the fallback fires - and calls withdraw() again. Since balances[msg.sender] hasn't been set to 0 yet, the require check passes again, and more ETH is sent. This loop continues until the Vault is empty or gas runs out.

contract Attacker {
    VulnerableVault public vault;

    constructor(address _vault) {
        vault = VulnerableVault(_vault);
    }

    function attack() public payable {
        vault.deposit{value: msg.value}();
        vault.withdraw();
    }

    // Fallback – called when ETH is received
    receive() external payable {
        if (address(vault).balance > 0) {
            vault.withdraw(); // Re‑enter before state updated!
        }
    }
}

The fix - Checks-Effects-Interactions pattern: Always update your contract's state before making external calls. This ensures that if the external call re-enters your contract, the state has already been updated, so the re-entrant call cannot exploit stale data.

// SECURE CONTRACT
contract SecureVault {
    mapping(address => uint256) public balances;

    function deposit() public payable {
        balances[msg.sender] += msg.value;
    }

    function withdraw() public {
        uint256 amount = balances[msg.sender];
        require(amount > 0, "No balance");

        // ✅ Step 1: CHECKS – validate conditions (done above)
        // ✅ Step 2: EFFECTS – update state FIRST
        balances[msg.sender] = 0;

        // ✅ Step 3: INTERACTIONS – make external call LAST
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success, "Transfer failed");
    }
}

Alternative fix - ReentrancyGuard: OpenZeppelin provides a ReentrancyGuard modifier that uses a state variable to lock the contract during execution. If a function with this modifier is re-entered, it will revert immediately.

import "@openzeppelin/contracts/utils/ReentrancyGuard.sol";

contract GuardedVault is ReentrancyGuard {
    mapping(address => uint256) public balances;

    function withdraw() public nonReentrant {
        uint256 amount = balances[msg.sender];
        require(amount > 0, "No balance");
        balances[msg.sender] = 0;
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success, "Transfer failed");
    }
}

Types of reentrancy: There are three main types. Single-function reentrancy (the classic DAO attack, where the same function is re-entered). Cross-function reentrancy (where one function makes an external call, and the callee calls a different function that reads the same state). Cross-contract reentrancy (where multiple contracts share state, and an external call to one contract triggers a call to another). The Checks-Effects-Interactions pattern protects against all three.

04   Integer Overflow and Underflow

When Numbers Break - The BeautyChain Hack

Integer overflow occurs when an arithmetic operation produces a value that exceeds the maximum representable value for the type. Before Solidity 0.8.0, uint256 values would wrap around: 2²⁵⁶−1 + 1 would become 0. This silent wrapping was the root cause of countless bugs and several major hacks.

The BeautyChain (BEC) Hack - 2018: The BeautyChain token contract had a batchTransfer function that multiplied amount × recipients without checking for overflow. An attacker crafted a call where the multiplication overflowed: a tiny amount multiplied by a huge number of recipients wrapped around to a very small number, but the individual transfer amounts were large. The attacker effectively created tokens from nothing, printing billions of dollars worth of BEC tokens in a single transaction.

// VULNERABLE (Solidity <0.8.0 without SafeMath)
function batchTransfer(address[] memory recipients, uint256 amount) public {
    uint256 total = amount * recipients.length; // ⚠️ OVERFLOW if too large!
    require(balances[msg.sender] >= total, "Insufficient balance");
    balances[msg.sender] -= total;
    for (uint256 i = 0; i < recipients.length; i++) {
        balances[recipients[i]] += amount;
    }
}

The solution - Solidity 0.8+ built-in checks: Since Solidity 0.8.0, the compiler automatically inserts overflow and underflow checks. If an operation overflows, the transaction reverts. This makes SafeMath unnecessary for most codebases. However, there are still cases where you might want unchecked arithmetic for gas optimization, but you must be absolutely certain no overflow can occur.

// Solidity 0.8+ – automatically safe
uint256 a = type(uint256).max;
uint256 b = a + 1; // ❌ REVERTS – overflow protection

// If you are CERTAIN no overflow is possible, use unchecked for gas savings
uint256 c;
unchecked {
    c = a + 1; // ⚠️ Wraps around – only use when safe!
}

Where overflow still matters in 0.8+: While simple arithmetic is protected, typecasting is not automatically checked. Downcasting from uint256 to uint128 can silently truncate values without reverting. Always check that the value fits in the smaller type before casting.

uint256 largeValue = 2**200;
uint128 smallValue = uint128(largeValue); // ⚠️ SILENT TRUNCATION – data loss!
// Always check: require(largeValue <= type(uint128).max, "Value too large");
05   Access Control - The Most Common Vulnerability in Audits

tx.origin, msg.sender, and the Functions You Forgot to Protect

Access control vulnerabilities are the single most common finding in smart contract audits. They range from completely missing access controls (anyone can call a critical function) to subtle misuses of tx.origin instead of msg.sender.

msg.sender vs tx.origin - the critical difference: msg.sender is the immediate caller of the current function. If Contract A calls Contract B, then inside Contract B, msg.sender is Contract A's address. tx.origin is the original EOA (externally owned account) that signed the transaction - it never changes, no matter how many contracts are in the call chain.

// VULNERABLE – Do not use tx.origin for access control!
contract VulnerableWallet {
    address public owner;

    constructor() { owner = tx.origin; }

    function withdraw() public {
        require(tx.origin == owner, "Not owner"); // ⚠️ tx.origin is DANGEROUS
        payable(msg.sender).transfer(address(this).balance);
    }
}

// Attacker's phishing contract:
contract PhishingAttack {
    VulnerableWallet wallet;
    constructor(address _wallet) { wallet = VulnerableWallet(_wallet); }
    function trick() public {
        // User calls this function, which calls withdraw
        // Inside withdraw, tx.origin is the USER (not the attacker!)
        // So the require passes, and funds go to the attacker!
        wallet.withdraw();
    }
}

The fix - always use msg.sender: The correct access control pattern uses msg.sender, which correctly identifies the immediate caller.

// SECURE
contract SecureWallet {
    address public owner;
    constructor() { owner = msg.sender; }

    modifier onlyOwner() {
        require(msg.sender == owner, "Not owner");
        _;
    }

    function withdraw() public onlyOwner {
        payable(owner).transfer(address(this).balance);
    }
}

Missing access control: The most basic and most devastating vulnerability. A function that should be restricted - like mint(), burn(), setFees(), or destroy() - is left with public visibility and no require check. In 2023 alone, missing access control accounted for over $100 million in losses across DeFi protocols. Always add a clear access modifier to every state-changing function that should not be callable by anyone.

06   Front-Running and MEV (Miner Extractable Value)

When Attackers See Your Transaction Before It Executes

Blockchain transactions are broadcast to a public mempool before they are included in a block. Anyone watching the mempool can see pending transactions and act on them first by submitting their own transaction with a higher gas price. This is front-running.

Example - DEX sandwich attack: You submit a transaction to buy a token on a decentralized exchange. An attacker sees your transaction, submits a buy order with higher gas to execute before you (driving the price up), then submits a sell order with slightly lower gas to execute after you (selling at the inflated price). You bought at a worse price, and the attacker profited from the spread.

Mitigations:

1. Commit-Reveal: Users first commit a hash of their action, and only reveal it in a second transaction after it's included in a block. This prevents front-runners from knowing what the action is until it's too late to front-run.

2. Flashbots: Send transactions directly to miners/validators through private relays, bypassing the public mempool entirely. This is now the dominant approach for MEV-sensitive transactions.

3. Fair ordering protocols: Some chains are experimenting with consensus-level fair ordering, where transactions are ordered by their arrival time rather than gas price. MST's Trika POSA consensus can explore these patterns.

07   Denial of Service (DoS) Attacks

Making Your Contract Unusable

1. Gas griefing: An attacker can force your transaction to fail by manipulating gas. For example, if your contract refunds gas to users, an attacker can consume all the gas in a loop, causing the refund to fail.

2. Block stuffing: An attacker fills blocks with their own high-gas transactions, preventing other transactions from being included. This is expensive but effective for short-term attacks on time-sensitive protocols like auctions.

3. Unbounded loops: If your contract has a for loop that iterates over a growing array (e.g., all users, all tokens), eventually the loop will consume more gas than the block gas limit, and the function will become permanently unusable. Always use mappings with separate counters instead of arrays for data structures that can grow indefinitely.

4. Owner-based DoS: If a critical function is onlyOwner and the owner loses their private key, the function is permanently frozen. Use multi-sig wallets and timelocks to reduce single-point-of-failure risks.

09   Putting It All Together - A Before-And-After Example

Fixing a Vulnerable Contract Step by Step

Below is a contract that contains multiple vulnerabilities. Study it carefully, then see the secure version with all fixes applied.

// VULNERABLE CONTRACT – Contains 5 security issues
contract BrokenAuction {
    address public owner;
    mapping(address => uint256) public bids;

    constructor() { owner = tx.origin; }

    function bid() public payable { bids[msg.sender] += msg.value; }

    function withdraw() public {
        uint256 amount = bids[msg.sender];
        require(amount > 0);
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success);
        bids[msg.sender] = 0;
    }

    function endAuction() public {
        require(tx.origin == owner);
        payable(owner).transfer(address(this).balance);
    }
}

Issues found: (1) Reentrancy in withdraw() - ETH sent before state update. (2) tx.origin used for ownership in both constructor and endAuction(). (3) endAuction() accessible by anyone because the require uses tx.origin which can be phished. (4) No event emissions, making frontend updates difficult. (5) No withdrawal limit or emergency pause mechanism.

// SECURE CONTRACT – All issues fixed
import "@openzeppelin/contracts/utils/ReentrancyGuard.sol";

contract FixedAuction is ReentrancyGuard {
    address public owner;
    mapping(address => uint256) public bids;
    bool public auctionEnded;
    event BidPlaced(address indexed bidder, uint256 amount);
    event Withdrawal(address indexed bidder, uint256 amount);

    constructor() { owner = msg.sender; }

    modifier onlyOwner() { require(msg.sender == owner, "Not owner"); _; }

    function bid() public payable {
        require(!auctionEnded, "Auction ended");
        bids[msg.sender] += msg.value;
        emit BidPlaced(msg.sender, msg.value);
    }

    function withdraw() public nonReentrant {
        uint256 amount = bids[msg.sender];
        require(amount > 0, "No bids");
        bids[msg.sender] = 0; // EFFECTS first
        (bool success, ) = msg.sender.call{value: amount}(""); // INTERACTIONS last
        require(success, "Transfer failed");
        emit Withdrawal(msg.sender, amount);
    }

    function endAuction() public onlyOwner {
        auctionEnded = true;
        payable(owner).transfer(address(this).balance);
    }
}
10   Assessment Preparation

Top Facts - Know These Cold

01
Reentrancy attacks exploit external calls made before state updates. The DAO hack ($60M) was a reentrancy attack. Fix with Checks-Effects-Interactions or ReentrancyGuard. Always update state before making external calls.
02
Integer overflow/underflow is protected in Solidity 0.8+ for arithmetic, but NOT for downcasting. Always validate values before casting to smaller types. Use SafeMath if supporting older Solidity versions.
03
Never use tx.origin for access control. tx.origin is the original EOA and can be phished through a malicious contract. Always use msg.sender, which correctly identifies the immediate caller.
04
Missing access control is the #1 finding in smart contract audits. Every state-changing function that should be restricted must have an explicit modifier or require check. Public by default is dangerous.
05
Front-running is possible because transactions are visible in the mempool before execution. Use Flashbots, commit-reveal schemes, or fair ordering protocols to mitigate MEV attacks on your users.
11   Glossary

Key Terms - Sub-Module 10.3

Reentrancy Attack
An exploit where an attacker recursively calls a function before the contract has updated its state, draining funds or corrupting data. The DAO hack was the most famous example.
Checks-Effects-Interactions
A design pattern that structures functions in three steps: validate conditions (Checks), update contract state (Effects), then make external calls (Interactions). Prevents reentrancy.
Integer Overflow
When an arithmetic result exceeds the maximum value representable by the type. Solidity 0.8+ automatically reverts on overflow; older versions wrapped silently.
tx.origin
A global Solidity variable that returns the original EOA that started the transaction. Never use it for access control - it can be phished through malicious contracts.
ReentrancyGuard
An OpenZeppelin modifier that uses a mutex (mutual exclusion lock) to prevent a function from being re-entered during execution.
Front-running
An attack where someone observes a pending transaction in the mempool and submits their own transaction first with a higher gas price to profit from the information.
Denial of Service (DoS)
Any attack that makes a contract unusable - gas exhaustion from unbounded loops, block stuffing, or owner-key loss that freezes critical functions.
MEV
Miner/Maximal Extractable Value - the profit that can be extracted by reordering, including, or censoring transactions within a block.
